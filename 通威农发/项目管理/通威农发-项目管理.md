# 通威农发-项目管理

> 项目：**通威农发** — 智能工厂生产业务平台项目管理方法论与实践

## 概述

通威农发智能工厂项目采用「WBS功能分解 + 接口分解 + 双周滚动迭代」三位一体计划体系，辅以日计划分解跟踪、问题看板、健康度评估会议三层管控机制，确保67家饲料分子公司（含海外8家）的数字化转型有序推进。项目从第5周（2023-12-15）启动双周迭代，至第22周（2024-04-18）完成18轮滚动，历经方案编制、蓝图评审、详细设计、开发测试四大阶段，逐步建立跨域协作秩序。

---

## 项目计划体系

### WBS功能分解

项目将9大业务域拆解为功能项清单，每个功能项定义以下属性：

| 维度 | 说明 |
|------|------|
| 业务域 | 计划/生产/物资/设备/质量/配方/无人过磅 |
| 一级分类 | 基础数据/核心业务/精益管理/工作台等 |
| 二级分类 | 细分子模块（如生产→工单管理/工单执行/精益管理） |
| 优先级 | P1（核心必做）/ P2（重要）/ P3（可延缓） |
| 客户端类型 | PC / 手机 / 平板(CS) / PDA |
| 操作者 | 分子公司岗位 / 总部岗位 / 系统 |

WBS覆盖约150+功能项，按业务域统计：

| 业务域 | 功能项数 | P1占比 | 核心关注点 |
|--------|---------|--------|-----------|
| 计划管理 | ~30 | 60% | 产能统筹、日计划排产、齐套检查、物料需求 |
| 生产管理 | ~25 | 65% | 工单执行、投料管理、交接班、异常管理 |
| 物资管理 | ~35 | 55% | 入库/出库/库内作业、盘点策略、紧急放行 |
| 设备管理 | ~20 | 55% | TPM维保、故障维修、机物料管理、事故管理 |
| 质量管理 | ~35 | 50% | 检验标准/方案/评审、巡检、SPC、分层审核 |
| 配方管理 | ~10 | 50% | 大小料分解、返工料添加规则、工单配方确认 |
| 无人过磅 | ~15 | 70% | 磅秤配置、登记/过磅/放行流程、硬件驱动 |

### 接口分解

接口清单独立于功能WBS，按外部系统分组：

| 外部系统 | 接口数量 | 方向 | 说明 |
|---------|---------|------|------|
| EBS | 31 | 双向 | 采购订单/销售订单/生产批/库存/主数据/成本 |
| 工厂MES | 7 | 双向 | 工单下发/投料数据/报警/停机时间/报修 |
| IOT | 3 | 单向→平台 | 设备状态/性能参数/报警信息 |
| 配方系统 | 1 | 单向→平台 | 配方明细同步 |
| FBC审批 | 2 | 双向 | 审批流触发与结果回写 |
| 预算系统 | 1 | 单向→平台 | 年度生产成本数据 |
| 近红外/DTS | 2 | 单向→平台 | 检测结果回写 |
| E采通 | 1 | 双向 | 磅单与出库结果 |

### 双周滚动迭代（W05→W22，18轮）

项目从2023年12月起以双周为迭代周期滚动推进，阶段演进如下：

**阶段一：方案编制与蓝图评审（W05–W11，2023.12–2024.01）**

此阶段聚焦蓝图方案编写与内部评审拉通。各业务域顾问完成现状调研、流程绘制、需求分析并输出蓝图初版，随后进行多轮内部评审与修改。关键里程碑：
- W05：启动双周滚动机制，各域开始现状调研
- W07：完成首批蓝图方案初稿，技术架构方案汇报
- W09：蓝图方案集中评审启动，流程图岗位名称标准化
- W11：完成业务蓝图封版1.0，月度方案汇报

**阶段二：详细设计与功能开发并行（W12–W18，2024.02–2024.03）**

蓝图评审通过后进入UI原型设计、详细设计文档编写阶段，同时P1功能启动开发。此阶段关键特征为设计与开发高度重叠，资源协调压力增大。关键里程碑：
- W12：UI原型设计启动，EBS接口开发进入批量输出期
- W14：功能清单分解完成并标注开发人天，P1功能开发全面铺开
- W16：技术方案（API/数据流转/高可用）定稿交付
- W18：质量基础数据结构设计完成，MES编程进入中期

**阶段三：集成测试与部署准备（W19–W22，2024.03–2024.04）**

重点转向系统联调测试、周边系统接口对接、无人过磅硬件部署准备。关键里程碑：
- W19：测试计划编写与培训，EBS集成测试启动
- W20：与周边系统配套计划确认，MES对点工作推进
- W21：无人地磅集成测试，详细方案材料汇总评审
- W22：详细方案评审完成，测试培训全面铺开

## 日常管理

### 日计划分解与跟踪

双周计划逐层分解为日计划，格式统一为：

| 字段 | 说明 |
|------|------|
| 类别 | 业务域（计划/生产/物资/设备/质量/EBS/IT/MES/硬件） |
| 负责人 | 具体执行人 |
| 双周计划 | 关联的双周任务编号 |
| 工作任务分解 | 当日具体交付物 |
| 计划完成日期 / 实际完成日期 | 偏差度量 |
| 任务状态 | 已完成 / 已延误 / 执行中 / 未开始 |
| 非正常原因 | 延误根因记录 |
| 改善措施 | 纠偏动作 |

日计划的核心价值在于**偏差即时可视**——当日标注「已延误」的任务立即触发改善措施跟踪，避免问题沉淀。

### 问题跟踪看板

项目维护统一的问题跟踪看板，字段涵盖：

| 字段 | 说明 |
|------|------|
| 领域 | 质量/计划/物资/设备/PMO/主数据/MES |
| 类别 | 问题 / 风险 |
| 事项分类 | 流程 / 接口 / 方案 / 进度 / 资源 / 其他 |
| 风险级别 | 低 / 中 / 高 |
| 问题描述 | 具体阻点 |
| 建议措施 | 初步解法提议 |
| 问题状态 | 处理中 / 已处理 / 待验证 |

典型高频问题模式：

1. **接口数据来源未确定**（计划域高频）：年度成本取数路径不清、历史销量数据归属不定，需专题决策
2. **跨域流程引用不规范**（质量域高频）：流程图引用他域泳道时名称不一致，依赖PMO统一标准化
3. **服务器/硬件阻塞**（MES域高频）：服务器阵列卡故障导致MES系统安装反复推迟
4. **主数据方案滞后**（PMO域高频）：主数据顾问到岗晚，拖累设备/机物料等依赖主数据的模块

### 健康度评估事项看板

项目定期开展健康度评估，识别各域风险并指派跟踪人。典型高风险项：

| 风险描述 | 根因 | 建议措施 |
|---------|------|---------|
| 蓝图变革点不够明确 | 顾问经验不足，方案多为线下搬线上 | 咨询组参与，挖掘行业创新价值 |
| 计划域人力瓶颈 | 仅1人负责全域，功能增加19项 | 协调增补资源 |
| 服务器选型采购延误 | 技术架构待确认项阻塞 | 技术架构师现场决策 |
| 机物料主数据方案未定 | 涉及4个业务域交叉 | 专题讨论+主数据人员到位 |
| MES工作范围未确认 | 集团侧数据协议接口表未定 | 领导层沟通确认 |

## 会议机制

### 双周例会

自2024年1月14日起固定双周例会，核心规则：
- **组织**：项目助理负责会议组织
- **资料**：执行经理输出会议资料
- **参会**：项目核心组成员持续参加
- **议程**：项目进展汇报 + 需协助事项 + 专题决策

### 跟进事项

首次双周例会识别8项跟进事项，覆盖四大类：

1. **方案质量提升**：顾问创新能力不足，需通过规划层面融入体现价值和高度；补充计划顾问、设备咨询顾问
2. **团队融合**：规划与实施两条产品线未充分融合，需现场顾问到位支持
3. **成本控制**：额外资源投入存在超责任制风险，需明确投入计划、推动研发项目立项
4. **客户关系**：明确蓝图汇报责任人，提前与高层做好沟通；拿出可复制的行业案例改善建议

## 业务域依赖关系

跨域依赖是项目协调的核心难点。依赖矩阵显示：

### 核心依赖链

```
EBS ←→ 生产管理 ←→ 工厂MES
  ↓         ↓           ↓
物资管理   质量管理    IOT
  ↓         ↓
计划管理   设备管理
```

### 高依赖域统计

| 业务域 | 对外依赖数 | 核心依赖方 | 典型依赖内容 |
|--------|-----------|-----------|-------------|
| EBS | 31 | 物资管理/计划/设备 | 采购订单、物料主数据、成本科目、子库 |
| 生产管理 | 20 | 计划/MES/仓库/质量/设备 | 工单获取→下发MES→投料→报工→完工→入库 |
| 物资管理 | 25 | EBS/无人过磅/质量/生产/中控 | 采购入库→过磅→质检→入库出库→EBS同步 |
| 设备管理 | 20 | EBS/IOT/MES/物资/生产 | 台账→维保→故障→报废全链路 |
| 质量管理 | 12 | 物资/无人过磅/FBC/MES/生产 | 原料入库感官→理化→批次→不合格评审 |

### 依赖方向模式

- **EBS→平台方向**：主数据（物料、客商、组织架构、子库、账户别名、物料成本）+ 业务订单（采购订单、销售订单、内部申请）
- **平台→EBS方向**：库存事务处理（入库明细、出库结果、子库转移、杂项出入库）+ 生产数据（投料消耗、完工入库、批次状态）
- **MES↔平台方向**：工单下发→工艺调整→投料记录→报工完工的双向数据流
- **IOT→平台方向**：设备状态采集、性能参数、报警信息的单向推送

关键设计决策：**平台作为数据中枢**，所有外围系统集成均通过平台中转，避免点对点直连带来的失控风险。

| 质量管理 | 31 | QC001–QC014 | 原料质检员、化验员、车间质检员、营养师 |
| 配方管理 | 6 | TH001–TH006 | 品管经理、营养师、中控员 |

### 角色-流程映射模式

项目识别出以下岗位参与多域流程的「交叉角色」，是协调重点：

| 角色 | 参与域 | 流程数 | 协调要点 |
|------|--------|-------|---------|
| 品管部经理 | 计划+质量+配方 | 5+ | 蓝图方案跨域拉通的关键接口人 |
| 中控员 | 计划+生产 | 4+ | 日计划执行的核心操作者，投料/换产/交接班 |
| 生产经理 | 生产+设备+质量 | 6+ | 异常审批、维保审核、绩效管理 |
| 原料保管员 | 物资+质量 | 3+ | 入库验收与质检发起的衔接 |

### 流程标准化实践

项目经历过流程图岗位名称不统一的问题，采取的治理措施：
1. 梳理完整的流程清单（从73支扩展至123支）
2. 统一流程标准名称和岗位名称
3. PMO不定期抽查，发现不规范立即要求组长改正
4. 各域涉及他域流程必须使用最新修订版名称

## 项目管理方法论提炼

### 计划方法论

1. **三层递进**：WBS功能分解（What）→ 接口分解（How to connect）→ 双周滚动（When），从结构到节奏逐步落地
2. **双周滚动节奏**：固定2周迭代周期，每轮包含计划→分解→执行→回顾闭环，避免长周期瀑布式失控
3. **日偏差可视**：日计划强制记录偏差原因与改善措施，形成可追溯的风险积累数据

### 协调方法论

1. **跨域依赖前置识别**：在蓝图阶段即梳理190+条跨域依赖关系，明确数据流向和对接模式
2. **交叉角色专项管理**：识别参与3个以上域的岗位作为协调关键点，减少信息传递损耗
3. **问题看板闭环**：问题跟踪看板覆盖提出→建议→处理→验证→闭环全链路，健康度评估高风险项指派专人跟踪

### 风险管控模式

| 风险类型 | 典型表现 | 应对策略 |
|---------|---------|---------|
| 资源瓶颈 | 单人负责全域，功能扩张后人力不足 | 及时协调增补，预留2周到位缓冲 |
| 技术阻塞 | 服务器故障/硬件到货延迟阻断关键路径 | 异常处理预案+并行推进不受阻工作 |
| 方案质量 | 顾问经验不足，方案缺乏变革深度 | 咨询参与+行业案例对标+原型验证 |
| 跨域冲突 | 流程命名不统一、审批流归属争议 | PMO标准化+专题决策会 |
| 进度偏差 | 开发与实施组节奏不一致 | 每周固定沟通日+功能清单联合评审 |
