# 通威农发-详细设计-功能设计

项目：**通威农发**

本文档汇总通威农发智能工厂项目详细设计阶段的核心功能设计，涵盖生产执行、仓储4大功能、EBS功能改造、开发规范共四个领域。来源为8份详细设计文档，已去除所有作者和出品机构信息。

## 生产执行

### 概述

生产执行模块面向饲料工厂的生产操作流程，覆盖从人员排班到工单完工的全链路管理。核心用户角色包括：中控技师、制粒膨化技师、工艺技师、打包员、班长。系统需同时兼容有工厂MES和无工厂MES两种场景。

### 人员管理

#### 生产排班

- **功能描述**：各公司按车间维度进行生产排班设置，定义班组与班制方案的绑定关系及开启时间
- **前置条件**：系统中提前维护班组信息
- **业务规则**：各班组根据生产排班起止时间进行产量统计；当班实际生产时间与排班计划时间不一致时需提前调整
- **核心字段**：车间名称、班制方案编码/名称、开启时间、创建人/时间

#### 生产排班表查询

- **功能描述**：根据班制方案、班次、班组信息生成日历式排班表，按月生成不跨月；支持班组工作时间调整、人员上岗/离岗及岗位调整
- **核心字段**：日期、每日排班信息（班制→班次→班组）

#### 人员维护

- **功能描述**：从人员档案同步"生产部"人员，维护岗位、车间、班组关联关系
- **核心字段**：工号、员工姓名、性别、性质（临时/正式）、状态（在职/离职）、岗位、部门、车间、班组

### 班制管理

- **功能描述**：定义公司车间班制方案（一班制/两班制/三班制）及班次信息（白班/晚班/工作时间）
- **后置条件**：为生产排班提供班方案数据
- **核心字段**：班制方案编码/名称、车间、状态、公司

### 任务管理

#### 预混稀释料管理

- **功能描述**：管理非微配生产方式的预混稀释料工单执行流程——品管员打印物料需求→预混稀释员领取物料→智能电子称精准配料→混合分装入库
- **前置条件**：提前维护混合机负载量、每批最小比例、混合时间
- **后置条件**：(1) 所有子批称重配盘后物料总用量传物资管理模块 (2) 工单报工后半成品数据传物资管理模块
- **核心字段**：工单编码、产品名称、计划数量、产量、主单位

#### 配料管理

- **功能描述**：预混稀释料称重配料管理界面。智能电子称自动称重，按混合子批自动记录称量数据；支持多个称重器具、多个容器切换、产前电子称校准
- **功能逻辑**：物料明细按计量(kg)值从大到小排序；子批号按子批序号从小到大排序
- **核心字段**：物料名称、理论量(kg)、定包重量(kg)、包数/计量(kg)、主单位、计划起止时间、实际开始时间、状态

### 生产批管理

#### 中控技师作业管理

中控技师是生产执行核心人员，承接计划模块下达的日工单进行生产执行。系统划分为三大功能模块：

| 模块 | 包含功能 |
|------|----------|
| 工单管理 | 工单拆分、工单合并、工单下发、工单撤消、清洗工单、产线指定、垛口指定、分装任务 |
| 工单执行 | 大料投料、小料投料、膏体投料、返工料添加、返工料入库 |
| 工单查询 | 历史工单信息检索查询 |

##### 工单拆分

- **功能描述**：将较大的工单计划数量拆分成较小的计划数量
- **前置条件**：子工单编码生成规则定义
- **后置条件**：子工单明细不传计划管理
- **业务规则**：已开启的工单不允许拆分
- **核心字段**：工单编码、拆分量、子工单编码、产品编码/名称、计划数量(T)、产线、计划起止时间、配方编码/版本、工艺路线编码/名称

##### 工单合并

- **功能描述**：对两个工单进行合并，生成新工单；前提是配方编码/版本号/工艺路线编码一致
- **业务规则**：用于相同产品、包装不同（仅编织带/标签不同）的合并生产
- **合并逻辑**：工单编码/产品/配方/工艺路线信息按计划数量最大的工单实例化；计划数量为所有工单汇总

##### 清洗工单

- **功能描述**：按照清洗规则约束创建清洗工单
- **后置条件**：清洗工单不传EBS和计划管理模块
- **清洗判定规则**：前/后工单中所含物料是否有"药物""色素"特殊属性
- **核心字段**：工单编码、清洗前/后生产工单及产品、清洗料产品名称/编码、计划用量/单位、工艺路线编码/名称/版本

##### 产线指定

- **功能描述**：中控技师为工单指定对应生产线体
- **核心字段**：车间、线体

##### 垛口指定

- **功能描述**：指定车间成品仓与垛口的绑定关系；一个成品仓可对应多个垛口
- **核心字段**：公司、车间、垛口编码/名称、成品仓编码/名称、创建/更新人员及时间

##### 分装任务

- **功能描述**：中控技师下发分装任务到指定岗位（小料投料岗/预混稀释料岗/投料岗），岗位操作员按任务进行物料分装准备
- **后置条件**：岗位操作员完成分装后，不触发物料领料单据到仓库管理模块
- **卡控逻辑**：计划数量-已分装量=计划分装量，下班中控技师录入数量大于计划分装量不允许确认
- **核心字段（主表）**：产品名称、计划数量、已分装量、计划分装量
- **核心字段（物料清单子表）**：物料编码/名称、物料分装总量（∑所有工单同一物料分装量）、主单位
- **核心字段（分装明细子表）**：产品/物料名称、计划数量、单吨用量(kg)、分装规格(kg/包)、包数

##### 返工料入库

- **功能描述**：管理在制品机尾料、清洗料、在制不合格品的入库流程
- **后置条件**：生成入库单传仓储管理模块
- **核心字段**：物料类型、工单编码、产品编码/名称、批次号、数量/单位、入库仓库、清洗流程、处置方式

##### 返工料添加

- **功能描述**：为在制工单添加返工料（清洗料/返工料）作为物料消耗
- **业务规则**：清洗料不参与品管部配方物料添加规则
- **后置条件**：返工料添加为班组工单物料消耗，传仓库管理模块
- **核心字段**：物料类型、工单编码、产品编码/名称、来源仓库、物料名称/编码/批次、数量/单位

#### 制粒膨化技师管理

- **功能描述**：制粒膨化技师操作界面，主要功能包括当单制粒膨化机工单执行、模具切换、岗位任务、设备运行异常监控、交接班、作业指导、异常发起
- **业务规则**：
  - 制粒膨化技师包含制粒技师和膨化技师，分别负责制粒工段和膨化工段
  - 一个制膨化技师可负责多条制粒线或膨化线
  - 换产任务和生产过程任务执行维度到设备
- **工单切换**：无工厂MES由技师手动选择；有工厂MES由施耐德系统自动切换推送

#### 交接班管理

交接班用于记录当班各岗位生产信息并转交下班人员，按岗位划分，存在PC端、一体机端、移动端三种操作终端：

| 岗位 | 终端 | 交接内容 |
|------|------|----------|
| 班长 | 移动端 | 工单完成/在制/未执行状态；生产异常（任务+设备）；注意事项 |
| 中控技师 | 桌面PC端 | 工单完成/在制/未执行状态；生产异常；注意事项；分装任务 |
| 制粒膨化技师 | 一体机端 | 在制工单进度；生产异常；注意事项 |
| 工艺技师 | 移动端 | 注意事项；清洁卫生 |
| 打包员 | 移动端 | 已完成/在制工单；注意事项 |

- **通用规则**：一个车间多个同岗人员时，任一人完成交接即算当班交接完成
- **打包员特殊逻辑**：提交接班后自动生成各在制工单生产入库单（含理论入库量）传WMS，WMS据此自动生成盘点单进行实物盘点后回传实际入库量

#### 作业记录

- **功能描述**：管理粉碎/超微粉碎工段作业记录。有工厂MES时自动接收施耐德启/停信息生成记录；无工厂MES时中控技师手动添加
- **计算逻辑**：粉碎效率 = 粉碎量 / (停止时间 - 启动时间)
- **核心字段**：设备名称、物料名称、起止时间、筛片规格(mm)、分级轮频率(mm)、喂料频率、主机电流、粉碎量(T)、粉碎细度

## 仓储4功能

### 定包接收

#### 功能描述

对预混料、编标物资进行定包方式接收入库。采购员在系统创建订单，通过汽运方式送货到厂，地磅计量只作为计量留痕。

#### 业务流程

```
勾选物料编码 → 根据车牌号绑定磅单 → 录入入库明细 → 确认入库
```

#### 功能逻辑

1. 将物料根据物料类别编码分为**原辅料**和**包装物**两个分区
2. 勾选需要绑定磅单的物料，点击【绑定磅单】：
   - 已绑定：显示"已绑磅单"信息
   - 未绑定：从"待绑磅单"栏选择，多个物料可绑定一个磅单，一个物料只绑定一个磅单
3. 录入入库明细（入库库位、垛型、总件数、总托数），点击【确认】新增
4. 【平均件重】：件重 = 净重/总件数；单位为件且件重非整数时提示"请确认净重"
5. 【确认入库】：
   - 单位为件时必须先进行平均件重
   - 编标物资单位分类为数量，只记件数不记重量
   - 无其他物料时返回主界面；多物料时弹窗选择下一入库物料

#### 核心字段

| 区域 | 字段 |
|------|------|
| 待绑磅单 | 物料编码、订单编号、供应商、物料名称、交货日期 |
| 磅单绑定 | 车牌号、磅单号、过磅时间 |
| 入库明细 | 物料名称/编码、车牌号、批次号、过磅单号、订单号、供应商、班组、毛重、均重、净重、单位、入库库位、垛型、总件数、总托数 |

### 散装销售

#### 功能描述

散装料销售出库。司机进厂提货经过无人过磅，填写纸质装货信息后仓储员安排实物装货，装货后司机二次过磅，市场后勤部文员根据两次磅差和装货信息进行开票并生成销售出库单扣减库存。

#### 业务流程

```
一次过磅 → 装车称重指定（选择库位）→ 封签采编 → 称重完成 → 回传EBS
```

#### 功能逻辑

1. 界面分四栏：**待发货**（一次过磅未装车）、**发货中**（装车过程中）、**已发货**（出厂已生成出库单，默认7天）、**未到场**（已开提货计划未一次过磅）
2. 一辆车多条提货计划，一条提货计划一个物料
3. 选中订单行→点击【称重指定】→查询物料库存→选择出库库位（单选）→【确认】生成无人过磅计量请求
4. 点击【封签】采集封签号：封签号可编辑、去重，一条物料至少一个封签号
5. 点击【称重完成】，回传EBS：重量以多次磅差汇总为准

#### 核心字段

| 区域 | 字段 |
|------|------|
| 出库任务 | 车牌号、过磅时间、过磅单号、一次过磅(重量)、计划订单数量 |
| 订单明细 | 车牌号、过磅时间、过磅单号、订单数量、一次过磅(重量)、计划总量、订单明细 |
| 称重指定 | 库位列表（有库存的库位） |
| 装车明细 | 出库库位、重量（不可编辑） |
| 封签采集 | 封签号列表 |

### 紧急放行（原料）

#### 功能描述

车间投料任务急需该物料，但理化检验结果未出时，由库管提出紧急放行申请。

#### 业务流程

```
选择待质检物料 → 创建紧急放行申请 → 审批 → 批次状态变为"紧急放行"
```

#### 功能逻辑

1. **待申请**：展示所有库存批次状态为"待质检"的物料，按批次号+物料编码汇总
2. 点击【创建】→输入申请原因（选填）→点击【确定】→提交申请
3. 审批通过：批次状态更新为"紧急放行"；审批驳回：保持"待质检"不变
4. **已审批**：默认展示近7天审批记录

#### 核心字段

物料名称、物料编码、子库、批次号、批次状态、件数、吨数、库位、供应商、入库时间

### 采购退货

#### 功能描述

对采购入库的不合格原材料进行退料。理化指标验证不合格后进行不合格评审，评审要求退货时生成退货申请单，审批结束后生成保管退货任务单；采购员通知供应商提货，司机入场过磅装车后仓储员做退货扣账处理。

#### 业务流程

```
不合格评审 → 生成退货任务 → 一次过磅 → 装车退货 → 二次过磅 → 确认退货 → 回传ERP
```

#### 功能逻辑

1. 展示统计：未退货/已退货数量（默认7天）
2. 任务明细字段：物料编码/名称、车牌号、合同编号、创建日期、计划重量、磅单号、供应商
3. 退货明细：按库位维度展示批次库存总量，支持下钻到对库明细，默认全部勾选
4. 退货量约束：实物库存 < 实际退货重量 < 实物库存 + 暂存区库存
5. 【确认退货】：车辆二磅后才能操作，处理账务回传ERP

#### 核心字段

| 区域 | 字段 |
|------|------|
| 退货任务 | 物料编码/名称、车牌号、合同编号、创建日期、计划重量、磅单号、供应商 |
| 退货明细 | 物料编码/名称、车牌号、批次号、过磅单号、供应商、合同编号、班组、实物库存、暂存区库存、毛重、净重、计划重量、实际重量 |

## EBS功能

### EBS现有功能改造

#### 业务背景

基于DIM平台上线，对现有业务进行操作系统切割，同时存在上线DIM与未上线DIM并行期的特殊情况，需要对EBS功能进行适配改造，使其既能兼容上线DIM工厂业务，也能兼顾未上线工厂业务。

#### 改造清单

| 序号 | 改造项 | 说明 |
|------|--------|------|
| 1 | 增加法人实体与库存管理弹性域 | 库存组织增加"是否已上线DIM"标识弹性域 |
| 2 | 销售订单值集列表修改 | 开票地点值列表修改为成品子库存对应地点+业务实体地点（已取消） |
| 3 | 销售订单头增加字段 | 收货单位、养殖单元、收货养户、退货申请号、调账标识、以销定采（行级复选框） |
| 4 | 销售订单车牌号 | 车牌号允许编辑，创建订单前必须维护磅单+车牌；车牌号为空不允许打印 |
| 5 | 销售订单增加打印 | 提货单打印模板聚合 |
| 6 | 销售订单执行状态查询接口修改 | DIM上线工厂查询逻辑调整：无发运数据取订单行状态，有发运数据返回"已完成" |
| 7 | 销售订单行增加装料顺序 | 新增LOADING_SEQ字段（NUMBER，必填） |
| 8 | 销售订单行待检量/数量控制 | 已上线DIM工厂从DIM平台取可销售库存、待检量，录入数量≤可销售库存 |
| 9 | 销售订单头/行取消 | 由提货计划创建的订单不允许手工取消；取消需调用DIM平台判定 |
| 10 | 销售订单头/行更新 | 发运前可在EBS修改多个字段；修改车牌需调DIM平台确认 |

### 销售提货计划

#### 业务背景

散装料需基于提货计划装车称重后，基于实际称重数据最近生成销售订单及财务数据；而现状称重前无系统支撑装货，称重后销售再按实际称重数量补销售订单。

#### 核心功能

创建提货计划，并基于提货计划装车结果自动生成销售订单。

#### 提货计划状态流转

```
已输入 → 已提交 → 散料称重完成 → 已创建订单 → 已登记订单（已收款） → 已发运 → 已关闭
                                                                   ↓
                                                              已取消
```

#### 提货计划头字段

| 字段 | 说明 |
|------|------|
| 业务实体 | 职责MO安全性配置文件下的OU，默认职责配置文件MO默认业务实体 |
| 提货计划单号 | 自动生成：4位OU短码+6位YYMMDD+4位流水 |
| 计划来源 | 手工创建默认"EBS"；外围系统导入为系统代码（YIDIANTONG/ISC） |
| 状态 | 已输入→已提交→散料称重完成→已创建订单→已登记订单→已发运→已关闭/已取消 |
| 客户名称 | 业务实体下有效客户 |
| 车牌 | 值列表：一次过磅无二次过磅的车牌；创建订单前可更新，修改需调DIM平台确认 |
| 订单类型 | 取系统参数默认订单类型 |
| 提货地点 | 成品子库存对应地点+业务实体地点 |
| 提货日期 | 默认创建日+1 |

#### 提货计划行字段

| 字段 | 说明 |
|------|------|
| 行号 | 自动序号 |
| 装料顺序 | 手工录入数值（必填） |
| 物料编码 | 业务类型控制范围值列表 |
| 数量 | 手工输入 |
| 辅助数量/辅助单位 | 物料启用辅助单位时可输入，自动换算 |
| 提货组织 | 业务实体下库存组织，默认物料默认发运组织 |
| 过磅单号/净重 | 取过磅表关联数据 |

#### 创建销售订单逻辑

- 状态为"散料称重完成"且自动创建复选框为N时可点击
- 按过磅传入净重创建销售订单，基于系统当前日期获取价目表价格
- 销售订单题头取值：订单类型取系统参数、发运组织取提货行提货组织、付款条件取提货计划题头
- 创建成功后将销售订单号回写提货计划题头

## 开发规范

### 通用规约

1. 命名严禁使用"拼音"或"拼音+英文"方式，杜绝不规范的缩写
2. 类名使用驼峰风格（DTO、VO等除外）
3. 抽象类以Abstract/Base开头，异常类以Exception结尾，测试类以Test结尾
4. 方法命名：get单对象、list多对象、count统计值、save/insert插入、remove/delete删除、update修改

### 代码规约

- 不允许未经预先定义的常量直接出现在代码中
- 大括号内为空简洁写{}，if/for/while/switch/do与括号之间加空格
- 4个空格缩进，禁止Tab字符
- 单行字符数限制不超过120个
- 迭代entrySet()获取Map的key和value
- 使用BigDecimal.valueOf()替代构造方法BigDecimal(double)
- 不允许使用java.sql.Date/Time/Timestamp
- 集合判空使用isEmpty()而非size()==0
- SimpleDateFormat线程不安全，不应定义为static变量
- 必须回收自定义ThreadLocal变量
- 字符串转化使用String.valueOf(value)
- 布尔类型变量不加is前缀，统一使用flag结尾
- 单个方法总行数不超过80行
- 不能使用过时的类或方法

### 数据库开发规范

1. 实体表使用英文单数，下横杠分割，小写命名；备份以bak_开头，分表以pt_开头，临时表以tmp_开头，日志表以log_开头
2. 非唯一索引命名为idx_列名1_列名2，唯一索引为uk_列名1_列名2
3. 字符集统一使用utf8mb4
4. 所有表和字段需添加注释
5. 小数精确计算使用decimal，大数据量考虑bigint
6. 时间类型统一用datetime，涉及时区用timestamp
7. 减少使用join/left join，拆分复杂查询
8. 减少使用视图、函数、触发器、存储过程

### 平台开发规范

#### 命名规范

- **app命名**：必须唯一，使用业务含义前缀
- **模型名**：必须唯一，表名转驼峰；@Model注解必写，需配置isAutoLog、isLogicDelete、isAbstract、description、name、tableName；类必须继承BaseModel
- **package**：新建时内部拉通确认
- **后端视图**：保证菜单ID唯一；按钮auth必须与模型auth方法一致，icon统一
- **前端扩展**：index.js写好备注；一个菜单一个文件夹；js文件命名为tiew_文件夹名_功能描述.js；自定义ID必须唯一且以菜单地址为前缀；必须手动加httpLoading

#### 开发规范

- 模型中不允许写方法（除可重写search和count）
- 有ER关系的主子列表，子表必须前端扩展
- 引擎版本2.1.0，新增逻辑删除字段（isLogicDelete=True），前后端交互ids改为String[]
- 避免循环里查询数据库

### 单元测试

1. 编写独立的单元测试，不依赖外部资源
2. 使用适当的测试框架（JUnit/TestNG）和断言库
3. 测试覆盖率达到预定目标
4. 每个单元测试只测试一个功能点或场景
5. 使用合适的测试数据覆盖正常/边界/异常情况
6. 验证预期的行为和结果

### 版本管理规范

- 使用Git进行代码版本管理
- 使用合适的分支策略（master/main、develop、feature、release、hotfix）
- 提交代码前进行代码审查
- 遵循代码合并流程，解决冲突并测试
- 建议使用Git Hooks进行代码质量检查
- 建议使用Git Flow等工具辅助分支管理
