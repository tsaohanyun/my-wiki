# 实战踩坑经验

> 项目：**通威农发** — 本页面记录项目实施中踩过的坑、遇到的问题和最终解决方案。

## SQL与存储过程

### 坑1：information_schema 引号转义陷阱

**症状**：从 `information_schema.ROUTINES.ROUTINE_DEFINITION` 获取存储过程定义后，重建时报语法错误。

**根因**：`ROUTINE_DEFINITION` 返回的是**已解析的文本**，存储过程里的 `''`（转义单引号）在查询结果中被还原成了 `'`。

```sql
-- 存储过程原始写法
WHEN 1 THEN ''合格''     ← 双写引号=一个引号

-- ROUTINE_DEFINITION返回
WHEN 1 THEN '合格'       ← 已还原，直接用于CREATE会报错
```

**解决方案**：
1. **首选**：用 `SHOW CREATE PROCEDURE proc_name` 获取带转义的完整定义
2. **备选**：用 `?` 占位符 + `EXECUTE USING @var` 替代CONCAT拼接

**来源**：2026-06-24 修改 processBatchQualityResult 存储过程

---

### 坑2：CONCAT参数结构被误改

**症状**：修改动态SQL时，把 `'',ids,''` 错误改成了 `'''', ids, ''''`，破坏了CONCAT的参数分割。

**根因**：没搞懂原始写法的意图就动手改了。`'',ids,''` 中的 `''` 是字符串的结束/开始引号，`,ids,` 是CONCAT的参数。

**教训**：
- 修改动态SQL前，**先理解CONCAT的字符串拼接结构**
- 不能想当然地全局替换

**解决方案**：彻底换掉CONCAT，改用 `?` 占位符 + `EXECUTE USING`

```sql
-- 旧写法（引号地狱）
SET @sql = CONCAT('...WHERE factory_id = ''', ids, '''...');

-- 新写法（干净安全）
SET @sql = '...WHERE factory_id = ?...';
PREPARE stmt FROM @sql;
EXECUTE stmt USING @ent_id;
```

**来源**：2026-06-24 修改 processBatchQualityResult 存储过程

### 坑3：存储过程修改的正确工作流

**错误流程**：
```
information_schema获取定义 → 修改 → 重建 → 报错 → 重试 → 循环...
```

**正确流程**：
```
SHOW CREATE PROCEDURE获取定义 → 理解原始逻辑 → 修改 → 语法预检 → 执行
```

**关键点**：
1. 用 `SHOW CREATE PROCEDURE` 而非 `information_schema`
2. 修改前先理解原始逻辑（特别是动态SQL的参数结构）
3. 输出前必须验证（引号匹配、括号匹配、占位符数量）

**来源**：2026-06-24 stored-procedure-safe-modify 技能创建

## 数据质量

### 坑4：lm_st_save_variable_cost 重复行问题

**症状**：BDSCFYJYE等指标计算结果偏大。

**根因**：源表存在两层问题叠加：
1. **精确重复行**：同一 `(company_id, company_name, date, cost_saving)` 组合出现多条（id不同但业务字段完全相同）
2. **同company_code关联多个company_id**：如 `2043`（越南公司）同时关联 `05bq93r29qzdw`（OU级）和 `0427l9ax70sui`（租户级），GROUP BY时被当作不同组分别聚合

**诊断SQL**：
```sql
-- 检查精确重复
SELECT company_id, company_name, date, cost_saving, COUNT(*) as cnt
FROM lm_st_save_variable_cost
WHERE company_code = '2043' AND date = '20251001'
GROUP BY company_id, company_name, date, cost_saving HAVING cnt > 1;

-- 检查同company_code多company_id
SELECT company_code, COUNT(DISTINCT company_id) as id_cnt
FROM lm_st_save_variable_cost GROUP BY company_code HAVING id_cnt > 1;
```

**修复方向**：源表去重 + ETL写入时统一company_id维度

**来源**：2026-06-17 绩效指标计算排查

## SP→Java迁移验证

### 坑5：UAT环境数据同步导致的假差异

**症状**：对比生产环境和UAT环境的存储过程结果时发现差异，怀疑Java代码有bug。

**根因**：UAT环境的源表数据与生产环境未完全同步，导致：
- `qms_sampleassay_son.result` 字段部分记录在PROD有值但在UAT为NULL
- 存储过程的 `WHERE result IS NOT NULL` 过滤条件放大了差异

**验证方法**：
1. 双库直连（pymysql同时连PROD和UAT）
2. Key级别逐条比对（业务key: inv_org_id + inv_item_id + material_batch + ...）
3. 源表追溯（逐层查batc_material → sampleassay_parent → sampleassay_son）

**结论**：
- fg管线差异0.14%，raw管线差异1.19%，process管线差异0.64%
- 差异全部来自UAT源表数据同步问题，Java代码逻辑正确

**正确验证方式**：在生产环境同时运行存储过程和Java代码，直接对比同库结果

**来源**：2026-06-25/26 fg/raw/process三条管线验证

## 环境与工具

### 坑6：MCP工具4000字符输出限制

**症状**：查询大表时返回结果被截断，只看到前4000字符。

**绕过方法**：
1. **LIMIT收窄**：`SELECT ... FROM big_table LIMIT 100`
2. **只查必要列**：`SELECT col1, col2` 而非 `SELECT *`
3. **聚合**：`SELECT COUNT(*), status GROUP BY status`
4. **分页**：`LIMIT 100 OFFSET 200`
5. **execute_code直连**：用Python pymysql绕过MCP限制

**来源**：2026-06-26 native-mcp 技能更新

## 通用教训

### 教训1：不理解就动手改 = 制造新bug

> 修改任何代码前，先理解原始逻辑的意图，特别是那些看起来"奇怪"的写法。

### 教训2：验证比生成更重要

> "生成SQL"不等于"完成任务"，"验证SQL"才是真正的完成。遵循 verification-before-completion 原则。

### 教训3：环境差异是最大的干扰项

> 对比两个环境的结果时，先排除数据同步问题，再怀疑代码bug。
