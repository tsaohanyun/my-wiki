# 第13章 数据质量管理 — 精读笔记

## 一句话总结

数据质量管理是一系列连续性职能活动，通过数据质量维度定义标准、嵌入质量要求于数据全生命周期，并以PDCA循环持续度量和改进，确保数据"适得其用"（Fit for Purpose）。

## 核心要点（7个关键点）

1. **数据质量管理是职能而非项目**：它是贯穿数据全生命周期的连续性工作，包含规划、实施和控制活动，不能仅靠一次性项目完成。持续运营需要持续资金保障。

2. **核心定义是"适得其用"**：数据质量定义为数据质量维度满足业务需求的程度。满足需求的数据即为有质量的数据，反之则为低质量数据。质量标准由数据消费者定义。

3. **预防优于纠正**：预防问题通常比纠正问题的成本低得多（有时低几个数量级）。核心原则是根因修正——理解问题根因并加以解决，而非停留在问题表象。"源解决"是最佳实践。

4. **数据质量维度提供标准化词汇**：常用维度包括有效性、完备性、一致性、完整性、及时性、唯一性、准确性、合理性。DAMA UK定义6个核心维度。维度框架（Strong-Wang、Redman、English等）为需求定义和持续度量提供基础。

5. **数据剖析是改进第一步**：使用统计技术了解数据集的真实结构、内容和质量，生成空值量、最大/最小值、字段分布等统计数据。剖析可识别潜在问题，但还需配合业务流程分析和血缘分析来分离根因。

6. **PDCA循环驱动持续改进**：计划（评估范围和优先级）→执行（解决根因）→检查（日常监测）→行动（处理新问题），形成数据质量改进的戴明环。当度量低于阈值、出现新数据集或规则变化时启动新循环。

7. **规则体系和SLA是制度保障**：数据质量规则应作为元数据管理，与业务影响挂钩，经领域专家确认。SLA规定关键边界数据质量问题的响应和改进期望，包括可接受阈值、通知机制和升级策略。

## 重要概念速查

| 术语 | 定义 |
|------|------|
| 数据质量 | 数据质量维度满足业务需求的程度，即"适得其用" |
| 关键数据元素（CDE） | 通过与业务驱动因素联系识别的、对组织和客户最重要的数据 |
| 数据质量维度 | 可度量的数据特征或特性，为定义数据质量需求提供标准化词汇 |
| 数据剖析（Profiling） | 使用统计技术检查数据集真实结构、内容和质量的分析形式 |
| 数据清洗（Cleaning） | 将数据转换为符合数据标准和领域规则的过程 |
| 数据质量指标（DQI） | 度量数据质量水平的量化指标，应具备可度量性、业务相关性等特性 |
| 根因分析（RCA） | 识别数据质量问题核心诱因的过程，常用帕累托、鱼骨图、5个Why等技术 |
| 数据质量SLA | 规定组织对关键边界数据质量问题响应和改进期望的协议 |
| Strong-Wang框架 | 聚焦数据消费者感知的4大类15维度框架（内在/依赖背景/表示性/可访问性） |
| 统计过程控制（SPC） | 通过分析过程输入输出变化来管理过程的方法，核心工具是控制图 |

## 方法论精华

### 数据质量改进生命周期（PDCA戴明环）

```
计划(Plan) → 执行(Do) → 检查(Check) → 行动(Act)
    ↑                                         │
    └─────────────────────────────────────────┘
```

- **计划**：评估已知问题范围、影响和优先级；基于根因分析制定解决计划
- **执行**：带头解决根因；非技术流程变更与技术变更协同实施
- **检查**：日常管理数据，主动监测质量；低于阈值时启动改进
- **行动**：处理新问题；新循环启动条件：低于阈值、新数据集、新质量要求、规则变化

### 数据质量评估流程（9步）

1. 明确评估目标 → 2. 识别待评估数据范围（聚焦小数据集）→ 3. 识别数据用法和消费者 → 4. 识别已知风险 → 5. 根据规则检查数据 → 6. 记录不一致程度 → 7. 深入分析（量化、排优先级、假设根因）→ 8. 与专家沟通确认 → 9. 使用评估结果作为规划基础

### 8项核心原则

| 原则 | 要点 |
|------|------|
| 聚焦关键数据 | 基于关键性及风险水平确定优先级 |
| 标准驱动 | 定义可度量的标准和期望 |
| 客观度量和透明 | 持续客观度量，与利益相关方共享方法 |
| 预防 | 重点放在防止数据出错上 |
| 根因修正 | 解决根因而非表象 |
| 嵌入业务流程 | 业务流程所有者对数据质量负责 |
| 体系化贯彻 | 系统所有者必须体系化落实质量要求 |
| 关联服务水平 | 报告和问题管理纳入SLA |

### 度量公式

```
有效数据质量 = (总测试次数 - 异常次数) / 总测试次数 × 100%
无效数据质量 = 异常次数 / 总测试次数 × 100%
```

## 实施要点

1. **从评估就绪状态开始**：评估组织对数据质量问题的认知、数据实际状况、相关风险、文化和技术准备情况。

2. **聚焦关键数据元素（CDE）**：主数据和参考数据通常被定义为关键数据。通过客户体验、效能效率、监管合规等维度识别CDE。

3. **IT/业务协作是关键**：业务人员了解数据含义，IT人员了解存储位置和方式。打破组织内部壁垒，建立协作机制。

4. **预防性措施优先**：建立数据录入控制、培训数据生产者、定义和执行规则（数据防火墙）、要求供应商提供高质量数据、实施数据治理责任制、建立正式变更控制。

5. **克服文化障碍**：应对"数据一直很糟糕"（依赖现状）、"他们需要处理"（责任推诿）、"没有我就做不了"（英雄主义）三种不良态度。

6. **资金保障**：改进项目资金应覆盖数据质量相关步骤；持续运营需要持续资金支持。记录成本和收益，优先进行数据改进。

7. **数据质量报告体系**：建立评分卡（高阶视图）、趋势报告、SLA指标、问题管理、治理合规、改进效果等多维度报告。

## 与其他章节的关联

- **第3章 数据治理**：数据质量问题通常是建立企业级数据治理的动因；治理组织提供优先事项指导、差异和冲突解决
- **第5章 数据建模与设计**：数据模型允许保留质量标志、关联业务规则；质量团队应与开发团队合作
- **第8章 数据集成与互操作**：数据流转过程是引发错误的常见源头；关键集成点监测可快速识别不良数据
- **第10章 主数据与参考数据**：主数据和参考数据是关键数据；劣质数据关键源头之一是主数据管理薄弱
- **第12章 元数据管理**：元数据定义数据表示含义，是明示数据质量期望的主要手段；质量度量结果应保存在元数据库中
- **第15章 成熟度评估**：数据质量成熟度评估有助于确定改进起点和推进路线图
- **第16章 组织与角色**：角色和责任的明确界定是质量管理有效实施的保障
- **第17章 变革管理**：质量改进方案的持续成功取决于组织文化的改变

## 考试/面试高频考点

1. **数据质量的8个维度**：有效性、完备性、一致性、完整性、及时性、唯一性、准确性、合理性——每个维度的定义和度量方法
2. **4大维度框架对比**：Strong-Wang（4类15维）、Redman（3类20+维）、English（固有+实用）、DAMA UK（6核心维）
3. **PDCA戴明环的4个阶段**及各阶段职责方和主要活动
4. **8项核心原则**（特别关注：预防优于纠正、根因修正、嵌入业务流程）
5. **根因分析的5种技术**：帕累托分析、鱼骨图、5个Why、追踪溯源、流程分析
6. **数据剖析与数据清洗的区别**：剖析是发现质量问题，清洗是纠正质量问题
7. **三种数据清洗方式**：自动化纠正、人工介入、手动纠正——各自的适用场景
8. **数据质量SLA应包含的8项内容**
9. **ROI分析的8个维度**：受影响数据关键性、数据量、可用期限、受影响流程、受影响人员、相关风险、修复根因成本、变通隐性成本
10. **统计过程控制（SPC）的三步应用**：控制→检测→改进
