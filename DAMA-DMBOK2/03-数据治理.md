# 数据治理

## 概述

数据治理（Data Governance, DG）是**对数据资产管理工作履行职权和实施管控（规划、实施、监督和执行）的行为**。无论组织是否正式确立了数据治理职能，它们都会对数据管理活动做出决策。但只有建立了正式数据治理职能的组织，才能更加有目的性地行使这些权力和实施管控，进而更好地从数据资产中获得价值。

数据治理职能可以**指导所有其他数据管理职能**。数据治理的目标是确保数据依照相关制度和最佳实践得到良好的管理。数据治理的重点不是直接管理数据本身，而是**如何对数据做出决策**，以及要求人员和流程在与数据相关的行为方面应该如何表现。

> **关键区分**：数据治理与数据管理的关系，类似于财务审计人员与财务管理的关系——数据治理的作用是确保数据被恰当地管理，而不是直接管理数据。数据治理代表**监督和执行之间固有的职责分离**。

数据治理职能独立于IT治理职能。IT治理专注于IT投资决策、IT应用组合、IT项目投资组合（硬件、软件及总体技术架构），以COBIT框架为标准。相比之下，数据治理只关注**数据资产管理及把数据作为资产来管理**。

## 核心概念

### 数据治理的目标

数据治理的目标是使组织能够将数据视为资产来管理。数据治理提供原则、制度、流程、框架、指标和监督，以便将数据作为资产管理并指导各层面的数据管理活动。

数据治理职能的具体范围和目标通常包括以下八个方面：

| 序号 | 范围 | 说明 |
|------|------|------|
| 1 | **战略** | 界定、沟通和推动数据战略和数据治理战略的执行 |
| 2 | **制度** | 制定和执行与数据及元数据生命周期（创建、访问、使用、安全、质量、文档化和处置）管理相关的制度或政策 |
| 3 | **标准** | 制定和执行数据管理标准 |
| 4 | **监督** | 在质量、隐私、安全和数据管理制度和规程等关键数据管理领域实际开展监察、审计和纠正工作 |
| 5 | **合规** | 确保组织能够满足数据相关法规的合规性要求 |
| 6 | **问题管理** | 识别、界定、提升和解决与数据安全、数据访问、数据质量、法规合规、数据所有权、制度、标准、术语或数据治理规程相关的问题 |
| 7 | **数据管理项目** | 发起改善数据管理实践的举措 |
| 8 | **数据资产估值** | 为评估数据资产的业务价值制定一致的标准和流程 |

为了实现上述目标，开展数据治理时应制定相应的制度和规程，在组织的不同层级推进数据管理实践活动，参与组织变革管理工作，积极向组织传达改善数据治理的优势，并宣传和倡导将数据作为资产进行成功管理所需的行为。

### 数据治理的业务驱动因素

最常见的数据治理驱动因素是**监管合规**，尤其是金融、医疗健康等强监管行业。数据治理的驱动力通常归结为**降低风险**或**优化流程**两大类。

**降低风险：**

- **一般风险管理**：监督风险数据对财务和声誉的影响，包括对法律（电子举证）和监管问题的应对
- **数据安全**：通过对数据的有效性、可用性、完整性、一致性、可审计性及安全性的控制来保护数据资产
- **隐私**：通过制度和合规性监控来控制隐私/机密/个人身份信息（PII）

**优化流程：**

- **监管合规**：对监管要求能够高效、一致地做出响应的能力
- **数据质量提升**：通过使数据更可靠，提高业务绩效的能力
- **元数据管理**：建立业务术语表以定义和定位组织内部数据，确保管理向组织提供广泛的其他元数据的能力
- **开发项目的效率**：改进软件开发生命周期（SDLC），以应对整个组织数据管理中的问题和机遇的能力，包括通过对数据生命周期的治理来管理数据特有的技术负债
- **供应商管理**：处理与数据相关的合约（如云存储、外部数据采购、数据产品销售和外包数据运营）并进行控制的能力

> **提示**：在组织内部，明确数据治理的特定业务驱动因素并将其与整体业务战略保持一致至关重要。过分强调"数据治理组织"往往会让部分领导层感到疏远。对组织文化的敏锐把握至关重要，需要使用恰当的描述、运营模式和角色。

数据治理是**持续性的**，并非一次性的，聚焦于从其数据中获取价值并降低相关风险。数据治理团队可以是一个具备特定责任的虚拟组织或垂直机构。数据治理职能应当考虑到特有的组织和文化问题，以及组织内部具体的数据管理挑战和机遇。

### 治理与管理的区别

| 维度 | 数据治理 | 数据管理 |
|------|----------|----------|
| **定义** | 对数据资产管理工作履行职权和实施管控（规划、实施、监督和执行）的行为 | 执行数据管理活动（创建、存储、维护、使用、处置等） |
| **角色定位** | 监督者——确保数据被恰当地管理 | 执行者——直接管理数据 |
| **核心职能** | 立法（制定制度、标准）、司法（问题管理与修复）、行政（保护和服务） | 执行具体的管理活动 |
| **典型活动** | 制定政策、审批标准、监督合规、问题升级 | 建模、存储、集成、质量检查、安全控制 |
| **决策层级** | 战略与政策层面 | 操作与执行层面 |

### 数据治理的六项原则

实施数据治理职能需要致力于变革，以下原则能够帮助数据治理奠定坚实的基础：

1. **领导力和战略**：成功的数据治理始于有远见、有担当的领导。企业业务战略驱动数据战略，数据战略引领数据管理活动
2. **业务驱动**：数据治理是一项业务职能，必须像治理与数据交互的业务一样治理与数据相关的IT决策
3. **共担责任**：在所有数据管理知识领域，数据治理是业务数据管理专员和技术数据管理专员的共同责任
4. **多层级的**：数据治理在企业全局或局部范围内都有发生，并且经常发生在中间各层级
5. **基于框架**：因为数据治理活动需要跨职能、跨领域进行协调，所以数据治理职能必须建立一个能界定责任与交互的运营框架
6. **基于原则**：指导原则是数据治理活动的基础，对于数据治理制度尤其如此。通常，组织会制定不含正式原则的制度——试图解决特定的问题。有时候原则能从制度中逆向构建。然而，作为制度工作的一部分，最好开始就能阐明一套核心原则和最佳实践。引用原则能够缓和潜在的阻力。随着时间的推移，额外的指导性原则还会在组织内部形成

数据治理职能必须具备三个核心特性：

- **可持续**：数据治理不是一个具有明确结束时间的项目，而是一个需要组织承诺能持续投入的企业职能。可持续的数据治理依赖于业务领导力、资金和所有权
- **可融合**：数据治理不是一个附加流程，需要融入软件开发方法、数据分析、主数据管理、风险管理及其他活动
- **可衡量**：良好的数据治理能够产生积极的财务影响，需要了解初始情况并规划可衡量的方案

### 以数据为中心的组织

以数据为中心的组织将数据视为资产，并在数据整个生命周期的所有阶段都对其进行管理，包括项目开发和持续运营。要转向以数据为中心，组织必须改变将战略转化为行动的方式。数据不应被当作业务流程和应用的附属，确保数据的高质量也是业务流程的目标之一。

人们很容易将数据、系统和基础设施混为一谈。要成为以数据为中心的组织，需要以不同的方式进行思考，并认识到管理数据与管理IT的不同。但要想完成上述转变并不容易，现行文化、内在政治、纠缠不清的所有权、预算竞争及遗留系统等，都是企业建立数据治理和数据管理愿景的巨大障碍。

寻求从自身数据中获取更多价值的组织通常认同：

1. 数据应当作为公司资产进行管理
2. 数据管理最佳实践应该在整个组织层面得到激励
3. 公司数据战略必须直接与整体业务战略保持一致
4. 数据管理流程应得到持续改进

### 数据制度

数据制度是将数据管理原则和管理意图整编为**指令性的基础规则**，用于管理数据和信息的创建、获取、完整性、安全性、质量和使用。

- 数据制度是全局性的，支持数据标准及与数据管理和使用相关的关键行为预期
- 数据制度描述了数据治理要"做什么"和"不做什么"，而数据标准和数据流程描述了数据治理"如何做"
- 数据制度的数量设置应当相对较少，陈述应简明扼要
- 必须有效地沟通、监测、强制执行和定期复评数据制度，数据治理委员会可授权给数据管理指导委员会

制度可以有不同的外在表现，示例如下：
1. 数据治理办公室（DGO）将保证组织的数据使用
2. 数据拥有者由数据治理办公室批准
3. 数据拥有者将依据其业务能力领域划分指定数据管理专员，数据管理专员将负责协调数据治理活动的日常职责
4. 最大限度地提供标准化报表/仪表盘/计分卡，以满足大多数业务需求
5. 授权用户将有权访问特定数据以进行临时/非标准报告
6. 定期评估所有关键数据，以评价其准确性、完整性、一致性、可访问性、唯一性、合规性和效率等

### 数据标准

标准提供了一种比较的方法，有助于定义质量和简化流程。通过采用标准，组织只需做一个决定并将其编入一套规则，而不需再为每个项目重新做一次相同的工作。

数据标准可以采取不同的形式：推断如何填充一个字段、管理不同字段间关系的规则、记录可接受和不可接受值的详细记录、格式等。数据标准通常由数据管理专业人员起草，由数据治理委员会或者指定工作组（如数据标准指导委员会）评审、批准和采用。

> **注意**：记录数据标准提供了一个捕捉细节和知识的机会。相较在前期就将其记录下来而言，通过再造或者反向工程来获取这项知识的代价更大。

对于数据标准，必须进行有效沟通、监测和进行周期性的评审与更新。最重要的是，必须采用适当的方法执行。数据治理委员会或者数据标准指导委员会依据合规标准，按既定日程或者按软件开发生命周期审批流程对数据管理活动进行审计。

**数据管理规程**是一种记录的方法、技术和步骤，用于完成特定活动，产生确定的结果和成果物。与制度和标准一样，不同组织的流程差异较大。

在数据管理知识领域内可以被标准化的概念包括：

| 知识领域 | 标准化内容 |
|----------|-----------|
| 数据架构 | 企业数据模型、工具标准、系统命名规则 |
| 数据建模和设计 | 数据模型管理规程、数据建模命名规则、定义标准、标准域和标准缩写 |
| 数据存储和运营 | 工具标准、数据库恢复标准、业务连续性标准、数据库性能、数据保留、外部数据获取 |
| 数据安全 | 数据访问安全标准、监测和审计规程、存储安全标准、培训需求 |
| 数据集成 | 数据集成和互操作的标准方法和工具 |
| 文档和内容 | 内容管理标准和规程、企业分类法、法律搜索支持、文档和邮件保留期限、电子签名、报告分发方法 |
| 参考数据和主数据 | 参考数据管理控制规程、记录系统、建立和强制使用声明、实体解析方案标准 |
| 数据仓库和商务智能 | 工具标准、处理标准和规程、报告和可视化格式标准、大数据处理标准 |
| 元数据 | 业务和技术元数据采集标准、元数据整合规程和使用情况 |
| 数据质量 | 数据质量规则、标准衡量方法、数据修复标准和规程 |
| 大数据和数据科学 | 数据源识别、权限、获取、记录系统、共享和更新 |

### 数据资产估值

数据资产估值是了解和计算数据对组织具有多少经济价值的过程。由于数据乃至商务智能都是抽象的概念，人们很难将其与经济影响联系起来。要理解无形资产（如数据）的价值，关键在于理解它的使用方式和使用价值。与许多其他资产（如货币、物理设备）不同，数据集不可替换（替代）。

数据生命周期的大部分阶段（包括获取、存储、管理及处置）都涉及成本。数据只有被使用才会产生价值。当数据被使用时，也会产生相应的风险管理成本。因此，当使用数据的收益超过获取、存储数据的成本，以及使用数据带来的风险时，使用数据就会产生价值。

衡量数据资产价值的方法：

| 方法 | 说明 |
|------|------|
| **替换成本** | 在灾难或数据泄露中丢失的数据的替换或恢复成本，包括组织内的交易、域、目录、文档和指标等数据 |
| **市场价值** | 企业并购时作为商业资产的价值 |
| **识别商机** | 通过利用数据（如在商务智能中）识别出的机会可以获得的收入价值，包括通过将数据用于交易或出售所获得的收入 |
| **销售数据** | 将数据打包后作为商品出售，或把从自身数据中获得的洞察作为商品出售 |
| **风险成本** | 根据可能受到的处罚、补偿成本、诉讼花费计算的价值，源于：①缺失必要的数据；②出现不该出现的数据（如发现法律规定不可留存的数据，应清除而未清除）；③不正确的数据导致客户、公司财产乃至声誉上的损失；④风险和风险成本的减少被用于改善和认证数据的运营干预成本所抵消 |

**定义数据资产估值方法**：数据治理委员会应该组织开展此工作并为此制定标准。一些组织通过估算因信息不充分造成的损失来计算数据资产的价值。信息鸿沟（需要什么信息与可以得到什么信息之间的差异）代表商业负债，弥合或者防止这一鸿沟的成本被用于估计缺失数据的业务价值。自此，组织能够开发模型用于估计现有信息的价值。价值估算可以被嵌入数据战略，这将为数据质量问题的根治方案和其他数据治理举措带来业务场景，从而论证这些方案和举措的合理性。

## 方法论

### 数据治理运营模型

数据治理运营模型分为三种类型：

| 模型 | 特征 | 适用场景 |
|------|------|----------|
| **集中式（Centralized）** | 一个数据治理机构监督所有主题域的全部活动 | 组织规模较小或治理初期 |
| **复制式（Replicated）** | 每个业务单元采用同样的数据治理运营模型和标准 | 多业务单元但流程一致的组织 |
| **联邦式（Federated）** | 一个数据治理组织与多个业务单元协调以维持一致的定义和标准 | 大型复杂企业 |

数据治理可借用政治治理来理解，拥有：
- **立法职能**：定义制度、标准、企业数据架构
- **司法职能**：问题管理和问题修复
- **行政职能**：保护和服务、行政责任

### 数据治理实施流程

数据治理的实施采用渐进式策略，核心流程包括四大阶段：

#### 阶段一：定义组织的数据治理

**1. 执行就绪评估**

| 评估类型 | 说明 |
|----------|------|
| 数据管理成熟度 | 了解组织使用数据做什么，衡量其当前数据管理能力和负荷。主要关注业务人员对公司在管理数据和使用数据发挥其优势的印象如何，如工具使用、报告水平等 |
| 变革能力 | 评估组织对数据治理所需变革的能力，识别潜在阻力。变革过程将评估现行组织架构、文化理念，以及变革管理过程本身 |
| 协作准备度 | 评估组织在管理和使用数据时的协作能力。管理职责的定义跨越了功能领域，因此具有协作性质。永远不要假设一家组织知道如何进行协作 |
| 业务一致性 | 检验组织在多大程度上将数据的使用和业务战略对齐。令人惊讶的是，发现与数据相关的活动往往是如此临时和随意 |

**2. 对齐业务执行探索**

数据治理职能必须通过识别并交付具体的利益（如减少向监管机构缴纳的罚款额）为组织做出贡献。探索活动包括：
- 识别和评估现行制度及指南的有效性——应对何种风险、鼓励何种行为、实施效果如何
- 数据质量分析：评估现有问题和困难，以及与低质量数据相关的影响和风险（见第13章）
- 数据管理实践评估：识别核心用户，创建潜在数据管理推动者的初始名单
- 从探索和协调一致的活动中得出数据治理需求清单

**3. 创建组织触点**

数据治理职能在更广泛的业务和数据管理战略背景下，通过以下触点与组织各领域建立连接：

| 触点领域 | 说明 |
|----------|------|
| 采购与合同 | 与供应商/合作伙伴管理或采购部门合作，针对数据管理合约（DaaS、云相关采购、第三方开发、内容采购/授权交易、IT工具采购和升级）制定和执行标准合约语言 |
| 预算与资金 | 作为防止重复投资和确保数据资产优化的协调中心 |
| 制度合规 | 了解法规和制度如何影响组织及其数据管理活动，持续监测、识别并跟踪新的及潜在的影响和要求 |
| 软件开发生命周期 | 在系统或应用开发生命周期中识别能够建立企业制度、流程和标准的控制点 |

#### 阶段二：制定数据治理战略

数据治理战略界定了治理任务的范围和目标，应全面制定并与总体业务战略、数据管理和IT战略共同发布。交付成果一般应包括：

1. **章程**：确定数据治理的业务驱动因素、愿景、使命和原则，包括准备度评估、内部流程发现和当前问题或成功标准
2. **运营框架和责任**：为数据治理活动定义组织架构和职责
3. **实施路线图**：制度和指令集推出的时间框架、业务术语表、架构、资产估值、标准和规程、预期的业务和技术流程变革、支持审计活动和法规遵从的交付成果
4. **规划运营目标**：描述可持续的数据治理活动的目标状态

**构建运营模式须考虑：**
- 数据对组织的价值：如果组织以出售数据为生，数据治理拥有巨大的商业影响力
- 商业模式：去中心化与中心化、本地化与全球化等是影响商业模式的主要因素
- 文化因素：纪律服从程度和变革适应能力。一些组织会以制度和原则为借口抵制治理的介入
- 监管影响：相对于弱监管的组织，受到强监管的组织将有不同的数据治理理念和运营模式

**制定目标、原则和制度**：通常由数据管理专业人员、业务制度人员或者这些人员的组合在数据治理委员会（DGC）的支持下起草，由数据管理人员和管理团队进行评审和优化，由数据治理委员会或类似机构做终审、修改和采纳。

**推动数据管理项目立项**：数据治理委员会负责界定业务场景，并监督项目状态和数据管理提升项目的进度。数据治理委员会要与项目管理办公室（PMO）协同分工。数据治理委员会也可能与企业级重大项目（如ERP、CRM、MDM等）进行协调以提升数据管理水平。

#### 阶段三：实施数据治理

早期优先活动：
1. 定义数据治理流程以满足高优先级目标
2. 建立业务术语表，记录术语和标准
3. 与企业架构和数据架构协调，以便更好地理解数据和系统
4. 为数据资产分配经济价值，以便更好地决策和理解数据如何促进组织的成功

**与架构团队协作**：数据治理委员会是数据架构的利益相关方，可能发起并批准数据架构组件，如面向业务的企业数据模型。数据治理委员会可能会与企业数据架构指导委员会或架构评审委员会（ARB）互动，以评审工程及其迭代项目。企业数据模型应当由数据架构师和数据管理人员在主题域团队中一同开发和维护。企业数据模型应当由数据治理委员会审查、批准和正式采用。该模型必须与关键业务战略、流程、组织和系统协调一致。在管理数据资产时，数据战略和数据架构是协调"把事情做正确"和"做正确的事情"的关键。

#### 阶段四：嵌入数据治理

数据治理组织的一个目标是将数据治理活动嵌入将数据作为资产的一系列相关流程。数据治理职能的持续运作需要规划。运营计划包含实施和运行数据治理活动所需的事件列表，概述了维持成功所需的活动、时间安排和技术。

可持续性意味着确保流程和资金到位，以保证组织数据治理框架持续运作。这一要求的核心是，组织接受数据治理，管理职能，监测和衡量结果，克服经常导致数据治理职能受阻和失败的障碍。

为了深化组织对于数据治理的总体理解，可以创建**数据治理兴趣社区**，以便数据治理应用的本地化和员工相互学习。这在实施数据治理的第一年特别有用，但会随着数据治理运营的成熟而逐步减弱。

### 问题管理流程

问题管理是识别、量化、划分优先级及解决数据治理相关问题的过程，包括：

| 问题类别 | 说明 |
|----------|------|
| 权力 | 关于决策权和程序的问题 |
| 变革管理升级 | 变革管理过程中引发的问题 |
| 合规 | 与满足合规需求相关的问题 |
| 冲突 | 制度、规程、业务规则、名称、定义、标准、架构、数据所有权等的冲突 |
| 一致性 | 与制度、标准、架构、规程一致性相关的问题 |
| 合同 | 商议和审查数据共享协议、采购和销售数据及云存储 |
| 数据安全和身份识别 | 隐私和保密问题，包括违规事件调查 |
| 数据质量 | 检测并解决数据质量问题，包括灾难事故、安全漏洞 |

问题升级路径：数据管理团队 → 业务单元数据治理委员会 → 数据治理委员会 → 数据治理指导委员会

数据治理需要以下控制机制和规程：
1. 识别、获取、记录、跟踪和升级问题
2. 分配和跟踪行动事项
3. 记录利益相关方的观点和可选的解决方案
4. 决定、记录和沟通问题解决方案
5. 促成客观、中立的讨论以便听取所有观点
6. 将问题升级到更高层级的权力机构

数据问题管理是非常重要的。它建立了数据治理团队的信誉，对数据消费者有直接、积极的影响，并且减轻了产品支持团队的负担。解决问题也证明数据能够得到管理，并且其质量可以得到提升。

### 变革管理

数据治理所需的正式性和纪律性往往与组织现有实践存在差异，因此变革管理是可持续发展的关键。对于许多组织而言，数据治理要求人们改变其行为和互动方式。一个正式的组织变革管理计划，加上合适的高管赞助，是推动所需行为改变的关键。

变革管理团队职责：
- **规划**：执行利益相关方分析、获取高层支持和建立一种克服抵触变革情绪的沟通方法
- **培训**：为数据治理职能创建和执行培训规划
- **影响软件系统的开发**：与项目管理办公室一道将数据治理步骤加入软件开发生命周期
- **制度实施**：沟通数据制度和组织对数据管理活动的承诺
- **沟通**：增强数据管理人员和其他数据治理专业人员的角色和责任意识，提升数据管理项目目标和期望

沟通对于变革管理过程非常重要，应关注以下方面：
1. **推广数据资产价值**：教育和告知员工关于数据在达成组织目标过程中的角色
2. **对反馈开展监测和行动**：积极寻求和使用利益相关方的反馈信息，有助于建立对数据治理目标的承诺
3. **开展数据管理培训**：组织各个层级的培训，以增强对数据管理最佳实践和流程的意识
4. **衡量变革管理的效果**（五大重点领域）：
   - 对变革需求的**意识**
   - 参与和支持变革的**愿望**
   - 如何变革的**知识**
   - 执行新技能和行为的**能力**
   - 保持变革有序的**力度**
5. **执行新指标和KPI**：对员工的激励应当与支持数据管理最佳实践的相关行为一致。激励机制也应当鼓励跨部门的活动和协作

### 法规遵从

全球性法规对数据管理实践有重大影响：

| 法规/标准 | 说明 |
|-----------|------|
| 会计标准 | GASB和FASB审计标准对信息资产管理方式有重大影响 |
| BCBS 239 / Basel Ⅱ | 巴塞尔银行业监管委员会有效风险数据加总和风险报告原则，欧盟金融机构需报告标准信息以证明流动性 |
| CPG 235 | 澳大利亚审慎监管局管理数据风险的标准，关注数据风险源头和全生命周期数据管理 |
| PCI-DSS | 支付卡行业数据安全标准 |
| Solvency Ⅱ | 欧盟保险行业偿付能力监管要求 |
| 隐私法案 | 地方、主权国家和国际法 |

合规评估核心问题：
1. 哪些方面的监管要求与组织相关？
2. 怎样合规？需要哪些制度和规程实现合规？
3. 何时要求合规？如何监测合规？
4. 组织能否采用行业标准达到合规？
5. 如何证明合规？
6. 违规的风险和处罚是什么？
7. 如何识别和报告违规行为？如何管理和整改违规情况？

数据治理监测组织应对监管需求或者涉及数据和数据实践的审计进行响应，如在合规报告中证明数据质量。

## 组织与角色

### 数据治理组织

每个组织应当采取一种能支持其业务战略，并有可能在其内部文化氛围背景下取得成功的数据治理模型。组织也应当做好准备，不断调整该模型以应对新的挑战。数据治理机构可以具有多个层级，以回应大型复杂企业内不同级别的关切——包括本地的、业务部门的和企业全局的。数据治理工作通常在多个委员会之间进行分工，每个委员会的目的和监督水平都不相同。

典型数据治理委员会/机构：

| 机构 | 层级 | 职能 |
|------|------|------|
| **数据治理指导委员会** | 企业级 | 最高决策机构，处理升级问题 |
| **数据治理委员会（DGC）** | 企业级 | 审查、批准和采用数据标准和架构，监督数据管理项目 |
| **数据管理指导委员会** | 企业级 | 由DGC授权，执行数据管理日常指导 |
| **架构评审委员会（ARB）** | 企业级 | 评审工程及迭代项目 |
| **业务单元数据治理委员会** | 业务单元级 | 处理本业务单元的数据治理问题 |
| **主题域团队** | 跨职能 | 数据架构师和数据管理人员共同开发和维护企业数据模型 |

> **说明**：上表并非某一个委员会/机构的组织架构图，而是用于诠释组织内各个机构如何通力合作共同执行数据治理职能。

### 关键角色与职能

#### 首席数据官（CDO）

- 数据治理职能的最高负责人
- 创建组织触点，连接采购与合同、预算与资金、制度合规、软件开发生命周期等领域
- 向"首席"高管（如首席风险官、首席财务官）汇报

#### 数据管理专员（Data Steward）

数据管理专员是为了组织的最高利益代表他人来管理数据资产的人。他们需要从企业全局角度确保数据的高质量和有效利用。优秀的数据管理专员对数据治理活动负有责任，并会定期投入时间专注参与这些活动。

数据管理专员的核心职责：

| 职责 | 说明 |
|------|------|
| 创建和管理核心元数据 | 定义和管理业务术语、有效数值和其他关键元数据。数据管理专员通常负责组织的业务词汇表，该词汇表将成为与数据相关的业务术语的权威来源 |
| 记录规则和标准 | 定义业务规则、数据标准和数据质量规则并将其文档化。用于界定预期的高质量数据的规则经常深藏于产生数据或者消费数据的业务流程之中 |
| 管理数据质量问题 | 参与识别和解决数据相关问题或促进解决过程 |
| 执行日常数据治理活动 | 确保数据治理制度和举措在日常工作和项目中得到实施。应当影响决策以确保数据管理的方式支持组织整体目标的实现 |

数据管理专员的类型：

| 类型 | 说明 |
|------|------|
| **首席数据管理专员（Chief Data Steward）** | 可代替CDO主持数据治理机构，或者可以在类似委员会的虚拟组织或其他分布式数据治理机构中充当CDO，也可能直接是管理层主管领导 |
| **高级数据管理专员（Executive Data Steward）** | 在数据治理委员会任职的高级管理人员 |
| **企业数据管理专员（Enterprise Data Steward）** | 对跨业务职能的数据域进行监督 |
| **业务数据管理专员（Business Data Steward）** | 业务专业人员，通常被认为是主题领域专家（SME），负责一个数据子集。与利益相关方一同定义并控制数据 |
| **数据所有者（Data Owner）** | 业务数据人员，拥有其管辖范围内有关数据的决策审批权 |
| **技术数据管理专员（Technical Data Steward）** | 在某一知识领域内进行操作的IT专业人员，如数据集成专家、数据库管理员、商务智能专家、数据质量分析员或者元数据管理员 |
| **协调型数据管理专员（Coordinating Data Steward）** | 领导并代表业务和技术数据管理专员团队参与跨团队讨论及与高层数据管理专员之间的讨论，在大型机构中尤为重要 |

> **DAMA经验**：DAMA-DMBOK 1指出"最优秀的数据管理专员通常是被发掘出来的，而非刻意培养的"——即使没有设立正式的数据治理职位，也会有人管理数据。正规化他们的管理职责和责任化他们正在进行的工作，能够使他们更成功。但数据管理专员也可以"培养出来"，人们可以接受培训从而成为数据管理专员。

#### 其他参与者

| 角色 | 职能 |
|------|------|
| 企业高管 | 提供战略方向和高层支持 |
| 主题领域专家 | 提供业务专业知识 |
| 成熟度评估师 | 评估组织数据管理成熟度 |
| 监管者 | 提供合规要求输入 |
| 企业架构师 | 协调数据架构 |

### CDO组织触点

数据治理职能部门通过以下触点与组织各领域建立连接：

- **采购与合同**：针对数据管理合约（DaaS、云相关采购、第三方开发等）制定和执行标准合约语言
- **预算与资金**：作为防止重复投资和确保数据资产优化的协调中心
- **制度合规**：持续监测、识别并跟踪新的法规影响和要求
- **软件开发生命周期**：在系统或应用开发生命周期中识别控制点

数据治理职能影响的触点支持了组织管理其数据的连贯性，因此可提高其使用数据的灵活度。本质上，这是一张关于组织如何感知到数据治理的蓝图。

## 工具与方法

数据治理基本上都是组织行为，但整个过程离不开工具的支持。数据治理需要持续沟通，应当充分持续利用现有沟通渠道以一致的方式传递关键信息，使得利益相关方知晓相关制度、标准和需求。

此外，数据治理职能必须有效管理其自身工作和自有数据。在选择特定功能的工具（如业务术语表解决方案）之前，组织应当定义其总体治理目标和需求，并着眼建造工具集。例如，一些业务术语表解决方案包括用于制度和工作流管理的附加组件。

### 核心工具

| 工具类型 | 说明 |
|----------|------|
| **在线展示/网站** | 提供文档库、搜索功能、简单工作流程管理，建立数据治理职能品牌 |
| **业务术语表** | 包含已达成统一定义的业务术语并与数据相关联的核心工具 |
| **工作流工具** | 管理流程（如实施新的数据治理制度），将流程与文档相连接 |
| **文档管理工具** | 协助管理制度和程序 |
| **数据治理计分卡** | 收集指标跟踪数据治理活动和制度合规情况 |

### 业务术语表

业务术语表的内容一般由业务数据管理人员负责。因为人们说话用词的习惯不同，所以制定业务术语表是非常必要的。由于数据代表的是其他事物，所以对数据进行清晰的定义特别重要。定义必须清楚、用词准确，并可解释任何特例、同义词或者变体。术语审批者应包括核心用户群体代表。数据架构通常可以提供主题域模型的定义草稿和类型分解。

业务术语表的目标：

1. 形成对核心业务概念和术语的共同理解
2. 降低由于业务概念理解不一致而造成的数据滥用风险
3. 提升技术资产（根据技术命名规则）与业务组织之间的协调性
4. 最大限度提升搜索能力使得组织知识得以访问

业务术语表不仅仅是术语和定义的列表，每个术语还与其他有价值的元数据相关联，如同义词、指标、血缘、安全和隐私等级、业务规则、负责该术语的管理员等。

### 数据治理网站内容

数据治理职能的网站应当包括：
1. 数据治理战略和职能章程（愿景、收益、目标、原则和实施路线图）
2. 数据制度和数据标准
3. 数据管理的角色和责任描述
4. 工作新闻发布
5. 数据治理论坛的链接
6. 有关数据治理主题的高管消息链接
7. 数据质量度量报告
8. 问题识别和调整更新的规程
9. 请求服务或问题反馈链接
10. 文档、演示文稿和培训计划，附带相关在线资源的链接
11. 数据治理职能的联系信息

## 度量指标

数据治理职能必须能够通过指标来衡量进展和成功，这些指标能够展示数据治理参与者如何增加业务价值并实现目标。除了有效性指标，增强数据治理的价值指标和那些验证组织在推广后仍有资源支持数据治理的持续性指标，对于维持数据治理职能也非常重要。

### 价值指标

| 指标 | 说明 |
|------|------|
| 对业务目标的贡献 | 数据治理活动对组织战略目标的直接贡献 |
| 降低风险 | 数据治理在减少数据相关风险方面的效果 |
| 提升运营效率 | 数据治理对业务流程效率的改善程度 |

### 有效性指标

| 指标 | 说明 |
|------|------|
| 目标达成情况 | 数据治理目标的实际达成程度 |
| 数据管理人员工具使用度 | 数据管理人员使用相关工具的程度 |
| 沟通有效性 | 沟通计划的执行效果 |
| 教育/培训有效性 | 培训计划的执行效果 |
| 变革速度 | 采取变革的速度 |

### 可持续性指标

| 指标 | 说明 |
|------|------|
| 制度和流程执行情况 | 制度和流程是否正常工作 |
| 标准和规程符合情况 | 是否遵守指导并且必要地改变行为 |

## 实施指南

### 组织和文化

有效且持久的数据治理职能需要在组织对数据的思维和行为上实现文化转变，需要一个持续的变更管理计划支持新的思维和行为，以及实施新的政策和流程，以实现预期的未来状态。不论数据治理战略多么精确或独特，只要忽视商业文化就会降低数据治理成功的概率。

变革管理是可持续发展的关键，变革的可持续性事关过程的质量，它可衡量过程持续增加价值的难易程度，保持数据治理功能需要规划变革。

### 渐进式实施策略

在大多数情况下，数据治理工作的实施会采取渐进式策略：
- 将其纳入大型项目（如主数据管理）一并实施
- 按区域或部门逐步推进
- 很少有组织在一开始就将数据治理工作部署到整个企业范围内

### 调整和沟通工具

| 工具 | 说明 |
|------|------|
| **业务/数据治理战略图** | 连接数据治理活动与业务需求，定期衡量和沟通数据治理如何赋能业务 |
| **数据治理路线图** | 与业务环境或者优先事项的改变相适应，不应当是僵硬死板的 |
| **持续的数据治理业务场景** | 定期调整业务场景以反映组织不断变化的优先级和财务状况 |
| **数据治理指标** | 随着数据治理职能的成熟增长和变化 |

### 关键成功因素

1. **高层支持**：需要"首席"高管（如首席风险官、首席财务官或首席数据官）的支持
2. **文化变革**：数据治理的努力需要包括文化变革的要素，需要强有力的领导支持
3. **可持续资金**：确保流程和资金到位，保证数据治理框架持续运作
4. **持续沟通**：持续利用现有沟通渠道以一致的方式传递关键信息
5. **行为变革**：当数据治理和数据管理需要重大的行为变革时，离不开正式的变革管理
