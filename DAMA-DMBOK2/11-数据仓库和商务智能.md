## 概述

数据仓库（Data Warehouse, DW）的概念起源于20世纪80年代，其核心价值在于将不同来源的数据集成到一个通用的数据模型中加以利用。集成后的数据不仅能为组织的业务运营流程提供深刻洞察，还能为企业智能决策、创新组织价值开辟全新途径。同样重要的是，数据仓库也是组织减少决策支持系统（DSS）重复建设的重要手段——这些系统大多依赖组织中相同的核心数据。企业级数据仓库（EDW）的理念可以减少数据冗余、提升信息一致性，以及支撑企业利用数据做出更精准的决策。

进入20世纪90年代，数据仓库建设开始受到前所未有的重视，特别是伴随商务智能（BI）作为业务决策主要驱动力的发展，数据仓库也随之确立了其"主流"地位。如今，大多数企业都建有数据仓库，并视其为数据管理不可或缺的核心组件。尽管数据仓库已非常成熟，但相关技术仍在不断发展。各种新形式的数据以日益迅猛的速度涌现，以及诸如数据湖等各种新概念应运而生，都将对数据仓库的未来产生深远影响（见第8章和第15章）。

**业务驱动因素：**
- 支撑业务运营和商务智能活动
- 满足监管合规要求（存储历史数据作为运营合规凭证）
- 提供对组织、客户及产品的深入洞察，提升运营效率和竞争优势
- 从回顾性评价发展到预测性分析能力

**建设目标：**
1. 支持商务智能系统的建设
2. 实现有效的业务分析和决策
3. 基于数据洞察寻找更多创新方法

**指导原则：**
1. **聚焦业务目标** — 确保数据仓库服务于组织的优先事项，并解决业务问题
2. **以终为始** — 以业务优先级和商务智能系统最终交付的数据范围来引领构建
3. **全局性视野设计，局部性行动建设** — 以最终愿景指导架构设计，以重点项目或冲刺项目的形式迭代构建、增量交付，从而实现更高效的即时投资回报
4. **先具体细化，后汇总优化** — 首先建立最细粒度的原子数据，逐层聚合以满足需求并确保性能，而不是替换明细数据
5. **倡导数据公开透明和自助服务** — 提供丰富上下文（包括各种元数据），赋予数据消费者更强大的数据洞察能力，同时确保利益相关方充分了解数据来源、内容及其集成过程
6. **在建设数据仓库的同时建设元数据** — 元数据应作为开发周期的一部分进行采集和管理，利用元数据准确地解释业务数据（如"为什么数据加总结果等于X""这是如何计算出来的""这个数据源自哪里"等基础问题）
7. **合作协同** — 与其他数据项目，特别是数据治理、数据质量和元数据管理项目展开密切协作
8. **不要千篇一律** — 为各类数据消费者提供适合他们的数据工具和产品

## 核心概念

### 数据仓库定义

数据仓库由两个核心组件构成：
- **集成数据库** — 用于决策支撑的集成数据存储
- **相关软件程序** — 用于采集、清洗、转换和存储来自各种内外部数据源的数据

为了支持历史的、分析类的和商务智能的需求，数据仓库建设还包括数据集市（数据仓库中部分数据子集的副本）。从广义上讲，数据仓库包括所有用于支持商务智能任意目标实现而提供数据的存储或抽取操作。

**企业级数据仓库（Enterprise Data Warehouse, EDW）** 是一种集中式的数据仓库，遵循企业级数据模型以确保整个组织决策支持活动的一致性。它能够满足整个组织的商务智能需求。

**数据仓库建设**指数据仓库中数据的抽取、清洗、转换、控制和加载等操作过程，侧重于通过强制执行业务规则和维护业务数据关系，在操作型数据基础上实现集成的、反映历史变化的数据集。数据仓库建设还包括与元数据库交互的过程。

传统数据仓库主要关注结构化数据（定义字段中的元素），但随着技术发展，如今已开始支持半结构化数据（如EDI数据传输、XML、HTML等）和非结构化数据（如电子邮件、自由格式文本、商业文档、视频、照片、网页等）。在数据仓库治理范围内定义一个可行的存储结构以支持非结构化数据分析工作负载一直是一个尚未解决的难题。

### 商务智能（BI）

"商务智能"具有双重含义：

1. **分析方法** — 旨在理解组织活动并发现潜在商业机会的数据分析方法，用于提高组织决策成功率。当人们说数据是竞争优势的关键要素时，实际上是在表达他们对商务智能的殷切期望：如果一个组织能向自己的数据提出"正确的问题"，就能获得关于产品、服务和客户方面的深刻洞察。
2. **技术栈** — 支撑上述数据分析活动的技术体系，支持数据查询、数据挖掘、统计分析、报表分析、场景建模（Scenario Modeling）、数据可视化和仪表盘等功能。

商务智能作为决策支持系统的进化产物，适用于预算编制、高级分析等多项工作。

### 多维建模

多维模型通常被称为**星型模型**，由**事实表**和**维度表**组成：

- **事实表**：包含业务流程的定量数据（如销售数据）
- **维度表**：存储与事实数据相关的描述性属性（如时间、产品、地点）

事实表与多个维度表连接，以图表展示时形状类似星星，故称为"星型模型"。多个事实表通过"数据总线"共享公共维度或一致性维度，这有些类似于计算机中的总线。通过接通到遵循一致性维度的数据总线，可以将多个数据集市在企业层面进行整合。

### 星型模型与雪花型模型

| 模型类型 | 特点 | 适用场景 |
|---------|------|---------|
| **星型模型** | 维度表直接连接到事实表，非规范化设计，查询性能优异 | 查询性能要求高、维度层次简单的场景 |
| **雪花型模型** | 维度表进一步规范化拆分，减少数据冗余，但增加查询复杂度 | 维度层次复杂、需要严格规范化的场景 |

（参见 [[DAMA-DMBOK2/05-数据建模和设计]] ）

### 数据集市

数据集市是数据仓库中部分数据子集的副本，专为支撑特定分析需求或满足特定数据消费者需求而设计。主要类型包括：

- **部门数据集市** — 面向特定主题域、单个部门或单个业务流程。通过汇总数据来支持快速分析，设计时通常使用维度建模（反范式技术）
- **操作型数据集市（OpDM）** — 专注于运营决策支持，直接从ODS获取数据，具有与ODS相同的特性：包含当前或近期的数据，内容经常变化

数据集市还可以作为虚拟化数据仓库的基础，合并的数据集市构成了最终的数据仓库实体。数据集成流程从持久层开始，刷新、更新或扩展内容至各数据集市。

### OLAP（在线分析处理）

OLAP是一种为多维分析查询提供快速性能的方法，其典型输出采用矩阵格式。维度形成矩阵的行和列，度量则为矩阵内的值，概念上可表示为一个数据立方体（Cube）。OLAP一词的起源在一定程度上是为了与联机事务处理（OLTP）明确区分。

在数据立方体中，分析师可以浏览数据库，筛选出特定的数据子集，改变数据的方向，并定义分析计算。立方体的多维分析在分析师希望以已知方式查看数据摘要的情况下特别有用。一个传统的应用是财务分析，分析师希望反复遍历已知的层次结构以分析数据，如日期（年、季度、月、周、日）、组织（地区、国家、业务单元、部门）和产品层次结构（产品类别、产品线、产品）。

**常见OLAP操作：**

| 操作 | 说明 |
|------|------|
| **切片（Slice）** | 多维数组的一个子集，固定一个或多个维度的单个值（如按时间切片，得到某年的切片数据） |
| **切块（Dice）** | 对两个以上维度的切片，或两个以上连续切片 |
| **向下/向上钻取（Drill Down/Up）** | 在不同数据粒度级别之间逐层探索，从最高度汇总（向上）到最细颗粒（向下） |
| **上卷（Roll-Up）** | 计算一个或多个维度的所有数据关系汇总，需要定义计算关系或公式 |
| **旋转（Pivot）** | 改变报表或页面展示的维度方向 |

**OLAP三种实施方法：**

| 方法 | 全称 | 说明 |
|------|------|------|
| **ROLAP** | 关系型联机分析处理 | 通过RDBMS中的二维表实现多维技术，星型模式架构是常用数据库设计技术 |
| **MOLAP** | 多维矩阵型联机分析处理 | 使用专门的多维数据库技术 |
| **HOLAP** | 混合型联机分析处理 | ROLAP与MOLAP的组合，部分数据以MOLAP存储，另一部分以ROLAP存储 |

OLAP工具和数据立方体的价值在于，通过将数据内容与分析师的心理模型对齐，降低混淆和错误解释的概率。为了满足所需的功能要求，可能需要将较大的维度拆分为多个单独的数据立方体，以适应存储、加载或计算需求。设定汇总级别可以确保在约定的响应时间内完成所需公式的计算和检索。在数据立方体中配置基于角色的安全性或多语言文本，一般需要额外的维度、附加功能、计算，有时甚至还需要创建单独的数据立方体结构。

通常，OLAP工具既有安装在桌面上的服务器组件，也有可以通过Web访问的面向终端用户的客户端组件。

### BI架构

数据仓库/商务智能环境的架构组件包括（参见下图）：

- **源系统** — CRM、财务系统、人力资源系统等业务系统，以及因行业而异的各种业务系统的数据。此外，还可能包括来自供应商和其他外部途径的数据，以及DaaS、网页内容和各种大数据计算的结果
- **数据集成层** — ETL、数据虚拟化及将数据转化为统一格式并放置到统一位置的其他技术。在SOA环境中，数据服务层也是该组件的一部分
- **数据存储区域** — 包括数据暂存区、主数据管理、中央数据仓库、ODS、数据集市、数据立方体等多个存储区域
- **数据访问工具** — 终端用户的分析和报表工具

**数据存储区域详细说明：**

| 存储区域 | 说明 |
|---------|------|
| **数据暂存区** | 介于原始数据源与集中式数据仓库之间的中间存储区域，用于转换、整合和预处理 |
| **主数据管理** | 参考数据和主数据存储在单独存储库中，数据仓库为主数据系统提供数据 |
| **中央数据仓库** | 经过转换和预处理后持久化保存所有历史原子数据及批处理运行后的最新实例化数据 |
| **ODS（操作型数据存储）** | 中央数据仓库的某个数据来源副本，支持更低延迟，仅包含某个时间窗口的数据 |
| **数据集市** | 支持数据仓库环境的展示层，呈现部门级或功能级子集 |
| **数据立方体** | 支持OLAP的经典模式，主要有关系型、多维型及混合型三种 |

**数据生命周期的流转：** 源系统 → 数据暂存区（清洗、集成、增强、标准化）→ 数据仓库/ODS → 数据集市/数据立方体 → 各类报表

**CIF与数据仓库/BI环境中的数据变化特征：**

| 变化维度 | 源系统侧 | 数据仓库/BI侧 |
|---------|---------|--------------|
| 目的 | 支撑业务功能执行 | 数据分析 |
| 终端用户 | 一线业务人员 | 企业决策者 |
| 系统使用 | 固定数据报表操作 | 即席分析结果查询 |
| 响应时间 | 要求快速 | 要求放宽（战略决策分析需要更多时间） |
| 数据量 | 每个操作涉及较少数据 | 每个查询或流程涉及更多数据 |

**数据仓库与应用程序中数据的差异：**

| 差异维度 | 数据仓库/数据集市 | 应用程序 |
|---------|-----------------|---------|
| 组织形式 | 按主题域 | 按功能需求 |
| 数据集成 | 经过集成整合 | 数据孤岛 |
| 时间特性 | 随时间变化的系列数据 | 当前时间的状态值 |
| 延迟 | 较高 | 较低 |
| 历史数据 | 包含大量历史数据 | 历史数据较少 |

大数据为企业数据流转增加了一条新路径：大部分数据仓库在将数据放入表之前会先进行数据集成，而大数据解决方案会在整合数据之前先接收数据。大数据商务智能除了能支持各种传统报表，还支持预测性分析和数据挖掘（见第14章）。

### ETL与数据集成

数据集成包括ETL、数据虚拟化及将数据转化为统一格式的各类技术。主要涉及两种数据加载类型：

**1. 历史数据加载：** 通常只加载一次，或在处理数据问题期间有限加载，之后不再加载。采集历史信息有多种方法，组织应该根据所需采集历史记录的具体需求进行设计。

**2. 持续增量更新：** 按照固定计划持续执行，确保数据保持最新状态。

**变更数据采集（CDC）技术比较：**

| 技术 | 相对复杂度 | 加载效率 | 重复列 | 删除列 | 说明 |
|------|-----------|---------|--------|--------|------|
| **数据库日志** | 低 | 高 | 否 | 否 | 通过数据库日志技术捕获变更，适用于供应商购买的应用程序 |
| **时间戳/日志表** | 中 | 中 | 是 | 否 | 最常见的技术方法，适用于有时间戳的系统 |
| **触发器** | 高 | 低 | 是 | 是 | 通过数据库触发器捕获变更，开销较大 |
| **全量加载** | 低 | 最低 | 是 | 否 | 适用于没有原生时间戳功能的遗留系统或批处理恢复条件 |

> 注："重复列"标识源系统变更与目标环境之间是否存在数据重复（当为"是"时，变更数据在目标端可能已存在）。"删除列"标识是否跟踪源系统中发生的删除操作（对不再使用的过期维度很有用）。

**准实时和实时数据采集方案：**

| 方案 | 数据积累位置 | 说明 |
|------|------------|------|
| **微批处理（源端累积）** | 源端 | 更频繁地执行批量加载（如每小时甚至每5分钟一次），或当达到某个阈值时执行。如果单个微批处理所耗费的时间比两次微批任务的间隔时间都长，下一个微批任务将被延迟 |
| **消息队列（总线累积）** | 总线 | 极小的数据包（消息、事件或事务）发布到消息总线，目标系统订阅并逐步加载。源系统和目标系统相互独立，事务隔离度极佳。经常在数据即服务（DaaS）中使用 |
| **流式处理（目标累积）** | 目标端 | 目标端系统接收数据后收集到缓冲区或队列，按顺序进行处理。聚合、交互等操作后的结果一同加载入数据仓库 |

**源到目标映射（Source-to-Target Mapping）：** 在各个实体和数据元素间建立从源系统到目标系统的转换规则，记录数据血缘关系。最困难的工作是确定多个系统中数据元素之间的链接有效性和等效性。为将不同系统中的数据元素映射到数据仓库的一致性结构，需要有可靠的分类法（通常是逻辑数据模型）。

**数据修正策略：**
- 针对已加载但又发现不正确的数据记录，将该条数据状态设置为"过期"并将正确的新数据加载进来，比删除旧记录更好（避免对相互关联的表和代理键造成破坏）
- 源系统应负责数据修复和更正，以降低目标系统的复杂性

**加载策略：**

| 策略 | 说明 | 适用场景 |
|------|------|---------|
| **乐观加载策略** | 在维度表中增加一条记录以加载无法关联的事实数据 | 数据质量较好，偶尔出现不匹配 |
| **悲观加载策略** | 创建数据回收区存储无法关联的事实数据，持续跟踪并在后续操作中重新加载 | 数据质量问题较多，需要严格管控 |

### 方法论

#### Kimball vs Inmon 方法论对比

数据仓库领域的两位巨擘——比尔·因蒙（Bill Inmon）和拉尔夫·金博尔（Ralph Kimball）开辟了两种主流的建模和实施流派。

| 对比维度 | Inmon（企业信息工厂 CIF） | Kimball（多维数据仓库） |
|---------|-------------------------|----------------------|
| **数据仓库定义** | 面向主题、集成、随时间变化、相对稳定的数据集合，支持管理决策过程 | 专为查询和分析定制的业务数据副本 |
| **建模方式** | 规范化关系模型（3NF） | 多维模型（星型模型） |
| **架构风格** | 自上而下（Top-Down），先建企业级数据仓库，再建数据集市 | 自下而上（Bottom-Up），先建数据集市，通过一致性维度整合 |
| **核心组件** | 应用程序、数据暂存区、集成与转换层、ODS、数据集市、OpDM、数据仓库、运营报表、参考数据/主数据/外部数据 | 业务源系统、数据暂存区、数据展示区、数据访问工具 |
| **数据存储** | 所有数据存储在单个数据仓库层，存储经过清洗、标准化和受控的原子级数据 | 数据仓库由部门数据集市组合而成，包含经过清洗、标准化和受控的数据，并在原子级别存储历史数据 |
| **集成机制** | 企业级数据模型驱动集成 | 数据仓库总线（一致性维度和一致性事实） |
| **扩展性** | 相对集中，扩展需遵循企业模型 | 更具扩展性，增量构建灵活 |
| **关键成功因素** | 拥有经过验证的企业级数据模型 | 共享/遵循一致性维度 |
| **数据流动** | 应用程序 → 数据暂存区 → 数据仓库 → 数据集市（单向流动） | 业务源系统 → 数据暂存区 → 数据展示区（数据集市组合） |
| **终端用户关注** | 侧重于企业管理数据范围 | 侧重于分析型数据的高效终端交付 |

**核心理念共通之处：**

1. 数据仓库存储的数据来自其他系统
2. 数据存储以增加数据价值为目标整合数据
3. 数据仓库使数据变得易于访问和分析
4. 构建目的是提供可靠、集成的数据给授权利益相关方
5. 用途广泛，从支撑工作流到运营管理，再到预测分析

#### 企业信息工厂（CIF）

比尔·因蒙提出的CIF架构包含以下组件：

| 组件 | 说明 |
|------|------|
| **应用程序** | 处理业务流程，产生的明细数据流入数据仓库和ODS |
| **数据暂存区** | 源数据库与目标数据仓库之间的中间数据库，用于ETL处理，对终端用户透明，大部分数据临时留存 |
| **集成与转换层** | 不同来源数据按标准企业模型转换并集成 |
| **操作型数据存储（ODS）** | 业务数据的集成数据库，包含当前或近期（30~90天）数据，数据变化较快 |
| **数据集市** | 专为数据分析提供数据，是数据仓库的子集，通过汇总数据支持快速分析 |
| **操作型数据集市（OpDM）** | 直接从ODS获取数据，支持运营决策，包含当前或近期数据 |
| **数据仓库** | 统一集成入口，支撑管理决策、战略分析和规划。数据先从应用程序和ODS流入，再流向数据集市（通常单向流动） |
| **运营报表** | 从数据存储中输出 |
| **参考数据、主数据和外部数据** | 理解业务所需的数据，直接访问可简化数据仓库的集成工作 |

**CIF数据仓库的特性：**
- **面向主题** — 按主要业务实体组织，而非功能或应用
- **集成** — 数据统一、内聚，键码结构、编码和解析规则、数据定义及命名规范保持一致。数据仓库中的数据不再是业务数据的副本，而是成为一个数据记录系统
- **随时间变化** — 存储数据在某一时间点的状态（快照），任何时候基于特定时间段的查询都会得到相同一致的结果
- **相对稳定** — 新数据追加而非频繁更新，一组记录可以代表同一事务的不同状态
- **包含明细和汇总数据** — 包含原子级业务明细及汇总数据。汇总可以是持久的（存储在表中）或非持久的（在视图中），取决于性能考量
- **反映历史变化** — 包含大量历史数据，通常要消耗很大的存储空间

#### 多维数据仓库（Kimball）

拉尔夫·金博尔提出的多维数据仓库是数据仓库开发的另一种主要建设模式。其架构以"棋子视图"展示，包含以下组件：

| 组件 | 说明 |
|------|------|
| **业务源系统** | 企业的操作型/事务型应用程序，生产数据后被集成到ODS和数据仓库中 |
| **数据暂存区** | 包括一系列用于集成和转换数据以便进行数据展示的处理流程 |
| **数据展示区** | 与CIF中的数据集市相似，主要区别在于采用"数据仓库总线"集成范式（通过共享/遵循一致性维度统一多个数据集市） |
| **数据访问工具** | 侧重于终端用户的数据需求，采用适当的数据访问工具 |

数据展示区是数据仓库环境中面向终端用户的主要接口，数据集市在此层面通过一致性维度实现企业级集成。

#### 数据仓库总线矩阵

数据仓库总线矩阵是Kimball方法论中的关键规划工具，展示生成事实数据的业务流程与表示维度的数据主题域之间的交汇。当多个流程使用相同的数据时，需要遵循维度的一致性。

| 维度/流程 | 日期 | 产品 | 店铺 | 供应商 | 仓库 |
|-----------|------|------|------|--------|------|
| 销售额 | ✓ | ✓ | ✓ | | |
| 库存 | ✓ | ✓ | ✓ | ✓ | ✓ |
| 订单 | ✓ | ✓ | | ✓ | |

> 日期、产品、店铺和供应商都是一致性维度的候选项。仓库维度不共享，因为它只供库存使用。

该矩阵独立于技术，可用于表示数据仓库/BI系统的长期数据内容需求，帮助组织界定可管理的开发工作范围，每次实施都是对整体架构的增量构建。在某种程度上，当存在足够多的维度模式时，就可以实现企业级数据仓库环境的集成。

#### 数据保管库（Data Vault）

数据保管库（DV）是另一种历史数据存储方法，在数据暂存过程中进行数据清洗和标准化。特点：
- 历史数据以标准化的原子结构存储，同时定义维度代理键、主键和备用键
- 确保业务键和代理键关系保持不变
- 事实表以原子结构形式持久化存储
- 通过数据集市为各种数据消费者提供数据
- 将历史数据保留在DV中，当后续增量引入粒度变化时，重新加载事实表即可
- 支持对表示层进行虚拟化，促进敏捷增量交付及与业务群体的协作开发
- 最后的物化过程可以采用比较传统的星型数据集市，供生产终端用户使用

### 数据仓库架构选择

| 架构因素 | 考量说明 |
|---------|---------|
| 时间延迟要求 | 实时/准实时/批量延迟需求决定技术选型 |
| 数据源可用性 | 源系统的可用窗口和访问能力 |
| 批处理窗口 | 可用的ETL执行时间窗口 |
| 维度一致性 | 一致性维度和事实的定义和管理 |
| 数据质量管控 | 数据修正、清洗、拒绝处理机制 |
| 变更数据采集 | CDC技术选择（日志、时间戳、触发器、全量） |
| 迟到维度处理 | 延迟到达的维度数据处理策略 |
| 可扩展性 | 架构是否支持增量扩展和业务增长 |

## 活动

### 理解需求

收集数据仓库/BI项目需求时应：
1. 明确业务目标和业务战略
2. 确定业务领域，界定业务范围
3. 采访相关业务人员，记录他们当下关心的问题和希望从数据中获得的答案
4. 厘清关键绩效指标（KPI）和统计口径，这些信息可以揭示业务规则，为数据质量自动化管控奠定基础
5. 将需求分类并排出优先级
6. 优先处理简单且有价值的需求，赢得速胜
7. 数据仓库/BI项目的需求文档应明确描述所涵盖业务领域和/或流程的整体背景和范围

### 定义和维护架构

数据仓库/BI架构应描述数据从哪里来、到哪里去、什么时候去、为什么要去，以及用什么样的方式流入数据仓库。其中，"用什么样的方式"包括相关的硬件和软件细节，以及将所有活动组合在一起的组织框架。技术要求应包括性能、可用性和时效性需求（见第4章和第8章）。

**架构最佳实践：**
- 建立数据原子化机制，提供原子化数据处理方式支撑事务级和运营级报表需求
- 概念模型架构是数据仓库/BI工作的起点
- 通过原型设计快速验证或驳回关键需求
- 通过授权需求变更管理团队，为企业界提供知识和采用计划

**发布管理：**
- 整合、协调运维流程以进行数据仓库/BI的生产管理
- 制定标准的发布计划，像管理软件版本那样对数据产品进行版本管理
- 制定版本发布时间计划表，包括年度需求、资源规划及标准版交付计划
- 建立有效的发布流程，确保管理层理解这是一个以数据产品为中心的主动发布过程

### 开发数据仓库和数据集市

数据仓库/BI项目通常有三条并行的开发主线：

| 开发主线 | 内容 | 责任领域 |
|---------|------|---------|
| **数据** | 识别数据来源，设计数据修复、转换、集成、存储和提供规则 | 数据管理团队 |
| **技术** | 支撑数据存储和传输的后端系统和流程 | 企业技术架构/应用架构 |
| **BI工具** | 数据消费者获取数据洞察所需的应用套件 | BI/分析团队 |

**关键步骤：**
1. 源到目标映射 — 建立数据元素间的转换规则和血缘关系
2. 数据修正和清洗 — 强制遵循标准，修正和增强域值
3. 数据转换 — 在技术系统中实施业务规则（定义正确的数据集成规则通常需要数据管理专员和其他业务领域专家的直接参与）
4. 数据加载 — 按定义的加载方法执行

### 实施商务智能产品组合

**用户分组策略：**

| 用户类型 | 需求特征 | 工具匹配 |
|---------|---------|---------|
| IT开发人员 | 聚焦数据提取，注重高级功能 | 高级分析工具，可编程接口 |
| 信息消费者 | 快速访问预定义报表 | 固定报表，简单交互（钻取、筛选、排序） |
| 高级用户/分析师 | 即席查询，数据探索 | 自助式分析工具，可视化探索 |
| 高管/经理 | 综合使用固定报表、仪表盘和计分卡 | 仪表盘，战略BI工具 |
| 外部客户 | 良好用户体验 | 嵌入式分析，门户 |

> 注意：随着用户技能的提高或需要执行不同的功能，他们可能会从一个用户群体转向另一个群体。

**BI工具选型考量：**
- 商业市场提供各种报表和分析工具，主流厂商提供经典且完美的报表功能
- 许多业务系统厂商提供嵌入式分析功能，包括从预填充的数据立方体或聚合表中提取标准内容
- 数据虚拟化技术模糊了本地数据源与外部购买或开放数据之间的界限
- 许多厂商通过合并、收购或全新开发，将相关BI工具组合成商务智能套件
- 每个BI工具都需要系统资源、技术支持和培训，以及架构集成

### 维护数据产品

采用增量开发方式对数据仓库进行扩展、增补或修改。

**发布管理流程示例：**
- 每年3次业务驱动发布 + 1次技术发布（基于技术的发布用于解决数据仓库内部需求）
- 使用MoSCoW列表管理工作范围（必须/应该/能够/不做）
- 采用时间盒管理
- 支持数据仓库增量开发和积压需求管理

**数据产品开发生命周期：**
- 参考业务部门的优先级，将迭代与发布任务有机结合
- 对于业务团队认为已准备好且可进一步研究的项目，部署到试点或沙箱环境中
- 试点项目表现好坏决定下一步行动，不能在不考虑下游数据质量或数据治理问题的情况下盲目推广
- 新的数据产品只有通过试点项目测试，业务部门和IT部门代表认为已做好投产准备，才可部署到生产环境

**监控和调优：**
- 监控整个系统中的数据加载过程，识别性能瓶颈和依赖关系
- 采用数据库调优技术，包括分区、备份调优和恢复策略
- 定义并展示面向客户的满意度指标（如平均查询响应时间、天/周/月活跃用户数等）
- 定期审查使用情况统计数据和使用方法
- 提供数据交付活动高级状态的仪表盘，兼具下钻功能
- 增加数据质量指标，使用热力图可视化基础设施的工作负载

## 方法

### 原型驱动需求

在开发产品之前，通过创建演示数据并在协同原型设计工作中采用需求挖掘方法，可以在实施活动开始之前快速确定需求及优先级。数据虚拟化技术的发展可以利用协同原型设计技术降低传统的实施难度。

**数据剖析的价值：**
- 有助于原型设计，降低与非预期数据的相关风险
- 数据仓库通常是最先暴露源系统或数据输入函数中数据质量缺陷的地方
- 揭示数据源之间的差异，这些差异可能会阻碍数据集成
- 对源数据状态的评估有助于更准确地估计数据集成的工作范围和可行性

### 自助式商务智能

自助服务是商务智能产品组合中的一种基本交付方式。一般是在受管理的门户中引导用户活动，根据用户的权限提供多种功能：
- 消息通知和警告
- 查看预定的生产报表
- 与分析报表互动
- 开发即席报表
- 仪表盘和计分卡

将协作工具扩展到用户社区，可以提供更丰富的自助服务，包括各种提示和技巧、数据加载状态、整体性能、数据发布进度的集成公告，甚至可以进行对话讨论。

可视化和统计分析工具使得数据探索和发现更加高效。以往这些技术只是IT和开发人员的领域，但现在业务团队也可以使用这些数据处理、计算和可视化技术。数据集成工作可以先通过业务用户进行可行性原型设计，再由IT部门进行实现和优化。

### 数据审计

为了维护数据血缘，所有结构和流程都应具备创建和存储审计数据的能力，以便用于跟踪和报告。允许用户查询这些审计数据，使用户能够自行验证数据的状况和到达情况，从而提高用户的信心。当数据出现问题时，审计信息也有助于更详细地进行故障排除。

## 组织与角色

### 专职团队

许多组织设有专门的全职团队负责管理数据仓库生产环境的日常运营：

- **前台支持团队** — 与业务用户互动，识别和报告产品缺陷，通知后端运维团队需要解决的所有产品缺陷问题
- **后端运维团队** — 确保生产环境配置已按要求执行，持续监控运行情况，升级警报并报告数据吞吐量状态

团队之间保持高频互动，确保即将发布的版本解决关键问题。专职团队会在固定周期内设定重复执行的任务计划，可以通过维护渠道看到待交付的成果及对应的工作负载峰值情况。

### 数据治理

受严格监管的行业（如金融行业）从治理良好的数据仓库中获益匪浅。数据仓库治理流程应与风险管理保持一致，且为业务驱动。不同类型的企业有不同的治理需求（如营销和广告公司使用数据的方式与金融机构不同）。治理流程应减少风险，而不是减少任务的执行。

最关键的功能是那些管理业务运营的发现或改进区域，以及那些确保数据仓库本身数据质量的功能。改进区域引领所有新方案的边界，因此需要通过数据交换和良好运行程序来实例化、操作、转移和丢弃这些区域的数据。数据归档和时间范围是边界协议中的关键要素，它们有助于避免数据需求蔓延。将数据加载到数据仓库中意味着分配时间、资源和编程工作，以便及时地将经过修复好的、可靠的、高质量的数据送达终端用户。

将一次性或有限使用的事件视为数据生命周期的一部分，并最大限度地将它们限制在试点区域或用户控制的"沙箱"范围内。实时分析过程可以通过自动化流程将同一时间范围的汇总结果反馈回数据仓库。运营策略是为实时环境中执行的业务流程而定义的，治理则适用于将结果引入数据仓库以供组织使用。

**关键治理活动：**
- 数据安全风险应对 — 通过风险暴露缓解矩阵管理，对被认为具有高暴露性、低缓解性或难以在早期检测的项目，需要充分发挥治理职能来限制和减少相关风险
- 数据归档和时间范围管理 — 避免数据需求蔓延
- 一次性/有限使用事件管控 — 限制在试点区域或沙箱范围内

### 获得业务认可

业务部门对数据的认可应是用户验收测试（UAT）的一部分。在初始加载和若干增量更新加载周期后，在商务智能工具中根据源系统中的数据进行结构化随机测试，以满足验收标准。

关键成功因素包括：
- 易于理解的数据
- 可验证的数据质量
- 可证明的数据血缘关系
- **概念数据模型** — 组织的核心信息有哪些？关键业务概念是什么？它们之间如何相互关联？
- **数据质量管控闭环** — 如何识别和修复问题数据？如何向问题来源系统责任人通报并让他们负责解决问题？
- **端到端元数据** — 架构如何支持端到端元数据集成流程？数据消费者如何回答"这个报表的含义是什么"或"这个指标是什么意思"等基本问题？
- **端到端可验证的数据血缘** — 业务用户能否以自动化、可维护的方式追溯到数据项对应的源系统？

### 客户/用户满意度

对数据质量的认知会影响客户满意度，但满意度还取决于其他因素，如数据消费者对数据的理解、运维团队对已知数据问题的响应速度。通过定期与用户代表召开会议，可以更容易地收集、理解和应对客户的反馈。这样的互动也有助于数据仓库团队向用户分享数据产品版本发布路线图，并了解数据消费者是如何利用数据仓库的。

### 服务水平协议

应在服务级别协议（SLA）中明确指定对数据仓库环境的业务和技术期望。通常情况下，不同类别的业务需求及其各自的支持系统（如ODS、数据仓库与数据集市）对响应时间、数据保留和可用性的要求差别很大。

### 度量指标

**1. 使用指标**

数据仓库中使用的度量指标通常包括已注册用户数、连接用户数和并发连接用户数。了解每个工具有多少授权许可账户是一个良好的开端，尤其是对于审计人员而言。但实际上有多少用户连接到该工具是一个更好的度量标准。另外，用户群体在每个时间段内发起了多少次查询（或类似查询的操作）对于系统容量规划来说是一个更好的技术参考指标。因此，数据仓库应从多角度、多指标分析使用情况，如审核用户数、用户查询量及活跃用户数等。

**2. 主题域覆盖率**

主题域覆盖率度量的是每个部门访问数据仓库（从数据拓扑的角度）的程度。它们还凸显了哪些数据是跨部门共享的、哪些数据本来应该共享但没有共享。将业务源映射到目标是另一种自然扩张，它强制执行并验证已经收集的元数据和血缘关系，并可以提供影响分析，以确定哪些源系统被哪些部门用于分析。这有助于通过减少对频繁使用的源数据的更改，将调优工作集中在那些具有高影响力的分析查询上。

**3. 响应时间和性能指标**

大多数查询工具都会记录响应时间。运维人员可以从工具中获取响应时间或性能指标，这些数据将提供关于用户数量和类型的指标情况。数据加载过程以原始格式收集每个数据产品的加载时间，这些加载时间还可以表示为预期支持的百分比。例如，一个数据集市的加载预计是每天刷新，加载时间窗口为4小时，如果该加载任务在4小时内完成，那么它就是100%得到支持的。同样，可以将此过程应用于所有下游数据抽取流程。

大多数工具都会在日志或存储库中保留提供给用户的对象相关查询记录、数据刷新和数据提取时间等信息。将此数据分为计划执行的对象和已执行的对象，并以原始计数方式表示尝试和成功的次数。需要关注高度受欢迎或查询表现不佳的对象，以避免满意度指标受损。如果一组对象经常出现故障，这些信息就可以用来指导开展缺陷分析、制订运维计划和容量规划。下一步自然是验证和调整服务级别，在下个版本中优化那些一直失败的项目，或者在缺少充足资金的情况下降低支持的服务级别。

## 工具

### 元数据库

元数据库的关键在于能将各种来源的元数据整合在一起，实现自动化和集成填充。核心组件包括：

**数据字典和术语表：** 用业务术语描述数据，包含使用该数据所需的其他信息（如数据类型、结构细节和安全限制等）。通常，数据字典的内容直接来自逻辑数据模型。在一些组织中，业务用户通过提供主题领域数据元素的定义，积极参与数据字典的开发（见第13章）。

**数据及数据模型血缘：** 记录数据从源到目标的完整流转路径。许多数据集成工具都提供血缘分析功能。记录下来的数据血缘有多种用途：
1. 调查数据问题的根本原因
2. 对系统变更或数据问题进行影响分析
3. 基于数据来源确定数据的可靠性

建议实施一个集成的影响分析和血缘分析工具，可以便于了解数据加载过程中涉及的所有动态部分，以及终端用户的报告和分析。

### 数据集成工具

数据集成工具用于加载数据进入数据仓库，核心功能包括：
- 流程审核、控制、重启和调度
- 选择性提取数据元素并传递给下游系统
- 控制操作执行，重启失败或中止的进程
- 与商务智能产品组合集成，支持工作流消息、电子邮件导入导出和语义层集成
- 工作流集成可以推动数据质量缺陷识别、解决和升级过程
- 能够将数据目标配置为语义层，可作为敏捷实施的数据虚拟化候选方案

### BI工具类型

| 工具类型 | 说明 | 典型用户 |
|---------|------|---------|
| **运营报表** | 分析短期（月度对比）和长期（年度对比）业务趋势，支持战术决策 | 一线业务人员 |
| **业务绩效管理（BPM）** | 组织目标一致性指标的正式评估，支持战略决策，包括预算、规划和财务合并 | 高管层 |
| **描述性自助分析** | 为业务一线提供运营分析，准实时指导决策 | 运营人员、分析师 |
| **即席查询工具** | 允许用户自行编写和生成报表，不是精确格式的文档，但支持图表和表格 | 高级用户、分析师 |
| **生产报表** | 跨DW/BI边界，查询业务系统生成可操作项（如发票、银行对账单） | IT人员 |
| **OLAP/数据立方体** | 多维分析，支持切片切块、钻取等操作 | 分析师、业务用户 |

**数据可视化方法：** 传统BI工具涵盖丰富的数据可视化方法，如表格、饼图、折线图、面积图、条形图、直方图、蜡烛图（K线图）等。可以静态格式（已发布的报表）或互动的在线格式提供。一些工具支持与终端用户交互，其中的钻取或筛选功能有助于在可视化中继续分析数据。

## 实施指南

### 就绪评估/风险评估

成功的数据仓库项目应做到：
1. 明确数据敏感级别和数据安全约束
2. 选择执行工具
3. 确保资源到位
4. 创建采集过程以评估和接收源数据

在数据仓库项目中识别和列出敏感或受限的数据元素，这些数据需要进行屏蔽或脱敏。如果考虑将实施或维护工作外包给其他团队，还需要应用其他约束以防数据泄露。在选择工具和分配资源之前，要考虑安全约束，确保已遵循数据治理的审查和批准流程。

对源数据状态的评估有助于更准确地估计数据集成的工作范围和可行性。

### 发布路线图

无论采用瀑布式、迭代式还是敏捷式实施方法，都需要"以终为始"的思维方式锚定期望的最终状态。数据仓库总线矩阵可作为沟通和推广工具在增量迭代过程中使用。利用业务确定的优先级和风险曝光指标确定每个增量版本的严谨性和开销。

每个增量版本将修改现有功能或添加全新的功能，这些功能通常与新上线的业务单元保持一致。维护一个待处理任务或工作项列表，以识别未完成的功能和面向业务的优先级。确定需要按不同顺序交付的所有技术依赖项，然后将这些工作成果打包成一个软件发布版本。

### 配置管理

配置管理与发布路线图保持一致，提供必要的后台调整和脚本，实现开发、测试到生产的自动化，并在数据库级别为模型打上发布版本的标签，以自动化方式将代码库与该标签绑定，确保手动编码、生成的程序和语义层内容在整个环境中协调一致，并对其进行版本控制。

### 组织与文化变革

在数据仓库/商务智能的整个生命周期中，始终保持一致的业务重点是项目成功的关键。梳理企业的业务价值链是了解业务背景的好方法，公司价值链中的特定业务流程提供了一个天然的商业导向背景，可以用来构建分析领域框架。

最重要的是，要根据实际的业务需求规划项目，并评估必要的业务支持，同时考虑以下关键成功因素：

1. **高层管理支持** — 管理层是否给予了足够的支持？是否有明确且积极参与的指导委员会及相应的资金支持。数据仓库/商务智能项目需要强有力的管理层支持
2. **目标范围界定** — 是否有明确的业务需求、业务目的和工作范围？
3. **业务部门参与** — 业务部门管理层是否承诺提供并安排合适的业务主题领域专家参与项目建设？缺乏业务部门深度参与和支持是项目常见的失败原因，也是暂停项目直至得到业务部门支持的充分理由
4. **合作伙伴准备** — 外部商业合作伙伴是否为长期增量交付做好了充足的准备？是否承诺建立卓越中心以便在未来版本中持续运维数据产品？目标群体的平均知识或技能差距有多大？能否在一个数据产品的增量版本中缩小这一差距？
5. **愿景目标统一** — IT战略对业务愿景及目标的支持力度有多大？确保所需的功能要求与当前的IT路线图中已规划或可持续的业务能力相对应。任何在建能力与业务愿景目标之间产生重大偏离或重大差距都可能导致项目停滞不前或完全停止

### 报表策略

报表策略应涵盖标准、流程、指南、最佳实践和程序，确保用户获取清晰、准确且及时的信息：

| 策略要素 | 说明 |
|---------|------|
| **安全访问** | 确保只有授权用户才能访问权限范围内的敏感数据 |
| **访问机制** | 建立用户如何对数据进行互动、报告、检查或查看等访问机制 |
| **适用工具** | 为各类用户群体提供适用的工具 |
| **报表属性** | 包括报表摘要、详细信息、例外情况，以及频率、时间、分布和存储格式 |
| **可视化** | 通过图形化输出发挥可视化功能的潜力 |
| **及时性与性能平衡** | 权衡数据新鲜度与查询性能 |

应定期评估标准报表，确保它们仍然具有业务价值，因为报表运行会产生数据存储和处理成本。为不同业务群体提供适当的报表工具是成功的关键因素。

研究中心可以提供培训、启动设置、设计最佳实践、数据源提示和其他具体解决方案或成果，帮助业务人员实现自助服务模式。

| 行业 | 典型应用场景 |

| **政府** | 公共政策分析、预算执行监控、服务效能评估 |

## 方案价值

### 风险分析

| 风险类别 | 风险描述 | 缓解措施 | 影响等级 |
|---------|---------|---------|---------|
| **业务参与不足** | 缺乏业务部门的深度参与和支持 | 建立业务-IT联合指导委员会，明确业务价值 | 高 |
| **数据质量问题** | 源系统数据质量问题影响数据仓库可靠性 | 实施数据质量管控闭环，建立数据剖析机制 | 高 |
| **架构扩展性不足** | 架构设计无法支撑业务增长和变化 | 采用企业级架构规划，分阶段迭代实施 | 中 |
| **技术选型失误** | 工具和技术与业务需求不匹配 | 原型验证，评估自建/购买/租赁（SaaS）策略 | 中 |
| **组织变革阻力** | 用户不采纳或抵制新系统 | 提供充分培训，建立卓越中心（BICC），推动数据文化 | 中 |
| **安全合规风险** | 敏感数据泄露或不合规 | 数据分类分级，安全约束嵌入架构设计 | 高 |
| **性能瓶颈** | 查询响应慢，影响用户体验 | 索引优化，聚合管理，性能监控与调优 | 中 |

### 投资回报（ROI）

数据仓库/BI项目的价值体现：

**有形价值：**
- 减少报表开发时间，提升IT生产力
- 降低数据存储和处理的重复成本
- 减少决策支持系统的重复建设

**无形价值：**
- 提升数据一致性和决策准确度
- 增强组织竞争优势和市场响应速度
- 满足监管合规要求，降低合规风险
- 推动数据文化建设和数据驱动决策

**度量指标：**

| 维度 | 指标示例 |
|------|---------|
| **使用指标** | 已注册用户数、活跃用户数、并发连接数、查询次数 |
| **主题域覆盖率** | 各部门访问数据仓库的程度、跨部门数据共享情况 |
| **响应时间** | 平均查询响应时间、天/周/月活跃用户数 |
| **性能指标** | 数据加载时间、加载支持百分比、SLA达成率 |
