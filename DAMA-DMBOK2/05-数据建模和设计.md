# 数据建模和设计

## 概述

数据建模（Data Modeling）是发现、分析和界定数据需求的过程，然后用被称为**数据模型**的精准形式来表示和传达这些需求。数据建模是数据管理的关键组成部分，建模过程要求组织发现并记录其数据是如何相互关联的，同时通过建模过程设计数据之间彼此的关联适配方式。

数据模型展示并帮助组织理解其数据资产，包含并承载着对数据消费者至关重要的元数据。在数据建模过程中产生的大部分元数据对于其他数据管理职能（如数据治理、数据仓库和数据分析等）都是重要且必要的。

用于表示数据的方案（Scheme）有很多种，最常用的6个方案是：关系型（Relational）、维度型（Dimensional）、面向对象型（Object-Oriented）、基于事实型（Fact-Based）、基于时序型（Time-Based）和非关系型（NoSQL）。这些方案的模型主要建立在3个细节层面：概念、逻辑和物理。每种模型都包含一组组件，如实体、关系、事实、键和属性。

本章将介绍数据模型的用途、数据建模的目标、原则及数据建模涉及的基本概念和通用术语，并使用一系列与教育相关的案例介绍不同数据模型的工作原理及其之间的差异。

## 核心概念

### 1. 数据模型的用途

数据模型对于有效的数据管理具有重要作用：

| 序号 | 用途 | 说明 |
|------|------|------|
| 1 | 提供通用术语 | 为组织内数据相关沟通建立共同语言 |
| 2 | 捕获显性知识 | 记录组织数据及系统的知识 |
| 3 | 沟通工具 | 在项目建设期间充当主要的沟通媒介 |
| 4 | 提供起点 | 为应用程序的定制、集成甚至替换提供基础 |

**目标和原则**：数据建模的目标是确认和记录不同视角对数据需求的理解，从而开发出更符合当前和未来业务需求的应用，同时为成功地完成广泛的数据应用项目和数据管理活动奠定基础，如主数据管理和数据治理职能。良好的数据建模能够降低维护费用，增加未来重复利用的可能性，进而降低构建新应用的成本。数据模型是元数据的一种重要形式。

确认和记录不同视角的理解有助于：

1. **形式化（Formalization）**：数据模型记录了对数据结构和数据关系的精确定义，能够评估业务规则实施后对数据的影响，无论是当前的状态还是期望的目标状态。形式化的定义为数据引入了规范的结构，从而降低在访问和保存数据时发生数据异常的概率。
2. **范围定义（Scope Definition）**：数据模型有助于解释数据的上下文边界，并帮助定义组织要采购的应用程序、项目、倡议或现有系统的实施范围。
3. **知识留存与文档化（Knowledge Retention/Documentation）**：数据模型通过捕获知识以显性形式保留企业记忆，它可作为未来项目使用的现状版本文档。数据模型就像地图一样，能够帮助业务专业人员、项目经理、分析师、建模师和开发人员了解环境中的数据结构。

### 2. 可被建模的数据类型

有4种主要类型的数据可以被建模（Edvinsson, 2013）。在所有给定组织中进行建模的数据类型反映了该组织的优先事项：

| 类型 | 说明 | 示例 | IT术语 |
|------|------|------|--------|
| 类别信息 | 用于对事物进行分类和分配类型的数据 | 市场类别、业务部门、产品颜色/型号 | 参考数据 |
| 资源信息 | 实施过程所需的基本资源 | 产品、客户、供应商、设施、组织 | 主数据/参考数据 |
| 业务事件信息 | 在业务操作过程中创建的数据 | 客户订单、供应商发货单、现金提款 | 交易型业务数据 |
| 详细交易信息 | 通过系统生成的大量详情数据 | POS系统、社交媒体、传感器数据、GPS、RFID、Wi-Fi | 大数据 |

> **静态数据**（Data at rest）和**动态数据**（Data in motion）都可以被建模。动态数据如协议、消息和事件等。

### 3. 数据模型组件

大多数数据模型都包含4个基础组件：**实体（Entity）、关系（Relationship）、属性（Attribute）和域（Domain）**。

#### 3.1 实体（Entity）

实体是组织收集信息的对象，是组织内存在的"名词"。一个实体可以被看作对一个基础问题的回答——谁、是什么、何时、何地、为什么、怎么办或是这些问题的综合。

**常用实体类别（表5-1）：**

| 类别 | 说明 | 示例 |
|------|------|------|
| 人物实体 | 与人相关的信息 | 学生、讲师、客户 |
| 地点实体 | 与位置相关的信息 | 学校、城市、地址 |
| 事物实体 | 具体或抽象的对象 | 课程、产品、设备 |
| 事件实体 | 发生的事情 | 订单、注册、交易 |
| 概念实体 | 抽象概念 | 专业、分类、状态 |

**实体、实体类型和实体实例（表5-2）：**

"实体类型"表示某种类型的东西。例如，"简·史密斯"属于"员工"，因此"简·史密斯"是实体实例，"员工"是实体类型。实体实例是具体实体的产生或值呈现。"学生"实体可能有"鲍勃·琼斯""乔·杰克逊""简·史密斯"等多个实例。

**实体别名（按建模方案）：**

| 建模方案 | 实体别名 |
|----------|----------|
| 关系模型 | 实体 |
| 维度模型 | 维度（Dimension）、事实表（Fact Table） |
| 面向对象模型 | 类（Class）、对象（Object） |
| 基于时序模型 | 中心（Hub）、卫星（Satellite）、链接（Link） |
| 非关系模型 | 文档（Document）、节点（Node） |

**实体别名（按抽象级别）：**

| 级别 | 别名 |
|------|------|
| 概念层 | 概念、术语 |
| 逻辑层 | 实体（根据数据模型方案不同而存在不同术语） |
| 物理层 | 数据表（Table） |

**实体的图形表示：** 在数据模型中，实体通常以矩形（或带有圆角的矩形）出现，内部标注它们的名称。例如："学生"（Student）、"课程"（Course）和"讲师"（Instructor）。

**高质量实体定义的3个特征：**

1. **清晰性**：定义应该易于阅读和理解。句子表达清晰，没有晦涩的缩写词或不明确的术语，如不会包括"有时"或"通常"这样的字眼。
2. **准确性**：定义是对实体进行精准和正确描述。相关业务领域的专家应审查有关定义，以确保它们是准确的。
3. **完整性**：定义的部分都应完整存在。例如，在定义代码时，应包括代码值。在定义标识符时，定义中应包含该标识符的唯一性范畴。

#### 3.2 关系（Relationship）

关系是实体之间的联系，捕获概念实体之间的抽象交互、逻辑实体之间的详细交互，以及物理实体之间的明确约束。

**关系别名：**

| 建模方案 | 关系别名 |
|----------|----------|
| 关系建模 | 关系（Relationship） |
| 维度建模 | 导航路径（Navigation Path） |
| 非关系建模 | 边（Edge）、连接（Link） |
| 物理层 | 约束（Constraint）、引用（Reference） |

**关系的图形表示：** 关系在数据建模图上显示为线条。例如，在信息工程符号中，"学生"和"课程"之间的关系捕获了"学生可能参加课程"的规则，"讲师"和"课程"之间的关系捕获了"讲师可能授课"的规则。线上的符号（称为基数）以精确的语法捕获规则。

**基数（Cardinality）：**

基数捕获一个实体实例与另一个实体之间的数量关系。基数由出现在关系连线两端的符号表示。数据规则通过基数进行指定和执行。

对于基数，选择很简单：**0、1或"多"**。关系的每一侧都可以具有0、1或"多"的任意组合。指定0或1可以捕捉实体实例是否在关系中是必需的。指定1或"多"可以捕捉在给定关系中有多少个特定实例参与其中。

**基数符号示例：**
- 每一名学生**可以**参加一门或多门课程
- 每一门课程**可以**被一名或多名学生参加

**关系的元数：** 关系中实体的数目被称为关系的"元数"，最常见的有一元、二元及三元关系。

| 类型 | 说明 | 示例 |
|------|------|------|
| 一元（递归）关系 | 只涉及一个实体。一对多的递归关系描述了一个层次结构，多对多的递归关系则描述了一种网络或图结构 | 课程先决条件关系 |
| 二元关系 | 涉及两个实体（最常见） | 学生与课程的参加关系 |
| 三元关系 | 涉及三个实体 | 学生在特定学期选择特定课程 |

**一元关系结构：**
- **层次结构**：一个实体最多拥有一个父实体（或称为更高阶的实体），子实体处于关系中"多"的一边，而父实体只处于关系中"1"的一边。例如，参加生物学研讨会需要首先参加生物学讲座和化学讲座，一旦参加生物学讲座被选为生物学研讨会的先决条件，则参加生物学讲座不可再作为其他课程的先决条件。
- **网络关系**：一个实体可以拥有多于一个的父实体。例如，允许参加生物学讲座作为多门课程的先决条件。

**外键（Foreign Key）：**

外键用于在物理模型中代表关系，在逻辑关系数据建模中有时也会用这种方法表示。定义两个实体之间关系的同时，就可能会有外键的建立。外键体现在关系中的"多"的一边的实体，即子实体中。例如，"注册"（Register）需要两个外键——"学生"的"学号"和"课程"的"课程号"。"学生"和"课程"是父实体，而"注册"是子实体。

---

#### 3.3 属性（Attribute）

属性用于定义、描述或度量实体的特征。属性可能具有域。属性在实体的物理模型表现为表中的列、字段、标签或节点。

**属性的图形表示：** 在数据模型中，属性一般被描述为实体矩形中的列表。例如，"学生"实体的属性包括"学号"（Student Number）、"学生姓氏"（Student First Name）、"学生名字"（Student Last Name）和"学生出生日期"（Student Birth Date）。

**标识符（键）分类：**

标识符（又称为键）是一个或多个属性组成的属性组，它可以唯一地界定某个实体的特定实例。

**按结构划分：**

| 键类型 | 说明 | 示例 |
|--------|------|------|
| 简单键 | 单个属性，可唯一界定实体实例 | UPC（通用产品码）、VIN（车辆识别号码） |
| 代理键 | 系统自动生成的计数器，无业务含义，通常是一个整数，其意义与其表的数值无关 | 自增整数ID |
| 复合键 | 由两个或多个属性组成的候选键，共同唯一标识一个实体 | 学号+学期 |
| 组合键 | 每个属性本身都是外键的复合键 | — |

**按功能划分：**

| 键类型 | 说明 |
|--------|------|
| 超级键 | 任意一组能够唯一标识实体的属性 |
| 候选键 | 一个或多个属性的最小属性组，可唯一标识实体实例（一个实体可以有多个候选键） |
| 主键 | 被选出的唯一识别标识符（一个实体只有一个主键） |
| 备用键 | 未被选为主键的候选键，仍然可以用于寻找某个特定的实例 |
| 业务键 | 业务专业人员用于检索实体实例的属性（有时被称为自然键） |

> 通常情况下，主键为代理键，备用键为业务键。

**标识关系与非标识关系：**

- **独立实体**：主键仅包含只属于该实体的属性。在关系模式中，大多数数据模型图用矩形符号表示独立实体。
- **非独立实体**：主键至少包含一个来自其他实体的属性。在关系模式中，用圆角矩形表示非独立实体。
- **标识关系**：父亲实体的主键转换成为子实体的主键的外键（如"学生"和"注册"之间、"课程"和"注册"之间的关系）。
- **非标识关系**：父亲实体的主键转换成子实体的非主键外键属性。

#### 3.4 域（Domain）

域是某个属性能够被赋予的全部可能值，主要作用是对属性的特征进行标准化。例如，日期的域包含所有可能的日期，适用于任何"日期"字段（聘用员工的日期、订单录入的日期、提交索赔的日期、课程开始的日期等）。

域中所有的值都被称为**有效值**，不在域中的值被称为**无效值**。属性的值不应超出其分配的域。

可以用附加的规则对域进行进一步限制，这些限制规则被称为**约束（Constraint）**。规则可以涉及域的格式、逻辑或两者皆有。

**域的定义方式：**

| 定义方式 | 说明 | 示例 |
|----------|------|------|
| 数据类型 | 指定属性能够采用的标准数据类型 | 整数、字符(30)、日期 |
| 数据格式 | 使用模板和掩码等格式 | 邮政编码、电话号码、仅用字母/数字代码 |
| 列表 | 包含有限值集合的域 | 订单状态码：开立/发货/结束/退货 |
| 范围 | 特定数据类型内，介于最小值和最大值之间所有的值 | 订单送货日期在下单后3个月内 |
| 基于规则 | 通过规则定义有效值 | 物品价格必须高于物品成本 |

### 4. 数据建模方案

最常见的6种数据模型方案，每种模式都有特定的图形符号（表5-3）：

| 方案 | 适用场景 | 主要特点 | 适用数据库 |
|------|----------|----------|------------|
| 关系型 | 事务处理系统 | 二维关系、减少冗余、精确表达 | RDBMS（概念/逻辑/物理模型均可） |
| 维度型 | 数据仓库/分析 | 事实表+维度表、利于查询分析 | RDBMS和MDBMS（三层模型均可） |
| 面向对象型 | 软件系统设计 | 类/对象、封装、继承 | RDBMS和对象数据库 |
| 基于事实型 | 概念建模 | 自然语言分析、对象角色建模 | 概念和逻辑模型可用于多种数据库 |
| 基于时序型 | 数据仓库 | 中心/卫星/链接、时间序列 | RDBMS（物理建模技术） |
| 非关系型 | 大数据/特定场景 | 文档/键值/列式/图数据库 | 依赖底层数据库结构（物理建模技术） |

> 即使在非传统的数据库中（如文档数据库），也可以在建立关系型概念数据模型（CDM）和逻辑数据模型（LDM）后建立物理数据模型（PDM）。

#### 4.1 关系型建模

关系理论的概念由爱德华·科德（Edward Codd）在1970年首次提出，它提供了一种组织数据从而表达其含义的系统性方法。这一方法有一个额外的好处，即减少了数据存储冗余。他认为数据在二维关系下能够实现最高效的管理。

关系模型的设计目标是对业务数据进行精确的表达，并且在一处只有一个数据（排除冗余）。关系模型是设计运行系统的理想模型，因为此类系统需要快速输入并精准存储信息。

**ER图符号系统：**

在关系模型中有几类不同的符号可以用来表达实体间的关系：

| 符号系统 | 英文名称 | 说明 |
|----------|----------|------|
| 信息工程 | Information Engineering (IE) | 最常见的符号系统，用三叉线或称"鸡爪印"（Crow's Feet）表示基数 |
| 信息建模的集成定义 | IDEF1X | 美国空军开发的建模方法 |
| 巴克符号 | Barker Notation | Oracle公司广泛使用的建模方法 |
| 陈氏符号 | Chen Notation | 由Peter Chen提出的实体关系建模方法 |

> 最常见的是信息工程符号，该方法用三叉线或称"鸡爪印"（Crow's Feet）表示基数。

#### 4.2 维度型建模

维度型建模的概念始于美国通用磨坊公司（General Mills）和达特茅斯学院在20世纪60年代的联合科研项目。在维度模型中，大量数据按照利于查询和分析的方式进行组织。相比之下，支持事务处理的数据库运营系统则面向快速处理单一交易进行优化。

维度模型用来聚焦处理特定业务过程中的业务问题。例如，用维度模型分析学生的录取情况，录取情况可以从学生来自的地域、学校名称、学期和学生是否接受经济援助等维度查看。导航可以从区域上升到地区和国家，从学期上升到学年。

> "轴符号"是用来建立维度模型的图形符号，它对于那些不习惯阅读传统的数据建模语法的人来说是一种非常有效的沟通工具。在关系模型中，关系连线说明业务规则；而在维度模型中，实体之间的连线表示导航路径，这些导航路径用于说明业务问题。

**核心组件：**

| 组件 | 说明 | 特点 |
|------|------|------|
| 事实表 | 每行代表具体度量值（金额、数量或计数），有些度量值是算法计算的结果 | 约占数据库总空间的90%，行数通常非常庞大 |
| 维度表 | 业务中的主要对象，含文字描述信息，用于"按条件查询"或"按条件报告" | 高度反规范化，约占总数据量的10% |

**维度表主键：**
维度表中作为主键的最主要的两种标识符是**代理键**和**自然键**。

**渐变维度（Slowly Changing Dimension, SCD）管理类型：**

| 类型 | 名称 | 说明 |
|------|------|------|
| 类型1 | 覆盖（O） | 新值覆盖旧值，直接更新原位置的数据 |
| 类型2 | 新的行（R） | 新值写入新行中，同时将旧行标记为非当前行 |
| 类型3 | 新的列（C） | 在同一行中为一个值系列设置多个列，新的值写入列的第一个位置，依次将旧值向后移动，最后一个被挤压出列的值会被丢弃 |

**雪花模型（Snowflake Schema）：**

将星型模型中平面的、单表的维度结构进行规范化处理，拆分成各自按照层次结构或网络结构的多个表。如果每个维度都被折叠成一个结构，生成的数据模型被称为**星型模式**；如果维度没有折叠（进行了规范化拆分），生成的数据模型就被称为**雪花模型**。

**粒度（Granularity）：**

事实表中的单行数据的含义或者描述，即任何一行包含的最详细的信息。定义事实表中的粒度是维度型建模的关键步骤之一。例如，如果一个维度模型用于度量学生注册过程，则粒度可能为学生、日期和班级。

**一致性维度（Conformed Dimension）：**

从整个组织的角度考虑而构建的，而不是基于某个特定的项目。由于它们具有一致的术语和值，这些维度能够在不同的维度模型中共享。例如，如果日历是一个一致性维度，则学期维度在计算学生学期内申请数的维度模型中与计算学生在学期内毕业人数的维度模型中应当具有相同的值和定义。

**一致性事实（Conformed Fact）：**

在跨单个数据集市时使用标准化的术语定义。不同的业务用户可能以不同的方式使用相同的术语。"客户增加"可能与"总额增加"或"调整增加"不同。开发者需要注意具有相同名称的事物在不同的组织中代表着不同的概念，或不同名称的事物在不同组织中代表的意义相同。

---

#### 4.3 面向对象建模（UML）

统一建模语言（Unified Modeling Language, UML）是一种建模软件的图形语言。UML根据数据库的不同有着不同种类的符号（类模型）。

**UML类模型特点：**

1. 类图（Class）类似于ER图，但在ER图中没有操作（Operation）或方法（Method）部分
2. 在ER图中，与操作最接近的是存储过程
3. 属性（Attribute）类型（如日期、分钟）是用可实现的应用程序代码语言表达的，而不是物理数据库可实现的术语
4. 默认值可以在符号中有选择地显示
5. 访问数据是通过类的公开接口，封装或数据隐藏是基于"局部影响"的

**类操作的可见性：**
- **公开的**：完全可见
- **保护的**：仅对子实体可见
- **私有的**：隐藏的

> 相比之下，ER物理模型仅提供公共访问途径；所有数据都同等地暴露在进程、查询或操作当中。

---

#### 4.4 基于事实的建模（FBM）

基于事实的建模（Fact-Based Modeling, FBM）属于概念建模语言，起源于20世纪70年代末。这些语言是基于对业务领域中可能出现的自然语言（模棱两可的句子）的分析。基于事实的语言通过对象、通过相关事实或这些对象的特征，以及每个对象在每个事实中所扮演的角色来描述世界。基于事实的模型不使用属性，它通过表示对象（实体和值）之间的精确关系减少直观或专家判断的需求。

**主要变体：**

- **对象角色建模（ORM/ORM2）**：模型驱动的工程方法，由特里·哈尔平（Terry Halpin）在1989年形式化为一阶逻辑。从典型的需求信息或查询的例子开始，在概念层次上用受控词表所表达的简单事实来描述这些例子。受控词表是自然语言的受限制的自然语义，它是无歧义性的，因此所表达的语义很容易被人理解；它也是形式化语言，因此可以自动将结构映射到低级操作上。
- **完全面向通信的信息建模（FCO-IM）**：关注话语世界中的沟通，利用自然语言构建概念信息模型，并结合具体实例。通过使用所谓的基本事实表达，FCO-IM为不同的沟通方式提供了广泛的支持，如相似的事实、同义词、同音异义词、广义对象类型及概念建模等方面。

---

#### 4.5 基于时序的建模

当数据必须按照时间顺序和特定的时间值相关联时，即使用基于时序的模式。

**数据保管库（Data Vault）：**

面向细节、基于时序的数据模型，由一组有连接关系的规范化表集合组成。综合了第三范式（3NF）和星型模式的优点，专为企业数据仓库设计，有3种类型的实体：

| 实体类型 | 说明 | 示例 |
|----------|------|------|
| 中心（Hub） | 业务实体，代表主键 | 学生、课程 |
| 链接（Link） | 关系，定义各中心之间的交易完整性 | 参加课程 |
| 卫星（Satellite） | 描述性属性，定义中心主键的上下文描述信息，支持不同类型的历史记录 | 学生联系方式、学生特点、课程描述 |

**锚建模（Anchor Modeling）：**

适用于结构和内容随时间变化的信息建模技术。它提供了一种图形符号，用于概念建模，类似于传统的数据建模，但扩展了处理时间数据的功能。4个基本的建模概念：

- **锚（Anchor）**：建模实体和事件
- **属性（Attribute）**：建模锚点的特性
- **连接（Tie）**：建模锚点之间的关系
- **节点（Knot）**：建模共享属性（如状态等）

---

#### 4.6 NoSQL建模

NoSQL数据库是一种基于非关系技术的数据库类别的名称。NoSQL这个名字与如何查询数据库关系不大，更侧重于数据的存储方式（涉及关系结构）。

NoSQL数据库有4种主要类型：

| 类型 | 说明 | 适用场景 | 特点 |
|------|------|----------|------|
| 文档数据库 | 将业务主题存储在一个被称为文档的结构中，而不是将业务主体拆分为多个关系结构 | 半结构化数据 | 如将学生、课程和注册信息存储于一个注册文档中 |
| 键值数据库 | 允许应用程序将其数据存储在两个列（"键"和"值"）中 | 简单和复杂信息存储 | 简单信息（日期、数字、代码）和复杂信息（文本、视频、音乐、照片）都存储在"值"列内 |
| 列式数据库 | 最接近RDBMS，但每个列存储在其独立的结构中 | 复杂数据类型 | 可以使用更加复杂的数据类型，包括未格式化的文本和图像（如Cassandra） |
| 图数据库 | 为那些可以用一组节点即可清晰地表示它们之间关系的数据设计 | 社交网络、路线图 | 节点之间的连接数不确定，适合遍历图以找到最短路线等操作（如Neo4J、Allegro、Virtuoso） |

---

### 5. 数据模型的层次细节

1975年，美国国家标准协会的标准规划和需求委员会发表了3种模式的数据库管理方法：

1. **概念性**：体现了在数据库中对企业真实世界视图的描述，代表当前最佳模式或企业经营方式
2. **外部模式**：数据库管理系统的不同用户对与他们的特定需求相关的企业模型的子集进行操作
3. **内部模式**：数据的"机器视图"，描述了存储的企业信息的图示

这3个层次通常分别转化为概念、逻辑和物理层面的细节。在项目中，概念数据建模和逻辑数据建模是需求计划和分析活动的一部分，而物理数据建模是一种设计活动。

#### 5.1 概念数据模型（CDM）

概念数据模型（Conceptual Data Model, CDM）用一系列相关概念的集合描述高阶数据需求。它只包含给定领域和函数中的基本和关键业务实体，并对每个实体和实体之间的关系进行描述。

**特点：**
- 最高层抽象
- 不包含属性细节
- 专注于业务实体和关系
- 用于确认范围和关键术语

**示例：** 如果要对学生和学校之间的关系进行建模，使用IE符号的关系概念数据模型：每所"学校"可能"包括"一名或多名"学生"，每名"学生"必须来自一所"学校"。此外，每名"学生"可以"提交"一个或多个"申请"，每个"申请"必须由一名"学生"提交。关系被用来在关系数据模型中定义业务规则。

#### 5.2 逻辑数据模型（LDM）

逻辑数据模型（Logical Data Model, LDM）是数据需求的详细描述，通常根据应用需求等特定的使用环境进行选择。逻辑数据模型仍然独立于任何技术或特定的实现约束。逻辑数据模型通常根据对概念数据模型的扩展开始。

**特点：**
- 中间抽象层
- 包含完整的属性定义
- 经过规范化处理
- 记录业务解决方案

**示例：** 在关系逻辑数据模型中，概念数据模型通过添加属性来扩展。对属性进行标准化后，将其分配给实体。每个属性和它所在实体的主键之间存在非常紧密的关系。例如，学校名称与学校代码有很紧密的关系，学校代码的每一个值都会最多显示学校名称的一个值。

> 逻辑关系数据模型描述业务流程的业务规则，逻辑维度数据模型描述业务问题，确定一个业务流程的运行状况和性能。

#### 5.3 物理数据模型（PDM）

物理数据模型（Physical Data Model, PDM）是一种详细的技术解决方案，通常以逻辑数据模型为基础，使其与一组硬件、软件和网络工具相匹配。物理数据模型与特定技术相关（如IBM DB2、Oracle、SQL Server等）。

**特点：**
- 最低抽象层
- 与特定数据库技术相关
- 包含性能优化（索引、分区、逆规范化等）
- 文档化技术解决方案

> 因为物理数据模型受实现技术约束，所以常常对结构进行组合（逆规范化）提高检索性能。

**物理设计关键技术：**

| 技术 | 说明 |
|------|------|
| 规范模型 | 一种变化的物理模型，用于描述系统之间的动态数据。描述了数据在系统之间作为信息包或消息进行传递的数据结构 |
| 视图 | 虚拟表，提供了一种方法来查看包含或引用了实际属性的一个或多个表中的数据。用于简化查询、控制数据访问和重命名列 |
| 分区 | 垂直（分隔列）或水平（分隔行）拆分表，便于归档和提高检索性能 |
| 逆规范化 | 刻意将范式化的逻辑数据模型实体转换成具有冗余或重复数据结构的物理表，以提高性能 |
| 索引 | 访问数据库中数据时提高查询性能的重要数据库对象 |

**视图（View）详细说明：**
- 标准视图在请求视图中的一个属性时会运行SQL来检索数据
- 实例化（通常称为"具体化"）视图在预定的时间运行
- 视图可以避免数据冗余及由于反规范化而导致的引用完整性缺失的问题

**分区（Partitioning）详细说明：**
- **垂直拆分**：可以减少查询集，创建包含部分列的子表（如根据字段是否主要是静态的，或主要是不稳定的）
- **水平拆分**：可以减少查询集，对列值进行细分创建子表（如创建只包含特定区域内的客户的区域客户表）

**逆规范化（Denormalization）详细说明：**

逆规范化是刻意将范式化的逻辑数据模型实体转换成具有冗余或重复数据结构的物理表。逆规范化的首要原因是提高性能：

1. 为了避免运行时连接的成本较高，预先将多个表中的数据组合在一起
2. 创建更小、预先进行筛选的数据副本，以减少运行时计算和/或大型表的全表扫描成本
3. 预先计算和存储高成本的数据计算，以避免运行时系统产生资源竞争
4. 逆规范化也可以通过将数据分割成多个视图或根据访问需求复制表的副本提高用户安全

> 逆规范化的风险在于由于重复而带来数据错误。当视图和分区在物理设计时缺乏效率时，通常会选择逆规范化处理。为确保正确地存储属性副本，一种好的方法是执行数据质量检查。
>
> 虽然在本节中使用了"逆规范化"这个术语，但这个过程并不仅仅适用于关系数据模型。例如，在文档数据库中，"逆规范化"则被称为"嵌入"。
>
> 在维度数据建模中，逆规范化被称为**折叠或合并**。如果每个维度都被折叠成一个结构，生成的数据模型被称为**星型模式**。如果维度没有折叠，生成的数据模型就被称为**雪花模型**。

---

### 6. 规范化

规范化（Normalization）是运用规则将复杂业务转化为稳定数据结构的过程。规范化的基本目标是确保每一个属性仅在某一个地方定义，以消除冗余及冗余导致的不一致性。这个过程需要深入理解每个属性和这些属性与主键之间的关系。

规范化规则根据主键和外键对属性进行分级。规范化规则分为多个范式级别，每个级别用不同的粒度和特性以正确搜索主键和外键。

**范式级别详解：**

| 范式 | 要求 | 详细说明 |
|------|------|----------|
| 第一范式（1NF） | 有效主键、属性原子性 | 确保每个实体都有一个有效的主键，且每个属性都依赖于主键；消除重复组，并确保每个属性的原子性和不可再分割（不包括多个值）。1NF通过额外的实体（通常称为关联实体）来解决多对多关系 |
| 第二范式（2NF） | 最小主键、属性依赖完整主键 | 在1NF基础上，确保每个实体的主键是最小的主键，且每个属性都依赖于完整的主键 |
| 第三范式（3NF） | 无隐藏主键、属性不依赖非键属性 | 在2NF基础上，确保每个实体没有隐藏的主键，每个属性都不依赖于键之外的任何属性（"非主键列之间不存在依赖关系"） |
| Boyce/Codd范式（BCNF） | 解决重叠复合候选键 | 候选键不是主键就是备用键。复合意味着不止一个（在实体的主键或备用键中有两个或多个属性），重叠意味着键之间存在隐藏的业务规则 |
| 第四范式（4NF） | 多对多关系分解为成对关系 | 将所有的多对多关系分解成成对的关系，直到它们不能被分解成更小的部分 |
| 第五范式（5NF） | 依赖关系分解为基本对 | 将实体间的依赖关系分解成基本对，所有的连接依赖项使用主键的部分 |

> **注意**：规范化模型一般指的是满足第三范式中的数据。需要满足Boyce/Codd范式、第四范式和第五范式的情况很少发生。

---

### 7. 抽象化

抽象化（Abstraction）指在保留概念或主题的重要属性和本质的前提下，移除细节，从而使其适用于更广泛的情况的一种过程。通过抽象，可以创建更通用的模型结构。建模者需要权衡开发和维护抽象结构的成本，以及在未来需要修改非抽象结构时所需的返工工作量。

**抽象类型：**
- **泛化（Generalization）**：将实体的公共属性和关系分组为超类实体
- **专门化（Specialization）**：将实体中的区分属性分离为子类实体，这种专门化通常基于各实体实例中的属性值

**子类关系特性：**
- 子类继承超类的所有属性
- 可减少冗余
- 传达看似独立实体之间的相似性
- 子类也可以通过角色或分类，将实体的实例按功能划分为不同的组（如"主体"含有"个体"和"组织"两个子类）

**示例：** "大学"和"高中"是"学校"的子类。

---

## 方法论

### 建模流程

数据建模和设计活动包括4个步骤：

1. **数据建模规划（P）**
2. **构建数据模型（D）**：概念数据建模 → 逻辑数据建模 → 物理数据建模
3. **评审数据模型（C）**
4. **维护数据模型（O）**

#### 1. 规划数据建模

数据建模工作计划主要包括：

| 步骤 | 内容 |
|------|------|
| 评估组织需求 | 了解业务背景和数据需求 |
| 确定建模标准 | 制定命名约定、符号规范等 |
| 明确存储管理 | 确定数据模型的存储和管理方式 |

**交付成果：**

1. **图表（Diagram）**：一个数据模型包含一到多个图表，以精确的方式描述对数据的需求，包括不同层级（概念、逻辑或物理模型）、具体的某种模式（关系、维度、对象、基于事实的、基于时序的或NoSQL），以及数据模式的表示（如IE、UML、ORM）。
2. **定义（Definitions）**：实体、属性和关系这3个对象的准确定义对于确保数据模型的精度而言至关重要。
3. **争议和悬而未决的问题（Issue and Outstanding Question）**：数据建模过程中经常会出现一些无法解决的争议和难以回答的问题，数据建模团队通常需要提交有关这些争议和悬而未决问题的文档。例如，对于一个"学生"模型而言，比较突出的问题可能是："如果一个学生离开后又返回，是为他们分配新的学号，还是保留使用原来的？"
4. **血缘关系（Lineage）**：有关数据是从哪里来、经过怎样的加工、变成了什么样的结果的脉络关系。一般而言，血缘关系会以来源/目标映射的形式呈现。血缘关系在数据建模过程中重要的两个原因：一是有助于数据建模人员深入理解数据需求，准确定位属性来源；二是确定属性在源系统中的情况，这是验证模型和映射关系准确性的有效工具。

#### 2. 构建数据模型

数据建模是一个不断迭代的过程。在建模过程中，首先要研究现有的数据模型和数据库，以及参考已发布的建模标准和数据标准，并搜集和考虑随时提出的新数据要求。在此基础上，建模人员先设计数据模型初稿，再与业务专家和业务分析师讨论及确认模型设计是否符合业务规则要求，同时提出修改建议，交由建模人员进行修改。如此反复进行，直至没有任何问题。

**正向工程流程：**

```
概念数据建模 → 逻辑数据建模 → 物理数据建模
```

**概念数据建模步骤：**
1. 选择模式（关系、维度、基于事实或非关系型）
2. 选择模型表示方式（IE、ORM等）
3. 完成初版概念数据模型——捕获特定用户群组的想法，不应试图去弄清其观点如何适应其他部门或整个组织的观点
4. 整合企业术语——利用既定的企业术语和建模规则实现数据建模工作
5. 签署确认模型——确认及审查定版的概念数据建模是否符合实际情况及是否满足客户需求

**逻辑数据建模步骤：**
1. 分析信息需求——识别业务流程中的信息要求，信息产品名称标识基本的业务词汇
2. 分析现有文档——参考已有的数据模型和数据库，注意确认每个细节的准确性和时效性
3. 添加关联实体——用于描述多对多关系，创建新的实体描述两边实体的关系
4. 添加属性——在概念实体中属性可以被再次拆分，但在逻辑数据模型中一个属性是原子性的，不可拆分
5. 分配域——保证属性内和属性间格式和数值集的一致性
6. 分配键——键属性用于唯一识别实体实例，分为主键和备用键

**物理数据建模步骤：**
1. 解决逻辑抽象实体——子类合并（将子类属性作为允许为空的列合并在超类表中）或超类拆分（将超类属性按不同子类拆分到不同表中）
2. 添加属性细节——添加技术名称、物理域、物理数据类型和长度、约束等
3. 添加参考数据对象——小型参考数据值集可通过3种方式实现：单独代码表、主共享代码表、嵌入规则
4. 分配代理键——分配业务不可见的特殊键值，确保原始主键上存在备用键
5. 逆规范化以提高性能
6. 添加索引以提高性能——基于频繁使用的列（特别是键）为大型表创建索引
7. 分区以提高性能——推荐使用日期键进行分区
8. 创建视图——控制访问、标准化常用对象或查询

**逆向工程：**

从现有数据库文档化的过程：**PDM → LDM → CDM**。大多数数据建模工具都支持各种数据库的逆向工程，但是逆向工程出来的模型是比较混乱的，为了提高其可读性，需要建模人员对其布局进行重新设计。

#### 3. 评审数据模型

数据模型需要质量控制和持续改进。诸如时间-价值（time-to-value）、支持成本和数据模型质量验证器（如数据模型计分卡）等技术都可用于评估模型的正确性、完整性和一致性。

评审会议的议程应包括评审初版模型（如果有）、对模型所做的更改、所有其他被考虑和拒绝的选项，以及新模型与现有建模或架构标准的符合程度。

设计评审应由一组具有不同背景、技能、期望和观点的主题专家进行。参与者必须能够讨论不同的观点并达成小组共识。每次设计评审应由一位领导主持，领导负责推动会议进行、创建并遵循议程、征求所有参与者的意见、保持秩序并总结小组的共识意见。

如果没有通过评审，建模人员必须进行修改以解决专家评审小组提出的所有问题。

#### 4. 维护数据模型

当需求变化时，需要对数据模型进行更新。某一层的改变通常需要相应更高层或更底层模型的改变。在每个开发迭代结束时，一个好的做法是对最新的物理数据模型进行逆向工程，并确保它与相应的逻辑数据模型保持一致。

### 数据模型计分卡

数据模型计分卡（Data Model Scorecard）用于衡量数据模型质量，包含10个质量指标：

| 序号 | 评价指标 | 说明 |
|------|----------|------|
| 1 | 模型满足需求的能力 | 确保数据模型满足已定义的需求。如果需求中要求包含订单信息，则应检查模型中是否包含订单信息 |
| 2 | 模型的完整性 | 包括需求完整性和元数据完整性。模型中包含所有已请求的需求，且只包含所要求的内容 |
| 3 | 模型与实例的匹配程度 | 确保模型的层级（概念、逻辑或物理）和实例（关系、维度、NoSQL）定义相匹配 |
| 4 | 模型结构的合理性 | 验证数据模型设计的落地性，确保能够直接构建数据库 |
| 5 | 模型的通用性 | 评审模型的扩展性或抽象程度 |
| 6 | 命名标准遵循情况 | 评审命名标准结构、术语和风格是否正确和一致地应用 |
| 7 | 模型的可读性 | 评审数据模型的易读性。将父实体放在子实体上方，将相关实体排放在一起，最小化关系线长度 |
| 8 | 对象定义情况 | 评审所有定义项是否清晰、完整和准确 |
| 9 | 与企业架构一致性 | 评审数据模型中的结构能否在更加广泛和一致的环境中应用 |
| 10 | 与元数据匹配程度 | 将存储模型结构中的数据和实际数据进行比对，确保一致性 |

**评分方法：**
- **模型评分列**：评审者按照每一项指标对模型评价后的得分
- **总分列**：每项指标的总分值
- **百分比列**：评审得分除以总分数，乘以100%
- **备注栏**：记录更详细的得分解释或模型修改事项
- **最后一行**：记录该模型评审后的总分

## 最佳实践

### 命名标准

ISO/IEC 11179元数据注册库是一项国际标准，用于组织中元数据的表示，其中包含与数据标准相关的多个部分，包括命名属性和编写定义的规则。

命名标准要点：
- 逻辑名称对业务用户有意义，尽可能使用完整的单词，避免使用最常见缩写之外的任何缩写
- 物理名称必须符合DBMS允许的最大长度，在必要时使用缩写
- 逻辑名称使用空格作为单词之间的分隔符，实际名称通常使用下划线作为单词分隔符
- 避免跨环境的名称变化（测试环境、质量评价环境或生产环境）
- 使用分类词（数量、名称和代码等属性名称中的最后一个术语）区分属性类型，显示哪些属性是定量的而不是定性的

### 数据库设计（PRISM原则）

| 原则 | 说明 |
|------|------|
| **P**erformance（性能和易用性） | 确保授权用户可以快速、便捷地访问数据，以可用且与业务相关的形式呈现数据 |
| **R**eusability（可重用性） | 确保数据库结构在适当的情况下能够被多个应用重复使用，并且可用于多种目的（业务分析、质量改进、战略规划、客户关系管理和流程改进） |
| **I**ntegrity（完整性） | 数据在任何情况下都应始终具有有效的业务含义和价值，以及反映业务的有效状态。执行数据的完整性约束检查 |
| **S**ecurity（安全性） | 真实和准确的数据应以高效、常态化的方式提供给授权用户，但只限于授权用户。必须满足隐私关切 |
| **M**aintainability（可维护性） | 以合理成本执行所有数据工作，确保数据的创建、存储、维护和使用成本不超过其对组织的价值，并能以最快的响应速度适应业务流程的变化 |

## 数据模型治理

### 数据模型和设计质量管理

数据分析人员和设计人员作为信息消费者（具有数据业务需求的人员）和数据生产者（以可用形式捕获数据）的中介，必须平衡信息消费者的数据需求和数据生产者的应用需求。

数据专业人士还必须平衡短期的商业利益和长期的商业利益。信息消费者需要及时获得数据以满足短期的商业需求，以及利用当前的商业机会。系统开发项目团队必须在限定的时间和预算内完成系统的开发任务，但必须满足所有利益相关方的长期利益，确保组织的数据存在于安全、可恢复、可共享和可重用的数据结构中。

### 数据模型和数据库设计标准应包括：

1. 数据建模和数据库设计的标准交付成果的清单和描述
2. 适用于所有数据模型对象的标准名称，可接受英文缩写和非常见单词的缩写规则列表
3. 所有数据模型对象的标准命名格式列表，包括属性命名和列命名
4. 设计和维护交付成果的标准方法的列表和描述
5. 数据建模和数据库设计角色和职责的列表和描述
6. 数据建模和数据库设计中涉及的所有元数据属性的列表和描述，包括业务元数据和技术元数据
7. 元数据质量的期望和要求
8. 使用数据建模工具的准则
9. 准备和指引设计评审的准则
10. 数据模型版本管理的指导原则
11. 禁止或需要避免的相关事项列表

### 版本管理

需要对数据模型和其他设计需求的变更进行详细管控。对于每个变更都应该予以记录：

1. **为什么要变更**，什么情况导致的变更——项目需求变化或者某些情况发生变化引起的变更
2. **哪些对象发生了变更**，发生了怎样的变更——包括新增了哪些表、修改或删除列等
3. **何时审批通过的这一变更**，这一变更何时应用到模型中
4. **谁执行的变更**
5. **此变更引起了模型中的哪些地方需要随之变更**

> 一些通用的数据建模工具提供数据模型版本管理和集成功能的模型管理存储库。如果不提供该项功能，则需要将数据模型导出为DDL或XML文件进行管理。
