# 数据存储和运营：概念与基础

> **所属知识体系**：[[DAMA-DMBOK2/06-数据存储和运营|DAMA-DMBOK2 第6章：数据存储和运营]]

---

## 概述

数据存储和运营（Data Storage and Operations）是DAMA-DMBOK2框架中专注于数据基础设施管理的知识领域。它涵盖对待存储数据的数据库设计、实现和技术支撑，确保数据在从创建/获取到处置的整个生命周期内实现价值最大化。

该知识领域包含两项核心子活动：
1. **数据库支撑**：侧重于数据生命周期相关的活动，从建立数据库环境开始，到获取、备份、清除数据，并确保数据库持续良好运行。
2. **数据库技术支撑**：界定满足组织需要的技术需求，确定技术架构，应用和管理相关技术，并解决与技术相关的问题。

在DAMA框架中，数据存储和运营是数据建模与设计的物理实现层，为数据安全提供基础设施支撑，并依赖数据架构进行整体规划。

**核心目标**：
1. 在数据的整个生命周期内管理其可用性
2. 确保数据资产的完整性
3. 管理数据事务的性能

### 业务驱动因素

公司运营越来越离不开各类信息系统。对依赖数据进行运营的公司而言，数据存储和运营的活动至关重要。**业务的连续性**是这些活动的主要驱动力。如果系统不可用，公司运营就可能受到影响甚至停摆。为了有效降低这类风险，公司需要为其建立一套可靠的数据存储基础设施。

### 指导原则

1. **识别并执行自动化的机会**：自动化数据库开发流程可以缩短开发周期，减少错误和返工。
2. **以重用为目标构建数据对象**：开发和使用可抽象和可复用的数据对象，避免应用程序与数据库模式紧密耦合。
3. **理解并适当运用最佳实践**：DBA应将数据库标准和最佳实践作为要求加以推广，但在有合理理由时也应灵活应对。
4. **数据库标准要与需求相适应**：SLA既要反映DBA的建议，也要考虑开发人员能否接受。
5. **明确DBA在项目工作中的角色期望**：在项目定义阶段就引入DBA。

## 核心概念

### 数据库基本术语

| 术语 | 定义 |
|------|------|
| **数据库（Database）** | 一系列被存储数据的集合，无论其结构或内容如何 |
| **实例（Instance）** | 通过数据库软件对数据库执行控制的某一特定存储区域 |
| **模式（Schema）** | 数据库或实例中包含的数据库对象的子集 |
| **节点（Node）** | 作为分布式数据库的一部分，承载计算或存储数据的单个计算机 |
| **数据库抽象（Database Abstraction）** | 使用通用API调用数据库函数，使应用程序可连接多个不同数据库 |

### 数据库架构类型

#### 集中式数据库

集中式数据库将所有数据存储在一个系统的一个地方，所有用户都通过该系统访问数据。对于受限数据可能是理想选择，但对需要广泛访问的数据存在风险——一旦系统不可用，用户将无法获得数据。

#### 分布式数据库

分布式数据库使得在大量节点上快速访问数据成为可能，主要特点包括：
- 基于普通商业硬件服务器，通过横向扩展从单服务器到数千台服务器
- 通过数据库管理软件在服务器间高效复制数据
- 自动处理节点故障，系统整体不太可能丧失全部服务能力
- 部分采用MapReduce计算范式进一步提高性能

**联邦数据库**：将多个自治的数据库系统映射成一个单一的联邦数据库，各成员数据库保持自治性但共享数据。

- **松耦合系统**：成员数据库构建自己的联邦模式，用户需直接掌握联邦模式构成
- **紧耦合系统**：各成员数据库共同构建统一的集成联邦模式
- 适用场景：企业信息集成、数据虚拟化、模式匹配和主数据管理

**区块链数据库**：联邦数据库的一种，最早为安全地管理不可伪造的金融交易设计。有两种类型结构：单个记录和区块。每笔交易包含一条记录，每个区块包含一组带时间戳的交易，使用哈希算法确保交易信息不可篡改。

#### 云计算平台部署

| 部署方式 | 说明 |
|---------|------|
| **虚拟机镜像** | 用户按时间偏好购买虚拟机实例，上传或使用预装数据库镜像 |
| **数据库即服务（DaaS）** | 平台负责安装和维护数据库，应用所有者按使用付费 |
| **云上托管数据库** | 数据库不作为服务提供，云服务提供商受委托管理数据库 |

### 数据库处理类型

数据库处理有两种基本类型：ACID和BASE。CAP定理可以用于定义一个分布式系统是接近强调一致性的ACID类型，还是更接近强调可用性的BASE类型。

#### ACID模型

ACID缩写词在20世纪80年代初出现，代表数据库事务可靠性不可或缺的4个约束条件：

| 属性 | 说明 |
|------|------|
| **原子性（Atomicity）** | 数据库所有操作要么都执行，要么都不执行 |
| **一致性（Consistency）** | 数据事务必须始终符合系统定义的所有规则 |
| **隔离性（Isolation）** | 一个事务的执行不受其他事务干扰 |
| **持久性（Durability）** | 一旦事务运行，对数据库的更改是永久的、不可撤销的 |

#### BASE模型

随着数据体量的空前增长和数据类型的不断增多，产生了存储非结构化数据、优化读取负载性能、增加横向扩展灵活性等全新需求：

| 属性 | 说明 |
|------|------|
| **基本可用（Basically Available）** | 即使某些节点出现故障，系统仍保证一定程度的数据可用性 |
| **软状态（Soft State）** | 数据处于不断变化状态，响应时不能保证数据是最新的 |
| **最终一致性（Eventual Consistency）** | 数据在所有节点的最终状态是一致的，但非每时每刻都一致 |

#### ACID与BASE差异对比

| 维度 | ACID | BASE |
|------|------|------|
| 设计理念 | 强调一致性 | 强调可用性 |
| 适用场景 | 关系数据库、事务处理 | 大数据、实时网络应用 |
| 一致性要求 | 强一致性 | 最终一致性 |
| 可用性 | 相对较低 | 基本可用 |
| 扩展性 | 垂直扩展为主 | 水平扩展为主 |
| 代表技术 | RDBMS（Oracle、MySQL等） | NoSQL（MongoDB、Cassandra等） |

#### CAP定理（布鲁尔定理）

该定理认为，分布式系统不可能同时满足ACID的所有要求：

- **一致性（Consistency）**：系统必须始终按设计和预期方式运行
- **可用性（Availability）**：系统必须时刻保持可用
- **分区容错性（Partition Tolerance）**：部分节点故障时系统仍能继续运行并提供服务

CAP定理指出，在任何共享数据的分布式系统里，这三种要求最多同时满足两个，即只能"三选二"。

### 数据存储介质

| 存储类型 | 特点 | 适用场景 |
|---------|------|---------|
| **SAN磁盘存储** | 稳定持久，可按访问频次使用不同磁盘 | 通用数据存储 |
| **内存数据库（IMDB）** | 所有数据处理在内存中进行，响应速度极快 | 实时数据分析处理 |
| **列压缩解决方案** | 列式数据库使用压缩技术存储列数据 | 大量重复数据字段的数据集 |
| **闪存/SSD** | 兼具内存访问速度和磁盘持久性两种优势 | 磁盘的高性价比替代品 |

---

### 数据库类型详述

#### 层次型数据库

最早的数据库模型，用于早期的大型机DBMS。数据组织成具有强制父子关系的树状结构。XML使用的也是这种层次模型。

#### 关系数据库（RDBMS）

基于集合论和关系代数，数据元素或属性（列）被关联成元组（行）。关系数据库是面向行的。

支持变体：
- **多维数据库**：允许多维度搜索，用于数据仓库和商务智能
- **时态数据库**：处理随时间变化的数据，包含有效时间和事务时间

#### 非关系数据库（NoSQL）

**面向列 vs 面向行的对比**：

| 场景 | 更高效的组织方式 | 原因 |
|------|----------------|------|
| 对很多行进行聚合计算 | 面向列 | 只需读取少数列，速度更快 |
| 获取某一行中的许多列 | 面向行 | 只需一次磁盘寻道就能检索整行 |

**主要NoSQL类型详述**：

| 类型 | 特点 | 典型应用 |
|------|------|---------|
| **列式数据库** | 压缩冗余数据 | 商务智能、OLAP |
| **空间数据库** | 存储和查询几何空间对象 | GIS、地图服务 |
| **对象/多媒体数据库** | 包含分层存储管理系统 | 多媒体应用、CAD/CAM |
| **平面文件数据库** | 行和列存在于单个文件中 | Hadoop数据存储 |
| **键值对数据库** | 数据项包含键标识符和键值 | 缓存、会话管理 |
| **文档数据库** | 由结构和数据组成的文档集合 | 内容管理 |
| **图数据库** | 关注节点之间的关系 | 社交网络、知识图谱 |
| **三元组数据库** | 由主-谓-宾组成的数据实体 | 语义网、知识管理 |

### 数据生命周期管理

DBA通过设计、实施和使用存储、处理或检索数据的系统，在数据的整个生命周期内维护和确保数据的准确性和一致性。

| 环节 | 说明 |
|------|------|
| **数据获取** | 从内部系统或外部来源获取数据并加载到数据库 |
| **数据迁移** | 将数据从一种存储类型、格式或系统转移到另一种 |
| **数据保留** | 按照法规和业务要求确定数据保持可用的时间长度 |
| **数据过期** | 数据价值降低，不再值得继续保留 |
| **数据处置** | 将数据从存储介质中完全删除并使其无法恢复 |

### 备份与恢复

#### 备份策略

- **增量备份**：常态进行，仅备份自上次备份后变化的数据
- **全库备份**：定期对每个数据库进行完整备份
- **热备份**：在应用程序运行时进行备份
- **冷备份**：数据库离线时进行备份

#### 恢复类型

| 恢复类型 | 说明 | 优先级 |
|---------|------|-------|
| **立即恢复** | 能够提前预测并自动解决的问题 | 最高 |
| **关键恢复** | 尽可能快地恢复系统 | 高 |
| **非关键恢复** | 等关键恢复完毕再进行恢复 | 低 |

#### 业务连续性规划

DBA必须确保所有数据库和服务器都有恢复计划，覆盖以下场景：
1. 物理数据库服务器失效
2. 一个或多个磁盘存储设备失效
3. 单个数据库失效
4. 数据库索引或数据页损坏
5. 数据库或日志段文件系统丢失
6. 数据库或事务日志备份文件丢失

---

### 数据归档

归档是将数据从可立即访问的存储介质迁移到检索性能较低的介质进行存档的过程。稳妥方法包括：
1. 创建辅助存储区域
2. 将数据库表分区成可以归档的单元
3. 将不常用数据复制到独立数据库中
4. 创建磁带或磁盘备份
5. 创建数据库作业定期清理不再使用的数据

---

### 数据库容量和增长预测

把数据库想象成一个盒子，数据是盒子中装的水果：
- **容量**：用多大的盒子才能装下所有水果和包装材料
- **增长速度**：有多少水果要放进盒子，放的速度有多快
- **取出速度**：从盒子里要能取出多少水果
- **增长预测**：盒子大小随时间是保持不变还是不断扩大
- **归档策略**：水果应该保存多久
- **数据清除**：水果腐烂至无法食用时就要清除

---

### 通用数据库过程

| 过程 | 说明 |
|------|------|
| **归档** | 将不常用数据迁移到成本较低的存储介质 |
| **容量预测** | 评估当前容量和未来增长 |
| **变更数据捕获（CDC）** | 检测数据变更并确保相关信息被适当记录 |
| **数据清除** | 将数据从存储介质完全删除 |
| **数据复制** | 在多个存储设备上存储相同数据 |
| **弹性和恢复** | 系统对错误条件的容忍度和恢复能力 |
| **数据留存** | 数据保持可用的时间长度 |
| **数据分片** | 将数据库中一部分独立出来进行独立更新 |

---

### 数据库弹性

数据库弹性用于衡量系统对错误条件的容忍度。提高弹性的常用方法：
- 及时发现并纠正导致错误的数据
- 监测并忽略导致错误的数据
- 在处理过程中对错误数据和采取的措施进行标记

---

### 数据复制模式

| 维度 | 选项 | 说明 |
|------|------|------|
| 复制方式 | 主动复制 | 每个副本重新创建和存储相同数据 |
|  | 被动复制 | 先在主副本上创建数据，再复制到其他副本 |
| 扩展方向 | 水平扩展 | 拥有更多的数据副本 |
|  | 垂直扩展 | 将副本放到更远的地理位置 |
| 实现方式 | 日志传送 | 辅助数据库定期接收并应用事务日志副本 |
|  | 镜像 | 主数据库的更新立即复制到辅助数据库 |

---

### 数据库环境

| 环境类型 | 用途 | 特点 |
|---------|------|------|
| **生产环境** | 实际业务正式运行 | 非常关键，停止运行会导致利润损失 |
| **开发环境** | 编写和测试代码 | 生产环境的精简配置版本 |
| **测试环境** | QAT、集成测试、UAT、性能测试 | 最好与生产环境拥有相同软硬件配置 |
| **沙箱/实验环境** | 验证假设和创新应用 | 只读访问生产数据，用户自行管理 |
