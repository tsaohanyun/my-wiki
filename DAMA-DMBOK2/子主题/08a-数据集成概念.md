# 数据集成概念

## 概述

**数据集成和互操作**（Data Integration and Interoperability, **DII**）描述了数据在不同数据存储库、应用程序和组织这三者内部和之间进行流动和整合的相关过程。

- **数据集成**：指数据存储、应用程序和组织内部及之间的数据流动和整合。数据集成将数据整合为一致的形式，可以是物理形式的一致，也可以是虚拟形式的一致。
- **数据互操作性**：指多个系统之间进行通信的能力。

### 核心职能

数据集成和互操作解决方案实现了大多数组织依赖的基本数据管理职能，包括：

| 序号 | 职能 | 说明 |
|------|------|------|
| 1 | 数据迁移和转换 | 在系统间移动并转换数据格式 |
| 2 | 数据整合 | 将数据整合到数据中心（Data Hub）或数据集市（Data Mart） |
| 3 | 供应商软件集成 | 将新供应商交付的软件集成到组织现有应用系统框架中 |
| 4 | 数据共享 | 在不同应用程序或组织之间共享数据 |
| 5 | 数据分发 | 跨数据存储库和数据中心分发数据 |
| 6 | 数据归档 | 将低频使用数据迁移至低成本存储 |
| 7 | 数据接口管理 | 管理系统间的数据接口 |
| 8 | 外部数据获取 | 获取和接收外部数据 |
| 9 | 异构数据集成 | 集成结构化和非结构化数据 |
| 10 | 智能支持 | 提供运营智能化和管理决策支持 |

### 目标

1. 以所需的格式和时间范围提供来自不同来源的数据
2. 通过开发共享模型和界面降低管理解决方案的成本和复杂性
3. 识别有意义的事件并自动触发警报和操作
4. 支持商务智能、分析、主数据管理和运营效率工作

### 基本原则

1. **采用企业视角**：确保具备未来可扩展的设计，通过迭代和增量实现交付
2. **平衡需求**：平衡当下与企业全局的数据需求、支撑与维护
3. **业务专家负责**：确保业务专家对数据集成和互操作性设计和活动负责，业务专家应参与数据转换规则的设计和修改

### 业务驱动因素

有效管理数据流动的需求是 DII 的主要驱动力：

- **规模驱动**：大多数组织拥有成百或上千个数据库和数据存储，管理数据在组织内部及与其他组织之间的流动过程已成为每个 IT 组织的核心职责
- **应用集成驱动**：随着组织越来越多地从软件供应商处购买应用程序，每个应用程序都带有自己的主数据存储、事务数据存储和报表数据存储，必须与其他数据存储集成
- **成本驱动**：企业级数据集成设计远远比分散的或点对点的解决方案效率更高、成本更低
- **合规驱动**：支持组织遵守数据处理标准和规则，企业级解决方案可重用代码，实现规则兼容性
- **维护驱动**：标准化工具可有效降低维护和人力成本，提高故障排除效率

> **案例说明**：即便是运行组织公共职能的企业资源规划（ERP）系统，也很少涵盖组织中所有的数据存储，它们同样需要与组织中的其他数据进行集成。在应用程序之间采用点对点的解决方案，可能会产生成千上万的接口，即使是最有效率的 IT 支撑组织都会被迅速拖垮。

### 技术驱动方法、工具与度量指标

| 类别 | 内容 |
|------|------|
| **技术驱动方法** | 松散耦合的应用程序、最小化接口数量、创建标准规范接口、服务编排 |
| **工具** | 数据转换引擎、业务规则引擎、数据虚拟化服务器、企业服务总线、数据和流程建模工具、数据分析工具、元数据存储库 |
| **度量指标** | 数据可用性、数据量和速度、解决方案成本和复杂性 |

## 核心概念

### ETL（抽取-转换-加载）

**ETL**（Extract, Transform, and Load）是数据集成和互操作的核心过程。无论是以物理方式还是虚拟方式执行，无论是批处理还是实时处理，ETL 都是在应用程序和组织之间移动数据的必要步骤。

#### 抽取（Extract）

抽取过程包括选择所需的数据并从其源数据中提取。被抽取的数据会在磁盘或内存中的物理数据存储库中存储。

- 在业务操作系统上执行时，应尽可能少地使用资源以避免对业务产生负面影响
- 在非高峰时间进行批处理是抽取的一个选项
- 需对已变更、待抽取的数据进行选择或识别操作
- 如果需要在磁盘上进行物理缓存，则缓存数据库可以考虑与源数据库或目标数据库合并

#### 转换（Transform）

转换过程是让选定的数据与目标数据库的结构兼容。转换包括多种类型：

| 转换类型 | 说明 | 示例 |
|----------|------|------|
| 格式变化 | 技术上的格式转换 | 从 EBCDIC 到 ASCII |
| 结构变化 | 数据结构的变化 | 从非规范化到规范化的转换 |
| 语义转换 | 数据值转换时保持语义一致化表达 | 源性别代码 0/1/2/3 → UNKNOWN/FEMALE/MALE/NOT PROVIDED |
| 去重 | 扫描目标、检测和删除重复记录 | 键值唯一性约束 |
| 重新排序 | 改变数据元素或记录的顺序 | 适应新定义的模式 |

转换可以批量执行，也可以实时执行。转换阶段所产生的数据应准备好与目标结构中的数据进行集成。

#### 加载（Load）

加载过程是在目标系统的物理存储中进行持久化或直接呈现转换结果。数据可能需要被进一步处理以便与其他数据集成，或者以最终形式呈现给消费者。

#### ELT（抽取-加载-转换）

当目标系统比源系统或中间应用系统具有更强的转换能力时，数据处理顺序可切换为 **ELT**（Extract, Load, and Transform）：

- 允许在数据加载到目标系统后再进行转换
- 源数据以原始数据的形式在目标系统上实例化，对其他进程有用
- 加载至数据湖的方式在大数据环境中很常见

> 参见：[[DAMA-DMBOK2-14-大数据和数据科学]]

#### ETL 与 ELT 对比

| 对比维度 | ETL | ELT |
|----------|-----|-----|
| **处理顺序** | 抽取 → 转换 → 加载 | 抽取 → 加载 → 转换 |
| **转换位置** | 在中间暂存区或 ETL 引擎中转换 | 在目标系统中转换 |
| **适用场景** | 数据仓库传统场景、数据量较小 | 大数据环境、数据湖、云数据仓库 |
| **目标系统要求** | 目标系统无需强大计算能力 | 目标系统需具备较强转换能力（如 MPP 数据库、Hadoop） |
| **数据保留** | 仅加载转换后数据 | 原始数据保留在目标系统，可供其他进程使用 |
| **延迟** | 转换在加载前完成，整体延迟可能较高 | 原始数据先加载，转换可异步进行 |
| **灵活性** | 转换逻辑集中在 ETL 工具中 | 转换逻辑可分散在目标系统中，更灵活 |
| **典型工具** | Informatica、DataStage、SSIS | dbt、Spark SQL、云数据仓库内置转换 |

#### 映射（Mapping）

映射是转换的同义词，既是从源结构到目标结构建立查找矩阵的过程，也是该过程的结果。映射定义了：
- 要抽取的源数据与抽取数据的识别规则
- 要加载的目标与要更新的目标行的识别规则
- 要应用的任何转换或计算规则

许多数据集成工具提供映射的可视化界面，开发人员可使用图形界面创建转换代码。

### 时延（Latency）

时延指从源系统生成数据到目标系统并使得该数据可用的时间差。不同的数据处理方法会导致不同程度的数据时延。

| 时延级别 | 处理方式 | 特征 |
|----------|----------|------|
| 高 | 批处理 | 定期调度，短时间窗口处理大量数据 |
| 较高 | 事件驱动/准实时 | 较小集合处理，系统负载较低 |
| 低 | 实时同步 | 源和目标之间不允许时间延迟 |
| 极低 | 流处理 | 事件发生后立即连续流出 |

#### 批处理（Batch Processing）

大多数数据在应用程序和组织之间以批量文件的形式移动：

- **快照（Snapshot）**：来自某一时刻的完整数据集
- **增量（Delta）**：自上次发送数据以来已发生变化的数据值
- **微批处理**：将批处理安排在比每天一次更高的频率上运行（如每5分钟一次）

**适用场景**：数据转换、迁移和归档，以及从数据仓库和数据集市中提取和加载数据

**风险**：在业务日结束时或夜间进行数据处理以最大限度减少应用程序更新的问题；应根据每日、每周或每月的报告时间表安排将数据移动到数据仓库的作业，以避免不完整的数据集。

#### 变更数据捕获（CDC）

变更数据捕获是一种通过增加过滤减少传送带宽需求的方法，只包含在特定时间范围内更改过的数据。

**基于数据的 CDC 技术**（3种）：

| 技术 | 说明 |
|------|------|
| 标记/时间戳 | 源系统在数据元素中填充时间戳、代码或标识作为变更指示符 |
| 变更列表 | 源系统在更改数据时将对象和标识符添加到简单对象列表中 |
| 变更对象 | 源系统在事务过程中将已更改的数据复制到单独的对象中 |

**基于日志的 CDC**：由数据库管理系统创建的数据活动日志被复制并处理，寻找特定更改后转换并应用到目标数据库。

> **注意**：基于数据的 CDC 技术利用源应用程序中内置的功能，可能会消耗大量资源，并需要有修改源应用程序的能力。

#### 准实时和事件驱动

- 数据在特定时间表内以较小的集合进行处理或在事件发生时处理
- 比批处理具有更低的延迟，系统负载通常较低
- 通常使用企业服务总线（ESB）实现
- 需监控状态信息和进程依赖，确保数据按正确顺序进入目标（如先处理主数据再处理交易数据）

#### 异步（Asynchronous）

- 提供数据的系统在继续处理新数据之前不会等待接收系统的更新确认
- 发送或接收系统都可以在一段时间内离线，而另一个系统能正常运行
- 源系统更新与中继到目标数据集之间的延迟通常为秒级或分钟级

#### 实时同步（Real-time Synchronous）

- 源数据和目标数据之间不允许存在时间延迟或其他差异
- 执行过程在执行下一个活动之前需等待来自其他应用程序或进程的确认
- 通过**两阶段提交**等数据库能力保持数据集同步
- 适用场景：金融机构使用两阶段提交确保财务交易表与财务平衡表完全同步

> **权衡**：同步集成方案只能处理更少的事务，因为必须等待确认。如果任何需要更新的应用程序不可用，主应用程序中的事务就无法完成，这可能使关键应用程序依赖不太重要的应用程序。

#### 低延迟或流处理

- 数据流在事件发生后立即从计算机系统实时连续地流出
- 包括使用固态硬盘等硬件解决方案或内存数据库等软件解决方案
- 异步通常用于低延迟解决方案
- 大规模多处理器或并行处理是常见配置

> **投资考量**：低延迟解决方案需要大量的硬件和软件投资。当组织需要非常快速地进行远距离数据移动时，额外成本才是合理的。传统磁盘驱动器的读写过程比内存或固态磁盘驱动器慢数千倍。

### 复制（Replication）

复制技术将分析和查询对主业务操作环境性能的影响降至最低：

- 在多个物理位置上维护数据集的精确副本以供就近访问
- 通常监视数据集的更改日志而非数据集本身，最大限度减少对应用操作的影响
- 标准复制解决方案是准实时的，副本间更新延时较小
- 使用数据库管理工具不需要大量编程工作，很少会有程序缺陷问题
- **限制**：源数据集和目标数据集应为彼此的精确副本；不适合多副本站点同时发生数据更改的场景

> 参见：[[DAMA-DMBOK2-06-数据存储和运营]]

### 归档（Archiving）

使用频率较低或未被积极使用的数据，可迁移到成本更低的替代数据结构或存储解决方案中：

- ETL 功能可用于将待归档数据传输到归档环境
- 适用于存储即将被淘汰的应用程序中的数据及长期未使用的生产运行系统中的数据
- **风险**：必须监控归档技术，确保数据在技术变更时仍然可访问；旧结构或格式可能无法被新技术读取，特别是对于仍需依规合法保存的数据

> 参见：[[DAMA-DMBOK2-09-文档和内容管理]]

### 企业消息格式/规范模型

规范模型是组织或数据集成团队用于标准化数据共享格式的通用模型：

- 在**中心辐射型**数据交互设计模型中，所有系统只需与中央信息枢纽交互
- 采用规范数据模型可减少数据交换所需的数据转换数量
- 每个系统仅需将数据转换为中央规范模型格式，无须针对每个系统的格式逐一转换
- 在管理3个以上系统时合理，超过100个应用系统时尤为重要

> **实践建议**：开发和商定共享的企业消息格式是一项重要任务。拥有规范数据模型可以显著降低企业中数据互操作的复杂性，从而大大降低维护成本。

### 交互模型

#### 点对点（Point-to-Point）

数据直接在系统间相互传递。适用于少量系统场景，但存在以下问题：

| 问题 | 说明 |
|------|------|
| 影响处理性能 | 提供数据的工作负载可能影响源系统正常业务性能 |
| 提高接口管理难度 | 接口数量接近系统数量的平方，管理维护工作量可能超过系统本身 |
| 带来潜在不一致性 | 多个系统需要不同版本或格式的数据时容易出现设计问题 |

#### 中心辐射型（Hub-and-Spoke）

通过一个中央数据枢纽（物理或虚拟）集中共享数据，供多个应用程序使用：

- **优点**：提供一致的数据视图、对源系统性能影响有限、减少提取次数、添加新系统只需建立与数据枢纽的接口
- **实现形式**：数据仓库、数据集市、操作型数据存储（ODS）、主数据管理（MDM）
- **企业服务总线（ESB）**：用于接近实时共享数据，数据枢纽是虚拟概念，定义共享数据的标准格式
- **局限**：部分模型存在不可接受的时延或性能问题；数据枢纽本身存在创建开销

#### 发布与订阅（Publish and Subscribe）

- 系统将数据推送（发布）出去，其他系统拉取（订阅）数据
- 在数据服务目录中列出推送数据的系统，希望使用的系统订阅这些服务
- 集中开发数据集并向所有需要者统一提供，确保一致性和及时性

### 数据集成和互操作架构概念

#### 应用耦合

| 耦合类型 | 特征 | 风险 |
|----------|------|------|
| **紧耦合** | 同步接口，一个系统需等待另一个系统的响应 | 一个系统不可用会导致另一个也受影响；业务连续性计划必须一致 |
| **松耦合** | 系统间传送数据不需要等待响应 | 一个系统不可用不会导致另一个也无法使用 |

松耦合可通过服务、API 或消息队列等技术实现。SOA 中的 ESB 是一种松散耦合的设计模式，其应用系统的替换通常不需要重写与其他系统的交互接口。

#### 编排和流程控制

**编排（Orchestration）**：描述多个流程在系统中的组织和执行方式。

**流程控制**组件包括：
- 数据库活动日志
- 批处理作业日志
- 警报
- 异常日志
- 作业依赖图（包含补救方案、标准响应）
- 作业的时钟信息（依赖作业的时间、预计作业时长、窗口时间）

#### 企业应用集成（EAI）

- 软件模块之间仅通过定义良好的 API 调用进行交互
- 数据存储只能通过自己的软件模块更新
- 基于面向对象概念，强调重用和替换任何模块而不影响其他模块

#### 企业服务总线（ESB）

- 在系统之间传递消息的中介
- 应用程序可通过 ESB 功能封装消息或文件，并在系统间发送和接收
- 充当两个应用程序之间的服务角色，是松散耦合的典型示例
- ESB 的中央处理器通常在独立于其他参与系统的服务器上实现
- 处理器跟踪哪些系统对什么类型的消息感兴趣，不断轮询每个参与系统传出的消息
- 因为数据从发送系统到接收系统需要几分钟的时间，这种模型被称为"准实时"型

#### 面向服务的架构（SOA）

- 在独立的软件模块之间定义良好的交互
- 每个模块提供其他模块或个人消费者执行的功能（服务）
- 服务没有调用应用程序的预先知识，实现是调用应用程序的黑匣子
- 可通过 Web 服务、消息传送、RESTful API 等多种技术实现
- 数据服务包括数据的添加、删除、更新和检索

> **治理要求**：必须围绕服务和 API 的设计及注册建立强大的治理模型。在开发新数据服务之前，确保不存在能满足请求的已有服务；新服务设计需考虑满足广泛需求以支持重用。

#### 复杂事件处理（CEP）

复杂事件处理将多个来源的数据进行合并，通过识别有意义的事件（如机会或威胁），为事件设置规则指导处理及路由，进而预测行为或活动，并根据预测结果自动触发实时响应。

**应用场景**：
- 预测消费者购买行为并自动推荐产品
- 检测可疑信用卡欺诈性扣款
- 监测传感器数据并触发预警

**数据挑战**：
- 有时事件发生的概率让在发生事件时检索必要的额外数据变得不切实际
- 高效处理通常要求预先在 CEP 引擎的内存中预存一些数据
- 需要能够集成各种类型数据的环境
- 常与大数据相关，需要支持超低时延要求的技术

> 参见：[[DAMA-DMBOK2-14-大数据和数据科学]]

#### 数据联邦和虚拟化

| 概念 | 说明 |
|------|------|
| **数据联邦**（Data Federation） | 无论各自结构如何，提供访问各个独立数据存储库组合的权限 |
| **数据虚拟化**（Data Virtualization） | 使分布式数据库及多个异构数据存储能够作为单个数据库来访问和查看 |

> 参见：[[DAMA-DMBOK2-06-数据存储和运营]]

**数据虚拟化详解**：

数据虚拟化服务器与传统的物理 ETL 工具不同，它对数据进行**虚拟抽取、虚拟转换和虚拟集成**，而非物理移动数据。

| 对比维度 | 物理集成（ETL） | 数据虚拟化 |
|----------|----------------|------------|
| **数据移动** | 物理抽取、转换并加载到目标 | 数据留在原处，通过虚拟层访问 |
| **数据存储** | 需要额外的目标存储空间 | 不需要额外存储，节省磁盘资源 |
| **实时性** | 通常为批处理，存在延迟 | 近实时访问最新数据 |
| **适用场景** | 大批量数据转换、历史数据仓库 | 跨源查询、实时报表、敏捷数据分析 |
| **性能影响** | 对源系统影响可控（可调度） | 查询时可能对源系统产生负载 |
| **数据一致性** | 转换后数据高度一致 | 依赖源系统数据一致性 |
| **典型工具** | Informatica、DataStage、SSIS | Denodo、Apache Tez、Presto |

数据虚拟化服务器可以将结构化数据和非结构化数据进行合并。数据仓库经常是数据虚拟化服务器的输入，但数据虚拟化服务器不会替代企业信息架构中的数据仓库。数据虚拟化适用于需要快速集成多个数据源、减少数据冗余和缩短交付周期的场景。

#### 数据即服务（DaaS）

- 从供应商获得许可并按需由供应商提供数据，而非存储和维护在被许可组织数据中心
- 典型示例：证券交易所出售证券和相关价格信息
- 也适用于组织内部，为各种功能和操作系统提供企业数据或数据服务
- 服务组织提供可用服务目录、服务级别和定价计划

#### 云化集成（IPaaS）

**云化集成**（Integration Platform-as-a-Service, IPaaS）是作为云服务交付的系统集成形式：

- 可用于处理数据、流程、SOA 和应用集成
- 在云计算出现之前：内部集成通过 ESB 管理；企业间集成通过 EDI 网关、VAN 等完成
- SaaS 模式为整合组织外部数据创造了新需求
- 云化集成解决方案通常作为 SaaS 应用程序在供应商数据中心运行

> 参见：[[DAMA-DMBOK2-06-数据存储和运营]]

### 数据交换标准

数据交换标准是数据元素结构的正式规则：

- **交换规范**：组织或数据交换团队用于标准化数据共享格式的通用模型
- **交换模式**：定义任何系统或组织交换数据所需的数据转换结构
- **NIEM**（国家信息交换模型）：为美国政府机构之间交换文件和交易而开发的数据交换标准，使用 XML 定义模式和元素

> 在系统之间达成一致的交换格式或数据布局可以大大简化企业中的数据共享过程，从而降低支撑成本，并使工作人员能更好地理解数据。

### 语义互操作

**语义互操作**（Semantic Interoperability）是指不同系统之间不仅能够交换数据，还能确保对数据含义的一致理解。它是数据集成中最高层次的互操作挑战。

| 互操作层次 | 说明 | 示例 |
|------------|------|------|
| **技术互操作** | 系统之间能够通过网络通信和交换数据 | HTTP 协议、TCP/IP 连接 |
| **语法互操作** | 数据格式和结构的一致性 | XML Schema、JSON 格式、CSV 结构 |
| **语义互操作** | 数据含义和业务规则的一致理解 | "客户"在不同系统中的定义一致性 |

**语义互操作的关键挑战**：

| 挑战 | 说明 | 应对策略 |
|------|------|----------|
| **术语差异** | 同一概念在不同系统中使用不同名称 | 建立企业数据字典和术语标准 |
| **含义差异** | 同一术语在不同系统中含义不同 | 定义统一的业务术语表和元数据 |
| **上下文差异** | 数据值依赖于业务上下文 | 记录业务规则和上下文元数据 |
| **粒度差异** | 数据的详细程度不同 | 建立数据粒度映射和聚合规则 |
| **时间差异** | 数据的时间参考点不同 | 统一时间戳标准和时区处理 |

**实现语义互操作的方法**：

1. **元数据管理**：维护完整的业务元数据和技术元数据，确保数据定义的一致性
2. **规范数据模型**：建立企业级规范模型，统一数据语义
3. **本体和语义标准**：使用 RDF、OWL 等语义 Web 技术定义数据关系
4. **数据治理**：通过数据治理确保语义规则的制定和执行
5. **数据目录**：建立数据目录，记录每个数据元素的业务含义和转换规则

> 参见：[[DAMA-DMBOK2-12-元数据管理]]

### 消息队列与事件流平台

消息队列和事件流平台是现代数据集成架构中的关键组件，用于实现系统之间的异步数据传递和解耦。

#### 消息队列（Message Queue）

消息队列是一种异步通信机制，允许生产者将消息发送到队列，消费者从队列中读取消息，实现系统之间的松耦合。

| 特性 | 说明 |
|------|------|
| **异步通信** | 生产者发送消息后无需等待消费者处理 |
| **解耦** | 生产者和消费者独立运行，互不影响 |
| **削峰填谷** | 缓冲突发流量，保护下游系统 |
| **可靠性** | 消息持久化，确保不丢失 |
| **典型实现** | RabbitMQ、ActiveMQ、IBM MQ |

#### 事件流平台（以 Apache Kafka 为例）

**Apache Kafka** 是一种分布式事件流平台，专为高吞吐量、低延迟的实时数据流处理而设计。

| 特性 | 说明 |
|------|------|
| **分布式架构** | 数据分布在多个节点上，支持水平扩展 |
| **高吞吐量** | 每秒可处理数百万条消息 |
| **持久化** | 消息持久化到磁盘，支持回溯消费 |
| **流处理** | 支持实时流处理（Kafka Streams、ksqlDB） |
| **事件溯源** | 支持事件驱动架构和事件溯源模式 |

**Kafka 核心概念**：

| 概念 | 说明 |
|------|------|
| **Topic** | 消息的逻辑分类，类似于数据库中的表 |
| **Partition** | Topic 的物理分区，支持并行处理 |
| **Producer** | 消息生产者，将事件发布到 Topic |
| **Consumer** | 消息消费者，从 Topic 读取事件 |
| **Consumer Group** | 消费者组，实现负载均衡和容错 |
| **Broker** | Kafka 集群中的服务器节点 |
| **Offset** | 消息在分区中的位置标识 |

**Kafka 在数据集成中的应用场景**：

| 场景 | 说明 |
|------|------|
| **实时数据管道** | 在系统之间构建实时数据流 |
| **变更数据捕获** | 结合 Debezium 等工具实现实时 CDC |
| **事件驱动架构** | 作为事件总线，驱动微服务间的事件通信 |
| **日志聚合** | 集中收集和处理分布式系统的日志 |
| **流式 ETL** | 实时数据转换和加载，替代传统批处理 ETL |

> **与传统 ESB 的对比**：Kafka 等事件流平台更侧重于高吞吐量的数据流处理，而 ESB 更侧重于服务编排和消息路由。现代架构中两者可以互补使用。

### API 管理

API（Application Programming Interface）管理是现代数据集成的重要组成部分，涉及 API 的设计、发布、监控和治理全生命周期。

#### API 类型

| API 类型 | 说明 | 适用场景 |
|----------|------|----------|
| **RESTful API** | 基于 HTTP 协议，使用 JSON/XML 格式 | Web 应用、移动端、微服务 |
| **GraphQL** | 查询语言，允许客户端精确请求所需数据 | 复杂数据查询、前端灵活需求 |
| **SOAP** | 基于 XML 的协议，严格的消息格式 | 企业级应用、金融行业 |
| **gRPC** | 基于 HTTP/2 的高性能 RPC 框架 | 微服务间通信、低延迟场景 |
| **WebSocket** | 全双工通信协议 | 实时数据推送、聊天应用 |

#### API 管理生命周期

| 阶段 | 活动 | 说明 |
|------|------|------|
| **设计** | API 定义、规范编写 | 使用 OpenAPI/Swagger 等标准定义 API |
| **开发** | API 实现、测试 | 编写代码、单元测试、集成测试 |
| **发布** | API 部署、文档发布 | 部署到 API 网关，发布开发者文档 |
| **运营** | 监控、分析、版本管理 | 监控 API 调用量、响应时间、错误率 |
| **退役** | API 下线、迁移 | 通知消费者，提供迁移路径 |

#### API 网关

API 网关是 API 管理的核心组件，提供以下功能：

| 功能 | 说明 |
|------|------|
| **路由** | 将 API 请求路由到对应的后端服务 |
| **认证与授权** | 验证调用者身份和权限 |
| **限流与节流** | 控制 API 调用频率，防止过载 |
| **缓存** | 缓存响应数据，提升性能 |
| **日志与监控** | 记录 API 调用日志，监控运行状态 |
| **协议转换** | 在不同协议之间进行转换 |
| **版本管理** | 支持 API 多版本并存 |

#### API 治理

API 治理确保 API 的设计和使用符合企业标准：

- **命名规范**：统一 API 命名风格（如 RESTful 资源命名）
- **版本策略**：定义 API 版本管理策略（URL 版本、Header 版本）
- **安全标准**：统一认证方式（OAuth2、JWT）、数据加密标准
- **文档标准**：要求所有 API 必须有完整的 OpenAPI 文档
- **服务目录**：建立 API 服务目录，促进服务发现和重用

> **治理要求**：必须围绕服务和 API 的设计及注册建立强大的治理模型。在开发新数据服务之前，确保不存在能满足请求的已有服务；新服务设计需考虑满足广泛需求以支持重用。

### 集成架构模式概览

以下是数据集成和互操作中常用的架构模式对比：

| 架构模式 | 说明 | 优势 | 劣势 | 适用场景 |
|----------|------|------|------|----------|
| **ETL** | 抽取-转换-加载，物理移动数据 | 数据质量可控、不影响源系统 | 延迟较高、需要额外存储 | 数据仓库、批量数据集成 |
| **ELT** | 抽取-加载-转换，在目标系统转换 | 原始数据保留、灵活性高 | 目标系统需强大计算能力 | 大数据、云数据仓库、数据湖 |
| **CDC** | 变更数据捕获，仅传输变更数据 | 带宽低、对源系统影响小 | 实现复杂、依赖源系统能力 | 准实时数据同步、数据复制 |
| **ESB** | 企业服务总线，消息中介 | 松耦合、服务编排 | 复杂性高、可能成为瓶颈 | 企业应用集成、SOA 架构 |
| **消息队列** | 异步消息传递 | 解耦、削峰、可靠性 | 消息顺序保证复杂 | 异步通信、事件驱动 |
| **事件流** | 实时事件流处理（如 Kafka） | 高吞吐、低延迟、可回溯 | 运维复杂、学习曲线高 | 实时数据管道、流式 ETL |
| **数据虚拟化** | 虚拟集成，不移动数据 | 实时访问、无冗余存储 | 性能依赖源系统 | 跨源查询、敏捷分析 |
| **API** | 通过接口访问数据和服务 | 标准化、可控、可监控 | 需要治理、版本管理 | 微服务、数据服务化 |
| **中心辐射型** | 中央数据枢纽集中共享 | 一致性好、接口少 | 枢纽创建开销、单点风险 | 多系统数据共享、MDM |
| **点对点** | 系统间直接交互 | 简单直接 | 接口爆炸、难以维护 | 仅适用于少量系统 |
