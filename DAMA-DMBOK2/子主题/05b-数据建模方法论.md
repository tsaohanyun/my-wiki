# 数据建模和设计：方法论与实践

> **所属知识体系**：[[DAMA-DMBOK2/05-数据建模和设计|DAMA-DMBOK2 第5章：数据建模和设计]]

---

## 方法论

### 建模流程

数据建模和设计活动包括4个步骤：

1. **数据建模规划（P）**
2. **构建数据模型（D）**：概念数据建模 → 逻辑数据建模 → 物理数据建模
3. **评审数据模型（C）**
4. **维护数据模型（O）**

#### 1. 规划数据建模

数据建模工作计划主要包括：

| 步骤 | 内容 |
|------|------|
| 评估组织需求 | 了解业务背景和数据需求 |
| 确定建模标准 | 制定命名约定、符号规范等 |
| 明确存储管理 | 确定数据模型的存储和管理方式 |

**交付成果：**

1. **图表（Diagram）**：一个数据模型包含一到多个图表，以精确的方式描述对数据的需求，包括不同层级（概念、逻辑或物理模型）、具体的某种模式（关系、维度、对象、基于事实的、基于时序的或NoSQL），以及数据模式的表示（如IE、UML、ORM）。
2. **定义（Definitions）**：实体、属性和关系这3个对象的准确定义对于确保数据模型的精度而言至关重要。
3. **争议和悬而未决的问题（Issue and Outstanding Question）**：数据建模过程中经常会出现一些无法解决的争议和难以回答的问题。
4. **血缘关系（Lineage）**：有关数据是从哪里来、经过怎样的加工、变成了什么样的结果的脉络关系。

---

#### 2. 构建数据模型

数据建模是一个不断迭代的过程。在建模过程中，首先要研究现有的数据模型和数据库，以及参考已发布的建模标准和数据标准，并搜集和考虑随时提出的新数据要求。

**正向工程流程：**

```
概念数据建模 → 逻辑数据建模 → 物理数据建模
```

**概念数据建模步骤：**
1. 选择模式（关系、维度、基于事实或非关系型）
2. 选择模型表示方式（IE、ORM等）
3. 完成初版概念数据模型——捕获特定用户群组的想法
4. 整合企业术语——利用既定的企业术语和建模规则
5. 签署确认模型

**逻辑数据建模步骤：**
1. 分析信息需求
2. 分析现有文档
3. 添加关联实体
4. 添加属性
5. 分配域
6. 分配键

**物理数据建模步骤：**
1. 解决逻辑抽象实体——子类合并或超类拆分
2. 添加属性细节——添加技术名称、物理域、物理数据类型和长度、约束等
3. 添加参考数据对象
4. 分配代理键
5. 逆规范化以提高性能
6. 添加索引以提高性能
7. 分区以提高性能
8. 创建视图

**逆向工程：**

从现有数据库文档化的过程：**PDM → LDM → CDM**。大多数数据建模工具都支持各种数据库的逆向工程，但是逆向工程出来的模型是比较混乱的，为了提高其可读性，需要建模人员对其布局进行重新设计。

---

#### 3. 评审数据模型

数据模型需要质量控制和持续改进。诸如时间-价值（time-to-value）、支持成本和数据模型质量验证器（如数据模型计分卡）等技术都可用于评估模型的正确性、完整性和一致性。

评审会议的议程应包括评审初版模型（如果有）、对模型所做的更改、所有其他被考虑和拒绝的选项，以及新模型与现有建模或架构标准的符合程度。

设计评审应由一组具有不同背景、技能、期望和观点的主题专家进行。参与者必须能够讨论不同的观点并达成小组共识。

---

#### 4. 维护数据模型

当需求变化时，需要对数据模型进行更新。某一层的改变通常需要相应更高层或更底层模型的改变。在每个开发迭代结束时，一个好的做法是对最新的物理数据模型进行逆向工程，并确保它与相应的逻辑数据模型保持一致。

| 工具类型 | 功能说明 |
|----------|----------|
| 数据建模工具 | 提供绘图、正向/逆向工程、DDL生成等功能 |
| 血缘分析工具 | 维护和展现数据属性来源，支持影响度分析 |
| 数据剖析工具 | 探索数据内容，识别数据质量差距和元数据缺陷 |
| 元数据库 | 存储数据模型描述性信息，提供元数据集成交换功能 |
| 数据模型模式 | 可重复使用的模型结构，分为基本模式、组装模式和集成模式 |
| 行业数据模型 | 为特定行业预先构建的数据模型，通常范围广泛且具体 |

### 数据模型计分卡

数据模型计分卡（Data Model Scorecard）用于衡量数据模型质量，包含10个质量指标：

| 序号 | 评价指标 | 说明 |
|------|----------|------|
| 1 | 模型满足需求的能力 | 确保数据模型满足已定义的需求 |
| 2 | 模型的完整性 | 包括需求完整性和元数据完整性 |
| 3 | 模型与实例的匹配程度 | 确保模型的层级和实例定义相匹配 |
| 4 | 模型结构的合理性 | 验证数据模型设计的落地性 |
| 5 | 模型的通用性 | 评审模型的扩展性或抽象程度 |
| 6 | 命名标准遵循情况 | 评审命名标准结构、术语和风格 |
| 7 | 模型的可读性 | 评审数据模型的易读性 |
| 8 | 对象定义情况 | 评审所有定义项是否清晰、完整和准确 |
| 9 | 与企业架构一致性 | 评审数据模型中的结构能否在更加广泛和一致的环境中应用 |
| 10 | 与元数据匹配程度 | 将存储模型结构中的数据和实际数据进行比对 |

**评分方法：**
- **模型评分列**：评审者按照每一项指标对模型评价后的得分
- **总分列**：每项指标的总分值
- **百分比列**：评审得分除以总分数，乘以100%
- **备注栏**：记录更详细的得分解释或模型修改事项

## 最佳实践

### 命名标准

ISO/IEC 11179元数据注册库是一项国际标准，用于组织中元数据的表示。

命名标准要点：
- 逻辑名称对业务用户有意义，尽可能使用完整的单词
- 物理名称必须符合DBMS允许的最大长度，在必要时使用缩写
- 逻辑名称使用空格作为单词之间的分隔符，实际名称通常使用下划线
- 避免跨环境的名称变化（测试环境、质量评价环境或生产环境）
- 使用分类词区分属性类型

### 数据库设计（PRISM原则）

| 原则 | 说明 |
|------|------|
| **P**erformance（性能和易用性） | 确保授权用户可以快速、便捷地访问数据 |
| **R**eusability（可重用性） | 确保数据库结构在适当的情况下能够被多个应用重复使用 |
| **I**ntegrity（完整性） | 数据在任何情况下都应始终具有有效的业务含义和价值 |
| **S**ecurity（安全性） | 真实和准确的数据应以高效、常态化的方式提供给授权用户 |
| **M**aintainability（可维护性） | 以合理成本执行所有数据工作，确保数据的创建、存储、维护和使用成本不超过其对组织的价值 |

## 行业应用

行业数据模型是为特定行业预先构建的数据模型，通常范围广泛且具体，一些模型包含数千个实体和属性。

| 行业 | 标准组织/模型 |
|------|--------------|
| 零售 | ARTS |
| 通信 | SID |
| 保险 | ACORD |
| 医疗保健 | 行业模型供应商 |
| 银行 | 行业模型供应商 |
| 制造业 | 行业模型供应商 |

> 所有购买的数据模型都需要裁剪定制以适应组织需求。

## 方案价值

### 风险与挑战

| 风险类型 | 说明 | 缓解措施 |
|----------|------|----------|
| 模型过时 | 业务变化导致模型与实际不符 | 建立定期维护和评审机制 |
| 过度规范化 | 影响查询性能 | 合理使用逆规范化 |
| 沟通障碍 | 业务与IT理解不一致 | 使用多种表示方式、加强培训 |
| 标准不一致 | 命名和设计风格混乱 | 制定并执行统一标准 |
| 知识流失 | 关键人员离开导致知识断层 | 完善文档化和知识传承 |

### ROI分析

| 投入 | 产出 | 价值 |
|------|------|------|
| 建模工具采购 | 标准化模型资产 | 提高开发效率 |
| 建模人员培训 | 高质量数据模型 | 降低维护成本 |
| 模型评审时间 | 减少设计缺陷 | 降低返工成本 |
| 元数据管理 | 数据血缘和可追溯性 | 支持合规和治理 |

## 数据模型治理

### 数据模型和设计质量管理

数据分析人员和设计人员作为信息消费者和数据生产者的中介，必须平衡信息消费者的数据需求和数据生产者的应用需求。

数据专业人士还必须平衡短期的商业利益和长期的商业利益。

### 数据模型和数据库设计标准应包括：

1. 数据建模和数据库设计的标准交付成果的清单和描述
2. 适用于所有数据模型对象的标准名称
3. 所有数据模型对象的标准命名格式列表
4. 设计和维护交付成果的标准方法的列表和描述
5. 数据建模和数据库设计角色和职责的列表和描述
6. 数据建模和数据库设计中涉及的所有元数据属性的列表和描述
7. 元数据质量的期望和要求
8. 使用数据建模工具的准则
9. 准备和指引设计评审的准则
10. 数据模型版本管理的指导原则
11. 禁止或需要避免的相关事项列表

### 版本管理

需要对数据模型和其他设计需求的变更进行详细管控。对于每个变更都应该予以记录：

1. **为什么要变更**
2. **哪些对象发生了变更**
3. **何时审批通过的这一变更**
4. **谁执行的变更**
5. **此变更引起了模型中的哪些地方需要随之变更**

> 一些通用的数据建模工具提供数据模型版本管理和集成功能的模型管理存储库。
