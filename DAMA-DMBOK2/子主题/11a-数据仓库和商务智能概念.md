# 数据仓库和商务智能：概念与基础

> **所属知识体系**：[[DAMA-DMBOK2/11-数据仓库和商务智能|DAMA-DMBOK2 第11章：数据仓库和商务智能]]

---

## 概述

数据仓库（Data Warehouse, DW）的概念起源于20世纪80年代，其核心价值在于将不同来源的数据集成到一个通用的数据模型中加以利用。集成后的数据不仅能为组织的业务运营流程提供深刻洞察，还能为企业智能决策、创新组织价值开辟全新途径。同样重要的是，数据仓库也是组织减少决策支持系统（DSS）重复建设的重要手段。企业级数据仓库（EDW）的理念可以减少数据冗余、提升信息一致性，以及支撑企业利用数据做出更精准的决策。

进入20世纪90年代，数据仓库建设开始受到前所未有的重视，特别是伴随商务智能（BI）作为业务决策主要驱动力的发展，数据仓库也随之确立了其"主流"地位。

**业务驱动因素：**
- 支撑业务运营和商务智能活动
- 满足监管合规要求（存储历史数据作为运营合规凭证）
- 提供对组织、客户及产品的深入洞察，提升运营效率和竞争优势
- 从回顾性评价发展到预测性分析能力

**建设目标：**
1. 支持商务智能系统的建设
2. 实现有效的业务分析和决策
3. 基于数据洞察寻找更多创新方法

**指导原则：**
1. **聚焦业务目标** — 确保数据仓库服务于组织的优先事项
2. **以终为始** — 以业务优先级和最终交付的数据范围来引领构建
3. **全局性视野设计，局部性行动建设** — 以最终愿景指导架构设计，迭代构建、增量交付
4. **先具体细化，后汇总优化** — 首先建立最细粒度的原子数据，逐层聚合
5. **倡导数据公开透明和自助服务** — 提供丰富上下文，赋予数据消费者更强大的数据洞察能力
6. **在建设数据仓库的同时建设元数据** — 元数据应作为开发周期的一部分进行采集和管理
7. **合作协同** — 与其他数据项目密切协作
8. **不要千篇一律** — 为各类数据消费者提供适合他们的数据工具和产品

## 核心概念

### 数据仓库定义

数据仓库由两个核心组件构成：
- **集成数据库** — 用于决策支撑的集成数据存储
- **相关软件程序** — 用于采集、清洗、转换和存储来自各种内外部数据源的数据

**企业级数据仓库（Enterprise Data Warehouse, EDW）** 是一种集中式的数据仓库，遵循企业级数据模型以确保整个组织决策支持活动的一致性。

**数据仓库建设**指数据仓库中数据的抽取、清洗、转换、控制和加载等操作过程，侧重于通过强制执行业务规则和维护业务数据关系，在操作型数据基础上实现集成的、反映历史变化的数据集。

传统数据仓库主要关注结构化数据，但如今已开始支持半结构化数据和非结构化数据。

### 商务智能（BI）

"商务智能"具有双重含义：

1. **分析方法** — 旨在理解组织活动并发现潜在商业机会的数据分析方法
2. **技术栈** — 支撑上述数据分析活动的技术体系，支持数据查询、数据挖掘、统计分析、报表分析、场景建模、数据可视化和仪表盘等功能

### 多维建模

多维模型通常被称为**星型模型**，由**事实表**和**维度表**组成：

- **事实表**：包含业务流程的定量数据（如销售数据）
- **维度表**：存储与事实数据相关的描述性属性（如时间、产品、地点）

### 星型模型与雪花型模型

| 模型类型 | 特点 | 适用场景 |
|---------|------|---------|
| **星型模型** | 维度表直接连接到事实表，非规范化设计，查询性能优异 | 查询性能要求高、维度层次简单的场景 |
| **雪花型模型** | 维度表进一步规范化拆分，减少数据冗余，但增加查询复杂度 | 维度层次复杂、需要严格规范化的场景 |

（参见 [[DAMA-DMBOK2/05-数据建模和设计]]）

### 数据集市

数据集市是数据仓库中部分数据子集的副本，专为支撑特定分析需求而设计：

- **部门数据集市** — 面向特定主题域、单个部门或单个业务流程
- **操作型数据集市（OpDM）** — 专注于运营决策支持，直接从ODS获取数据

### OLAP（在线分析处理）

OLAP是一种为多维分析查询提供快速性能的方法，其典型输出采用矩阵格式。

**常见OLAP操作：**

| 操作 | 说明 |
|------|------|
| **切片（Slice）** | 固定一个或多个维度的单个值 |
| **切块（Dice）** | 对两个以上维度的切片 |
| **向下/向上钻取（Drill Down/Up）** | 在不同数据粒度级别之间逐层探索 |
| **上卷（Roll-Up）** | 计算一个或多个维度的所有数据关系汇总 |
| **旋转（Pivot）** | 改变报表或页面展示的维度方向 |

**OLAP三种实施方法：**

| 方法 | 全称 | 说明 |
|------|------|------|
| **ROLAP** | 关系型联机分析处理 | 通过RDBMS中的二维表实现多维技术 |
| **MOLAP** | 多维矩阵型联机分析处理 | 使用专门的多维数据库技术 |
| **HOLAP** | 混合型联机分析处理 | ROLAP与MOLAP的组合 |

### BI架构

数据仓库/商务智能环境的架构组件包括：

- **源系统** — CRM、财务系统、人力资源系统等业务系统
- **数据集成层** — ETL、数据虚拟化及将数据转化为统一格式的技术
- **数据存储区域** — 包括数据暂存区、主数据管理、中央数据仓库、ODS、数据集市、数据立方体
- **数据访问工具** — 终端用户的分析和报表工具

**数据存储区域详细说明：**

| 存储区域 | 说明 |
|---------|------|
| **数据暂存区** | 介于原始数据源与集中式数据仓库之间的中间存储区域 |
| **主数据管理** | 参考数据和主数据存储在单独存储库中 |
| **中央数据仓库** | 经过转换和预处理后持久化保存所有历史原子数据 |
| **ODS（操作型数据存储）** | 中央数据仓库的某个数据来源副本，支持更低延迟 |
| **数据集市** | 支持数据仓库环境的展示层 |
| **数据立方体** | 支持OLAP的经典模式 |

**数据生命周期的流转：** 源系统 → 数据暂存区（清洗、集成、增强、标准化）→ 数据仓库/ODS → 数据集市/数据立方体 → 各类报表

**数据仓库与应用程序中数据的差异：**

| 差异维度 | 数据仓库/数据集市 | 应用程序 |
|---------|-----------------|---------|
| 组织形式 | 按主题域 | 按功能需求 |
| 数据集成 | 经过集成整合 | 数据孤岛 |
| 时间特性 | 随时间变化的系列数据 | 当前时间的状态值 |
| 延迟 | 较高 | 较低 |
| 历史数据 | 包含大量历史数据 | 历史数据较少 |

大数据为企业数据流转增加了一条新路径：大部分数据仓库在将数据放入表之前会先进行数据集成，而大数据解决方案会在整合数据之前先接收数据。
