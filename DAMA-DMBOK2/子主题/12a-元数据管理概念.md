# 元数据管理：概念与基础

> **所属知识体系**：[[DAMA-DMBOK2/12-元数据管理|DAMA-DMBOK2 第12章：元数据管理]]

---

## 概述

**元数据管理（Metadata Management）** 是数据管理知识体系中的第12个核心职能领域。元数据通常被定义为"关于数据的数据"，这一简洁的定义背后，涵盖了对技术、业务流程、数据规则、约束及逻辑与物理数据结构等极为广泛的信息范围。

元数据描述了数据本身（如数据库、数据元素、数据模型）、数据所表示的概念（如业务流程、应用系统、软件代码、技术基础设施），以及数据与概念之间的联系（关系）。元数据帮助组织理解自身的数据、系统和工作流，促进数据质量评估，对数据库和其他应用的管理至关重要。它有助于提升组织**处理、维护、集成、保护、审计和治理数据**的能力。

可以类比一座拥有成千上万书籍却缺乏目录卡片的大型图书馆——没有元数据的组织，就如同没有目录卡片的图书馆。若缺乏可靠的元数据，组织无从知晓自身拥有什么数据、数据表示什么、数据来自何处、数据如何在系统中流转、谁有权访问这些数据，也无法明确保持高质量数据的重要意义。

> 要成为数据驱动型组织，就必须先实现元数据驱动。

### 元数据管理定义

**元数据管理**：规划、实施和控制那些有助于处理、维护、集成、保护、审计和管理其他数据的活动。

### 目标

1. 提供对业务术语和使用的组织理解，包括技术沿袭
2. 收集和整合不同来源的元数据
3. 提供标准的元数据访问方式，并确保数据交换中的信任水平
4. 确保元数据的质量、一致性、时效性和安全性

### 业务驱动因素

可靠、管理良好的元数据有助于：

- **提高数据可信度**：通过提供上下文语境和开展数据质量度量
- **提升战略数据价值**：通过强化多元化应用，如主数据
- **提高运营效率**：通过识别冗余数据和流程
- **防止使用过时或不正确的数据**
- **减少查找和分析数据所需的时间**
- **提高沟通成效**：在数据使用者和IT专业人员之间
- **减少项目失败风险**：通过创建准确的影响分析
- **提高产品上市速度**：通过缩短系统开发生命周期
- **降低培训成本**：通过全面记录数据上下文语境、历史和来源
- **满足监管合规要求**

元数据有助于采用一致的方式表示信息，尤其当需要满足监管合规要求时，能够简化工作流程，以及保护敏感信息。

反之，管理不善的元数据会导致：冗余的数据和流程、重复的数据字典和元数据库、不一致的数据元素定义、元数据版本间相互冲突，以及数据和元数据的可靠性受到质疑。

### 目标和原则

成功实施元数据解决方案应遵循以下指导原则：

| 原则 | 说明 |
|------|------|
| **组织承诺** | 确保组织对元数据管理的承诺（包括企业高层管理和经费支持），将元数据管理作为企业级数据资产管理战略的一部分 |
| **战略规划** | 制定元数据战略，说明如何创建、维护、集成和访问元数据，战略应推动需求定义并与业务优先级保持一致 |
| **企业视角** | 从企业视角确保未来的可扩展性，但通过迭代和增量方式交付实施以带来业务价值 |
| **宣导推广** | 宣传元数据的必要性及不同类型的元数据用途，鼓励更多业务部门使用元数据 |
| **访问便利** | 确保员工能够了解如何访问和使用元数据 |
| **保证元数据质量** | 让员工意识到元数据通常通过现有流程生成，流程所有者应对相关元数据质量负责 |
| **审计元数据标准** | 制定元数据标准、强制执行并审计标准使用情况，以便利数据集成和应用 |
| **持续改进** | 创建反馈机制，以便数据使用者可以将错误或过时的元数据反馈给管理团队 |

## 核心概念

### 元数据与数据的关系

元数据本身也是一种数据，同样应作为数据进行管理。许多组织都面临一个相似的问题，即如何在元数据和非元数据之间划分界限。从概念上讲，该界限与数据代表的抽象层级有关。从经验而言，**某一个人的元数据可能是另一个人的数据**。即使有些数据看起来像元数据（如多个列的名称列表），也可能只是普通数据——例如，该数据作为某个分析的输入，而该分析的目标是洞悉那些跨组织流转的数据内容。

组织无须纠结于哲理上的差别，而应聚焦于根据元数据的用途（创建新数据、理解既有数据、促成系统间流转、访问数据、共享数据）来确定元数据需求，并确认可以满足这些需求的源数据。

### 元数据的三种类型

元数据一般分为3种类型，这种分类有助于理解元数据框架包含的信息范围以及生成元数据对应的职能边界。

| 类型 | 定义 | 示例 |
|------|------|------|
| **业务元数据** | 关注数据的内容和状态，包含数据治理相关的详细信息 | 数据集定义与描述、业务规则与转换规则、数据模型、数据质量规则、数据血缘、数据标准、有效值约束、安全与隐私级别 |
| **技术元数据** | 提供数据的技术细节、存储系统及系统内/间流转流程 | 物理数据库表名和字段名、字段属性、访问权限、ETL作业详情、文件格式模式定义、源到目标映射、数据血缘文档、恢复和备份规则 |
| **操作元数据** | 描述数据处理和访问的细节 | 批处理作业执行日志、抽取历史记录、调度异常处理、错误日志、查询访问模式与频率、备份与保留策略、SLA要求和规定 |

> 分类的最佳依据是**元数据的生成方式**而非用途。从使用角度看，这些类型并非严格区分——技术人员和操作人员也会用到"业务"元数据。

除了信息技术，图书管理或信息科学等领域也采用其他分类体系：

- **描述元数据**（如标题、作者和主题）：描述资源、确定标识及检索
- **结构元数据**（如页码、章节编号）：描述资源内部或资源之间所含内容的相互关系
- **管理元数据**（如版本号、归档日期）：用于资源的生命周期管理

#### 业务元数据详情

业务元数据主要关注数据的内容和状态，包含与数据治理相关的详细信息。包括与技术无关的名称、概念定义、主题域、实体和各个属性；每个属性的数据类型和其他特征；业务范围描述、计算公式、算法及业务规则、值域的具体值及其定义等。完整示例包括：

1. 数据集、表和字段的定义和描述
2. 业务规则、转换规则、计算和推导公式
3. 数据模型
4. 数据质量规则和检核结果
5. 数据更新计划
6. 数据原生地和数据血缘
7. 数据标准
8. 为数据元素指定的记录系统
9. 有效值约束
10. 利益相关方的联系信息（如数据所有者、数据管理专员）
11. 数据的安全及隐私级别
12. 已知的数据问题
13. 数据使用说明

#### 技术元数据详情

技术元数据提供数据的技术细节、存储数据的系统，以及数据在系统内、系统间流转的流程。完整示例包括：

1. 物理数据库表名和字段名
2. 字段属性
3. 数据库对象属性
4. 访问权限
5. 数据的CRUD（创建、替换、更新、删除）规则
6. 物理数据模型，包括数据表名、键和索引
7. 数据模型与各类实物资产间的对应关系记录
8. ETL作业详情
9. 文件格式模式定义
10. 源到目标的映射文档
11. 数据血缘文档，包括上游和下游变更影响的信息
12. 程序和应用的名称和描述
13. 内容周期性更新的作业调度计划和任务依赖
14. 恢复和备份规则
15. 数据访问的权限、组和角色

#### 操作元数据详情

操作元数据描述了数据处理和访问的细节。完整示例包括：

1. 批处理程序的作业执行日志
2. 抽取过程和结果的历史记录
3. 调度异常处理
4. 审计、核对和控制测量的结果
5. 错误日志
6. 报告和查询的访问模式、频率与执行时间
7. 补丁和版本维护计划和执行情况，以及当前的补丁级别
8. 数据备份、保留策略，创建日期，灾难恢复预案
9. 服务水平协议（SLA）的要求和规定
10. 容量和使用模式
11. 数据归档、保留规则和相关归档文件
12. 清除标准
13. 数据共享规则和协议
14. 技术人员的角色、责任和联系方式

### ISO/IEC 11179 元数据注册标准

ISO的元数据注册标准提供了定义元数据注册的框架，旨在基于数据的精确定义，从数据元素开始，实现元数据驱动的数据交换。标准包含六个部分：

| 部分 | 内容 |
|------|------|
| **第1部分** | 数据元素的生成和标准化框架 |
| **第2部分** | 数据元素分类 |
| **第3部分** | 数据元素的基本属性 |
| **第4部分** | 数据定义的制定规则和指南 |
| **第5部分** | 数据元素的命名和识别原则 |
| **第6部分** | 数据元素的注册 |

### 元数据标准（Dublin Core / CWMM）

元数据标准在与业务贸易伙伴交换数据时至关重要。组织已意识到与客户、供应商、合作伙伴和监管机构共享信息的价值，为支持共享信息的最佳使用，需要共享公共元数据，这催生了许多专业领域的标准。在规划早期应采用基于行业或行业特有的元数据标准，并使用这些标准来评估元数据管理技术。

#### Dublin Core 元数据标准

**Dublin Core（都柏林核心）** 是国际上应用最广泛的描述性元数据标准之一，由OCLC（联机计算机图书馆中心）和NCSA（国家超级计算应用中心）于1995年联合发起。Dublin Core提供了一组精简的、跨领域的元数据元素，用于描述网络资源和其他数字资源。

Dublin Core包含15个核心元素：

| 元素 | 说明 | 类型 |
|------|------|------|
| **Title（标题）** | 资源的名称 | 内容 |
| **Creator（创建者）** | 对资源内容负主要责任的实体 | 知识产权 |
| **Subject（主题）** | 资源内容的主题关键词 | 内容 |
| **Description（描述）** | 资源内容的文本说明 | 内容 |
| **Publisher（出版者）** | 使资源可用的责任实体 | 知识产权 |
| **Contributor（贡献者）** | 对资源内容有贡献的实体 | 知识产权 |
| **Date（日期）** | 资源生命周期中相关事件的日期 | 日期 |
| **Type（类型）** | 资源内容的性质或类型 | 内容 |
| **Format（格式）** | 资源的物理或数字格式 | 内容 |
| **Identifier（标识符）** | 资源的唯一标识 | 内容 |
| **Source（来源）** | 派生当前资源的相关资源 | 内容 |
| **Language（语言）** | 资源内容的语言 | 内容 |
| **Relation（关联）** | 与其他资源的关联关系 | 内容 |
| **Coverage（覆盖范围）** | 资源内容的空间或时间范围 | 内容 |
| **Rights（权限）** | 资源上的权限声明 | 知识产权 |

Dublin Core的简单性和通用性使其成为跨领域数据交换和数字图书馆建设的基础标准，广泛应用于政府数据开放、数字档案管理和网络资源描述等场景。

#### CWMM（企业Web元数据模型）

**CWMM（Corporate Web Metadata Model，企业Web元数据模型）** 是面向企业级Web内容和数据资产管理的元数据框架。CWMM旨在为企业提供统一的Web资源元数据描述规范，支持企业内部和外部数据的互操作性。

CWMM的核心设计理念包括：

- **层次化结构**：支持从企业级到部门级再到项目级的多层次元数据管理
- **可扩展性**：允许企业根据自身业务需求扩展标准元数据元素
- **语义互操作**：通过统一的语义定义确保不同系统间的元数据理解一致
- **生命周期管理**：覆盖元数据从创建、使用到归档和退役的完整生命周期

CWMM通常在企业数据治理实践中与Dublin Core等通用标准结合使用，以实现从通用描述到企业特定需求的无缝衔接。

#### 其他相关标准

除了Dublin Core和CWMM，组织在实施元数据管理时还应关注以下标准：

| 标准 | 适用领域 | 说明 |
|------|----------|------|
| **ISO/IEC 11179** | 元数据注册 | 数据元素的标准化、分类和注册框架 |
| **Dublin Core** | 数字资源描述 | 15个核心元素的通用描述框架 |
| **CWMM** | 企业Web资源 | 企业级Web内容元数据管理模型 |
| **METS** | 数字图书馆 | 编码和传输数字图书馆元数据的标准 |
| **PREMIS** | 数字保存 | 数字对象保存元数据标准 |
| **DDI** | 社会科学数据 | 调查和统计数据的文档化标准 |

> 元数据的ISO标准为工具开发人员提供了指导。无论如何，了解这些标准及其影响仍然是有益的。

### 非结构化数据的元数据

所有数据都有结构，但并非都以关系数据库的形式存在。元数据对非结构化数据的管理同样重要，甚至更加重要。回顾图书馆的比喻——图书和杂志是典型的非结构化数据，目录卡片中的元数据就成为最主要的可用工具。非结构化数据的元数据包括：

- **描述元数据**：目录信息、分类词库关键词
- **结构元数据**：标签、字段结构和格式
- **管理元数据**：来源、更新计划、访问权限、导航信息
- **书目元数据**：图书馆目录条目
- **记录保留元数据**：保留策略
- **保存元数据**：存储、归档条件和保护规则

> 大多数关于非结构化数据的元数据论断都与传统的内容管理相关，但也出现许多针对数据湖中非结构化数据进行管理的新实践。那些希望利用Hadoop等大数据平台发挥数据湖优势的组织发现，必须对采集的数据进行目录编制以便促进后续访问。针对每个入湖对象，都应收集一组基本的元数据项（如名称、格式、来源、版本、接收日期等），借此形成数据湖的内容目录。

参考：[[DAMA-DMBOK2/09-文档和内容管理]]、[[DAMA-DMBOK2/精读笔记/14-大数据精读]]

### 元数据来源

元数据可以从多种渠道收集。如果应用程序和数据库中的元数据管理得当，就可以直接进行采集和集成。然而，大多数组织在应用程序层面并没有很好地管理元数据，因为元数据通常只是应用程序处理的副产品，而非最终产品。与其他形式的数据一样，在集成元数据之前需要做大量的准备工作。

大部分操作元数据是在数据处理过程中产生的。使用这些元数据的关键在于将其收集为可用的形式，并确保负责解释元数据的人拥有所需的工具。例如，解析错误日志等内容本身也需要描述这些日志的元数据。同样，大部分技术元数据可以从数据库对象中提取。通过对现有系统实施逆向工程，可以获取数据的相关知识，并从现有的数据字典、数据模型和流程文档中提取业务元数据。但这种方式存在一些风险，其中最大的风险在于无法确定这些定义最初是如何开发和完善的。

> 有意识地开发定义元数据比简单接受现有定义更为可取。由于开发定义需要时间和特定的技能（如写作和协调技能），业务元数据的开发需要通过设定专职责任去管理和维护。

项目工作是收集和开发所需技术元数据和业务元数据的良机——数据建模的过程需要讨论数据元素的含义及其相互关系，这些知识应被记录并整理，用于数据字典、业务术语表和其他存储库的构建。定义良好的业务元数据可在不同项目之间重复使用，并推动数据集中业务概念一致性的理解。

以下按字母顺序列出主要来源：

| 来源 | 说明 |
|------|------|
| **应用程序元数据库** | 内置于数据建模工具、BI工具等应用中的物理元数据表，组织日趋成熟后希望将这些不同来源的元数据进行集成管理 |
| **业务术语表** | 记录和存储组织的业务概念、术语体系、定义及关系，支持业务术语的全生命周期管理 |
| **商务智能工具** | 生成与BI设计相关的元数据，包括概述信息、类、对象、派生信息和计算项、过滤器、报表、报表字段、报表布局、报表用户、报表告发频率和报表发布渠道 |
| **配置管理工具(CMDB)** | 管理IT资产元数据及其关系、合同细节，每个资产被称为配置项（CI），可与变更管理流程集成 |
| **数据字典** | 定义数据集的结构和内容，适用于单个数据库或应用，管理每个数据元素的名称、描述、结构、特性、存储要求、默认值、关系、唯一性等属性 |
| **数据集成工具** | 记录数据在不同系统间迁移时的血缘关系，提供API允许外部元数据库提取血缘关系信息 |
| **数据库管理和系统目录** | 描述数据库内容、大小、版本、状态和运行信息，关系数据库将数据管理为表、列、索引、约束、视图和存储过程 |
| **数据映射管理工具** | 在项目分析和设计阶段将需求转化为映射规范，可直接被数据集成工具使用或由开发人员生成代码 |
| **数据质量工具** | 通过验证规则评估数据质量并提供质量分数，大多数提供与其他元数据库交换质量分数和质量概况的功能 |
| **字典和目录** | 包含术语、表和字段的详细信息及数据位置，对数据管理团队和数据分析师尤为有用 |
| **事件消息工具** | 在系统间移动数据时产生大量元数据，包括图形接口管理数据移动逻辑 |
| **建模工具和存储库** | 生成概念、逻辑和物理模型相关的设计元数据，包括主题域、逻辑实体、逻辑属性、表、字段、索引、主键和外键等 |
| **参考数据库** | 记录枚举数据的业务值和描述，管理同一领域内或跨领域之间各种编码值之间的关系 |
| **服务注册库** | 管理SOA服务和服务端点的技术信息，包括定义、接口、操作、输入和输出参数、策略、版本等 |
| **其他元数据存储** | 事件注册表、源列表、接口列表、代码集、领域词汇表、空间和时间模式、空间参考系统等 |

#### 业务术语表详细要求

业务术语表的作用是记录和存储组织的业务概念、术语体系、定义及这些术语之间的关系。对于大多数组织而言，业务术语表可能仅需一个电子表格就可以搞定。然而，随着组织的日渐成熟，企业通常会购买或自建业务术语表管理系统，系统里包含丰富的业务术语信息，系统也具备对业务术语的全生命周期管理的能力。

业务术语表管理系统需要满足3类核心用户的功能需求：

| 用户类型 | 说明 |
|----------|------|
| **业务用户** | 数据分析师、研究分析师、业务主管和高管，使用业务术语表来理解术语和数据 |
| **数据管理专员** | 使用业务术语表管理术语和定义其生命周期，并通过将数据资产与术语表相关联来提升知识管理水平。提出术语和使用方面的议题，帮助解决组织内的认知差异 |
| **技术用户** | 使用业务术语表进行架构、系统设计和开发决策，并进行影响分析 |

业务术语表应包含的业务术语属性：

1. 术语名称、定义、首字母缩写或简称，及其同义词
2. 负责管理与术语相关的数据的业务部门或应用程序
3. 维护术语的人员姓名和更新日期
4. 术语的分类或分类间的关联关系（业务功能关联）
5. 需要解决的存在冲突定义、问题性质、行动时间表
6. 对术语的常见误解
7. 支持定义的算法
8. 血缘
9. 支持该术语的官方或权威数据来源

> 每个业务术语表管理系统的实施都应该有一组支持其治理过程的基本的报告。建议组织不要"打印术语表"，因为业务术语表的内容不是静态的。数据管理专员通常负责术语表的开发、使用、运营和报告。报告包括跟踪尚未审核的新术语和定义，处于待定状态的术语和定义，以及缺少定义或其他属性的术语。业务术语表的搜索便捷性越高，越容易推广使用，但最重要的特点是它包含了丰富完整和高质量的信息。

#### 数据目录

**数据目录（Data Catalog）** 是元数据管理的核心组件之一，用于对企业内所有数据资产进行系统化的发现、编目和管理。数据目录不仅是字典或术语表的集合，更是面向数据发现和数据治理的综合平台。

数据目录的核心功能包括：

| 功能 | 说明 |
|------|------|
| **数据发现** | 自动扫描和发现企业内各系统中的数据资产，包括结构化、半结构化和非结构化数据 |
| **元数据编目** | 为发现的数据资产建立统一的元数据目录，记录技术元数据、业务元数据和操作元数据 |
| **数据血缘** | 记录和展示数据从源头到消费端的完整流转路径 |
| **数据分类** | 基于业务规则和标准对数据资产进行分类和标签化 |
| **搜索与浏览** | 提供类似搜索引擎的数据资产查找能力，支持按业务术语、技术名称、标签等多维度检索 |
| **数据治理集成** | 与数据治理流程集成，支持数据质量评估、数据所有权确认和访问权限管理 |
| **协作与注释** | 允许用户对数据资产添加业务注释、评分和使用说明 |

数据目录对数据管理团队和数据分析师来说十分有用，他们可以借此了解企业中的数据范围，不管是研究问题，还是寻找新应用程序的数据源信息。在数据湖和大数据环境中，数据目录的作用尤为关键——没有目录编制，数据湖就容易退化为"数据沼泽"。

#### 数据血缘追踪

**数据血缘（Data Lineage）** 描述了数据从其源头（官方系统或记录系统）到最终使用位置的完整流转路径，包括数据在各系统间经过的转换、加工和聚合过程。

发现和记录有关数据资产的元数据的一个重要意义是提供了有关数据如何在系统间进行流动的信息。许多元数据工具中存储了关于数据在某个环境中的现状信息，并提供跨系统或应用程序接口的查看血缘的功能。

数据血缘的两种类型：

| 血缘类型 | 说明 | 适用场景 |
|----------|------|----------|
| **实现态血缘（As Implemented Lineage）** | 基于编程代码当前版本的已实现血缘 | 用于理解数据在生产环境中的实际流转 |
| **设计态血缘（As Designed Lineage）** | 映射规格说明书中描述的设计阶段血缘 | 用于验证实际实现是否符合设计意图 |

数据血缘创建的局限性在于元数据管理系统的覆盖范围。特定功能的元数据库或数据可视化工具可以提供在其交互环境范围内的数据血缘信息，但是无法提供其交互环境范围之外的数据血缘。将数据血缘的各个部分连接起来的过程称为**"拼接"**。拼接的结果展示了数据从其原始位置移动到其最终位置的全景可视化视图。

**数据血缘的两种分析焦点**：

| 焦点 | 方法 | 适用对象 |
|------|------|----------|
| **业务焦点** | 从目标位置追溯到数据起源的具体源系统，将血缘发现限制在业务优先级较高的数据元素上 | 业务用户，帮助理解特定数据元素在系统中迁移时发生了什么 |
| **技术焦点** | 从源系统开始，识别所有直接的数据消费者，然后识别所有间接的数据消费者，重复此过程直到所有相关系统都被识别 | 技术用户，帮助回答"社会安全号码在哪里？"或"如果修改指定字段的长度，哪些系统会受到影响？" |

> 如果与数据质量监测结合，血缘关系可以用来定位系统设计如何对数据质量产生了负面影响。许多数据集成工具提供了数据血缘分析功能，该功能不仅包括开发好的代码，也包括数据模型和物理数据库。一些工具提供面向业务用户的Web界面，以监控和更新元数据定义。记录的血缘关系有助于业务和技术人员使用数据——如果没有它，用户将会浪费大量时间来检查异常现象、潜在的变更影响或未知结果。

#### 影响分析

**影响分析（Impact Analysis）** 是数据血缘的核心应用之一，用于评估数据结构、业务规则或系统变更可能产生的连锁影响。影响分析报告概述了潜在变更可能影响的组件，从而加快和简化了估算和维护任务。

影响分析的关键应用场景：

| 场景 | 说明 | 示例 |
|------|------|------|
| **数据结构变更** | 评估字段名称、类型或长度变更的影响范围 | "如果修改指定字段的长度，哪些系统会受到影响？" |
| **业务规则变更** | 评估计算公式或转换规则变更的影响 | 修改利润计算公式后，哪些报表需要更新？ |
| **数据源变更** | 评估源系统升级、迁移或退役的影响 | 源系统数据库版本升级后，ETL作业是否兼容？ |
| **数据质量改进** | 评估数据修复操作的影响范围 | 修正客户地址数据后，哪些下游报表会受影响？ |
| **系统退役** | 评估停用某个系统的影响 | 停用旧CRM系统后，哪些数据集成作业需要调整？ |

希望能够实现这样一个集成的影响分析和数据血缘工具，它能够理解加载过程中所涉及的所有移动部分，以及终端用户的报告和分析。

### 元数据架构的类型

元数据也有生命周期。所有元数据管理解决方案都包含对应生命周期的架构层次：创建和采集 → 存储 → 集成 → 交付 → 使用 → 控制和管理。

| 架构类型 | 描述 | 优点 | 缺点 |
|----------|------|------|------|
| **集中式** | 单一的元数据库存储来自各种来源的元数据副本 | 高可用性、快速检索、数据结构独立、可提升质量 | 需复杂同步流程、维护成本高、需自定义模块 |
| **分布式** | 无持久化存储库，实时从源系统检索 | 始终最新、分布式查询高效、实施工作量小 | 不支持手动条目、标准化挑战、依赖源系统可用性 |
| **混合式** | 结合集中式与分布式特性 | 近实时检索、可扩充元数据、降低手动干预 | 不提高可用性、需额外系统开销 |
| **双向** | 允许在架构任何部分更改元数据并同步回源 | 反馈闭环、元数据双向同步 | 强制同步、变更冲突处理复杂、维护接口多 |

#### 集中式架构详解

集中式存储库的内部存储库通过定期从各种工具中导入元数据来完成加载。然后，集中存储库提供一个门户供终端用户提交查询请求。由于在集中存储库中已经收集了各种工具的元数据，可以直接实现跨元数据的全局搜索。

- **适用场景**：追求在公共元数据库中实现组织元数据高度一致性管理的组织
- **优点**：
  1. 高可用性，因为它独立于源系统
  2. 快速的元数据检索，因为存储库和查询通常集成在一起
  3. 解析的数据库结构不受第三方或业务系统的专有属性所影响
  4. 提取元数据时可以进行转换、自定义或者使用其他源系统中的元数据进行补充增强，从而提高了元数据的质量
- **缺点**：
  1. 必须采取复杂的流程来确保元数据源头中的更改能够快速地同步到集中式存储库中
  2. 维护集中式存储库可能成本高昂
  3. 元数据的抽取可能需要自定义模块或中间件支持
  4. 验证和维护自定义代码可能会增加对内部IT人员和软件供应商的额外要求

#### 分布式架构详解

完全分布式的架构维护了一个单一接入点，元数据检索引擎通过实时从源系统中检索数据来响应用户请求。元数据管理环境维护了处理用户查询和搜索所需的必要源系统目录和查找信息，并通过通用对象请求代理或类似的中间件协议访问这些源系统。

- **限制**：不具备进行跨各种元数据源的全局搜索能力
- **优点**：
  1. 元数据始终尽可能保持最新且有效，因为它是从其数据源中直接被检索的
  2. 查询是分布式的，可能会提高响应和处理的效率
  3. 来自专有系统的元数据请求仅限于查询处理，而不需要详细了解专有数据结构，最小化了实施和维护所需的工作量
  4. 开发自动化元数据查询处理可能更简单，只需要很少的手动干预
  5. 减少了批处理，无须进行元数据复制或同步处理
- **缺点**：
  1. 无法支持用户定义或手动插入的元数据条目，因为没有存储库可用来放置这些新增条目
  2. 需要通过统一的、标准化的展示方式呈现来自不同系统的元数据，存在标准化挑战
  3. 如果参与的源系统不可用，查询能力将直接受到影响
  4. 元数据的质量完全取决于源系统

#### 混合式架构详解

混合架构中，存储库的设计仅考虑用户添加的元数据、关键的标准化元数据和来自手动处理的数据源的元数据。该架构得益于从源头近实时地检索元数据，并根据需要扩充元数据，以最有效地满足用户需求。混合方法降低了手动IT干预和自定义编码访问专有系统的功能的工作量。

- **适用场景**：具有快速变化的操作元数据的组织，需要一致、统一的元数据的组织，以及经历了元数据和元数据源大幅增长的组织
- **不适用**：拥有更多静态元数据和元数据增量较小的组织

#### 双向架构详解

双向元数据架构允许元数据在架构的源、数据集成、用户界面等任何部分进行更改，而后将变更从存储库（代理）同步到其原始源以实现反馈。该设计强制要求元数据库包含最新版本的元数据，并强制保证更改能够同步到元数据源。必须系统地捕获并解决变更；必须构建和维护一系列处理接口用于将存储库的内容写回至元数据源。
