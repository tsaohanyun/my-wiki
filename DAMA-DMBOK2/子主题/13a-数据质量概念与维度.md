## 概述

数据质量管理是数据管理体系中一项核心职能（Function），而非单一的方案（Program）或项目（Project）。它是一系列连续性工作，包含规划、实施和控制活动，旨在确保数据宜于消费并满足数据消费者的需求。数据质量管理贯穿数据全生命周期，通过制定标准、嵌入质量要求于数据创建/转换/存储过程，并依据标准进行度量，实现对数据质量的持续管理。

数据质量可定义为**数据质量的维度满足业务需求的程度**，更简洁的定义是**"适得其用"（Fit for Purpose）**。满足需求的数据即为有质量保障的数据，反之则为低质量数据。

高质量数据因使用情境而异。同样的数据可能在组织内的某些领域被视为高质量数据，而在其他领域则被视为低质量数据。极少有组织能拥有完美的业务流程、技术流程或数据管理实践。影响数据质量的因素很多：对数据质量影响组织成功因素的理解不足、领导不重视数据质量、规划不善、"烟囱式"的系统设计、不一致的开发流程、不完整的文档、缺乏标准或缺乏数据治理等。

> 正规的数据质量管理类似于产品制造领域的持续质量管理。通过制定标准、把质量要求嵌入数据创建、转换和存储数据过程，以及依据标准进行数据度量，实现数据全生命周期的质量管理。通常需要一个承担数据质量职能的团队（Data Quality Function Team），该团队负责聚集业务和技术方面的数据管理专家，推动质量管理技术的应用，确保数据满足各种目的的消费需求。

**数据质量管理语境关系图要素：**

| 要素 | 内容 |
|------|------|
| **定义** | 数据质量的规划、实施和控制活动，应用数据收集和处理技术，确保数据满足企业和本地消费者的需求，并适合使用 |
| **目标** | 1. 基于数据消费者的需求，制定受控的方法，使数据适合其目的；2. 在数据生命周期中定义数据质量控制的标准、要求和规范；3. 定义并实施测量、监控和报告数据质量水平的流程；4. 通过流程和系统改进，识别并倡导提升数据质量的机会 |
| **业务驱动输入** | 数据制度和标准、数据质量期望、业务要求、业务规则、数据要求、业务元数据、技术元数据、数据源和数据存储、数据沿袭 |
| **供给者** | 业务管理、主题领域专家、数据分析师、数据架构师、系统专家、数据管理员、业务流程分析师、数据供应商 |
| **方法** | 数据质量指标、数据剖析、预防和纠正措施、根本原因分析 |

## 业务驱动因素

通过规范的数据质量管理改善数据可信度，主要业务驱动因素包括：

1. **提升利益相关方体验和组织声誉**
2. **提高组织效能**
3. **降低与劣质数据有关的风险和成本**
4. **改进组织效率和生产力**

体系化方法达成数据质量管理成效的路径：
- 理解驱动数据需求的场景（适用性）
- 进行数据质量需求与场景的匹配（目标）
- 借助固化的流程确保持续达到质量要求

组织拥有高质量数据的切实收益：

| 收益类别 | 具体表现 |
|----------|----------|
| **客户信任** | 客户相信数据正确，有信心与组织打交道 |
| **员工效率** | 员工可更快速、更一致地解答问题，减少数据确认时间 |
| **收入增长** | 通过识别业务机会和成交生意获取收益 |
| **运营成本降低** | 减少客服话务量，及时解决呼叫需求 |
| **决策质量提升** | 员工更多时间用于借助数据进行洞察和正确决策 |
| **合规与信用** | 组织符合监管要求并提升信用评级 |

劣质数据充满风险，可能会损害组织声誉，导致罚款、损失收入、客户流失及负面的媒体曝光。

## 目标与原则

数据质量管理专注于以下目标：
1. 根据数据消费者的要求，制定数据匹配用途的方法
2. 作为数据生命周期的一部分，定义数据质量控制的标准和规范
3. 定义并实施用于度量、监测和报告数据质量水平的流程
4. 识别并倡导通过改进流程和系统来提高数据质量的机会

数据质量改进方案应遵循以下**核心原则**：

| 原则 | 说明 |
|------|------|
| **聚焦关键数据（Criticality）** | 聚焦于对企业和客户最为关键的数据，基于关键性及风险水平确定优先级 |
| **标准驱动（Standards-Driven）** | 为利益相关方的数据质量诉求定义可度量的标准和期望 |
| **客观度量和透明（Objective Measurement and Transparency）** | 持续进行客观度量，与利益相关方共享度量活动和方法 |
| **预防（Prevention）** | 重点放在防止数据出错和导致质量下降的条件上，而不应只关注纠正记录 |
| **根因修正（Root Cause Remediation）** | 理解数据质量问题的根因并加以解决，而非停留在问题表象；通常需要改造流程和支持流程的系统 |
| **嵌入业务流程（Embedded in Business Processes）** | 业务流程所有者对流程产生的数据质量负责，且必须在其流程中强力贯彻数据质量标准 |
| **体系化贯彻（Systematically Enforced）** | 系统所有者必须体系化落实数据质量要求 |
| **关联服务水平（Connected to Service Levels）** | 数据质量报告和问题管理应纳入服务水平协议（SLA） |

## 关键数据（Critical Data）

数据质量管理的首要原则是将改进工作聚焦于对组织及客户最重要的数据。**关键数据元素（Critical Data Element, CDE）**的识别通过与业务驱动因素（客户体验、效能和效率）的联系实现。缺少特定的数据元素将会严重影响组织的成功，这种影响程度为特定数据元素的数据质量管理级别提供了业务依据。

关键数据通常被用于：
- 监管、财务或管理报表
- 业务运营需求
- 度量产品质量和客户满意度
- 商业战略与差异化竞争

主数据和参考数据通常被定义为关键数据。数据是否为关键数据，依据使用数据的流程、数据出现在报告中的性质，或当数据出现问题时对组织的财务、监管或声誉产生的风险进行判定。

> 示例："如果客户的电子邮件地址字段数据不完整，我们将无法通过电子邮件向客户发送产品信息，从而失去潜在销售机会。众所周知，我们每发送一封电子邮件会带来平均1美元的收入。"这个简单示例说明了价值驱动因素与改进数据质量之间的关联关系。

## 数据质量维度详解

数据质量维度是可度量的数据特征或特性，使用"维度"一词是为了与物理对象的度量维度（如长、宽、高）建立联系。数据质量维度为定义数据质量需求提供了一套标准化词汇。基于维度可以定义初始数据质量评估的结果并进行持续度量。维度为可度量的规则提供了基础，而这些规则应直接与关键流程中的潜在风险相关联。

数据质量维度是推动"标准驱动""客观度量和透明"原则的关键概念。

### 六大数据质量核心维度（DAMA UK 2013）

DAMA UK 2013年发布的白皮书定义了6个核心维度，这是业界最为广泛采用的维度体系：

**（1）准确性（Accuracy）**

数据正确描述"真实世界"对象或事件的程度。准确性是最基本也是最重要的数据质量维度。

| 要素 | 说明 |
|------|------|
| **定义** | 数据值与真实世界实体或事件的实际状态相匹配的程度 |
| **度量方法** | 将某个数据值与记录系统或其他已验证数据源（如从供应商购买的营销数据）中的对应数据值相比较，验证这些值是否准确 |
| **业务规则示例** | 客户地址必须与实际居住地址一致；交易金额必须与实际发生金额匹配 |
| **常见问题原因** | 数据录入错误、系统映射错误、过时数据未更新 |

**（2）完整性（Completeness）**

已存数据与潜在数据量的百分比。完整性关注数据是否缺失。

| 要素 | 说明 |
|------|------|
| **定义** | 数据记录中必填字段和可选字段被填充的程度 |
| **度量方法** | 值存在检查（Value Presence）、记录完整性检查（Record Completeness） |
| **业务规则示例** | 如果国家被划分为州/省，则所有地址必须包含州/省；客户必须有有效的邮寄地址 |
| **常见问题原因** | 用户跳过非必填字段、数据采集流程不完善、系统默认值未设置 |

**（3）一致性（Consistency）**

将一个事物的两个或多个表示与定义进行比较时，不存在差异。

| 要素 | 说明 |
|------|------|
| **定义** | 数据在不同数据集、系统或表示之间保持逻辑一致的程度 |
| **度量方法** | 跨字段关系断言、定义一致性检查、算法一致性验证 |
| **业务规则示例** | 邮政编码必须与州/省匹配；不同系统中同一客户的姓名必须一致 |
| **常见问题原因** | 系统间数据同步延迟、业务规则执行不一致、字段重载 |

**（4）时效性（Timeliness）**

从要求的时点看，数据及时表达现实的程度。

| 要素 | 说明 |
|------|------|
| **定义** | 数据在需要时可用且反映当前状态的程度 |
| **度量方法** | 时间阶段监测、全周期附属数据检查、数据更新频率评估 |
| **业务规则示例** | 参考值将定期更新，当前地址需要与最新名称保持同步；报告数据必须在T+1日内可获取 |
| **常见问题原因** | 数据处理延迟、系统间传输时间过长、更新机制不完善 |

**（5）唯一性（Uniqueness）**

根据识别方式，实体实例（物体）不会被多次记录。

| 要素 | 说明 |
|------|------|
| **定义** | 在数据集中，每个现实世界实体有且仅有一条对应记录 |
| **度量方法** | 重复值可能性评估、唯一标识检查、模糊匹配分析 |
| **业务规则示例** | 每个客户必须有唯一的客户ID；可能搜索出地址相同、行为方式雷同的人以评估是否为同一人 |
| **常见问题原因** | 缺乏参照完整性、数据录入时未检查已有记录、系统合并时未去重 |

**（6）有效性（Validity）**

如果数据符合已定义的对应语法（格式、类型、范围），则认为数据有效。

| 要素 | 说明 |
|------|------|
| **定义** | 数据值符合预定义的格式、类型、范围和业务规则的程度 |
| **度量方法** | 格式合规检查、值域成员检查、范围一致检查、映射一致检查 |
| **业务规则示例** | 所有州/省名称必须是参考表中的值；电话号码必须符合标准格式；数值范围必须大于0且小于100 |
| **常见问题原因** | 数据录入界面缺少编辑检查、下拉列表配置不当、字段重载导致数据混乱 |

### 有效性规则的四种类型

| 规则类型 | 说明 | 示例 |
|----------|------|------|
| **格式合规（Format Compliance）** | 一种或多种给数据元素赋值的模板 | 电话号码的格式标准 |
| **值域成员（Value Domain Membership）** | 数据元素的值应在值域的枚举范围内 | 美国邮政编码的州字段约定为双字符编码 |
| **范围一致（Range Conformance）** | 数据元素的值必须在定义的数值、词典或时间范围内 | 数值范围必须大于0且小于100 |
| **映射一致（Mapping Conformance）** | 数据元素的值必须与不同值域做等价映射 | "AL""01""Alabama"为同一州的不同表示 |

### 常用数据质量维度完整列表

| 维度 | 定义 | 度量方法 |
|------|------|----------|
| **有效性（Validity）** | 数据符合已定义的语法（格式、类型、范围） | 格式合规检查、值域成员检查、范围一致检查 |
| **完备性（Completeness）** | 已存数据与潜在数据量的百分比 | 值存在检查、记录完整性检查 |
| **一致性（Consistency）** | 数据表示间不存在差异 | 跨字段关系断言、定义一致性检查 |
| **完整性（Integrity）** | 数据的准确、一致和可靠性 | 唯一性检查、关系有效性检查、列规则符合检查 |
| **及时性（Timeliness）** | 数据及时表达现实的程度 | 时间阶段监测、全周期附属数据检查 |
| **唯一性（Uniqueness）** | 实体实例不会被多次记录 | 重复值可能性评估、唯一标识检查 |
| **准确性（Accuracy）** | 数据正确描述真实世界对象或事件的程度 | 与记录系统或其他已验证数据源比较 |
| **合理性（Reasonableness）** | 数据值在合理预期范围内 | 相对度量（随时间统计）、绝对度量（固定阈值） |

### 主要数据质量维度框架

**（1）Strong-Wang框架（1996年）**

聚焦于数据消费者对数据的感知，划分为4大类共15个维度：

| 大类 | 维度 |
|------|------|
| **内在数据质量** | 准确性、客观性、可信度、声誉度 |
| **依赖背景的数据质量** | 增值性、相关性、及时性、完备性、适量性 |
| **表示性数据质量** | 可解释性、易理解性、表示一致性、简洁性 |
| **可访问性数据质量** | 可访问性、访问安全性 |

**（2）Redman框架**

Thomas Redman提出基于数据结构的数据质量维度，将数据对象定义为"可表示的三元组"（来自实体的某个数据元素域的某个值），涵盖3大类20多个维度：

- **数据模型**：内容（相关性、获取价值的能力、定义清晰度）、详细程度（颗粒度、精度）、构成、自然性、可识别性（同质性、最小必要冗余性）、一致性（语义一致性、结构一致性）、应变性（健壮性、灵活性）
- **数据值**：准确性、完备性、时效性、一致性
- **数据表示**：适当性、可解释性、可移植性、格式精确性、格式灵活性、表示空值的能力、存储的有效利用、物理实例与格式的一致性

**（3）English框架（Larry English）**

分为固有质量特性和实用质量特性。固有特性独立于数据使用，实用特性与数据呈现相关且是动态的——其价值（质量）取决于数据的使用方式：

| 类别 | 维度 |
|------|------|
| **固有质量特性** | 定义一致性、值的完备性、有效性/业务规则一致性、数据源的准确性、现实映射的准确性、精确性、非冗余性、冗余或分布式数据的等效性、冗余或分布式数据的并发性 |
| **实用质量特性** | 可访问性、及时性、背景清晰性、可用性、衍生完整性、正确性或事实完备性 |

**（4）DAMA UK核心维度（2013年）**

DAMA UK发布的白皮书描述了6个核心维度：完备性、唯一性、及时性、有效性、准确性、一致性。

此外还描述了对质量产生影响的其他特性：

| 特性 | 说明 |
|------|------|
| **可用性** | 数据是否易于理解、简明、相关、可访问、易维护且精度适中？ |
| **时间问题** | 数据是否稳定，且能响应合理的变更请求？ |
| **灵活性** | 数据是否具有可比性且与其他数据兼容？是否具有实用的分组和分类？是否可以复用？是否易操作？ |
| **置信度** | 是否已建立数据治理、数据保护和数据安全的管理流程？数据的声誉如何？是否已被验证或可验证？ |
| **价值** | 数据是否具有良好的成本收益比？是否得到了最佳应用？是否危及人们的安全或隐私或企业的法律责任？ |

## 数据质量业务规则

业务规则描述了组织应如何在内部运作，以便在与外部世界的互动中取得成功并实现合规。数据质量规则应与质量维度对齐，用于描述数据质量的要求。

### 常见业务规则类型

| 规则类型 | 说明 | 示例 |
|----------|------|------|
| **有效性规则** | 格式合规、值域成员、范围一致、映射一致 | 州/省名称必须在参考表中 |
| **完备性规则** | 定义可接受或不可接受数据缺失的条件 | 如果国家划分为州/省，则所有地址必须包含州/省 |
| **一致性规则** | 两个或多个数据元素之间的条件性断言 | 邮政编码是否与特定的州或省匹配 |
| **定义一致性规则** | 确认在整个组织的流程中对数据定义有相同理解并正确使用 | 计算字段的算法一致性、汇总和状态相互依赖规则 |
| **完整性规则** | 确保数据准确、一致和可靠 | 每行唯一、表间关系有效、列数据符合预定义规则 |
| **及时性规则** | 数据可访问性和可用性期望 | 将流程分解为多个及时性监测阶段，确保附属类数据也满足及时性要求 |
| **唯一性规则** | 实体必须有唯一标识 | 同一现实对象有且仅有一条对应记录；评估重复数值可能性 |
| **合理性规则** | 聚合函数检验 | 相对度量（随时间统计评估记录计数）；绝对度量（通过固定阈值验证平均金额） |
| **准确性规则** | 与记录系统或其他已验证数据源比较 | 将数据值与权威来源对比验证 |

### 业务规则实施注意事项

1. 并非每个维度每次都适用
2. 通常业务规则会被压缩以涵盖多个数据质量维度和多个数据元素
3. 数据质量规则可以通过多种技术实现，包括数据输入选项列表、参考数据查找和质量检查报告
4. 数据质量规则通常通过软件实现，或使用数据输入的文档模板实现

> 州/省名称管理的业务规则示例：**有效性**——所有州/省名称必须是参考表中的值；**完备性**——如果国家被划分为州/省，则所有地址必须包含州/省；**完整性**——所有州/省名称必须与地址的国家名称关联；**时效性**——参考值将定期更新，当前地址需要与最新名称保持同步。
