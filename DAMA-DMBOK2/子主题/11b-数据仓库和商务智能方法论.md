> **所属知识体系**：[[DAMA-DMBOK2/11-数据仓库和商务智能|DAMA-DMBOK2 第11章：数据仓库和商务智能]]

## ETL与数据集成

数据集成包括ETL、数据虚拟化及将数据转化为统一格式的各类技术。主要涉及两种数据加载类型：

**1. 历史数据加载：** 通常只加载一次，或在处理数据问题期间有限加载。

**2. 持续增量更新：** 按照固定计划持续执行，确保数据保持最新状态。

**变更数据采集（CDC）技术比较：**

| 技术 | 相对复杂度 | 加载效率 | 重复列 | 删除列 | 说明 |
|------|-----------|---------|--------|--------|------|
| **数据库日志** | 低 | 高 | 否 | 否 | 通过数据库日志技术捕获变更 |
| **时间戳/日志表** | 中 | 中 | 是 | 否 | 最常见的技术方法 |
| **触发器** | 高 | 低 | 是 | 是 | 通过数据库触发器捕获变更 |
| **全量加载** | 低 | 最低 | 是 | 否 | 适用于没有原生时间戳功能的遗留系统 |

**准实时和实时数据采集方案：**

| 方案 | 数据积累位置 | 说明 |
|------|------------|------|
| **微批处理（源端累积）** | 源端 | 更频繁地执行批量加载 |
| **消息队列（总线累积）** | 总线 | 极小的数据包发布到消息总线 |
| **流式处理（目标累积）** | 目标端 | 目标端系统接收数据后收集到缓冲区或队列 |

**源到目标映射（Source-to-Target Mapping）：** 在各个实体和数据元素间建立从源系统到目标系统的转换规则，记录数据血缘关系。

**数据修正策略：**
- 将不正确的数据状态设置为"过期"并将正确的新数据加载进来
- 源系统应负责数据修复和更正

**加载策略：**

| 策略 | 说明 | 适用场景 |
|------|------|---------|
| **乐观加载策略** | 在维度表中增加一条记录以加载无法关联的事实数据 | 数据质量较好 |
| **悲观加载策略** | 创建数据回收区存储无法关联的事实数据 | 数据质量问题较多 |

## Kimball vs Inmon 方法论对比

| 对比维度 | Inmon（企业信息工厂 CIF） | Kimball（多维数据仓库） |
|---------|-------------------------|----------------------|
| **数据仓库定义** | 面向主题、集成、随时间变化、相对稳定的数据集合 | 专为查询和分析定制的业务数据副本 |
| **建模方式** | 规范化关系模型（3NF） | 多维模型（星型模型） |
| **架构风格** | 自上而下（Top-Down） | 自下而上（Bottom-Up） |
| **数据存储** | 所有数据存储在单个数据仓库层 | 数据仓库由部门数据集市组合而成 |
| **集成机制** | 企业级数据模型驱动集成 | 数据仓库总线（一致性维度和一致性事实） |
| **扩展性** | 相对集中 | 更具扩展性，增量构建灵活 |
| **关键成功因素** | 拥有经过验证的企业级数据模型 | 共享/遵循一致性维度 |

**核心理念共通之处：**
1. 数据仓库存储的数据来自其他系统
2. 数据存储以增加数据价值为目标整合数据
3. 数据仓库使数据变得易于访问和分析
4. 构建目的是提供可靠、集成的数据给授权利益相关方
5. 用途广泛，从支撑工作流到运营管理，再到预测分析

### 企业信息工厂（CIF）

比尔·因蒙提出的CIF架构包含以下组件：

| 组件 | 说明 |
|------|------|
| **应用程序** | 处理业务流程，产生的明细数据流入数据仓库和ODS |
| **数据暂存区** | 源数据库与目标数据仓库之间的中间数据库 |
| **集成与转换层** | 不同来源数据按标准企业模型转换并集成 |
| **操作型数据存储（ODS）** | 业务数据的集成数据库，包含当前或近期数据 |
| **数据集市** | 专为数据分析提供数据，是数据仓库的子集 |
| **操作型数据集市（OpDM）** | 直接从ODS获取数据，支持运营决策 |
| **数据仓库** | 统一集成入口，支撑管理决策、战略分析和规划 |
| **运营报表** | 从数据存储中输出 |
| **参考数据、主数据和外部数据** | 理解业务所需的数据 |

**CIF数据仓库的特性：**
- **面向主题** — 按主要业务实体组织
- **集成** — 数据统一、内聚，键码结构、编码和解析规则保持一致
- **随时间变化** — 存储数据在某一时间点的状态（快照）
- **相对稳定** — 新数据追加而非频繁更新
- **包含明细和汇总数据** — 包含原子级业务明细及汇总数据
- **反映历史变化** — 包含大量历史数据

### 多维数据仓库（Kimball）

拉尔夫·金博尔提出的多维数据仓库包含以下组件：

| 组件 | 说明 |
|------|------|
| **业务源系统** | 企业的操作型/事务型应用程序 |
| **数据暂存区** | 包括一系列用于集成和转换数据的处理流程 |
| **数据展示区** | 采用"数据仓库总线"集成范式 |
| **数据访问工具** | 侧重于终端用户的数据需求 |

### 数据仓库总线矩阵

数据仓库总线矩阵是Kimball方法论中的关键规划工具：

| 维度/流程 | 日期 | 产品 | 店铺 | 供应商 | 仓库 |
|-----------|------|------|------|--------|------|
| 销售额 | ✓ | ✓ | ✓ | | |
| 库存 | ✓ | ✓ | ✓ | ✓ | ✓ |
| 订单 | ✓ | ✓ | | ✓ | |

### 数据保管库（Data Vault）

数据保管库（DV）是另一种历史数据存储方法，在数据暂存过程中进行数据清洗和标准化。特点：
- 历史数据以标准化的原子结构存储
- 确保业务键和代理键关系保持不变
- 事实表以原子结构形式持久化存储
- 通过数据集市为各种数据消费者提供数据
- 支持对表示层进行虚拟化，促进敏捷增量交付

## 数据仓库架构选择

| 架构因素 | 考量说明 |
|---------|---------|
| 时间延迟要求 | 实时/准实时/批量延迟需求决定技术选型 |
| 数据源可用性 | 源系统的可用窗口和访问能力 |
| 批处理窗口 | 可用的ETL执行时间窗口 |
| 维度一致性 | 一致性维度和事实的定义和管理 |
| 数据质量管控 | 数据修正、清洗、拒绝处理机制 |
| 变更数据采集 | CDC技术选择 |
| 迟到维度处理 | 延迟到达的维度数据处理策略 |
| 可扩展性 | 架构是否支持增量扩展和业务增长 |

## 活动

### 理解需求

收集数据仓库/BI项目需求时应：
1. 明确业务目标和业务战略
2. 确定业务领域，界定业务范围
3. 采访相关业务人员
4. 厘清关键绩效指标（KPI）和统计口径
5. 将需求分类并排出优先级
6. 优先处理简单且有价值的需求，赢得速胜

### 定义和维护架构

**架构最佳实践：**
- 建立数据原子化机制
- 概念模型架构是数据仓库/BI工作的起点
- 通过原型设计快速验证或驳回关键需求
- 通过授权需求变更管理团队

**发布管理：**
- 整合、协调运维流程
- 制定标准的发布计划
- 建立有效的发布流程

### 开发数据仓库和数据集市

数据仓库/BI项目通常有三条并行的开发主线：

| 开发主线 | 内容 | 责任领域 |
|---------|------|---------|
| **数据** | 识别数据来源，设计数据修复、转换、集成、存储和提供规则 | 数据管理团队 |
| **技术** | 支撑数据存储和传输的后端系统和流程 | 企业技术架构 |
| **BI工具** | 数据消费者获取数据洞察所需的应用套件 | BI/分析团队 |

**关键步骤：**
1. 源到目标映射
2. 数据修正和清洗
3. 数据转换
4. 数据加载

### 实施商务智能产品组合

**用户分组策略：**

| 用户类型 | 需求特征 | 工具匹配 |
|---------|---------|---------|
| IT开发人员 | 聚焦数据提取，注重高级功能 | 高级分析工具 |
| 信息消费者 | 快速访问预定义报表 | 固定报表 |
| 高级用户/分析师 | 即席查询，数据探索 | 自助式分析工具 |
| 高管/经理 | 综合使用固定报表、仪表盘和计分卡 | 仪表盘，战略BI工具 |
| 外部客户 | 良好用户体验 | 嵌入式分析，门户 |

### 维护数据产品

采用增量开发方式对数据仓库进行扩展、增补或修改。

**发布管理流程示例：**
- 每年3次业务驱动发布 + 1次技术发布
- 使用MoSCoW列表管理工作范围
- 采用时间盒管理

**监控和调优：**
- 监控整个系统中的数据加载过程
- 采用数据库调优技术
- 定义并展示面向客户的满意度指标
- 定期审查使用情况统计数据

## 方法

### 原型驱动需求

在开发产品之前，通过创建演示数据并在协同原型设计工作中采用需求挖掘方法，可以在实施活动开始之前快速确定需求及优先级。

**数据剖析的价值：**
- 有助于原型设计，降低与非预期数据的相关风险
- 数据仓库通常是最先暴露源系统数据质量缺陷的地方
- 揭示数据源之间的差异

### 自助式商务智能

自助服务是商务智能产品组合中的一种基本交付方式：
- 消息通知和警告
- 查看预定的生产报表
- 与分析报表互动
- 开发即席报表
- 仪表盘和计分卡

### 数据审计

为了维护数据血缘，所有结构和流程都应具备创建和存储审计数据的能力。
