## 概述

数据安全是[[DAMA-DMBOK2-01-数据管理|数据管理]]的核心知识领域之一，包括安全制度和规程的**规划、建设与执行**，为数据和信息资产提供正确的身份验证、授权、访问和审计。

虽然数据安全的具体内容（如哪些数据需要被保护）会因行业和国家/地区而异，但数据安全实践的核心目标始终一致：**保护数据资产，并确保其符合隐私和保密法律法规要求、合同协议约定及业务需求。**

这些需求来自以下几个方面：

| 需求来源 | 说明 |
|---------|------|
| **利益相关方** | 组织必须认识到客户、病人、学生、公民、供应商和商业伙伴等在隐私保护和数据保密方面的需求，组织内每个人都必须成为对利益相关方数据负责任的受托人 |
| **政府法规** | 出发点是保护利益相关方的利益，不同法规目标各异：有些限制信息的随意获取，有些确保数据公开、透明和有人担责 |
| **特有业务关切点** | 每个组织都有保护其特有数据的需要，有效利用可提高客户洞察能力、提升竞争优势 |
| **合法的访问需求** | 在保护数据安全的同时，必须确保数据能够被合法访问 |
| **合同义务** | 合同和保密协议规定了数据安全制度，不能阻止用户完成工作 |

### 业务驱动因素

数据安全活动由两大主要驱动因素推动：

| 驱动因素 | 说明 |
|---------|------|
| **降低风险** | 合规性、受托责任、声誉保护、隐私保护。组织可能因违法或不遵守合同义务而被处罚，数据泄露会导致组织美誉度降低和客户信心下降 |
| **促进业务增长** | 运营成功、客户信任、竞争优势、电子商务。可信的电子商务推动了业务和利润的增长，强大的信息安全有助于建立客户信任并促进交易的达成 |

> 如果将降低风险和发展业务的目标整合到一个连贯的信息管理和保护战略中，这些目标就可以互为补充和相互支持。

#### 降低风险的要求

随着数据相关法规不断增多（通常是为了应对数据被窃取和泄露），合规性要求也随之增加。安全部门通常不仅负责管理IT合规性要求，还负责管理整个组织的制度、实践、数据分类分级和访问授权规则。

与数据管理的其他职责类似，数据安全最好在企业级层面开展。如果缺乏步调一致的协调，让各业务单元各自寻找其安全解决方案，不但会导致总体成本增加，还可能会由于不一致的保护措施反而让安全性进一步降低。

信息安全管理首先要对组织的数据进行分类分级，过程包括：

1. **识别和分类分级敏感数据资产** — 根据行业和组织不同，敏感数据范围有所差别（包括个人标识信息、医疗、财产等）
2. **探明整个企业敏感数据** — 根据数据存放位置不同，安全性要求也有所不同。大量敏感数据存储在一个地方，如果该地方遭到破坏，将会带来很高风险
3. **确定每项数据资产需要如何保护** — 根据数据内容和技术类型不同，确保采取针对性的安全措施
4. **识别信息与业务流程如何交互** — 确定在什么条件下，什么样的数据访问是被允许的

除了对数据本身进行分类分级，还需对外部威胁（如来自黑客和犯罪分子的威胁）和内部风险（如由员工和流程产生）进行评估。大量数据通过员工的无知或绕过安全政策而被丢失或暴露。

> 一项调查显示："70%的IT专业人士认为，使用未经授权的程序导致了公司一半的数据丢失事件。" Ponemon集团和Symantec的一份报告显示："2012年三分之二的数据泄露事件是由人为错误和系统问题造成的。"

#### 业务增长

信息技术已广泛应用于办公室、市场和家庭。台式计算机、笔记本电脑、智能手机、平板电脑及其他设备已成为大多数企业和政府运作的重要组成部分。电子商务的爆炸式增长改变了组织提供商品和服务的方式。

产品和服务质量与信息安全有直接关系：**强大的信息安全有助于建立客户信任并促进交易的达成。**

#### 将安全视为资产管理

敏感数据的一种管理方法是通过**元数据**。数据安全的分类分级和监管的敏感性可以在数据元素和数据集层面进行捕获。现有的技术可以为数据打标，并使得元数据跟随数据一起在企业内流转。

如果强制实施统一的标准，则这种方法可以使多个部门、业务单元和供应商都能够使用相同的元数据。标准的安全元数据可以优化数据保护，引导业务的应用和技术支持的流程，从而降低成本。当敏感数据被正确识别时，组织能够与客户和合作伙伴建立信任。与安全相关的元数据本身也成为一种战略资产，可以提高交易、报告和业务分析的质量，同时降低保护成本和信息丢失或被盗带来的相关风险。

### 目标

数据安全活动的目标包括：

1. **赋能适当访问** — 允许对企业数据资产进行适当访问，并防止不当访问
2. **确保法规合规** — 确保符合隐私、保护和保密性的法规和政策
3. **保护利益相关方** — 确保满足利益相关方对数据隐私、数据机密性的要求

### 原则

| 原则 | 英文 | 说明 |
|------|------|------|
| 协同合作 | Collaboration | IT安全管理员、数据管理专员、数据治理团队、内外部审计和法律部门协同完成 |
| 企业统筹 | Enterprise Approach | 数据安全标准和制度必须在整个组织内一致地应用 |
| 主动管理 | Proactive Management | 成功源于管理的主动性和动态性，引入所有利益相关方的努力，管理变革，克服组织或文化瓶颈 |
| 明确的问责制 | Clear Accountability | 角色和职责必须明确界定，包括跨组织和跨角色的数据"监管链" |
| 元数据驱动 | Metadata-Driven | 数据元素的安全分类分级是数据定义的基础组成部分 |
| 减少暴露面 | Reduce Risk by Reducing Exposure | 最小化敏感/机密数据的扩散，尤其是在非生产环境中 |

## 核心概念

### 数据安全定义

> **数据安全**（Data Security）：包括安全制度和程序的规划、建设与执行，以在文化和监管考虑范围内为数据和信息资产提供正确的**身份验证、授权、访问和审计**。

### 漏洞与威胁模型

#### 漏洞（Vulnerability）

漏洞是系统中的弱点或缺陷，使其能够被成功攻击和破坏——本质上是组织防御中的一个"漏洞"，但有时候这些漏洞容易被黑客"漏洞利用"。

常见的漏洞示例：

| 漏洞类型 | 说明 |
|---------|------|
| 过期安全补丁 | 存在未更新安全补丁的网络计算机 |
| 弱密码保护 | 不受强密码保护的网页或系统 |
| 培训不足 | 用户未接受过忽略未知发件人邮件附件的培训 |
| 缺乏技术保护 | 不受技术保护的公司软件，使攻击者能够控制系统 |
| 默认密码 | 安装软件时使用缺省密码，黑客迟早会发现 |
| 后门 | 开发人员出于维护目的创建的隐藏入口 |

> 在多数情况下，非生产环境比生产环境更脆弱。因此，将生产数据控制在生产环境之内非常重要。

#### 威胁（Threat）

威胁是针对组织采取攻击的**潜在行动**。威胁可以是内部的或外部的，不一定总是恶意的（如穿着制服的内部人员可能在毫不知情的情况下采取攻击行动）。威胁可能与特定漏洞相关，因此可以优先考虑修复这些漏洞。

对每种威胁，都应匹配一种相应的抵御能力，以防止或限制威胁可能造成的损害。威胁的发生也被称为**攻击面**。

威胁的例子包括：
- 带有病毒感染附件的电子邮件
- 使服务器不堪重负无法执行业务的进程（拒绝服务攻击）
- 对已知漏洞的利用

#### 威胁评估要素

| 要素 | 说明 |
|------|------|
| **外部威胁** | 来自黑客、犯罪分子、竞争对手的攻击，近年变得越来越复杂和有针对性 |
| **内部风险** | 由员工和流程产生（故意或无意），大量数据通过员工无知或绕过安全政策而被丢失或暴露 |
| **攻击面分析** | 评估所有可能被攻击者利用的入口点和漏洞 |

### 风险与风险分类

#### 风险（Risk）

风险指造成损失的可能性，也指可能造成损失的事物或条件。可通过以下因素计算每个威胁的风险：

| 风险计算因素 | 说明 |
|-------------|------|
| 威胁概率与频率 | 威胁发生的概率及其发生的频率 |
| 损害类型与程度 | 每次威胁事件发生时可能造成的损害类型和程度（包括声誉损害） |
| 业务影响 | 损害将对收入或业务运营产生的影响 |
| 修复成本 | 发生损害后的修复成本 |
| 预防成本 | 预防威胁的成本，包括漏洞修复成本 |
| 攻击者意图 | 潜在攻击者的目标或意图 |

> 可按潜在损害程度或发生的概率确定风险优先级。容易被利用的漏洞具有更高概率发生风险。风险的优先级必须由各利益相关方通过正式流程来确定。

#### 风险分类

风险分类描述了数据的敏感性及出于恶意目的对数据进行访问的可能性。分类用于确定谁（什么角色的人员）可以访问数据。用户权限内所有数据中的最高安全分类决定了整体的安全分类。

| 分类级别 | 英文 | 说明 |
|---------|------|------|
| **关键风险数据** | Critical Risk Data (CRD) | 由于个人信息具有很高的直接财务价值，内外部各方可能费尽心思地寻求未经授权使用。滥用会导致严重处罚、增加挽留客户/员工成本、损害品牌与声誉 |
| **高风险数据** | High Risk Data (HRD) | 为公司提供竞争优势，具有潜在直接财务价值。滥用可能导致商业机密泄露、财务损失、法律风险、监管处罚及品牌声誉受损 |
| **中等风险数据** | Moderate Risk Data (MRD) | 对于未经授权方来说几乎没有实际价值的公司信息，但未经授权使用也可能对公司产生负面影响 |

### 安全分类与保密等级

数据安全的两个驱动因素：**保密等级**和**监管要求**。

| 维度 | 保密等级（内部驱动） | 监管要求（外部驱动） |
|------|-------------------|-------------------|
| **来源** | 源自组织内部 | 由外部定义（法律、条约、行业法规） |
| **基础** | "需要知道"的基础上共享 | "允许知道"的基础上共享 |
| **数量限制** | 任何数据集只能有一个密级（基于最敏感数据项） | 可附加多个监管类别 |

#### 保密等级分类

| 级别 | 说明 |
|------|------|
| **面向普通受众** | 任何人都可以获得的信息 |
| **仅内部使用** | 仅限员工或组织内部成员使用，信息分享风险很小。仅供内部使用、可在组织外部展示或讨论，但不得复制 |
| **机密** | 若无恰当保密协议或类似协议，不得在组织以外共享。不得与客户共享其他客户机密信息 |
| **受限机密** | 要求个人通过许可才能获得访问资格，仅限于特定"需要知道"的个人 |
| **注册机密** | 机密程度非常高，所有信息访问者都必须签署法律协议才能访问数据，并承担保密责任 |

> 保密级别并不意味着由于监管要求而受到任何限制。例如，不会通知数据管理者不得在数据来源国之外公开数据。

#### 监管类别（外部驱动）

由于有许多法规存在重叠，按主题域归纳为几个法规类别或法规系列。法规的执行类别的最佳数量建议**不超过9个**。

**法规系列举例：**

| 系列 | 说明 | 涉及法规 |
|------|------|---------|
| **个人标识信息 (PII)** | 可识别个人的信息（单独或一组信息可标识一个人），如姓名、地址、电话号码、身份证号码、账号、年龄、种族、宗教、生日、家庭成员姓名、就业信息、薪酬等 | 欧盟隐私指令、加拿大PIPEDA、日本PIP法案、PCI标准、美国FTC要求、GLBA等 |
| **财务敏感数据** | 所有财务信息，包括"股东"或"内部"数据及尚未公开披露的当前财务信息，未公布的未来业务计划、计划中的并购或分拆等 | 内幕交易法、SOX、GLBA |
| **医疗敏感数据 (PHI)** | 有关个人健康或医疗的所有信息 | HIPAA（美国）及各国相关法律 |
| **教育记录** | 有关个人教育的所有信息 | FERPA（美国） |
| **支付卡行业 (PCI-DSS)** | 可识别金融机构账户的个人信息，如姓名、信用卡号、银行账号、账户到期日期等 | PCI-DSS合同义务（全球基本统一） |
| **商业秘密** | 使用专有方法、组合、方案、来源、设计、工具、配方或操作技术取得竞争优势的信息 | 行业法规和知识产权法 |
| **合同约束** | 合同中规定如何使用或不使用的信息，如环境记录、危险材料报告、客户密码等 | 合同协议 |

> 保护机密客户数据的合同义务与美国、日本和加拿大政府保护个人可标识信息的法规非常相似，也与欧盟隐私合规要求类似。当列出并比较各种法规的可审计合规行为时，很容易看出这种模式。

#### 安全分级工作成果

分级的工作成果是一组经正式批准的安全分级和监管类别，以及从中央存储库中获得此类元数据的流程，使得业务和技术员工了解他们正在处理、传送和授权信息的敏感性。

**关键原则**：大多数信息可以聚合，从而使其具有更高或更低的敏感性。开发人员需要知道聚合如何影响整体安全分级和监管类别。

### 安全分类分级流程

信息安全管理首先要对组织的数据进行分类分级。以下是完整的分类分级流程：

| 步骤 | 活动 | 说明 |
|------|------|------|
| 1 | 识别和分类分级敏感数据资产 | 根据行业和组织不同，识别敏感数据范围（PII、医疗、财务等） |
| 2 | 探明整个企业敏感数据 | 确定敏感数据存储位置，评估不同位置的安全性要求 |
| 3 | 确定每项数据资产的保护措施 | 根据数据内容和技术类型，采取针对性安全措施 |
| 4 | 识别信息与业务流程交互 | 分析业务流程，确定在什么条件下允许何种数据访问 |
| 5 | 分配保密等级 | 数据管理专员评估和确定适当的数据密级 |
| 6 | 分配监管类别 | 根据适用法规对信息进行分类 |
| 7 | 记录元数据 | 在元数据库中记录安全性和监管分类 |

> 文档和报告的分级应基于文档中发现的最高密级信息。在每个页面或每个屏幕的页眉或页脚中标记分级。假定任何未标记的产品都是面向一般受众的。

### 访问控制（4A+1E模型）

数据安全需求和规程分为4个方面，简称**"4A"**，加上一个**"E"**。信息分类、访问权限、角色组、用户和密码是实施安全策略和满足4A的常用手段。

| 要素 | 英文 | 详细说明 |
|------|------|---------|
| **访问** | Access | 允许有授权的个人及时连接到信息系统并使用数据。做动词用时意味着主动连接并使用数据；做名词用时指此人具有有效的授权 |
| **审计** | Audit | 审查安全操作和用户活动，确保符合法规并遵守公司制度和标准。信息安全专业人员定期查看日志和文档以验证合规性 |
| **验证** | Authentication | 验证用户身份是否属实。除密码方式外，更严格的方法包括安全令牌、回答问题、指纹等。验证过程中所有传输信息都经过加密 |
| **授权** | Authorization | 依据个人角色授予其访问特定数据视图的权限。系统在每次用户登录时检查是否拥有有效的授权令牌（Token），这是公司活动目录（Active Directory）中的一个条目 |
| **权限** | Entitlement | 由单个访问授权决策向用户公开的所有数据元素的总和。在生成授权请求之前，管理人员必须决定某人"有权"访问这些信息 |

#### 身份验证方法

| 方法 | 说明 |
|------|------|
| 密码 | 最基本的验证方式，需遵循强密码标准 |
| 安全令牌 | 专用硬件设备用于登录 |
| 问题回答 | 回答预设的安全问题 |
| 生物特征 | 指纹、面部识别、视网膜扫描等 |
| 多因素认证 | 结合两种以上验证方式，所有具有高度敏感信息权限的用户都应使用 |

#### 密码标准

| 标准 | 说明 |
|------|------|
| 密码设置 | 每个用户账户必须有密码，用户自行设置满足安全标准的"强"密码 |
| 临时密码 | 创建新用户账户时，临时密码应设置为首次使用立即过期 |
| 禁止空白密码 | 不允许使用空白密码 |
| 定期更改 | 建议每隔45～180天更改一次，具体频率取决于系统性质和数据敏感程度 |
| 风险平衡 | 密码更改过于频繁也会带来风险，因为员工可能会将密码记录在不设防的记事本上 |

#### 监控机制

安全监控对于保障其他流程的正常运行至关重要。监控和审计都需持续地或定期地进行。正式审计必须由第三方主持开展才能视为有效。

| 类型 | 说明 | 特点 |
|------|------|------|
| **主动监控** | 实时监控保密信息系统，提醒安全管理员可疑活动或不当访问；主动中断不符合特定访问配置文件的活动 | **检测机制** |
| **被动监控** | 定期对系统进行快照并与基准进行比较，跟踪系统随时间变化情况；报告发送给数据管理员 | **评估机制** |

#### 凭据安全

| 方面 | 说明 |
|------|------|
| **身份管理系统** | 用户只需在登录工作站时输入一次密码，之后所有身份验证通过企业用户目录执行（"单点登录"） |
| **用户ID标准** | 电子邮件域中用户ID应唯一。不鼓励使用包含系统员工ID号的信息作为ID |
| **多因素识别** | 包括移动设备回拨、语音代码、硬件设备、生物特征因素等 |

### 访问控制模型（RBAC/ABAC）

数据访问控制可根据需要在用户或组级别上进行管理。小型组织在用户级别管理数据访问是可接受的，但大型组织使用**基于角色的访问控制（RBAC）**更加有益。

#### 基于角色的访问控制（RBAC）

角色组使得安全管理员能够按角色定义权限，并通过在适当的角色组中添加用户实现授权。

**关键原则：**
- 可将一个用户添加到多个组中，但这种做法可能使授予该用户的权限难以理解
- 尽量将每个用户只分配到一个角色组内
- 需要创建特定数据权限的不同用户视图，以满足法律法规要求

**角色定义方法：**

| 方法 | 方向 | 说明 |
|------|------|------|
| **网格方法** | 从数据开始 | 基于数据机密性、法规和用户职责，映射数据-角色关系。公共用户角色可访问公开级别数据；营销角色可访问某些PII但不能访问受控数据 |
| **层次结构方法** | 从用户开始 | 在工作组或业务单元级别构建群组定义，子角色权限受限于父角色权限。需要能细粒度下钻到单个用户权限的报告系统 |

**角色分配网格示例：**

| 角色 | 公开数据 | 内部数据 | PII | 受控数据 | 机密数据 |
|------|---------|---------|-----|---------|---------|
| 公共用户 | ✓ | - | - | - | - |
| 一般员工 | ✓ | ✓ | - | - | - |
| 营销角色 | ✓ | ✓ | ✓ | - | - |
| 管理角色 | ✓ | ✓ | ✓ | ✓ | - |
| 高级管理 | ✓ | ✓ | ✓ | ✓ | ✓ |

#### 基于属性的访问控制（ABAC）

在RBAC基础上，ABAC进一步考虑了以下属性维度：

| 属性维度 | 说明 |
|---------|------|
| **用户属性** | 用户角色、部门、职位、安全许可级别等 |
| **资源属性** | 数据分类级别、所属业务域、敏感度标签等 |
| **环境属性** | 访问时间、地理位置、设备类型、网络环境等 |
| **操作属性** | 读取、写入、删除、导出等操作类型 |

> 查询级访问控制是解决过度授权的方案，将控制机制设置为数据库权限限制为最低需要的SQL操作和数据范围。数据访问控制粒度要从表级访问深入到特定行和列。

#### 用户和角色管理中的挑战

| 挑战 | 说明 |
|------|------|
| **数据一致性** | 用户信息（姓名、职务、员工ID）不得不冗余存储在多个位置，数据孤岛常常相互冲突 |
| **解决方案** | 集中管理用户身份数据和角色组成员关系，变更应通过变更管理系统进行跟踪 |
| **角色定义要求** | 基于角色的安全依赖于明确地定义和一致地分配角色 |

### 加密技术

加密（Encryption）是将明文数据转换为复杂编码，以隐藏特权信息、验证传输完整性或验证发送者身份的过程。加密数据不能在没有解密密钥或算法的情况下被读取。解密密钥或算法通常单独存储，且不能根据同一数据集中的其他数据元素计算出来。

#### 三种常见加密算法

| 类型 | 原理 | 常见算法 | 适用场景 | 特点 |
|------|------|---------|---------|------|
| **哈希** | 使用算法将数据转换为数学表示 | MD5（Message Digest 5）、SHA（Secure Hashing Algorithm） | 信息传输完整性验证、身份验证 | 必须知道确切算法和加密顺序才能反转 |
| **对称加密** | 使用一个密钥加密和解密，发送者和接收者共享同一密钥 | DES、3DES（Triple DES）、AES（Advanced Encryption Standard）、IDEA、Twofish、Serpent | 逐字符或按块加密，适合大量数据加密 | 速度较快，但密钥分发是挑战 |
| **非对称加密** | 使用一对密钥：公钥加密，私钥解密 | RSA（Rivest-Shamir-Adelman）、Diffie-Hellman、PGP（Pretty Good Privacy） | 将多数据源以受保护方式发送给少数接收者（如提交到清算中心） | 安全性高，但计算开销大 |

> **注意**：使用Simple DES是不明智的，因为它容易受到许多简单的攻击。

#### 数据存储与传输安全

| 类型 | 说明 | 保护措施 |
|------|------|---------|
| **数据存储安全** (Data-at-Rest) | 保护静态存储中的数据 | 加密存储、访问控制、备份加密 |
| **数据传输安全** (Data-in-Motion) | 保护系统之间通过网络传输的数据 | VPN隧道、HTTPS加密、传输层加密 |

### 数据脱敏

数据脱敏（Data Masking/Fuzzification）是一种以数据为中心的安全机制。数据可以通过混淆（使之模糊或不清晰）或掩码来降低可用性，删除、打乱或以其他方式改变数据的原始外观，但不丢失数据的含义或数据与其他数据集之间的关系。属性中的值可能会改变，但新值对于这些属性仍然是有效的。

#### 脱敏类型

##### 静态数据脱敏（Persistent Data Masking）

永久且不可逆转地修改数据，通常用于从生产环境向开发或测试环境提供数据。脱敏后的数据仍可用于测试流程、应用程序、报表等。

| 子类型 | 说明 | 特点 |
|--------|------|------|
| **不落地静态脱敏** (In-flight Persistent Masking) | 数据在源和目标之间移动时进行脱敏或混淆处理 | 非常安全，不会留下未脱敏的临时中间文件或数据库；如遇到问题可以重新运行 |
| **本库静态脱敏** (In-place Persistent Masking) | 数据源和目标位置相同，脱敏后覆盖原始数据 | 存在一定风险，脱敏失败难以将数据还原为可用格式。适用于小众业务场景，一般不落地脱敏更安全 |

##### 动态数据脱敏（Dynamic Data Masking）

不更改基础数据，只在终端用户或系统中展示时改变数据外观。

> **示例**：社会保障号码 `123456789` 经动态脱敏后显示为 `***-**-6789`，呼叫中心人员可验证通话对象但无法获取完整号码。

#### 脱敏方法

| 方法 | 说明 | 示例 |
|------|------|------|
| **替换** (Substitution) | 用查找表中的字符或整个值或标准模式替换原始数据 | 用列表中的随机值替换名字 |
| **混排** (Shuffling) | 在记录中交换相同类型的数据元素，或在不同行之间交换同一属性的数据元素 | 在供应商发票中混排供应商名称 |
| **时空变换** (Temporal Variance) | 把日期前后移动若干天，变动小到足以保留特征，大到无法识别原始数据 | 日期偏移处理 |
| **数值变换** (Value Variance) | 使用随机参数（如正负百分比），变动小到保持特征，大到无法识别原始数据 | 数值范围偏移 |
| **置空或删除** (Nulling or Deleting) | 移除不应出现在测试系统中的数据 | 删除敏感字段 |
| **随机选择** (Randomization) | 将部分或全部数据元素替换为随机字符或一串单个字符 | 全部替换为随机字符串 |
| **加密** (Encryption) | 通过密码技术将可识别、有意义的字符流转换为不可识别的字符串，就地进行混淆 | 原地加密混淆 |
| **表达式脱敏** (Expression Masking) | 将所有值更改为一个表达式的结果 | 将自由格式字段改为"这是个注释字段" |
| **键值脱敏** (Key Masking) | 对数据库键字段（或有关联关系的相似字段）脱敏，需满足脱敏数据唯一性和可重复性 | 对测试中保持数据一致性极为重要 |

### 数据完整性

数据完整性（Data Integrity）是对数据整体状态的要求，保护数据免于遭受不当的添加、删除或修改。

> **示例**：美国的《萨班斯-奥克斯利法案》（SOX）主要规定了财务信息的创建和编辑的规则，以保护财务信息的完整性。SOX法案限制和管理谁可以更改财务数据，从而确保数据完整性。

### 网络安全术语

| 术语 | 说明 |
|------|------|
| **后门** (Backdoor) | 计算机系统或应用程序中被忽视或隐藏的入口，允许未授权用户绕过密码等限制获取访问权限。后门通常是开发人员出于维护系统目的而创建的，任何后门都是安全风险 |
| **机器人/僵尸** (Bot/Zombie) | 被恶意黑客使用特洛伊木马、病毒、网络钓鱼或下载文件感染所接管的计算机。机器人网络（Bot-Net）是由被感染机器组成的网络 |
| **Cookie** | 网站在计算机硬盘上安放的小数据文件，用于识别用户并分析偏好。有时会被间谍软件利用，引发隐私问题 |
| **防火墙** (Firewall) | 过滤网络流量的软件/硬件，保护计算机或网络免受未授权访问。对进出通信数据进行扫描、限制或监管，防止未许可信息通过 |
| **边界** (Perimeter) | 组织系统与外部系统之间的边界，通常将防火墙部署在内部和外部环境之间 |
| **DMZ** | "非军事区"（De-militarized Zone），位于组织边界区域，用防火墙部署在其与组织之间。DMZ环境与互联网之间始终设有防火墙 |
| **超级用户账户** | 具有系统管理员或Root访问权限的账户，仅在紧急情况下使用。凭据要求高度安全性，使用时需适当的文档和批准，短时间内过期 |
| **键盘记录器** (KeyLogger) | 攻击软件，记录键盘上所有按键行为并发送到互联网。捕获每个密码、备忘录、公式、文档和Web地址 |
| **渗透测试** (Penetration Testing) | "白帽"黑客模拟恶意黑客从外部侵入系统，以识别系统漏洞。发现的漏洞应在应用程序正式发布前解决 |
| **VPN** | 虚拟专用网络（Virtual Private Network），使用不安全互联网创建安全"隧道"。通过多因素身份验证连接到防火墙，对所有传送数据进行强加密 |

### 恶意软件

恶意软件（Malware）指为损坏、更改或不当访问计算机或网络而创建的软件。任何未经授权安装的软件都可视为恶意软件。

| 类型 | 说明 |
|------|------|
| **广告软件** (Adware) | 从互联网下载，监控计算机使用（如访问了哪些网站），收集浏览和购买习惯个人资料并出售给营销公司 |
| **间谍软件** (Spyware) | 未经同意潜入计算机以跟踪操作。搭载在其他软件程序上，记录用户按键以窃取个人信息（信用卡号、银行账户、密码等） |
| **特洛伊木马** (Trojan Horse) | 通过伪装或嵌入合法软件进入计算机系统的恶意程序。安装后可删除文件、访问个人信息、安装恶意软件、重新配置计算机、安装键盘记录器，或将计算机用作僵尸 |
| **病毒** (Virus) | 依附到可执行程序或有漏洞的应用程序上，能造成从让人讨厌到极具破坏性的后果。病毒程序总是需要依附于另一个程序 |
| **蠕虫** (Worm) | 可在网络中自行复制和传播的程序。主要功能是通过消耗大量带宽危害网络，甚至导致网络中断 |

#### 恶意软件来源

| 来源 | 说明 |
|------|------|
| **即时消息 (IM)** | 许多IM系统在添加安全功能方面进展缓慢，成为传播病毒、间谍软件、网络钓鱼和蠕虫的途径 |
| **社交网络** | Facebook、Twitter、LinkedIn等社交网站已成为在线罪犯、垃圾邮件发送者和身份盗窃者的目标。还会带来员工发布企业敏感信息的风险 |
| **垃圾邮件 (Spam)** | 未经请求而大批量发送的商业电子邮件。检测方法包括：已知发送域名、抄送地址超限、正文只含超链接图片、特定文本模式等 |

> 垃圾邮件也可能是互联网恶作剧或包含恶意软件的附件。回复垃圾邮件可使发件人得到有效的电子邮件地址，并可能收到更多垃圾邮件。

### 社会工程与钓鱼

社会工程（Social Engineering）和钓鱼（Phishing）是通过欺骗方式获取敏感信息的攻击手段。

#### 社会工程

指恶意黑客试图通过欺骗方式使人们提供信息或访问信息的方法。黑客利用所获得的各种信息说服其他员工（具备合法请求身份的）。有时黑客会依次联系几个人，每一步都收集有用信息，以获得更高级别员工的信任。

#### 钓鱼攻击

| 类型 | 说明 |
|------|------|
| **普通钓鱼** | 通过电话、即时消息或电子邮件诱使接收方提供有价值的信息或个人隐私。通常伪装为折扣或降低利率的销售宣传 |
| **鲸钓鱼叉** (Whaling) | 专门针对高管的虚假电子邮件，极具针对性 |
| **伪装攻击** | 黑客亲自前往目标地点直接与员工交谈，有时伪装成或冒充供应商获取敏感信息 |

### 数据安全类型

数据安全不仅涉及防止不当访问，还包括允许对数据的适当访问。应通过授予权限（Opt-in）控制对敏感数据的访问。**"最小化授权"**是一项重要的安全原则，仅允许用户、进程或程序访问其合法目的所允许的信息。

#### 设施安全

设施安全是抵御恶意行为人员的第一道防线。设施上至少应具有一个"上锁"的数据中心，仅限授权员工访问。社会工程威胁将"人"视为设施安全中最薄弱的环节。

#### 设备安全

移动设备（笔记本电脑、平板电脑、智能手机）由于可能丢失、被盗及遭受黑客攻击，本质上都有安全风险。

**设备安全标准包括：**

| 标准 | 说明 |
|------|------|
| 访问策略标准 | 使用移动设备连接的访问策略 |
| 数据存储标准 | 在便携式设备上存储数据的标准 |
| 数据清除标准 | 设备的数据清除和处理符合记录管理制度 |
| 加密和反恶意软件 | 安装加密和反恶意软件的标准 |
| 安全漏洞告知 | 安全漏洞的报告和通知机制 |

#### 电子通信安全

用户必须接受培训，避免通过电子邮件或即时通信应用发送个人信息或任何受限制的/机密的公司信息。这些不安全的通信方式可被外部读取或拦截。社交媒体（博客、门户、Wiki、论坛等）应被视为不安全的通信方式，使用时不应包含机密或受限制的信息。
