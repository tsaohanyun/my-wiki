# 参考数据和主数据管理：方法论与流程

> **所属知识体系**：[[DAMA-DMBOK2/10-参考数据和主数据管理|DAMA-DMBOK2 第10章：参考数据和主数据管理]]

---

## 主数据管理（MDM）

DAMA将MDM定义为：对主数据值和标识符进行控制，以确保在各系统中一致地使用最准确和最新的关键业务实体数据。Gartner将主数据管理定义为："一个技术驱动的知识领域，由业务和技术部门共同协作，以确保企业共享主数据资产的统一性、准确性、可管理性、语义一致性并支持问责制。主数据是由标识符和扩展属性组成的一致且统一的数据集，描述了企业的核心实体，包括客户、潜在客户、员工、供应商、地理位置、组织架构和会计科目等。"

Gartner的定义强调主数据管理是一个由人、流程和技术组成的知识领域，而不是一个特定的应用系统解决方案。然而，主数据管理的缩写"MDM"却常用于特指主数据管理系统或产品。主数据管理系统可以简化主数据管理的部分工作，能提升效率，并且很有效，不过仅仅依靠主数据管理系统并不能保证主数据得到良好的管理并满足组织需求。

> **注意：** 在整个DAMA-DMBOK中，MDM指管理主数据的整体过程，而不仅仅是用于管理这些数据的工具。

**主数据管理面临的根本挑战：** 人们选择不同的方式表示相似的概念，而将这些表述协调一致并不容易。同样重要的是，信息会随着时间的推移而变化，系统性地解释这些变化需要具备计划、数据知识和技术技能。任何认识到需要进行主数据管理的组织大多已经有一个复杂的系统环境，有多种途径可以采集和存储对客观世界实体的引用。这种系统复杂性会随着时间的推移或发生企业并购等行为而不断加剧。

考虑到这种复杂性，在实施时最好每次只处理主数据管理的一个数据域，可以从小处着手，先梳理属性较少的，然后不断扩展。

### MDM规划基本步骤

在某个领域内规划主数据管理工作包括以下基本步骤：

1. 识别能提供主数据实体全面视图的候选数据源
2. 制定准确匹配和合并实体实例的规则
3. 建立识别和恢复不当匹配和合并数据的方法
4. 建立分发可信主数据到企业内所有系统的方法

### MDM全生命周期关键活动

主数据管理需要形成一个全生命周期的闭环管理，关键活动包括：

1. **建立主数据实体的上下文**：包括相关属性的定义及其使用条件，并加以治理
2. **识别同一实体的多个实例**：构建和维护标识符和交叉参考以支持主数据集成
3. **跨来源匹配和集成数据**：确定主数据权威数据源，提供主记录和事实；合并多来源记录，提供跨系统的信息合并视图，并尝试解决属性命名和数据值不一致的问题
4. **识别未能正确匹配或合并的实例**：确保它们得到修正，并正确关联到标识符
5. **实现跨应用系统的可信主数据访问**：通过直接读取、使用数据服务或复制主数据到业务系统、数据仓库或其他分析型数据存储
6. **在组织内强制使用主数据**：深入开展数据治理和变革管理，以确保在企业级层面共享主数据

### MDM关键处理步骤

主数据管理的关键处理步骤包括数据模型管理、数据采集、数据验证/标准化/增强、实体解析、数据共享和管理责任：

```
数据模型管理 → 数据采集 → 数据验证/标准化/增强 → 实体解析 → 数据共享 → 管理责任
```

**1. 数据模型管理**

清晰且一致的逻辑数据定义至关重要。主数据模型帮助组织克服"系统方言"难题——系统中使用的术语和定义在其范围内有意义，但不一定在整个企业中通用。主数据的属性定义和相关数据值的粒度必须在整个组织中合乎情理。

不同候选源系统里可能拥有相同的属性名称，但其数据值在企业全局层面可能处于完全不同的语境中。类似地，不同源系统可能存在不同名称的属性，在企业全局层面可以合并为单个属性，并且数据值处于适当的语境。

**2. 数据采集**

即使在特定数据源中，表示同一实体实例的数据看起来也可能不同。规划、评估新数据源并将其合并到MDM解决方案中必须是一个可靠的、可重复的过程。

|------|---|---|---------|------|-----|------|------|

数据采集活动包括：
- 接收并响应新数据源采集需求
- 使用数据清洗和数据分析工具执行快速、及时、一致和高阶的数据质量评估
- 评估数据集成的复杂性并告知需求方
- 尝试采集数据，评估其对匹配规则的影响
- 确定新数据源的数据质量量化指标
- 确定新数据源质量监控和维护工作的责任人
- 完成与整体数据管理环境的集成

**3. 数据验证、标准化与数据增强**

要实现实体解析，必须使数据尽可能保持一致。标准化和增强过程包括：

- **验证**：识别那些明显错误、可能不正确或默认的数据
- **标准化**：确保数据内容符合标准参考数据值、标准格式或字段
- **数据增强**：添加可以改进实体解析服务的属性

**4. 实体解析与标识符管理**

实体解析（Entity Resolution）是确定两个对客观世界对象的引用指向的是同一对象还是不同对象的过程（Talburt, 2011）。实体解析包括一系列活动：**实例提取、实例准备、实例解析、身份管理、关系分析**。

**（1）匹配与候选识别**

匹配（Matching）或候选识别（Candidate Identification）是确认不同记录与单个实体如何相关联的过程。此过程的风险包括：

- **假阳性（False Positives）**：两个不同实体链接到同一个标识符上（一码多物）
- **假阴性（False Negatives）**：同一个实体的两个对象并没有链接到同一个标识符下（一物多码）

| 算法类型 | 说明 | 优点 | 缺点 |
|---------|------|------|------|
| **确定性算法（Deterministic）** | 依赖定义的匹配模式和匹配推理规则分配权重得出分值 | 可预测，比较简单，开箱即用 | 不会超出规则制定人员的预期 |
| **概率算法（Probabilistic）** | 利用统计算法评估两条记录代表相同实体的可能性 | 不依赖于预设的规则，可自我优化 | 结果存在不确定性，需要训练数据 |

**（2）身份解析示例**

身份解析是利用多个字段精确匹配数据到同一实体的方法。典型的冲突场景包括地址变更、姓氏变更、标识符未知等情况。

**（3）匹配工作流程/调节类型**

| 类型 | 说明 | 适用场景 |
|------|------|----------|
| **多重识别匹配规则（Duplicate Identification）** | 聚焦一组能唯一标识一个实体的特定数据元素，识别多条数据合并的可能性 | 高价值、高风险匹配 |
| **匹配链接规则（Match-link）** | 标识出可能与主记录相关的记录，只建立交叉引用关系的链接 | 需要灵活关联的场景 |
| **匹配合并规则（Match-merge）** | 匹配相关记录并将这些记录合并为唯一、统一、一致和全面的记录 | 需要权威唯一版本的场景 |

**（4）主数据ID管理**

| 标识符类型 | 说明 |
|-----------|------|
| **全局ID（Global ID）** | MDM解决方案分配和维护的唯一标识符，附加到已匹配的记录上 |
| **交叉引用（X-Ref）** | 源ID和全局ID之间映射关系的管理，包括维护映射历史记录 |

**（5）隶属关系管理**

| 关系类型 | 说明 | 特点 |
|---------|------|------|
| **隶属关系（Affiliation Relationships）** | 通过编程逻辑提供最大的灵活性 | 父子层次结构也可以用该关系类型展示数据 |
| **父子关系（Parent-Child Relationships）** | 数据结构已隐含父子层次关系 | 只需要较少的程序编码；但如果关系发生变化可能影响数据质量 |

**5. 数据共享与管理责任**

尽管MDM的大部分工作可通过工具自动化，但仍需数据管理人员手动处理匹配错误的数据。在理想情况下，从管理过程中吸取的经验教训可以用来改进匹配算法，减少手工操作的次数（见 [[DAMA-DMBOK2/03-数据治理]] 和 [[DAMA-DMBOK2/08-数据集成和互操作]]）。

## 数据共享架构

参考数据和主数据的集成有多种基本的架构方法。每个主数据主题域一般都有自己的记录系统。

| 架构模式 | 说明 | 优点 | 缺点 |
|----------|------|------|------|
| **注册模式（Registry）** | 注册表是向多种记录系统中主数据记录的索引 | 相对容易实现，只需要对记录系统进行少量更改 | 跨系统组合需要非常复杂的查询 |
| **事务中心模式（Transaction Hub）** | 应用系统与事务中心交互，实现主数据的访问和更新 | 更好地管理主数据，对外提供一致的主数据源 | 从现有记录系统中删除更新主数据功能的成本很高 |
| **混合模式（Consolidated）** | 注册模式和事务中心模式的结合 | 在提供企业视图的同时，减少了对记录系统的影响 | 需要跨系统复制数据，并且会出现数据延迟问题 |

**架构选择建议：**
- 小型组织可以有效地利用事务中心模式
- 拥有多个系统的全球性组织更建议选择注册模式
- 拥有"烟囱式"的业务部门和各种源系统的组织，采用混合模式才是正确选择
- 业务领域专家、数据架构师和企业架构师应该就主数据集成及架构方法提出自己的观点

| **P（规划）** | 2. 评估和评价数据源 | 了解数据结构和内容、收集或创建数据的过程、数据质量及其根本原因 |

| **C（控制）** | 6. 制定治理制度，强制使用主数据 | 建立路线图以让各系统把主数据值和标识符作为流程输入 |
| **D、O（开发、运营）** | 7. 实现数据共享/集成服务 | 获取共享数据源并发布参考数据和主数据 |

### 实施指南

1. **遵循主数据架构**：建立并遵循适当的参考架构对于在整个组织中管理和共享主数据至关重要
2. **监控数据移动**：确保数据及时提取和分发，展示共享和使用情况，识别血缘，辅助根本原因分析
3. **管理参考数据变更**：参考数据属于共享资源，不能随意更改
4. **数据共享协议**：建立服务水平协议（SLA）和指标来衡量共享数据的可用性和质量
5. **组织和文化变革**：参考数据和主数据管理需要人们放弃对本地数据和流程的控制

### 参考数据变更管理

由于参考数据属于共享资源，不能随意更改。成功管理参考数据的关键在于组织愿意放弃在本地控制共享数据。

**参考数据变更类型：**

| 变更类型 | 说明 |
|---------|------|
| 外部参考数据集特定数据行级别的变更 | 如行业代码的新增、修改或废弃 |
| 外部参考数据集的结构变化 | 如ICD-9到ICD-10的结构升级 |
| 内部参考数据集特定数据行级别变更 | 如内部状态代码的新增或修改 |
| 内部参考数据集的结构变化 | 如参考数据层级关系的调整 |
| 创建新的参考数据集 | 应充分考虑原始需求方要求之外的潜在需求 |

**ICD代码变更案例：**

| 对比维度 | ICD-9 | ICD-10 |
|---------|-------|--------|
| 数据格式 | 不同于ICD-10 | 不同于ICD-9 |
| 编码粒度 | 较粗 | 更加具体，单个编码传递更多信息 |
| 编码数量（截至2015年） | 约13,000个 | 约68,000个 |

**参考数据变更请求流程：**

```
接收变更请求 → 通知利益相关方 → 评估变更影响 → 审批决策 → 执行变更 → 通知变更完成
```

## 参考数据与主数据治理

参考数据和主数据是共享资源，需要进行治理。并不是所有的数据不一致问题都可以实现自动化处理，有些需要人们相互沟通协作才能解决。如果缺乏数据治理，参考数据和主数据解决方案将只是新增的数据集成实用系统，无法发挥其全部潜力。

治理过程需确定以下事项：
1. 需要集成的数据源
2. 需要落实的数据质量规则
3. 应遵循的使用规则条件
4. 需要监控的活动和监控频率
5. 数据管理工作的优先级和响应级别
6. 如何展示数据以满足利益相关方的需求
7. 参考数据和主数据管理部署中的标准审批标准和预期
