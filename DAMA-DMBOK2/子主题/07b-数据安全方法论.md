## 安全框架

数据安全活动包括：**确定需求 → 评估当前环境差距或风险 → 实施安全工具与流程 → 审核数据安全措施**。

### 识别数据安全需求

区分业务需求、外部监管要求和应用软件产品的规则很重要。应用程序系统通常具有超出业务流程所需的数据安全要求。

| 需求来源 | 说明 | 方法 |
|---------|------|------|
| **业务需求** | 分析业务规则和流程，确定安全接触点 | 数据到流程和数据到角色的关系矩阵 |
| **监管要求** | 创建包含所有数据相关法律法规及受影响数据主题的完整清单 | 法规清单管理（见表7-1） |

#### 法律法规清单

| 地区 | 法规 |
|------|------|
| **美国** | SOX（2002）、HITECH（2009）、HIPAA安全条例（1996）、Gramm-Leach-Bliley I/II、SEC法律和公司信息安全责任法、国土安全法案和美国爱国者法、FISMA、加州SB1386 |
| **欧盟** | 数据保护指令（EU DPD 95/46/EC）、巴塞尔协议II、GDPR |
| **加拿大** | 加拿大281法案、PIPEDA |
| **澳大利亚** | CLERP法案 |
| **国际/行业** | PCI-DSS（适用于所有信用卡公司）、FTC保护客户信息标准 |

### 制定数据安全制度

需要IT安全管理员、安全架构师、数据治理委员会、数据管理专员、内外部审计团队及法律部门之间的协作。

| 制度层级 | 内容 | 审批方 |
|---------|------|--------|
| **企业安全制度** | 员工访问设施的全局制度、电子邮件标准、基于职位的安全访问级别、安全漏洞报告制度 | 高级管理层 |
| **IT安全制度** | 目录结构标准、密码管理制度、身份管理框架 | IT指导委员会 |
| **数据安全制度** | 针对各个应用程序、数据库角色、用户组和信息敏感性的分类 | 数据治理委员会 |

> 数据治理委员会是数据安全制度的**审查方和批准方**。数据管理主管是制度的**主管方和维护方**。员工需要了解并遵从数据安全制度。安全制度应便于供应商、消费者和其他利益相关方轻松访问。

### 定义数据安全细则

细则是对制度的补充，提供满足制度意图的更详细信息：

1. **定义数据保密等级** — 创建或采用满足业务需求的分级方案，包含从最低机密到最高机密的一系列等级
2. **定义数据监管类别** — 分析类似法规并分组归类，将各种法律法规整理到同一法规类别中，既提供统一实施标准，也可确保全球任何地方的合规性
3. **定义安全角色** — 基于角色的访问控制（RBAC）

### 评估当前安全风险

从以下3个方面对每个系统进行评估：

1. 存储或传输数据的敏感性
2. 保护数据的要求
3. 当前的安全保护措施

> 记录调查结果，为将来的评估创建基线。此文档也是隐私合规的要求。在大型组织中，可能会聘用"白帽"黑客来评估漏洞。

### 实施控制措施和规程

控制措施和程序应涵盖：

| 控制领域 | 具体内容 |
|---------|---------|
| 权限生命周期 | 用户如何获取和终止对系统的访问权限 |
| 角色管理 | 如何将用户分配给角色或从角色中移除 |
| 权限监控 | 如何监控权限级别 |
| 变更管理 | 如何处理和监控访问变更请求 |
| 数据分类 | 如何根据机密性和适用法规对数据进行分类 |
| 事件响应 | 一旦检测到数据泄露后如何处理 |

### 系统安全风险识别

| 风险类型 | 说明 | 应对措施 |
|---------|------|---------|
| **滥用授权** | 用户被授予超出工作需要的特权（过度授权）。仅由于管理用户权限具有挑战性，许多用户被赋予通用默认访问权限 | 查询级访问控制（行级、列级）；自动化工具管理用户权限 |
| **滥用合法特权** | 用户滥用合法数据库权限进行未授权行为。如工作人员使用Excel连接数据库绕过Web应用限制，查询并保存所有患者记录 | 数据库访问控制（时间、位置监控和下载信息量） |
| **非法权限升级** | 攻击者利用数据库平台软件漏洞，将权限从普通用户提升为管理员。存储过程、内置函数、协议实现甚至SQL语句中都可能存在漏洞 | IPS + 查询访问控制入侵防护相结合 |
| **服务/共享账户滥用** | 使用服务账户或共享账户增加泄露风险。有些组织将监控系统配置为忽略与这些账户相关的任何警报，进一步增加风险 | 限制使用为特定系统上的特定命令；服务账户每次使用时分配新密码；默认不使用共享账户 |
| **平台入侵** | 数据库资产软件更新和入侵防护不足。软件更新通常按维护周期实施而非尽快实施，在补丁可用和实际更新之间数据库不受保护 | 定期软件升级 + 部署专用IPS |
| **SQL注入** | 攻击者将未授权SQL语句注入易受攻击的数据通道（存储过程、Web应用输入框），可能获得对整个数据库的无限制访问 | 在上传服务器处理前清理所有输入数据 |
| **默认密码** | 软件安装过程中创建的默认账户密码。SAP在安装时创建多个默认数据库用户 | 清除默认密码是每次实施的重要安全步骤 |
| **备份数据滥用** | 备份介质丢失导致数据泄露 | 对所有数据库备份进行加密；安全地管理解密密钥；密钥异地存储以助灾难恢复 |

## NIST与ISO27001映射

### NIST风险管理框架

美国国家标准与技术研究院（NIST）风险管理框架的第一步就是对所有企业信息进行分类。建立企业数据模型对于这个目标的达成至关重要。

**NIST框架核心步骤：**

| 步骤 | 活动 | 说明 |
|------|------|------|
| 1 | **分类** | 对所有企业信息进行分类分级 |
| 2 | **选择** | 选择适当的安全控制措施 |
| 3 | **实施** | 实施选定的安全控制 |
| 4 | **评估** | 评估控制措施的有效性 |
| 5 | **授权** | 授权信息系统的运行 |
| 6 | **监控** | 持续监控安全控制措施 |

### ISO 27001/27002

ISO 27001是信息安全管理体系（ISMS）的国际标准，ISO 27002是其配套的实践准则。

| 标准 | 内容 | 与数据安全的映射 |
|------|------|-----------------|
| **ISO 27001** | 信息安全管理体系要求 | 建立、实施、维护和持续改进ISMS |
| **ISO 27002** | 信息安全控制实践准则 | 涵盖访问控制、加密、审计、物理安全等具体控制措施 |

**ISO 27001/27002与数据安全的关键对应领域：**

| ISO 27002域 | 数据安全对应内容 |
|------------|-----------------|
| **A.5 信息安全策略** | 数据安全制度的制定和维护 |
| **A.6 组织信息安全** | 数据安全组织架构、角色和职责 |
| **A.8 资产管理** | 数据资产分类分级、资产清单 |
| **A.9 访问控制** | 访问控制策略、用户权限管理、RBAC/ABAC |
| **A.10 密码学** | 加密技术、密钥管理 |
| **A.12 操作安全** | 恶意软件防护、备份、日志管理 |
| **A.13 通信安全** | 网络安全管理、数据传输保护 |
| **A.14 系统获取开发和维护** | 项目安全需求、安全开发生命周期 |
| **A.16 信息安全事件管理** | 安全事件检测、响应和报告 |
| **A.17 业务连续性** | 灾难恢复、业务连续性计划 |
| **A.18 合规性** | 法律法规遵从、审计管理 |

> 参考文献：Calder, Alan, and Steve Watkins. *IT Governance: An International Guide to Data Security and ISO27001/ISO27002*. 5th ed. Kogan Page, 2012.

## 实施流程

| 步骤 | 活动 | 模式 |
|------|------|------|
| 1 | 确定相关数据安全要求 | 规划 (P) |
| 2 | 定义数据安全制度 | 控制 (C) |
| 3 | 定义数据安全标准 | 开发 (D) |
| 4 | 评估当前安全风险 | 规划 (P) |
| 5 | 实施控制和规程 | 运营 (O) |

### 方法与技术

| 方法 | 说明 |
|------|------|
| **应用CRUD矩阵** | 创建数据-流程矩阵和数据-角色关系（CRUD/CRUDE）矩阵，映射数据访问需求 |
| **即时安全补丁** | 建立在所有计算机上尽快安装安全补丁的流程 |
| **元数据中的安全属性** | 在元数据库中记录安全性和监管分类，标记到数据上 |
| **项目需求中的安全要求** | 在分析阶段就详细确定数据和应用程序的安全要求 |
| **加密数据的高效搜索** | 用相同加密方法加密搜索条件，用密文查找匹配项以减少解密成本 |
| **文件清理** | 在文件共享前清理文件元数据（如历史变更记录），降低机密信息被共享风险 |

### 管理安全制度合规性

管理安全制度合规性包括确保遵守制度并有效维护控制的日常活动：

1. **管理监管合规**
   - 衡量授权标准和程序的合规性
   - 确保所有数据需求都是可度量的、可审计的
   - 确保受监管数据在存储和使用时使用标准工具和流程进行保护
   - 当发现潜在不合规问题或违规时，使用升级和通知机制

2. **审计数据安全和合规活动**
   - 内部审计活动的目的是确保数据安全和监管合规被常规和定期地执行
   - 审计师必须独立于审计中涉及的数据和/或流程
   - 审计不是为了发现错误，而是为管理层和数据治理委员会提供客观、公正的评估及合理建议

**审计检查内容包括：**

| 审计项目 | 说明 |
|---------|------|
| 制度评估 | 评估制度和标准，确保明确定义合规控制并满足法规要求 |
| 程序分析 | 分析实施程序和用户授权实践，确保符合监管目标 |
| 授权评估 | 评估授权标准和程序是否充分且符合技术要求 |
| 升级评估 | 评估违规事件的升级程序和通知机制 |
| 外包审查 | 审查外包合同、数据共享协议和监管合规义务 |
| 成熟度评估 | 评估组织内安全实践成熟度，报告"监管合规状态" |
| 变革建议 | 建议合规制度变革和运营合规改进 |

### 安全意识与培训

| 措施 | 说明 |
|------|------|
| **培训** | 通过组织各级安全举措培训推广安全标准。通过在线测试等评估机制跟踪培训，培训和测试应是强制性的，作为员工绩效评估的前提条件 |
| **制度一致性** | 为工作组和各部门制定补充企业制度的数据安全制度，采取"因地制宜"的方式 |
| **度量安全收益** | 将数据安全收益同组织计划联系起来，在平衡计分卡中包括数据安全活动的客观指标 |
| **供应商安全要求** | 在SLA和外包合同义务中包括数据安全要求 |
| **增强紧迫感** | 强调法律、合同和监管要求，以增强数据安全管理的紧迫感 |
| **持续沟通** | 支持持续的员工安全培训计划，向员工通报安全计算实践和当前威胁 |

### 用户数据权限可见性

必须在系统实施期间审查每个用户的数据授权（通过单个授权提供的所有数据的总和），以确定是否包含受监管的信息。要了解谁可以访问哪些数据，就需要管理描述数据机密性和监管分类的元数据，以及权限和授权本身。监管敏感性分类应该是数据定义过程的标准部分。

## 度量指标

度量信息保护过程并确保该过程按要求运行至关重要。更需要关注可操作的度量指标——几个关键性指标比一大堆明显不相干的指标更易于管理。

### 安全实施指标

| 指标 | 说明 |
|------|------|
| 安装最新安全补丁的企业计算机百分比 | 衡量补丁管理效果 |
| 安装并运行最新反恶意软件的计算机百分比 | 衡量恶意软件防护覆盖 |
| 通过背景调查的新员工百分比 | 衡量人员安全管控 |
| 年度安全实践测验中得分超过80%的员工百分比 | 衡量安全意识培训效果 |
| 已完成正式风险评估分析的业务单位百分比 | 衡量风险管理覆盖 |
| 成功通过灾难恢复测试的业务流程百分比 | 衡量业务连续性准备 |
| 已成功解决审计发现的问题百分比 | 衡量审计整改效果 |

**可跟踪趋势的统计指标：**

| 指标类别 | 具体指标 |
|---------|---------|
| 系统性能 | 所有安全系统的性能指标 |
| 人员审查 | 背景调查和结果 |
| 应急准备 | 应急响应计划和业务连续性计划状态 |
| 犯罪事件 | 犯罪事件和调查 |
| 合规整改 | 合规的尽职调查及需要解决的调查结果数量 |
| 风险管理 | 已执行的信息风险管理分析及导致可操作变更的数量 |
| 制度审计 | 制度审计的影响和结果（如清洁办公桌制度检查） |
| 物理安全 | 安全运营、物理安全和楼宇防护统计 |
| 制度文档 | 文档化的、可访问的安全标准 |
| 合规动机 | 可被度量的相关方遵守安全制度的动机 |
| 行为风险 | 业务行为和声誉风险分析，包括员工培训 |
| 数据卫生 | 基于财务、医疗、商业秘密和内幕信息等特定类型数据的商业卫生和潜在内幕风险 |
| 信心指标 | 管理者和员工之间的信心和影响力指标 |

### 安全意识指标

| 指标 | 说明 |
|------|------|
| **风险评估结果** | 提供定性数据，反馈给相关业务单位以增强责任意识 |
| **风险事件和画像** | 识别需要纠正的未管理风险敞口，通过后续测试了解信息传达情况 |
| **正式反馈调查和访谈** | 确定安全意识水平，衡量成功完成安全意识培训的员工数量 |
| **事故复盘和经验教训** | 为安全意识缺口提供信息来源，具体指标可包括减少了多少漏洞 |
| **补丁有效性审计** | 涉及使用机密和受控信息的计算机，评估安全补丁的有效性 |

### 数据保护指标

| 指标 | 说明 |
|------|------|
| **关键性排名** | 对特定数据类型和信息系统的关键性排名 |
| **年度化损失预期** | 与数据丢失、泄露或损坏相关的事故、黑客攻击、盗窃或灾难的年度化损失预期 |
| **特定数据丢失风险** | 与某些类别受监管信息相关的特定数据丢失风险及补救优先级排序 |
| **风险映射** | 数据与特定业务流程的风险映射（如销售点设备风险包含在金融支付系统风险预测中） |
| **威胁评估** | 对有价值数据资源及其传播媒介遭受攻击的可能性进行评估 |
| **漏洞评估** | 对业务流程中敏感信息可能有意或无意暴露的特定部分进行评估 |

### 安全事件指标

- 检测并阻止入侵的尝试
- 通过防止入侵节省的安全成本投资回报

### 机密数据扩散指标

> 机密数据存储的位置越多，泄露风险越大。应度量机密数据的副本数量以减少扩散。
