# 数据存储和运营

## 概述

数据存储和运营（Data Storage and Operations）是DAMA-DMBOK2框架中专注于数据基础设施管理的知识领域。它涵盖对待存储数据的数据库设计、实现和技术支撑，确保数据在从创建/获取到处置的整个生命周期内实现价值最大化。

该知识领域包含两项核心子活动：
1. **数据库支撑**：侧重于数据生命周期相关的活动，从建立数据库环境开始，到获取、备份、清除数据，并确保数据库持续良好运行。在此过程中，要特别重视对数据库的持续监测和调优。
2. **数据库技术支撑**：界定满足组织需要的技术需求，确定技术架构，应用和管理相关技术，并解决与技术相关的问题。

在DAMA框架中，数据存储和运营是[[DAMA-DMBOK2-05-数据建模与设计|数据建模与设计]]的物理实现层，为[[DAMA-DMBOK2-07-数据安全|数据安全]]提供基础设施支撑，并依赖[[DAMA-DMBOK2-04-数据架构|数据架构]]进行整体规划。

**核心目标**：
1. 在数据的整个生命周期内管理其可用性
2. 确保数据资产的完整性
3. 管理数据事务的性能

### 业务驱动因素

公司运营越来越离不开各类信息系统。对依赖数据进行运营的公司而言，数据存储和运营的活动至关重要。**业务的连续性**是这些活动的主要驱动力。如果系统不可用，公司运营就可能受到影响甚至停摆。为了有效降低这类风险，公司需要为其建立一套可靠的数据存储基础设施。

### 指导原则

1. **识别并执行自动化的机会**：自动化数据库开发流程可以缩短开发周期，减少错误和返工，并将对开发团队的影响降至最低。DBA可以适应更敏捷的应用程序开发方法。
2. **以重用为目标构建数据对象**：开发和使用可抽象和可复用的数据对象，避免应用程序与数据库模式紧密耦合（即"对象关系阻抗失配"）。可使用数据库视图、触发器、函数及存储过程、应用程序数据对象和数据访问层、XML和XSLT等机制。
3. **理解并适当运用最佳实践**：DBA应将数据库标准和最佳实践作为要求加以推广，但在有合理理由时也应灵活应对偏离标准的情况。坚持数据库标准必须以不影响项目成功交付为前提。
4. **数据库标准要与需求相适应**：SLA既要反映DBA的建议，也要考虑开发人员能否接受。
5. **明确DBA在项目工作中的角色期望**：在项目定义阶段就引入DBA，有助于在项目整个开发生命周期中提供支持。

## 核心概念

### 数据库基本术语

| 术语 | 定义 |
|------|------|
| **数据库（Database）** | 一系列被存储数据的集合，无论其结构或内容如何，大型数据库还包括实例和模式 |
| **实例（Instance）** | 通过数据库软件对数据库执行控制的某一特定存储区域，企业通常同时运行多个实例 |
| **模式（Schema）** | 数据库或实例中包含的数据库对象的子集，用于将对象组织成更易于管理的部分。通常具有所有者和访问列表，可将敏感数据与一般用户群隔离 |
| **节点（Node）** | 作为分布式数据库的一部分，承载计算或存储数据的单个计算机 |
| **数据库抽象（Database Abstraction）** | 使用通用API调用数据库函数，使应用程序可连接多个不同数据库，如ODBC。好处是程序的可移植性，不足是无法使用特定数据库函数 |

### 数据库架构类型

#### 集中式数据库

集中式数据库将所有数据存储在一个系统的一个地方，所有用户都通过该系统访问数据。对于受限数据可能是理想选择，但对需要广泛访问的数据存在风险——一旦系统不可用，用户将无法获得数据。

#### 分布式数据库

分布式数据库使得在大量节点上快速访问数据成为可能，主要特点包括：
- 基于普通商业硬件服务器，通过横向扩展从单服务器到数千台服务器
- 通过数据库管理软件在服务器间高效复制数据
- 自动处理节点故障，系统整体不太可能丧失全部服务能力
- 部分采用MapReduce计算范式进一步提高性能：数据请求被划分为许多小的工作模块，每个模块都可以在集群中任何节点上执行或重复执行

**联邦数据库**：将多个自治的数据库系统映射成一个单一的联邦数据库，各成员数据库保持自治性但共享数据。与非联邦数据库不同，联邦方式并没有将各自治数据库中的数据在物理上整合到一起，而是通过数据互操作将数据联邦视为一个大型对象进行管理（见[[DAMA-DMBOK2-08-数据集成与互操作|数据集成与互操作]]）。

- **松耦合系统**：成员数据库构建自己的联邦模式，用户需直接掌握联邦模式构成，失去了位置透明性
- **紧耦合系统**：各成员数据库共同构建统一的集成联邦模式，用户无须进行数据复制
- 适用场景：企业信息集成、数据虚拟化、模式匹配和主数据管理

**区块链数据库**：联邦数据库的一种，最早为安全地管理不可伪造的金融交易设计，现在也广泛应用于合同管理或健康信息交换等。有两种类型结构：单个记录和区块。每笔交易包含一条记录，每个区块包含一组带时间戳的交易，使用哈希算法确保交易信息不可篡改。一旦产生新的区块，所有旧区块中的哈希值就不再变化，篡改交易行为将被所有用户发现。

#### 云计算平台部署

| 部署方式 | 说明 |
|---------|------|
| **虚拟机镜像** | 用户按时间偏好购买虚拟机实例，上传或使用预装数据库镜像 |
| **数据库即服务（DaaS）** | 平台负责安装和维护数据库，应用所有者按使用付费 |
| **云上托管数据库** | 数据库不作为服务提供，云服务提供商受委托管理数据库 |

DBA需要与网络和系统管理员协调，建立体系化的集成项目方法：
- **标准化/整合**：通过数据整合减少存储位置数量，制定标准规程（识别关键业务数据、设定数据保留期限、配置数据加密和数据复制策略等）
- **服务器虚拟化**：通过虚拟化技术替换或整合服务器设备，降低资金和运营成本、减少能源消耗
- **数据自动化**：包括数据供应、配置、修正、版本管理和合规等一系列自动化任务
- **数据安全**：不仅要关注物理基础设施安全，还要管理云环境下的数据安全（见[[DAMA-DMBOK2-07-数据安全|数据安全]]）

### 数据库处理类型

数据库处理有两种基本类型：ACID和BASE。正如在pH酸碱度范围中酸和碱分别位于对立的两端，ACID和BASE也代表数据库处理两个截然不同的方向。CAP定理可以用于定义一个分布式系统是接近强调一致性的ACID类型，还是更接近强调可用性的BASE类型。

#### ACID模型

ACID缩写词在20世纪80年代初出现，代表数据库事务可靠性不可或缺的4个约束条件：

| 属性 | 说明 |
|------|------|
| **原子性（Atomicity）** | 数据库所有操作要么都执行，要么都不执行。如果事务某部分失败，所有事务都会失败 |
| **一致性（Consistency）** | 数据事务必须始终符合系统定义的所有规则，未完成事务必须回滚 |
| **隔离性（Isolation）** | 一个事务的执行不受其他事务干扰，每个事务独立进行，从而保持数据的完整性和一致性 |
| **持久性（Durability）** | 一旦事务运行，对数据库的更改是永久的、不可撤销的，即使系统发生故障也不会丢失 |

在关系数据库中，ACID技术是最具主导性的工具，通常采用SQL作为接口。

> 注：吉姆·格雷提出了这个概念。Haerder和Reuter（1983）首次创造了ACID这个词。

#### BASE模型

随着数据体量的空前增长和数据类型的不断增多，产生了存储非结构化数据、优化读取负载性能、增加横向扩展灵活性等全新需求，导致数据库技术走向了与ACID相反的方向：

| 属性 | 说明 |
|------|------|
| **基本可用（Basically Available）** | 即使某些节点出现故障，系统仍保证一定程度的数据可用性（虽然数据可能已经过时） |
| **软状态（Soft State）** | 数据处于不断变化状态，响应时不能保证数据是最新的（允许不同节点的数据副本存在数据延时） |
| **最终一致性（Eventual Consistency）** | 数据在所有节点的最终状态是一致的，但非每时每刻都一致 |

在大数据环境中，组织通常使用BASE类型的数据库系统（如大型互联网公司和社交媒体公司），因为这些业务场景不需要实时准确地拿到所有数据。

#### ACID与BASE差异对比

| 维度 | ACID | BASE |
|------|------|------|
| 设计理念 | 强调一致性 | 强调可用性 |
| 适用场景 | 关系数据库、事务处理 | 大数据、实时网络应用 |
| 一致性要求 | 强一致性 | 最终一致性 |
| 可用性 | 相对较低 | 基本可用 |
| 扩展性 | 垂直扩展为主 | 水平扩展为主 |
| 代表技术 | RDBMS（Oracle、MySQL等） | NoSQL（MongoDB、Cassandra等） |

#### CAP定理（布鲁尔定理）

CAP定理是在集中式系统朝着分布式系统发展过程中被提出的（Brewer, 2000）。该定理认为，分布式系统不可能同时满足ACID的所有要求，系统规模越大，满足的要求点越少。因此，分布式系统必须在各种要求之间进行权衡：

- **一致性（Consistency）**：系统必须始终按设计和预期方式运行
- **可用性（Availability）**：系统必须时刻保持可用，对每次请求做出响应
- **分区容错性（Partition Tolerance）**：部分节点故障时系统仍能继续运行并提供服务

CAP定理指出，在任何共享数据的分布式系统里，这三种要求最多同时满足两个，即只能"三选二"。

> 注：[[DAMA-DMBOK2-14-大数据与数据科学|第14章]]将讨论的Lambda架构设计就是该定理的应用——当用户更看重可用性和分区容错性时采用快速流处理路径；当更看重一致性和可用性时采用批处理路径。

### 数据存储介质

| 存储类型 | 特点 | 适用场景 |
|---------|------|---------|
| **SAN磁盘存储** | 稳定持久，可按访问频次使用不同磁盘（低频数据用低速廉价磁盘，高频数据用高性能磁盘）。磁盘阵列可组成SAN，数据直接基于背板光纤传输 | 通用数据存储 |
| **内存数据库（IMDB）** | 从永久存储加载到内存，所有数据处理在内存中进行，响应速度极快。为防止意外宕机具有数据持久性配置功能 | 实时数据分析处理 |
| **列压缩解决方案** | 列式数据库使用压缩技术存储列数据，状态存储为指向状态表的指针，显著压缩主表，降低I/O带宽 | 大量重复数据字段的数据集 |
| **闪存/SSD** | 兼具内存访问速度和磁盘持久性两种优势 | 磁盘的高性价比替代品 |

其他存储方式还包括：虚拟存储区域网络（VSAN）、基于云的存储解决方案、射频识别技术（RFID）、数字钱包、数据中心和私有云/公有云/混合云存储等（见[[DAMA-DMBOK2-14-大数据与数据科学|第14章]]）。

### 数据库类型详述

#### 层次型数据库

最早的数据库模型，用于早期的大型机DBMS，是最严格的结构之一。数据组织成具有强制父子关系的树状结构，每个父级可以有多个子级，但每个子级只有一个父级（一对多关系）。目录树就是层次数据结构的典型例子。XML使用的也是这种层次模型。

#### 关系数据库（RDBMS）

基于集合论和关系代数，数据元素或属性（列）被关联成元组（行），表是具有相同结构的关系集。集合运算（并集、交集和差集）以SQL形式组织或检索数据。要写入数据必须提前知道表结构（写入模式）。关系数据库是面向行的。

支持变体：
- **多维数据库**：允许多维度搜索，用于数据仓库和商务智能。使用多维表达式（MDX）访问数据，它是SQL的一种变体
- **时态数据库**：处理随时间变化的数据，包含两种基本时间：
  - **有效时间**：现实世界中事实或实体真实发生的时间范围
  - **事务时间**：对数据库对象进行操作的时间，记录修改或更新的操作历史
  - 除了这两种时间，还可以有决策时间，此时称为多时态数据库

#### 非关系数据库（NoSQL）

数据存储为简单字符串或完整文件，读取时进行处理的模式（"读取时进行处理的模式"）。非关系数据库提供了一种存储和检索数据的新机制，与传统关系数据库相比使用的一致性模型约束较少。采用这种方法的动机包括设计简单、横向扩展和更好地控制可用性。

**面向列 vs 面向行的对比**：

| 场景 | 更高效的组织方式 | 原因 |
|------|----------------|------|
| 对很多行进行聚合计算 | 面向列 | 只需读取少数列，速度更快 |
| 同时为所有行更新某一列 | 面向列 | 可有效写入某列而不必访问其他列 |
| 获取某一行中的许多列（行较小） | 面向行 | 只需一次磁盘寻道就能检索整行 |
| 同时提供所有行数据时插入新行 | 面向行 | 只需一次磁盘寻道即可写入整行 |

- **面向行的数据库**：非常适用于联机事务处理（OLTP），处理重点是交互式事务
- **面向列的数据库**：适用于联机分析处理（OLAP），如数据仓库，通常需要对海量数据进行少量高度复杂的查询

**主要NoSQL类型详述**：

| 类型 | 特点 | 典型应用 |
|------|------|---------|
| **列式数据库** | 压缩冗余数据，如状态ID列只有唯一值 | 商务智能、OLAP |
| **空间数据库** | 存储和查询几何空间对象，使用空间索引快速查找。支持空间测量、空间函数、空间谓词、几何构造、观察者函数等操作 | GIS、地图服务 |
| **对象/多媒体数据库** | 包含分层存储管理系统和数据对象类集合 | 多媒体应用、CAD/CAM |
| **平面文件数据库** | 行和列存在于单个文件中，不依赖特定应用格式 | Hadoop数据存储、数据传输 |
| **键值对数据库** | 数据项包含键标识符和键值 | 缓存、会话管理 |
| **文档数据库** | 由结构和数据组成的文档集合，可使用XML或JSON结构 | 内容管理 |
| **图数据库** | 关注组成图的节点之间的关系而非节点本身 | 社交网络、知识图谱 |
| **三元组数据库** | 由主-谓-宾组成的数据实体，基于RDF术语 | 语义网、知识管理 |

**三元组数据库分类**：
1. **原生三元组数据库**：从零开始实现，利用RDF数据模型高效存储和访问RDF数据
2. **RDBMS支持的三元组数据库**：通过向现有RDBMS添加RDF描述层来构建
3. **NoSQL三元组数据库**：目前还处在研究阶段，将来可能成为RDF的存储管理器

三元组数据库最适合分类法和词库管理、关联数据集成和知识门户等应用。

### 专用数据库

有些特殊应用需要专用数据库，其管理方式有别于传统关系数据库：

| 应用领域 | 数据库类型 | 说明 |
|---------|-----------|------|
| **CAD/CAM** | 对象数据库 | 计算机辅助设计与制造，大多数嵌入式实时应用也需要对象数据库 |
| **GIS** | 地理空间数据库 | 参考数据至少每年更新一次，用于公用事业（电网、煤气管道等）、电信网络管理或航海导航 |
| **购物车功能** | XML数据库 | 在线零售网站暂时存储客户订单数据，社交媒体数据库可能实时使用这些数据投放广告 |

> 注：这类数据库中的部分数据会被复制到传统的OLTP数据库或数据仓库中。许多供应商应用程序可能使用自己专有的数据库模式。

### 数据生命周期管理

DBA通过设计、实施和使用存储、处理或检索数据的系统，在数据的整个生命周期内维护和确保数据的准确性和一致性。DBA是所有数据库变更的管理者，需要对数据库变更进行精准管理，严格实施并控制变更行为。

数据生命周期管理包括数据获取、迁移、保留、过期和处置的策略和过程。DBA在将数据库变更部署到质量控制和生产环境时，应使用严格可控制、可记录和可审核的流程。在数据库变更时，DBA应制订回滚计划，确保当变更出现问题时能够将数据库还原到先前版本。

| 环节 | 说明 |
|------|------|
| **数据获取** | 从内部系统或外部来源获取数据并加载到数据库 |
| **数据迁移** | 将数据从一种存储类型、格式或系统转移到另一种 |
| **数据保留** | 按照法规和业务要求确定数据保持可用的时间长度 |
| **数据过期** | 数据价值降低，不再值得继续保留 |
| **数据处置** | 将数据从存储介质中完全删除并使其无法恢复 |

### 备份与恢复

#### 备份策略

- **增量备份**：常态进行，仅备份自上次备份后变化的数据
- **全库备份**：定期对每个数据库进行完整备份
- **热备份**：在应用程序运行时进行备份，当备份过程中产生数据更新时，要么前推到完成，要么在重新加载备份时回滚
- **冷备份**：数据库离线时进行备份，如果系统需要持续在线则不可行
- 备份文件应与数据库分开存放到另一个文件系统，并备份到SLA指定的独立存储介质
- 每天备份的数据库副本要存储在远离生产环境的安全场所
- 对于OLTP数据库，事务日志备份频率取决于数据更新速度和数据量

#### 恢复类型

| 恢复类型 | 说明 | 优先级 |
|---------|------|-------|
| **立即恢复** | 能够提前预测并自动解决的问题，如故障切换到备份系统，通过系统设计来立即恢复 | 最高 |
| **关键恢复** | 尽可能快地恢复系统，最小化业务延迟或中断风险 | 高 |
| **非关键恢复** | 等关键恢复完毕再进行恢复 | 低 |

相比关系数据库，文档数据库中的数据更容易恢复。如果从日志而不是完全备份中恢复，关系数据库可能需要更新的目录信息，增大了恢复难度。

#### 业务连续性规划

DBA必须确保所有数据库和服务器都有恢复计划，覆盖以下场景：
1. 物理数据库服务器失效
2. 一个或多个磁盘存储设备失效
3. 单个数据库失效（包括DBMS主数据库、临时存储数据库、事务日志等）
4. 数据库索引或数据页损坏
5. 数据库或日志段文件系统丢失
6. 数据库或事务日志备份文件丢失

DBA要评估每个数据库的重要性，据此确认恢复时的优先级。组织的管理层和业务连续性小组应审查并批准数据恢复计划。恢复计划副本、DBMS所需的所有软件、说明书和安全代码应保存在生产环境之外的另一个安全之处。

**定期恢复测试非常重要**：如果备份不可用或不可读，就没有办法从灾难中恢复任何系统。恢复测试可以在与生产系统相同基础架构和配置的非生产系统副本环境下进行。

### 数据归档

归档是将数据从可立即访问的存储介质迁移到检索性能较低的介质进行存档的过程。归档后的数据可以恢复到原系统供临时使用。稳妥方法包括：
1. 创建辅助存储区域，最好在辅助数据库服务器上
2. 将数据库表分区成可以归档的单元
3. 将不常用数据复制到独立数据库中
4. 创建磁带或磁盘备份
5. 创建数据库作业定期清理不再使用的数据

**归档与技术变更**：当生产系统的技术或架构发生变化时，需同时评估存档的可恢复性：
- 确定是否保留及保留多少历史归档，不再需要的可以销毁
- 技术发生重大变化时，先将存档数据恢复到原系统，一起升级或迁移到新系统，然后重新归档
- 源数据库结构发生更改时，高价值存档数据要先恢复到原数据库，一起更改结构，然后重新归档
- 对不经常访问的存档数据，可保留旧系统的小版本供有限数据访问

> 使用当前技术无法恢复的档案是毫无意义的，保留旧机器以读取无法以其他方式读取的档案既不高效又不具成本效益。

### 数据库容量和增长预测

把数据库想象成一个盒子，数据是盒子中装的水果，管理成本（索引等）是包装材料：
- **容量**：用多大的盒子才能装下所有水果和包装材料
- **增长速度**：有多少水果要放进盒子，放的速度有多快
- **取出速度**：从盒子里要能取出多少水果，取的速度有多快
- **增长预测**：盒子大小随时间是保持不变还是不断扩大
- **归档策略**：水果应该保存多久？蔫掉的水果是否放到另一个盒子长期储存？是否需要放回原盒子？
- **数据清除**：水果腐烂至无法食用时就要清除

### 通用数据库过程

| 过程 | 说明 |
|------|------|
| **归档** | 将不常用数据迁移到成本较低的存储介质，定期进行恢复测试 |
| **容量预测** | 评估当前容量和未来增长，平衡成本和效率 |
| **变更数据捕获（CDC）** | 检测数据变更并确保相关信息被适当记录，通常基于日志的复制，是非侵入性方法。两种方法：数据版本管理（评估时间戳/版本号/状态标识列）和读取日志复制 |
| **数据清除** | 将数据从存储介质完全删除并使其无法恢复，降低成本和风险。维护数据的成本不应超过它产生的价值 |
| **数据复制** | 在多个存储设备上存储相同数据，确保业务连续性 |
| **弹性和恢复** | 系统对错误条件的容忍度和恢复能力 |
| **数据留存** | 数据保持可用的时间长度，需符合法规要求。某些数据出于法律原因需要留存特定时间段，超过规定时间则变成负担 |
| **数据分片** | 将数据库中一部分独立出来进行独立更新，分片通常很小，用户可直接刷新或重写 |

### 数据库弹性

数据库弹性用于衡量系统对错误条件的容忍度。如果数据库能够检测异常并提前中止或能从常见错误（如失控查询）中自动恢复，则认为它是有弹性的。

数据处理错误包括数据加载失败、查询返回失败、ETL错误和其他处理错误。提高弹性的常用方法：
- 及时发现并纠正导致错误的数据
- 监测并忽略导致错误的数据
- 在处理过程中对错误数据和采取的措施进行标记，避免重启流程时重复处理

对于非常关键的数据，DBA需要把数据转移到远程服务器上的另一个数据库副本中，如果当前数据库发生故障，应用程序就可以切换到远程数据库继续处理业务。

### 数据复制模式

| 维度 | 选项 | 说明 |
|------|------|------|
| 复制方式 | 主动复制 | 每个副本重新创建和存储相同数据，无主副本 |
|  | 被动复制 | 先在主副本上创建数据，再复制到其他副本 |
| 扩展方向 | 水平扩展 | 拥有更多的数据副本 |
|  | 垂直扩展 | 将副本放到更远的地理位置 |
| 实现方式 | 日志传送 | 辅助数据库定期接收并应用事务日志副本，可复制到多个辅助数据库 |
|  | 镜像 | 主数据库的更新立即复制到辅助数据库，通常只有一台辅助数据库，成本更高 |

- **多主机复制**：可以先将数据更新提交到任何数据库节点，然后复制到其他节点，但会增加复杂性和成本
- **数据复制透明性**：多个副本的存在对用户或应用程序是透明的，用户无须关心也不必知道使用的是哪个副本
- 选用哪种复制方式取决于数据的重要程度及业务连续时效性要求

### 数据库环境

在系统开发生命周期中，各种环境下都会使用数据库。当系统发生变更时，DBA要介入开发环境的数据结构设计；在测试环境中DBA团队负责实施所有变更；在生产环境中必须严格遵守标准流程和程序。

| 环境类型 | 用途 | 特点 |
|---------|------|------|
| **生产环境** | 实际业务正式运行 | 非常关键，停止运行会导致利润损失并影响客户。从业务角度看才是"真实"环境 |
| **开发环境** | 编写和测试代码 | 生产环境的精简配置版本，配置较少的磁盘空间、CPU和内存。任何系统补丁或更新应首先在此测试 |
| **测试环境** | QAT、集成测试、UAT、性能测试 | 最好与生产环境拥有相同软硬件配置，对性能测试尤为重要。可从生产系统读取数据，但绝不能写入 |
| **沙箱/实验环境** | 验证假设和创新应用 | 只读访问生产数据，用户自行管理。用户拥有完整的CRUD权限，DBA负责构建环境、赋予权限和监控使用 |

**测试环境的细分用途**：
1. **质量保证测试（QAT）**：根据需求进行功能测试
2. **集成测试**：将独立开发的多个模块作为整体系统进行测试
3. **用户验收测试（UAT）**：从用户视角测试系统功能，需编制足够的测试用例
4. **性能测试**：执行大业务量或高复杂度的测试，可在任何时间执行

> 注：如果单个实例的许可成本在允许范围内，沙箱环境可以采用虚拟机（VM）。如果沙箱区域位于生产数据库系统内，必须严格隔离，杜绝沙箱环境的数据回写到生产系统。

## 方法论

### 存储策略选择

#### DBMS选择评估因素

选择战略性DBMS软件时需要考虑的技术因素：
1. 产品架构和复杂性
2. 容量和速度限制，包括数据流量限制
3. 应用类别（事务处理、商务智能、个人办公）
4. 特定功能需求（如是否需要支持时间类数据计算）
5. 硬件平台和操作系统
6. 软件支持工具的可用性
7. 性能基准，包括实时统计数据
8. 可扩展性
9. 软件、内存和存储要求
10. 韧性（错误处理和报告）

还需要考虑的组织因素：
- 企业对技术风险的偏好
- 是否配备训练有素的专业技术人员
- 使用成本（许可费、维护费、计算资源成本）
- 供应商声誉和支持策略、版本更新计划
- 其他客户的使用情况

企业要考虑成本效益原则，产品总费用（包括维护管理费用、软件许可费用和技术支持费用）不应超过该产品带来的经济价值。在真正引进产品并全面投产之前，最好在小型试点项目或POC中对产品进行验证。

### 运营管理流程

#### 管理数据库技术

| 阶段 | 主要活动 | 关键交付物 |
|------|---------|-----------|
| 理解 | 了解业务数据需求，将工具知识与业务需求结合 | 技术需求分析报告 |
| 评估 | 评估DBMS产品，进行概念验证（POC） | 技术选型报告、POC结果 |
| 部署 | 与业务用户和开发人员合作，探索新技术应用 | 部署流程文档 |
| 监控 | 定期备份恢复测试，性能监控，培训计划 | 监控报告、优化建议 |

#### 管理数据库运营

| 阶段 | 主要活动 | 关键交付物 |
|------|---------|-----------|
| 需求理解 | 定义存储需求、识别使用模式、定义访问要求 | 存储需求文档、使用模式分析 |
| 业务连续性规划 | 制定备份策略、恢复计划、灾难恢复方案 | 业务连续性计划、恢复计划 |
| 实例创建 | 安装DBMS、维护多环境版本、管理物理存储 | 实例配置文档、SCM记录 |
| 性能管理 | SLA设置、可用性管理、运行监控、问题修复 | 性能报告、SLA合规报告 |
| 测试数据管理 | 生成和管理测试数据、数据脱敏 | 测试数据集、脱敏规则 |
| 数据迁移 | 数据迁移规划、执行和验证 | 迁移方案、验证报告 |

#### 识别使用模式

数据库有以下几种基本使用模式：
1. 事务处理
2. 大型数据集的数据读写
3. 数据处理时间分布不均匀（如月末压力大、周末压力轻）
4. 数据处理位置分布不均匀（如人口密集地区有更多交易）
5. 数据处理优先级不同（某些部门或批处理有更高优先级）

DBA要能够预测数据库使用模式的变动转换，建立适当的流程应对高峰期（如查询管理或优先级管理）、利用低谷期（将需要大量资源的操作放到低谷期运行）。

#### 数据访问方法

从数据库访问数据可以采用多种标准语言、方法和格式：
- **ACID类型**：SQL、ODBC、JDBC、XQJ、ADO.NET、XML、X Query、XPath和Web Services
- **BASE类型**：C、C++、REST、XML和Java
- 有些标准语言可以将非结构化数据（如HTML或自由文本文件）转换为结构化数据（如XML或SQL）

#### 物理存储管理（SCM流程）

| 步骤 | 说明 |
|------|------|
| **配置识别** | 与数据管理专员、架构师、建模师一起识别配置属性并记录在基线中 |
| **配置变更控制** | 更改配置属性和重新确定基线的流程和审批步骤 |
| **配置状态报告** | 在任何时间点记录和报告每个配置属性的基线 |
| **配置审计** | 物理配置审计（符合设计文档）和功能配置审计（达到性能属性） |

#### 数据库访问控制管理

DBA负责管理数据资产的受控环境：
- **受控环境**：与NSA合作管理网络角色和权限、全天候监控、防火墙管理、补丁管理等
- **物理安全**：基于SNMP的监控、数据审计日志、灾难管理和数据库备份计划
- **监控**：通过对关键服务器不间断的硬件和软件监控确保持续可用
- **控制**：通过访问控制、数据库审计、入侵检测和漏洞评估工具维护数据安全

#### 数据库性能管理

数据库性能取决于可用性和速度两个相互依存的方面。DBA和NSA通过以下方式管理：
1. 设置和调优操作系统及应用程序参数
2. 管理数据库连接，符合基于组织标准和协议的数据库链接策略
3. 持续调优操作系统、网络和事务处理中间件
4. 选择适当的存储设备，使用存储管理软件优化不同存储技术的使用
5. 有效预测数据库容量增长，支持数据生命周期管理活动
6. 提供运营工作负荷和基准，支持SLA管理、收费计算、服务器容量及规划

**数据库可用性管理**：可用性指系统可以用于生产工作的时间百分比。影响可用性的4个因素：
1. **可管理性**：创造和维护数据库运行环境的能力
2. **可恢复性**：在服务中断后重新提供服务的能力
3. **可靠性**：在规定时间内提供约定水平服务的能力
4. **可维护性**：发现问题、诊断原因和维修/解决问题的能力

**导致数据库无法使用的情况**：

| 类别 | 具体情况 |
|------|---------|
| 计划内停机 | 出于维护的考虑、出于升级的考虑 |
| 计划外停机 | 服务器硬件故障、磁盘硬件故障、操作系统故障、DBMS软件故障、数据中心站点故障、网络故障 |
| 应用程序问题 | 安全和授权错误、严重的性能问题、恢复失败 |
| 数据问题 | 数据损坏（缺陷、设计不当或用户失误）、数据库对象丢失、数据丢失、数据复制错误 |
| 人为失误 | 各类人为操作错误 |

**数据库性能低下的常见原因**：
1. **内存分配或竞争**：需要为数据分配缓冲区或缓存
2. **锁定和阻塞**：进程锁定数据库资源（如表或数据页），阻塞其他进程。两个进程可能出现"死锁"
3. **数据库统计信息不准确**：查询优化器依赖的统计数据应经常更新，否则导致查询性能低下
4. **编码不当**：糟糕的SQL代码是最常见的性能不佳原因
5. **低效的复杂表连接**：应使用视图预定义复杂表联接，避免在数据库函数中使用复杂SQL
6. **索引不足**：要充分创建支持查询的索引，但不要在更新频繁的表上创建过多索引
7. **应用程序活动**：应用程序应在独立于DBMS的服务器上运行，避免与数据库竞争资源
8. **服务器过载**：新增数据库可能对现有数据库性能产生不利影响，应创建新服务器或迁移
9. **数据库波动性**：短时间内大量插入和删除操作可能导致统计信息不准确
10. **失控查询**：用户无意中提交占用过多资源的查询，可使用排名或查询管理器阻止或暂停

**问题修复后的措施**：
- 与应用开发人员合作改进和优化数据库代码
- 归档或删除应用程序不再需要的数据
- 对OLTP数据库，可考虑与数据建模人员协商重构受影响部分的数据库模型（需慎重，应先尝试创建视图和索引、重写SQL代码）
- 只读型和分析型数据库例外：去范式化是常规做法，有助于提高性能和访问便捷性

#### 管理测试数据集

软件测试占系统开发成本的近一半。测试数据可以是完全编造的、使用无意义数值生成的、或样本数据（实际生产数据的子集）。如果生产数据包含受保护或受限数据，则必须对样本数据进行**掩码处理**。

测试数据可以以集中或系统的方式生成（功能测试），也可以使用统计或过滤器，或不太集中的方法（大量随机自动化测试）。测试数据可以保存下来短期内重复使用（支持回归测试），也可以使用一次后删除。DBA应监控项目测试数据，确保定期清除过时的测试数据以节约数据库容量。

#### 管理数据迁移

数据迁移是将数据从一种存储类型、格式或计算机系统转移到另一种的过程，同时尽可能减少信息损耗。迁移过程中通常会执行自动化的和手动的数据修复，以提高数据质量、消除冗余或过时信息，并适应新系统的要求。

数据迁移的原因包括：服务器或存储设备的更换或升级、网站整合、服务器维护或数据中心搬迁。大多数实施方式允许以不中断的方式进行迁移。

利用数据迁移技术，存储管理员可以同时完成许多日常任务：
1. 将数据从过度使用的存储设备移至独立环境
2. 根据需要将数据转移到速度更快的存储设备
3. 实施数据生命周期管理策略
4. 将数据从旧存储设备迁移到离线存储或云存储

### 工具与技术

| 工具类别 | 功能 |
|---------|------|
| **数据建模工具** | 自动化建模任务，生成DDL，支持逆向工程，验证命名标准、存储元数据 |
| **数据库监控工具** | 自动监控容量、可用性、缓存性能、用户统计等关键指标，可同时监控多种类型数据库 |
| **数据库管理工具** | 配置、补丁、升级安装、备份恢复、数据库克隆、测试管理、数据清理 |
| **开发人员支持工具** | 连接数据库和执行命令的可视化界面 |

### 最佳实践

1. **识别并执行自动化机会**：缩短开发周期，减少错误和返工
2. **以重用为目标构建数据对象**：避免应用程序与数据库模式紧密耦合
3. **理解并适当运用最佳实践**：在合理理由时灵活应对偏离标准的情况
4. **数据库标准与需求相适应**：SLA既要反映DBA建议，也要考虑开发人员接受度
5. **明确DBA在项目中的角色期望**：在项目定义阶段就引入DBA
6. **在低阶环境下测试**：所有变更先在开发环境测试，再逐级升级到生产环境
7. **所有变更操作脚本化**：避免直接在数据库中更改数据，先将更改放入更新脚本文件在非生产环境测试
8. **采用物理命名标准**：遵循ISO/IEC 11179元数据注册库第5部分——命名和标识原则

## 组织与角色

### 关键角色

#### 数据库管理员（DBA）

DBA是数据管理专业中最常见、最被广泛认可的角色，在数据存储和运营方面发挥主导作用，在数据安全（见[[DAMA-DMBOK2-07-数据安全|第7章]]）和数据建模（见[[DAMA-DMBOK2-05-数据建模与设计|第5章]]）中发挥关键作用。DBA为开发、测试、质量保证和其他特殊数据库环境提供支持。

DBA并非独立执行所有活动，数据管理专员、数据架构师、网络管理员、数据分析师和安全分析师会与其协同工作。

**DBA细分角色**：

| 角色 | 职责 | 详细说明 |
|------|------|---------|
| **生产DBA** | 数据运营管理的主要责任 | 确保数据库性能和可靠性、建立备份恢复机制、建立集群和容错机制、数据归档 |
| **应用程序DBA** | 负责所有环境中的一套或多套数据库 | 为开发人员提供专业服务，专注于特定数据库 |
| **过程和开发DBA** | 审查和管理程序数据库对象 | 管理存储过程、触发器、UDF等程序逻辑；开发DBA专注于数据设计和特殊用途数据库（如沙箱） |
| **网络存储管理员（NSA）** | 关注支撑数据存储阵列的软件和硬件 | 多元化类型的网络存储阵列各有不同需求特性和监控要求 |

**生产DBA的可交付成果**：
1. 生产数据库环境，包括DBMS实例，具有充足资源和容量，配置适当的安全性、可靠性和可用性级别
2. 在生产环境中建立控制数据库变更的机制和流程
3. 建立确保数据可用、完整和可恢复的机制（应对数据丢失或损坏情景）
4. 建立错误监测机制，实时监测和报告数据库、DBMS或数据服务器中出现的错误
5. 提供与SLA相匹配的数据库服务（数据可用性、可恢复性、数据性能等）
6. 建立数据库性能监控机制和流程，及时反映工作负载和数据量的变化

**应用程序DBA的优势与不足**：
- **优势**：被视为应用程序支撑维护团队中不可或缺的成员，通过专注于特定数据库可为开发人员提供更好的专业服务
- **不足**：容易视野受限，难以把握组织的整体数据需求，也容易忽视数据库管理员的通用实践。应与数据分析师、建模师和架构师密切协作

#### 其他协作角色

| 角色 | 协作内容 |
|------|---------|
| 数据管理专员 | 识别配置属性、制定保留策略 |
| 数据架构师 | 制定存储标准、数据复制策略 |
| 数据建模师 | 建立物理数据模型 |
| 网络管理员 | 协调网络和存储基础设施 |
| 安全分析师 | 制定数据安全策略 |

### 组织结构

- 生产DBA通常隶属于生产基础设施或应用程序操作支持组
- 应用程序DBA、过程和开发DBA常整合到应用开发团队中
- 网络存储管理员通常属于基础设施组

## 实施指南

### 就绪评估/风险评估

就绪评估和风险评估围绕两个核心思想展开：

**1. 数据丢失风险**
数据可能会因技术或程序错误或恶意行为而丢失。组织需要制定策略来降低这些风险：
- SLA应制定对数据保护的一般要求，得到详细文档化记录程序的支持
- 网络威胁在不断演变，需要对数据丢失风险进行持续评估
- 建议进行SLA审计和数据审计，以评估和规划风险缓解措施

**2. 与技术准备相关的风险**
NoSQL、大数据、三元组存储等新技术要求IT人员具备相应的技能和经验。许多组织缺乏利用这些新技术所需的技能。DBA、系统工程师和应用开发人员及业务用户必须做好准备，以便在商务智能和其他应用中利用这些技术优势。

### 组织与文化变革

DBA往往难以有效提升自身工作对组织的价值。他们需要认识到数据所有者和数据使用者的真正关心所在，平衡短期和长期的数据需求。

**常见问题**：
- 应用开发人员经常将数据管理视为开发的障碍
- DBA在适应技术变化（如XML、对象和SOA）和新的开发方法（如敏捷开发、XP和Scrum）方面进展缓慢
- 过度强调数据安全、可重复使用性、数据质量对组织的长期价值，在组织中很难被认可

**克服障碍的指导原则**：
1. **主动沟通**：与项目团队保持密切沟通，审查开发团队编写的数据访问代码、存储过程、视图和数据库函数
2. **站在对方的立场上沟通**：与业务人员谈论业务需求和投资回报率，与开发人员讨论面向对象、松散耦合和易于开发的工具
3. **坚持以业务为中心**：应用程序开发的目标是满足业务需求并从项目中获得最大价值
4. **乐于助人**：总是对人说"不"会促使他们忽视标准并另谋他路
5. **不断学习**：评估项目期间遇到的挫折并从中吸取教训

## 方案价值

### 风险与挑战

| 风险类别 | 具体风险 | 影响程度 | 缓解措施 |
|---------|---------|---------|---------|
| **数据丢失** | 技术或程序错误导致数据丢失 | 极高 | 定期备份、恢复测试、SLA审计 |

| **安全威胁** | 网络攻击和数据泄露 | 极高 | 访问控制、加密、持续安全评估 |
| **技术人才短缺** | 缺乏新技术所需的技能和经验 | 高 | 培训计划、交叉培训、外部招聘 |
| **合规风险** | 未遵守数据保留和清除法规 | 高 | 制定清晰的数据留存策略、定期审计 |
| **成本超支** | 存储和许可成本超出预算 | 中 | TCO分析、存储优化、云迁移 |
| **组织文化障碍** | DBA被视为开发障碍 | 中 | 主动沟通、以业务为中心、乐于助人 |

## 数据存储和运营治理

### 度量指标

**数据存储度量指标**：
1. 数据库类型的数量
2. 汇总事务统计
3. 容量指标
4. 已使用的存储容量
5. 存储容器数量
6. 已提交和未提交数据块或页面的数据对象数量
7. 队列中的数据
8. 存储服务使用情况
9. 针对存储服务提出的请求数量
10. 使用数据存储服务应用程序性能的提升

**性能度量指标**：
1. 事务频率和数量
2. 查询性能
3. API服务性能

**运营度量指标**：
1. 数据检索时间的汇总统计信息
2. 备份大小
3. 数据质量评估
4. 可用性

**服务度量指标**：
1. 按类型划分的问题提交、解决和升级数量
2. 问题解决的时间

> 数据库管理员需要与数据架构师和数据质量团队一起讨论采用哪些度量指标。

### 信息资产跟踪

数据存储管理工作还涉及确保组织遵守所有许可协议和监管要求。组织应对软件许可、年度支持费用、服务器租赁协议和其他固定成本等仔细跟踪和按年进行审计。不遵守许可协议会给企业带来严重的财务风险和法律风险。

审计数据可以帮助确定每种类型的技术和产品的总拥有成本（TCO）。组织要定期评估过时、过保、用处不大或成本过高的技术和产品。

### 数据审计和数据验证

**数据审计**：根据既定标准对数据集进行评估，通常是为了调查特定关注点，确定数据存储是否符合合同和方法论要求。

**数据验证**：根据既定验收标准对存储数据进行评估，以确定其质量和可用性。

DBA通过以下方式支持数据审计和数据验证工作：
1. 帮助制定和审查方法
2. 进行初步数据筛选和审查
3. 开发数据监控方法
4. 应用统计、地理统计和生物统计技术优化数据分析
5. 支持采样和分析
6. 审核数据
7. 为数据发现提供支持
8. 担任与数据库管理相关问题的主题领域专家（SME）
