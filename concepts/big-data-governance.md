# 大数据治理

## 定义

大数据治理是在数据的全生命周期内，对其进行管理的原则性方法，目标是确保数据的**安全、及时、准确、易用**。数据生命周期包括：创建→处理→存储→查看→报告和使用→保留和检索→销毁。

## 治理的必要性

| 维度 | 问题描述 |
|------|----------|
| 数据规模 | 数据量增加至传统技术无法处理的量级 |
| 时效性 | 业务与数据使用方对数据延迟更加敏感 |
| 多样性 | 结构化、半结构化、无结构化数据并存 |
| 使用门槛 | 缺乏有效的元数据标签解释 |
| 价值密度 | 海量数据中有价值的信息比例低 |
| 真实性 | 数据质量参差不齐，多拷贝导致不一致 |
| 有效性 | 数据合规性与隐私保护要求；2021年《数据安全法》正式通过 |

## 核心技术架构

### 全链路大数据平台

数据处理的六个阶段：

1. **数据上报** — Log Server、binlog 等原始数据产出
2. **数据采集** — Kafka 消息队列批量/流式采集接入
3. **数据同步** — 跨系统、跨环境的数据同步，支持 HIVE
4. **数据加工** — 离线/实时 ETL 处理，数据清洗与转换
5. **数据分发** — 分发至数仓（DORIS、DRUID、ClickHouse 等）
6. **数据服务** — API 化的数据访问服务，对接报表平台、App分析、AB系统等

### 消息队列与流式计算

- **Kafka** — 极强的集成能力，一套技术满足流式和批式两种处理方式，使业务和开发团队有较好的技术积累
- **Flink** — 流批一体的计算框架，流式处理平台和批处理平台采用统一平台，降低成本和复杂度

技术选型重点：支持流批一体（Kafka + Flink）。

### 数据湖

数据湖是一套数据存储、处理的解决方案，包含流式和批式（离线）。与数仓分层关联，作为数据一站式开发平台的底层支撑。^[raw/articles/data-governance-overview.md]

### 数据API服务

- 隐藏复杂业务逻辑
- 屏蔽数据的差异性
- 访问控制与审计

### 数据资产监控和管理

| 监控维度 | 内容 |
|---------|------|
| 数据使用监控 | 冷热数据管理、安全审计需求 |
| 数据生产质量监控 | SLA定义、全链路监控覆盖 |
| 数据源集成监控 | 数据源状态跟踪、数据质量监控 |
| 数据容量管理和监控 | 数据存储分层管理、脏数据清理 |

### 开源工具：Apache Atlas

Apache Atlas 是开放式元数据管理和治理工具，核心功能包括：

1. **元数据类型&实例** — 类型合一，具有原始属性、复杂属性、对象引用；可继承其他类型；捕获元数据对象详细信息及关系
2. **分类** — 动态创建分类，可包含更多属性
3. **血缘** — API 更新/访问、UI 查看
4. **搜索和发现** — UI 查询、类 SQL 查询

## 成本管理 ^[raw/articles/big-data-governance-day2.md]

### 成本结构

```
大数据引擎
├── 存储引擎：HDFS、HIVE 等
├── 计算引擎：Flink、Hive、MapReduce 等
└── 数据服务引擎：应用引擎

成本组成：存储成本 + 计算成本 + 数据服务成本
```

### 成本公式

**总成本 = 不可控成本 + 可控成本**

不可控成本（基础设施等）与可控成本（存储策略、计算优化等）的分离是成本优化的起点。

### 存储优化策略

- **减表**：清理无用表
- **减副本**：降低冗余副本数
- **减列**：移除无用字段
- **减行**：清理过期数据
- **减分区**：合并或删除过期分区

### 生命周期矩阵

数据分层对应不同生命周期策略：DOG（原始）→ DWD → DWS → TDMC → APP → 临时表 → 开发环境，层级越低保留期越短。

### 治理实施方法

PDCA 循环 + 元数据驱动 + 专项数据治理 + 自驱式数据管理

## 数据规范治理 ^[raw/articles/big-data-governance-day3.md]

### 面临的问题

数据孤岛、数据不一致、数据难用、成本高

### 治理思路

1. 制定模型规范和标准
2. 确定治理目标和方法
3. 元数据驱动治理

### 模型分层规范

**业务数据化 → 数据资产化（面向实事建模） → 数据业务化（面向领域建模）**

### 业界模型建设方法论

- **范式建模** — 第三范式，减少冗余
- **维度建模** — 维表+事实表，大厂常用，常结合范式建模
- **DATA Vault** — 面向企业数据仓库的建模方法
- **Anchor** — 超级规范化建模

### 指标定义规范 ^[raw/articles/big-data-governance-day3.md]

核心原则：**唯一性、可扩展性、易理解**

指标派生路径：
- 数据域 → 维度 → 维度模型 → 统计周期/粒度/业务限定 → **派生指标**
- 数据域 → 业务过程 → 事实模型 → 原子指标 → 衍生指标 → **派生指标**

### 治理体系 ^[raw/articles/big-data-governance-day3.md]

**四大目标**：问题标准化、治理可量化、过程策略化、运营有抓手

**资产健康评估**：
- 模型：问题诊断 → 评估打分
- 质量：问题诊断 → 评估打分
- 成本：问题识别 → 评估打分

结合治理评估体系的运营机制，实现治理收益的持续评估。
