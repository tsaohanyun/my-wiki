# 大数据治理

## 定义

大数据治理是在数据的全生命周期内，对其进行管理的原则性方法，目标是确保数据的**安全、及时、准确、易用**。数据生命周期包括：创建→处理→存储→查看→报告和使用→保留和检索→销毁。

## 治理的必要性

| 维度 | 问题描述 |
|------|----------|
| 数据规模 | 数据量增加至传统技术无法处理的量级 |
| 时效性 | 业务与数据使用方对数据延迟更加敏感 |
| 多样性 | 结构化、半结构化、无结构化数据并存 |
| 使用门槛 | 缺乏有效的元数据标签解释 |
| 价值密度 | 海量数据中有价值的信息比例低 |
| 真实性 | 数据质量参差不齐，多拷贝导致不一致 |
| 有效性 | 数据合规性与隐私保护要求；2021年《数据安全法》正式通过 |

## 核心技术架构

### 全链路大数据平台

数据处理的六个阶段：

1. **数据上报** — Log Server、binlog 等原始数据产出
2. **数据采集** — Kafka 消息队列批量/流式采集接入
3. **数据同步** — 跨系统、跨环境的数据同步，支持 HIVE
4. **数据加工** — 离线/实时 ETL 处理，数据清洗与转换
5. **数据分发** — 分发至数仓（DORIS、DRUID、ClickHouse 等）
6. **数据服务** — API 化的数据访问服务，对接报表平台、App分析、AB系统等

### 消息队列与流式计算

- **Kafka** — 极强的集成能力，一套技术满足流式和批式两种处理方式，使业务和开发团队有较好的技术积累
- **Flink** — 流批一体的计算框架，流式处理平台和批处理平台采用统一平台，降低成本和复杂度

技术选型重点：支持流批一体（Kafka + Flink）。

### 数据湖

数据湖是一套数据存储、处理的解决方案，包含流式和批式（离线）。与数仓分层关联，作为数据一站式开发平台的底层支撑。^[raw/articles/data-governance-overview.md]

### 数据API服务

- 隐藏复杂业务逻辑
- 屏蔽数据的差异性
- 访问控制与审计

### 数据资产监控和管理

| 监控维度 | 内容 |
|---------|------|
| 数据使用监控 | 冷热数据管理、安全审计需求 |
| 数据生产质量监控 | SLA定义、全链路监控覆盖 |
| 数据源集成监控 | 数据源状态跟踪、数据质量监控 |
| 数据容量管理和监控 | 数据存储分层管理、脏数据清理 |

### 开源工具：Apache Atlas

Apache Atlas 是开放式元数据管理和治理工具，核心功能包括：

1. **元数据类型&实例** — 类型合一，具有原始属性、复杂属性、对象引用；可继承其他类型；捕获元数据对象详细信息及关系
2. **分类** — 动态创建分类，可包含更多属性
3. **血缘** — API 更新/访问、UI 查看
4. **搜索和发现** — UI 查询、类 SQL 查询

## 成本管理 ^[raw/articles/big-data-governance-day2.md]

### 成本结构

```
大数据引擎
├── 存储引擎：HDFS、HIVE 等
├── 计算引擎：Flink、Hive、MapReduce 等
└── 数据服务引擎：应用引擎

成本组成：存储成本 + 计算成本 + 数据服务成本
```

### 成本公式

**总成本 = 不可控成本 + 可控成本**

不可控成本（基础设施等）与可控成本（存储策略、计算优化等）的分离是成本优化的起点。

### 存储优化策略

- **减表**：清理无用表
- **减副本**：降低冗余副本数
- **减列**：移除无用字段
- **减行**：清理过期数据
- **减分区**：合并或删除过期分区

### 生命周期矩阵

数据分层对应不同生命周期策略：DOG（原始）→ DWD → DWS → TDMC → APP → 临时表 → 开发环境，层级越低保留期越短。

### 治理实施方法

PDCA 循环 + 元数据驱动 + 专项数据治理 + 自驱式数据管理

## 数据规范治理 ^[raw/articles/big-data-governance-day3.md]

### 面临的问题

数据孤岛、数据不一致、数据难用、成本高

### 治理思路

1. 制定模型规范和标准
2. 确定治理目标和方法
3. 元数据驱动治理

### 模型分层规范

**业务数据化 → 数据资产化（面向实事建模） → 数据业务化（面向领域建模）**

### 业界模型建设方法论

- **范式建模** — 第三范式，减少冗余（详见下方"数据库范式"章节）
- **维度建模** — 维表+事实表，大厂常用，常结合范式建模
- **DATA Vault** — 面向企业数据仓库的建模方法
- **Anchor** — 超级规范化建模

### 数据库范式（Database Normalization）

> 设计关系数据库时，遵从不同的规范要求，设计出合理的关系型数据库，这些不同的规范要求被称为不同的范式。各种范式呈递次规范，越高的范式数据库冗余越小。关系数据库共有六种范式：1NF → 2NF → 3NF → BCNF → 4NF → 5NF，一般工程实践只需满足第三范式（3NF）。

#### 第一范式（1NF）— 原子性

**定义**：关系模式中所有域都应是原子性的，即数据库表的每一列都是不可分割的原子数据项，不能是集合、数组、记录等非原子数据项。

**核心要求**：
- 每个字段值只能是单一值，不可包含多个值
- 不允许有重复的列（属性）
- 每一行必须唯一（需要主键）

**违反示例**：
```
| 员工ID | 姓名 | 联系电话           |
|--------|------|-------------------|
| 001    | 张三 | 13800001111,13900002222 |  ← 一个字段存了两个号码
```

**修正后**：
```
| 员工ID | 姓名 | 联系电话     |
|--------|------|-------------|
| 001    | 张三 | 13800001111 |
| 001    | 张三 | 13900002222 |
```

> 第一范式是关系模式设计的基本要求，几乎所有关系数据库都必须满足。但有些场景（如 NoSQL）会突破 1NF 限制，称为非1NF模型。

#### 第二范式（2NF）— 消除部分函数依赖

**定义**：在 1NF 的基础上，消除非主属性对候选码的部分函数依赖，即所有非主属性必须完全依赖于整个主键。

**核心要求**：
- 必须先满足 1NF
- 每个非主属性必须完全依赖于主键，不能只依赖主键的一部分
- 适用于复合主键的场景

**违反示例**（复合主键：学号+课程号）：
```
| 学号  | 课程号 | 成绩 | 课程名称   | 学分 |
|-------|--------|------|-----------|------|
| S001  | C01    | 90   | 数据库    | 4    |
| S002  | C01    | 85   | 数据库    | 4    |  ← "课程名称""学分"只依赖课程号，不依赖学号
```

**修正后**（拆分为两个表）：
```
选课表（学号 + 课程号 → 成绩）：
| 学号  | 课程号 | 成绩 |
|-------|--------|------|
| S001  | C01    | 90   |

课程表（课程号 → 课程名称、学分）：
| 课程号 | 课程名称 | 学分 |
|--------|----------|------|
| C01    | 数据库   | 4    |
```

> 简而言之：第二范式 = 第一范式 + 属性完全依赖于主键。单列主键的表天然满足 2NF。

#### 第三范式（3NF）— 消除传递函数依赖

**定义**：在 2NF 的基础上，消除非主属性对候选码的传递函数依赖，即任何非主属性不依赖于其他非主属性。

**核心要求**：
- 必须先满足 2NF
- 非主属性之间不能有依赖关系
- 即：非主属性 → 非主属性 的依赖不允许存在

**违反示例**：
```
| 员工ID | 姓名 | 部门编号 | 部门名称 | 部门地址   |
|--------|------|----------|----------|-----------|
| E001   | 张三 | D01      | 研发部   | 3号楼5层  |  ← "部门名称""部门地址"依赖"部门编号"，形成传递依赖
```

传递依赖链：员工ID → 部门编号 → 部门名称/部门地址

**修正后**（拆分为两个表）：
```
员工表：
| 员工ID | 姓名 | 部门编号 |
|--------|------|----------|
| E001   | 张三 | D01      |

部门表：
| 部门编号 | 部门名称 | 部门地址  |
|----------|----------|----------|
| D01      | 研发部   | 3号楼5层 |
```

> 简而言之：第三范式 = 第二范式 + 非主属性不得传递依赖于主属性。即"不要在一张表里存可以通过外键关联查到的信息"。

#### 范式递进关系

```
1NF（原子性）→ 2NF（消除部分依赖）→ 3NF（消除传递依赖）→ BCNF（消除主属性子集依赖）→ 4NF → 5NF
```

| 范式 | 解决的问题 | 核心规则 |
|------|-----------|---------|
| 1NF  | 字段不原子 | 每列不可再分 |
| 2NF  | 部分函数依赖 | 非主属性完全依赖主键 |
| 3NF  | 传递函数依赖 | 非主属性不依赖非主属性 |
| BCNF | 主属性子集依赖 | 每个决定因素都是候选键 |

#### 范式与反范式

范式的优点：减少冗余、保持一致性、避免插入/删除/更新异常。但范式越高，表划分越细，查询时需要更多 JOIN 操作，可能严重影响性能。

**反范式（Denormalization）** 是在满足范式的基础上，为了提升查询性能而适度引入冗余的策略。适用于：
- 读多写少的场景
- 对查询响应时间敏感的业务
- 避免多表 JOIN 的性能开销

> 实践建议：数据库设计先满足 3NF，再根据性能需求在特定场景下做适度反范式。

### 指标定义规范 ^[raw/articles/big-data-governance-day3.md]

核心原则：**唯一性、可扩展性、易理解**

指标派生路径：
- 数据域 → 维度 → 维度模型 → 统计周期/粒度/业务限定 → **派生指标**
- 数据域 → 业务过程 → 事实模型 → 原子指标 → 衍生指标 → **派生指标**

### 治理体系 ^[raw/articles/big-data-governance-day3.md]

**四大目标**：问题标准化、治理可量化、过程策略化、运营有抓手

**资产健康评估**：
- 模型：问题诊断 → 评估打分
- 质量：问题诊断 → 评估打分
- 成本：问题识别 → 评估打分

结合治理评估体系的运营机制，实现治理收益的持续评估。
