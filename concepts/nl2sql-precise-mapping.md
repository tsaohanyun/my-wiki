# NL2SQL 精确映射

## 定义

NL2SQL 精确映射是一种**受约束的 Text2SQL 系统架构**，核心思想是不让 LLM 自由生成 SQL，而是通过多层管道将自然语言逐步转化为可执行且可信的 SQL 查询。与 [[text2kpi-optimization]] 的 NL2API 路线不同，NL2SQL 面向的是直接查询数据库的场景。

```
❌ 用户问题 → LLM → SQL（自由生成，不可控）
✅ 用户问题 → 语义解析 → Schema召回 → 术语映射 → SQL生成 → SQL校验 → 执行反馈
```

一句话概括：**不要做"自由生成 SQL"，要做"受约束的语义映射系统"。** ^[raw/articles/nl2sql-precise-mapping.md]

## "精准"的四个层次

| 层次 | 含义 | 示例 |
|------|------|------|
| 语义理解准 | 正确解析意图、指标、维度、过滤 | "上个月华东区销量前10客户" → 时间+区域+指标+排序+数量 |
| 字段映射准 | 业务词准确映射到数据库字段 | "销量" → `sales_amt` 而非 `order_amount` |
| SQL语法准 | 适配目标方言（MySQL/Hive/SparkSQL等） | 日期函数、limit 语法差异 |
| 结果可信 | SQL 能跑、结果不歪、不误查不漏查 | 执行成功 + 业务校验通过 |

## 六层架构 ^[raw/articles/nl2sql-precise-mapping.md]

### 第一层：语义解析层

将自然语言抽成结构化意图 JSON（metric / dimensions / filters / sort / limit）。**先做业务语义理解，再做 SQL 生成**，准确率显著提升。

### 第二层：Schema 检索层

不要把全库 schema 喂给模型（300张表会导致上下文爆炸、误选字段）。先做 **RAG for SQL**，召回相关表、字段、注释、指标口径、join 关系。

### 第三层：术语映射层

这是"精准"的关键。建立**业务语义字典**，包含：

- 同义词与别名（"销量"="销售数量"= `sum(sale_qty)`）
- 指标定义（含 SQL 表达式和所属表）
- 过滤维度定义（维度值映射，如"华东"→ `region_name = '华东区'`）
- 时间口径（"上月"→ 动态时间表达式）
- 枚举值映射

### 第四层：受约束 SQL 生成层

不给模型自由发挥，严格约束：
1. 只允许使用召回到的表和字段
2. 必须遵守 join 路径
3. 指标必须从指标字典选取
4. 时间条件必须标准化
5. 输出固定 JSON 中间结构，再由程序渲染 SQL

### 第五层：SQL 校验层

| 校验类型 | 工具/方法 |
|----------|-----------|
| 语法校验 | sqlglot / Apache Calcite / JSQLParser |
| Schema 校验 | 表存在性、字段存在性、join 合法性、聚合与 group by 匹配 |
| 业务规则校验 | 指标口径一致性、禁止 `select *` |
| 风险校验 | 无 where 禁全表扫描、敏感表拦截、delete/update 默认禁用 |

### 第六层：执行反馈层

SQL 执行失败后，将错误信息 + 当前 schema + 原始问题回传模型做修正（**self-correction**），形成纠错闭环。

## 五个关键手段 ^[raw/articles/nl2sql-precise-mapping.md]

### 手段一：规则化抽取优先

时间表达式、排序、TopN、聚合词、比较词等**规则稳定的内容优先规则抽取**，比纯 LLM 更准。

### 手段二：给字段补业务描述

每个字段补充：表说明、字段中文名、字段含义、枚举值解释、示例数据、join 键说明。对准确率提升非常明显。

### 手段三：建立指标层而非裸表

类似 BI 的语义模型层——指标定义 + 维度定义 + 时间口径 + join 关系 + 可用过滤条件。模型在"指标语义层"上生成查询，而非在裸表上拼凑。

### 手段四：Few-shot 按领域分类

按业务域（销售/库存/财务/用户）组织示例，每条含：用户问题 + 相关表 + 映射理由 + 标准 SQL。

### 手段五：歧义显式暴露

精准系统不默认乱猜，输出歧义标记（如"指标未明确：订单数/订单金额"），或配置默认策略。

## 工程方案

```
┌──────────────────────────────────────────────────┐
│                  元数据中心                        │
│  表名/字段/中文解释/主外键/join/指标/同义词/敏感标记  │
│  存储：MySQL / ES / 向量库 / YAML配置中心          │
└─────────────────┬────────────────────────────────┘
                  ↓
┌──────────────────────────────────────────────────┐
│                  检索模块                         │
│  BM25关键词 + 向量召回 + 混合召回 + rerank         │
└─────────────────┬────────────────────────────────┘
                  ↓
┌──────────────────────────────────────────────────┐
│            NL2DSL → DSL2SQL                       │
│  LLM生成DSL → 程序化渲染SQL（可控/可校验/易debug）  │
└─────────────────┬────────────────────────────────┘
                  ↓
┌──────────────────────────────────────────────────┐
│          校验与修复 → 结果解释                      │
│  SQL Parser / Schema校验 / explain / 错误重试      │
│  向用户解释：指标、过滤条件、所用表、映射理由         │
└──────────────────────────────────────────────────┘
```

## MVP 路径

1. **单域起步** — 先只支持一个业务域（如销售分析）
2. **整理资产** — 表结构说明 + 指标字典 + 20~50 条高质量问句-SQL 样本
3. **跑通管道** — 用户问题 → 检索 schema → LLM 生成 DSL → 渲染 SQL → 校验 → 返回
4. **数据驱动迭代** — 收集用户原问题、生成 SQL、执行结果、用户修正记录

## 五个常见坑

| # | 坑 | 后果 |
|---|----|------|
| 1 | 全库 schema 喂模型 | 上下文太大、误选表字段、准确率下降 |
| 2 | 无统一指标口径 | "销售额""GMV""订单金额"混淆 |
| 3 | 直接生成 SQL 不做中间结构 | 出错难定位是理解错/映射错/拼接错 |
| 4 | 无结果校验 | SQL 看似正确但业务上错误 |
| 5 | 缺中文业务别名体系 | 最常错的是业务词到字段的映射 |

## 技术选型

| 模块 | 推荐方案 |
|------|----------|
| 后端框架 | Spring Boot（API/路由/权限/审计） |
| LLM 推理 | Python 服务（推理 / rerank / embedding） |
| 缓存 | Redis（schema 与映射结果） |
| 元数据存储 | MySQL / PostgreSQL |
| 关键词召回 | Elasticsearch |
| 向量召回 | Milvus / pgvector |
| SQL 解析校验 | sqlglot / Apache Calcite |
| DSL 渲染 SQL | Jinja2 / 程序模板 |
| 数据治理 | dbt 或自定义语义层 |
