# 系统设计面试

## 概述

系统设计面试考察候选人设计大规模、可扩展系统的能力。这类面试通常持续 45-60 分钟，要求你在白板/画图工具上从零设计一个完整系统。

---

## 一、系统设计框架（RESHADED）

### Step 1: 需求澄清（Requirements）

**功能性需求：**
- 系统需要做什么？
- 核心功能有哪些？

**非功能性需求：**
- 用户规模（DAU/MAU）
- 读写比例
- 延迟要求
- 一致性要求（强一致 vs 最终一致）
- 可用性要求

> 💡 **技巧：** 在面试前 5 分钟内完成需求澄清，面试官会提供更多上下文。

### Step 2: 容量估算（Estimation）

**估算方法：**

```
假设：
- DAU = 1亿
- 每用户每天读 50 次，写 10 次
- 读 QPS = 1亿 × 50 / 86400 ≈ 57,870 ≈ 60K QPS
- 写 QPS = 1亿 × 10 / 86400 ≈ 11,574 ≈ 12K QPS
- 峰值 = 平均 × 3 ≈ 180K 读 QPS, 36K 写 QPS

存储估算：
- 每条记录 1KB
- 每天新增 = 1亿 × 10 × 1KB = 1TB/天
- 每年 = 365TB（考虑冗余 × 3 = 约 1PB）

带宽估算：
- 读带宽 = 60K × 1KB = 60MB/s
- 写带宽 = 12K × 1KB = 12MB/s
```

### Step 3: API 设计

定义关键 API 接口：

```
POST /api/v1/tweet  → 创建推文
GET  /api/v1/timeline → 获取时间线
GET  /api/v1/feed    → 获取推荐流
POST /api/v1/follow  → 关注用户
```

### Step 4: 数据模型设计

**选择存储：**
- 关系型数据库（MySQL/PostgreSQL）：强一致、事务
- NoSQL（MongoDB/Cassandra）：水平扩展、灵活Schema
- 缓存（Redis）：低延迟读取
- 搜索引擎：全文检索
- 对象存储（S3）：图片/视频等大文件

### Step 5: 高层架构图

画出核心组件及数据流：

```
客户端 → CDN → 负载均衡 → API网关 → [应用服务集群] 
                                          ↓
                              [缓存层] ← [消息队列]
                                          ↓
                              [主从数据库] + [搜索引擎] + [对象存储]
```

### Step 6: 详细设计

深入关键模块：
- 分库分表策略
- 缓存策略（Cache Aside / Write Through / Write Behind）
- 消息队列选型与使用
- 数据一致性方案
- 搜索与推荐

### Step 7: 可扩展性与可靠性

- 水平扩展
- 数据分片
- 读写分离
- 容灾与故障转移
- 监控与告警

### Step 8: 识别瓶颈与权衡

- 单点故障（SPOF）
- 数据一致性 vs 可用性（CAP定理）
- 延迟 vs 成本
- 简单性 vs 灵活性

## 二、经典系统设计题

### 2.1 设计短链服务（URL Shortener）

**关键点：**
- 生成策略：自增ID + Base62 编码 vs 哈希
- 重定向：301（缓存） vs 302（可统计）
- 存储选择：MySQL（自增ID）vs NoSQL
- 扩展：分片（按ID范围/哈希分片）

**架构概览：**
```
用户 → 负载均衡 → Web服务器 → ID生成服务 → Base62编码
                                    ↓
                                MySQL/Redis
```

### 2.2 设计 Twitter / 新鲜事系统（News Feed）

**关键点：**
- Feed 流模型：推模式 vs 拉模式 vs 推拉结合
- 推模式：写时扩散到粉丝收件箱（适合读多写少）
- 拉模式：读时实时聚合（适合大V不推送）
- 推拉结合：普通用户推，大V拉

**核心组件：**
```
推文服务 → 消息队列 → 扇出服务 → [用户收件箱(Redis)]
                                      ↓
时间线服务 ← [推文存储] ← [粉丝关系存储]
```

### 2.3 设计消息系统（如 WhatsApp/微信）

**关键点：**
- 长连接管理（WebSocket / 自定义协议）
- 消息投递保证（At-least-once）
- 在线状态管理
- 消息存储与同步
- 端到端加密

### 2.4 设计 KV 存储 / 缓存系统

**关键点：**
- 一致性哈希做数据分布
- 虚拟节点解决数据倾斜
- 副本机制（NWR 模型 / Quorum）
- Gossip 协议做节点发现
- 数据修复（Read Repair / Anti-Entropy）

### 2.5 设计限流器（Rate Limiter）

**算法对比：**

| 算法 | 优点 | 缺点 |
|-----|------|------|
| 固定窗口 | 简单 | 临界突刺 |
| 滑动窗口 | 平滑 | 内存较高 |
| 令牌桶 | 允许突发 | 需调参 |
| 漏桶 | 匀速 | 不允许突发 |

**分布式限流：**
- Redis + Lua 脚本原子操作
- 滑动窗口日志 / 计数器

### 2.6 设计文件存储系统（如 Google Drive / Dropbox）

**关键点：**
- 文件分块上传（Chunk Upload）
- 秒传（内容哈希去重）
- 增量同步
- CDN 分发
- 权限控制

### 2.7 设计搜索引擎

**关键点：**
- 爬虫系统
- 倒排索引
- 分词与同义词
- 排序算法（TF-IDF / PageRank / 机器学习排序）
- 分布式索引（分片 + 副本）

### 2.8 设计电商系统

**关键点：**
- 商品目录与搜索
- 库存管理（防超卖）
- 订单系统（分布式事务）
- 支付系统（幂等性）
- 秒杀系统（限流 + 缓存 + 异步）

### 2.9 设计排行榜系统（Leaderboard）

**关键点：**
- Redis Sorted Set
- 分区排行榜
- 实时更新与定期快照
- 近似排名（近似算法）

### 2.10 设计评论系统

**关键点：**
- 评论树结构存储（邻接表 / 路径枚举 / 嵌套集）
- 分页策略（游标分页 vs 偏移分页）
- 热门排序算法
- 反垃圾

## 三、核心概念速查

### 3.1 CAP 定理

| 选择 | 含义 | 适用场景 |
|-----|------|---------|
| CP | 一致性 + 分区容忍 | 金融系统 |
| AP | 可用性 + 分区容忍 | 社交网络 |
| CA | 一致性 + 可用性 | 单机系统 |

### 3.2 一致性模型

- **强一致：** 读操作总是返回最新写入值
- **最终一致：** 经过一段时间后最终一致
- **因果一致：** 因果关系的操作有序
- **读己写：** 用户总能读到自己的写入

### 3.3 分片策略

| 策略 | 优点 | 缺点 |
|-----|------|------|
| 范围分片 | 范围查询高效 | 热点问题 |
| 哈希分片 | 均匀分布 | 范围查询困难 |
| 一致性哈希 | 增删节点影响小 | 实现复杂 |
| 目录服务 | 灵活 | 元数据服务单点 |

### 3.4 缓存模式

| 模式 | 描述 | 适用场景 |
|-----|------|---------|
| Cache Aside | 先查缓存，未命中查DB并回填 | 通用 |
| Read Through | 缓存代理读DB | 读密集 |
| Write Through | 同时写缓存和DB | 数据一致要求高 |
| Write Behind | 异步写DB | 写密集，容忍延迟 |

### 3.5 常用数据

| 指标 | 数值 |
|-----|------|
| 单机 QPS | ~1,000 (普通应用) |
| Redis QPS | ~100,000 |
| MySQL QPS | ~5,000 (读) |
| 网络延迟(同城) | ~1ms |
| 网络延迟(跨国) | ~100-200ms |
| SSD 随机读 | ~16,000 IOPS |
| 1Gbps 带宽 | ~125MB/s |

## 四、面试技巧

### 4.1 时间分配（45分钟）

| 阶段 | 时间 | 内容 |
|-----|------|------|
| 需求澄清 | 5 min | 功能与非功能需求 |
| 容量估算 | 5 min | QPS、存储、带宽 |
| API设计 | 5 min | 核心接口定义 |
| 架构设计 | 10 min | 高层架构图 |
| 详细设计 | 15 min | 深入关键模块 |
| 权衡讨论 | 5 min | 瓶颈与改进 |

### 4.2 沟通技巧

1. **主动提问：** 不要假设需求，问清楚
2. **边画边说：** 使用画图工具，保持可视化
3. **先广后深：** 先画高层架构，再深入细节
4. **说明权衡：** 每个设计决策都要说为什么
5. **承认局限：** 不知道就说不知道，不要瞎编

### 4.3 常见加分项

- 主动讨论监控和告警
- 考虑安全性和权限控制
- 提到容量规划和成本
- 讨论可测试性和可运维性
- 提出渐进式改进方案
