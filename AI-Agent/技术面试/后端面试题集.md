# 后端面试题集

## 概述

本文档整理后端开发面试中的高频题目，涵盖数据库、缓存、消息队列、微服务架构和分布式系统等核心领域。

## 一、数据库

### 1.1 MySQL 索引

**索引数据结构：B+ 树**

B+ 树特点：
- 非叶子节点只存储索引，不存储数据
- 叶子节点存储所有数据，且通过链表相连
- 查询性能稳定（每次查询都需要走到叶子节点）
- 范围查询高效（叶子节点有序链表）

**为什么不用 B 树 / 红黑树 / Hash？**

| 结构 | 问题 |
|-----|------|
| B 树 | 非叶子节点也存数据，扇出小，树更高 |
| 红黑树 | 二叉树，树太高，磁盘 IO 多 |
| Hash | 不支持范围查询和排序 |

**聚簇索引 vs 非聚簇索引：**

| 类型 | 特点 | InnoDB | MyISAM |
|-----|------|--------|--------|
| 聚簇索引 | 叶子节点存完整数据行 | 主键索引 | ❌ |
| 非聚簇索引 | 叶子节点存主键值（需要回表） | 二级索引 | 所有索引 |

**联合索引与最左前缀原则：**

```sql
-- 联合索引 (a, b, c)
-- ✅ 能用到索引
WHERE a = 1
WHERE a = 1 AND b = 2
WHERE a = 1 AND b = 2 AND c = 3

-- ❌ 不能用到索引
WHERE b = 2
WHERE b = 2 AND c = 3
WHERE a = 1 AND c = 3  -- 只能用 a
```

**索引失效场景：**
- `WHERE` 中对索引列使用函数/计算
- `LIKE '%xxx'`（左模糊）
- 隐式类型转换
- `OR` 连接的条件中有非索引列
- `NOT IN`, `!=`, `IS NOT NULL`（部分场景）

### 1.2 MySQL 事务与锁

**ACID 特性：**

| 特性 | 含义 | 实现机制 |
|-----|------|---------|
| 原子性 | 事务要么全部成功要么全部失败 | undo log |
| 一致性 | 事务前后数据一致 | 应用层 + 其他三个特性 |
| 隔离性 | 并发事务互不干扰 | 锁 + MVCC |
| 持久性 | 提交后数据永久保存 | redo log |

**隔离级别：**

| 隔离级别 | 脏读 | 不可重复读 | 幻读 |
|---------|------|-----------|------|
| Read Uncommitted | ✅ | ✅ | ✅ |
| Read Committed | ❌ | ✅ | ✅ |
| Repeatable Read（默认） | ❌ | ❌ | ❌（InnoDB） |
| Serializable | ❌ | ❌ | ❌ |

**MVCC（多版本并发控制）：**
- 每行数据有隐藏字段：`trx_id`（事务ID）、`roll_pointer`（指向 undo log）
- Read View 在事务开始时创建，决定能看到哪些版本
- RC 级别：每次 SELECT 都创建新的 Read View
- RR 级别：事务开始时创建一次 Read View

**锁的分类：**

```
按粒度：
  - 表锁：开销小，加锁快，并发低
  - 行锁：开销大，加锁慢，并发高
  - 页锁：介于两者之间

按类型：
  - 共享锁（S锁）：读锁，SELECT ... LOCK IN SHARE MODE
  - 排他锁（X锁）：写锁，SELECT ... FOR UPDATE
  
按算法：
  - Record Lock：锁定索引记录
  - Gap Lock：锁定索引间隙（防止幻读）
  - Next-Key Lock：Record + Gap（左开右闭）
```

### 1.3 SQL 优化

**慢查询优化步骤：**
1. `EXPLAIN` 分析执行计划
2. 检查 `type` 字段（system > const > eq_ref > ref > range > index > ALL）
3. 检查是否使用索引（`key` 字段）
4. 检查扫描行数（`rows` 字段）
5. 优化索引 / 重写 SQL

**常见优化技巧：**
- 避免 `SELECT *`，只查需要的列
- 大分页优化：`WHERE id > last_id LIMIT 10` 代替 `LIMIT 100000, 10`
- `JOIN` 时小表驱动大表
- 避免 `OR`，可用 `UNION ALL` 替代
- 批量插入代替循环单条插入

### 1.4 分库分表

**垂直拆分：**
- 垂直分库：按业务模块拆分数据库
- 垂直分表：将宽表拆分为多个窄表

**水平拆分：**
- 水平分库：同一个表的数据分散到不同数据库
- 水平分表：同一个表的数据分散到多张表

**分片策略：**
- 取模分片：`user_id % N`
- 范围分片：按 ID/时间范围
- 一致性哈希：增删节点影响小

**分库分表后的问题：**
- 跨库 JOIN → 应用层组装 / 冗余字段
- 分布式事务 → Saga / TCC / 本地消息表
- 全局唯一 ID → 雪花算法（Snowflake）
- 跨库分页 → 各库查询后合并（复杂）

## 二、Redis 缓存

### 2.1 Redis 数据结构与应用

| 类型 | 底层结构 | 典型应用 |
|-----|---------|---------|
| String | SDS | 缓存、计数器、分布式锁 |
| Hash | ziplist / hashtable | 对象存储 |
| List | quicklist（ziplist + 链表） | 消息队列、最新列表 |
| Set | intset / hashtable | 去重、交集/并集 |
| ZSet | ziplist / skiplist + dict | 排行榜、延迟队列 |
| Stream | radix tree | 消息队列（支持消费者组） |

### 2.2 缓存三大问题

| 问题 | 描述 | 解决方案 |
|-----|------|---------|
| 缓存穿透 | 查询不存在的数据，穿透到DB | 布隆过滤器 / 缓存空值 |
| 缓存击穿 | 热点 Key 过期，大量请求涌入DB | 互斥锁 / 热点 Key 永不过期 |
| 缓存雪崩 | 大量 Key 同时过期 | 过期时间加随机值 / 多级缓存 |

### 2.3 缓存一致性策略

**Cache Aside（旁路缓存）——最常用：**
```
写：先更新DB，再删除缓存
读：先查缓存，未命中查DB并回填
```

**延迟双删：**
```
1. 删除缓存
2. 更新DB
3. 延迟N毫秒后再删除缓存
```

**为什么是删除缓存而不是更新缓存？**
- 避免并发写入导致的数据不一致
- 懒加载思想，减少不必要的缓存更新
- 部分缓存值计算复杂

### 2.4 Redis 持久化

| 方式 | 原理 | 优点 | 缺点 |
|-----|------|------|------|
| RDB | 定时快照（fork 子进程） | 文件小、恢复快 | 可能丢数据 |
| AOF | 追加命令日志 | 数据安全 | 文件大、恢复慢 |
| 混合 | RDB + AOF 增量 | 兼顾两者 | 配置复杂 |

### 2.5 Redis 分布式锁

```lua
-- 加锁（原子操作）
-- SET key value NX PX 30000

-- 解锁（Lua 脚本保证原子性）
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
```

**注意事项：**
- 设置过期时间防止死锁
- value 设为唯一标识（UUID），解锁时验证
- 锁续期：Redisson 的 WatchDog 机制
- Redlock：多个 Redis 实例投票（争议较大）

## 三、消息队列

### 3.1 消息队列核心作用

1. **异步解耦：** 生产者不需要等待消费者处理
2. **削峰填谷：** 高峰期请求积压在队列中，消费者按自己的节奏处理
3. **系统解耦：** 生产者和消费者不需要直接依赖

### 3.2 主流 MQ 对比

| 特性 | Kafka | RocketMQ | RabbitMQ |
|-----|-------|----------|----------|
| 吞吐量 | 极高（百万级/s） | 高（十万级/s） | 中（万级/s） |
| 延迟 | 毫秒级 | 毫秒级 | 微秒级 |
| 可靠性 | 高 | 非常高 | 高 |
| 消息顺序 | 分区内有序 | 支持严格顺序 | 不保证 |
| 事务消息 | ❌ | ✅ | ❌ |
| 适用场景 | 大数据/日志流 | 金融/电商 | 企业应用 |

### 3.3 消息可靠性

**三个保证阶段：**

1. **生产端不丢：**
   - 同步发送 + 确认机制（ACK）
   - 失败重试
   - 事务消息

2. **Broker 不丢：**
   - 消息持久化（磁盘刷盘）
   - 主从复制
   - ACK 确认机制

3. **消费端不丢：**
   - 手动 ACK（处理完成后再确认）
   - 幂等消费

### 3.4 消息幂等性

**重复消息原因：** 网络重传、消费失败重试、生产端重试

**去重方案：**
- 数据库唯一索引
- Redis 去重（消息ID + TTL）
- 状态机校验（业务状态判断）
- 乐观锁（版本号）

### 3.5 消息积压处理

**紧急处理：**
1. 增加消费者实例
2. 临时增加消费者线程数
3. 如果消息可以丢弃，设置 TTL 过期

**根本解决：**
1. 优化消费者处理逻辑
2. 消息分类，不同优先级不同处理
3. 异步处理 + 批量消费

## 四、微服务

### 4.1 微服务架构核心组件

```
客户端 → API 网关 → [服务注册发现] → 微服务集群
                        ↑
              [配置中心] [链路追踪] [监控告警]
```

| 组件 | 职责 | 常见实现 |
|-----|------|---------|
| 服务注册与发现 | 服务上下线自动感知 | Eureka, Nacos, Consul, etcd |
| API 网关 | 统一入口、路由、鉴权、限流 | Kong, APISIX, Spring Cloud Gateway |
| 配置中心 | 集中配置管理 | Nacos, Apollo, Spring Cloud Config |
| 服务调用 | 服务间通信 | REST, gRPC, Dubbo |
| 负载均衡 | 请求分发 | Ribbon, 客户端负载均衡 |
| 熔断降级 | 故障隔离 | Hystrix, Sentinel, Resilience4j |
| 链路追踪 | 请求链路可视化 | Jaeger, Zipkin, SkyWalking |
| 消息总线 | 配置广播 | Spring Cloud Bus |

### 4.2 服务注册与发现

**CP vs AP：**
- **CP（如 etcd, Consul, Zookeeper）：** 保证一致性，注册数据强一致
- **AP（如 Eureka, Nacos AP 模式）：** 保证可用性，允许短暂数据不一致

**Eureka 自我保护机制：**
- 当丢失过多心跳时（>15%），进入保护模式
- 不再剔除服务实例，防止网络分区导致的误剔除

### 4.3 熔断与降级

**熔断器三态：**

```
Closed（正常） → 错误率超阈值 → Open（熔断）
                                      ↓ 等待恢复时间
                              Half-Open（半开探测）
                                ↓        ↓
                          成功→Closed   失败→Open
```

**关键参数：**
- 错误率阈值（如 50%）
- 熔断持续时间（如 5 秒）
- 半开状态探测请求数（如 3 个）

**降级策略：**
- 返回默认值
- 返回缓存数据
- 简化逻辑（如关闭推荐，只返回核心数据）

## 五、分布式系统

### 5.1 分布式事务

| 方案 | 一致性 | 性能 | 复杂度 | 适用场景 |
|-----|-------|------|-------|---------|
| 2PC | 强一致 | 低 | 中 | 数据库层面 |
| TCC | 强一致 | 中 | 高 | 金融交易 |
| Saga | 最终一致 | 高 | 中 | 长事务 |
| 本地消息表 | 最终一致 | 高 | 低 | 异步场景 |
| 事务消息 | 最终一致 | 高 | 中 | MQ 支持的场景 |

**TCC 流程：**
```
Try（预留资源） → Confirm（确认提交） / Cancel（回滚释放）
```

**Saga 模式：**
```
正向操作: T1 → T2 → T3 → T4
补偿操作: C1 ← C2 ← C3 ← C4（某步失败时逆序补偿）
```

### 5.2 分布式 ID

| 方案 | 原理 | 优缺点 |
|-----|------|-------|
| UUID | 随机生成 | 无序、占空间大 |
| 数据库自增 | `AUTO_INCREMENT` | 单点瓶颈 |
| 号段模式 | 批量取号 | 需要数据库支持 |
| Snowflake | 时间戳+机器ID+序列号 | 时钟回拨问题 |
| Redis INCR | 原子自增 | 依赖 Redis |

**Snowflake 结构（64 bit）：**
```
| 1 bit 符号位 | 41 bit 时间戳 | 10 bit 机器ID | 12 bit 序列号 |
```

### 5.3 一致性哈希

**解决的问题：** 节点增删时最小化数据迁移

**原理：**
1. 将哈希空间组织成环（0 ~ 2³²-1）
2. 节点和数据都映射到环上
3. 数据顺时针找到最近的节点
4. 虚拟节点解决数据倾斜

```
节点增删时，只影响相邻节点之间的数据段
普通取模: 增删节点 → 几乎所有数据要重新分配
一致性哈希: 增删节点 → 只影响 1/N 的数据
```

### 5.4 Raft 共识算法

**核心机制：**
- **Leader 选举：** 随机超时选举，获得多数票成为 Leader
- **日志复制：** Leader 接收写请求 → 复制到 Follower → 多数确认后提交
- **安全性：** 已提交的日志不会丢失

**三个角色：**
- **Leader：** 处理所有客户端请求
- **Follower：** 被动接收 Leader 的请求
- **Candidate：** 选举中的候选者

### 5.5 分布式锁对比

| 实现 | 优点 | 缺点 |
|-----|------|------|
| Redis（SETNX） | 性能高 | 主从切换可能丢锁 |
| Zookeeper | 可靠性高 | 性能较低 |
| etcd | 性能与可靠性兼顾 | 运维成本 |
| Redisson | 功能完善 | 依赖 Redis |

## 六、设计模式与编码

### 6.1 常考设计模式

| 模式 | 类型 | 应用场景 |
|-----|------|---------|
| 单例模式 | 创建型 | 全局唯一实例（配置、连接池） |
| 工厂模式 | 创建型 | 对象创建解耦 |
| 策略模式 | 行为型 | 算法切换（支付方式选择） |
| 观察者模式 | 行为型 | 事件通知（消息总线） |
| 装饰器模式 | 结构型 | 功能扩展（IO 流） |
| 代理模式 | 结构型 | AOP、远程代理 |
| 责任链模式 | 行为型 | 请求过滤（中间件链） |
| 模板方法 | 行为型 | 流程固定，步骤可变 |

### 6.2 单例模式（双检锁）

```java
public class Singleton {
    private static volatile Singleton instance;

    private Singleton() {}

    public static Singleton getInstance() {
        if (instance == null) {
            synchronized (Singleton.class) {
                if (instance == null) {
                    instance = new Singleton();
                }
            }
        }
        return instance;
    }
}
// volatile 防止指令重排序（new 操作非原子）
```

## 七、面试高频综合题

### Q1: 如何设计一个秒杀系统？

```
1. 前端层：页面静态化 + 按钮防重复提交 + 倒计时
2. 网关层：限流（令牌桶） + IP 黑名单 + 验证码
3. 应用层：库存预检查 + 异步下单 + 防超卖
4. 存储层：Redis 预扣库存 + 数据库乐观锁兜底
5. 消息队列：削峰，异步创建订单
6. 一致性：Redis Lua 原子扣减 + DB 乐观锁兜底
```

### Q2: 如何保证接口幂等性？

```
1. 天然幂等：GET / PUT / DELETE
2. 唯一索引：数据库层面防重
3. Token 机制：请求前获取 Token，请求时携带并验证
4. 状态机：业务状态校验
5. 乐观锁：version 字段
6. 分布式锁：Redis 锁防止并发
```

### Q3: 如何排查线上 OOM？

```
1. 查看监控告警，确认 OOM 时间和频率
2. jmap -dump 导出堆快照
3. MAT / VisualVM 分析大对象和内存泄漏
4. jstack 查看线程栈，排查死锁/阻塞
5. 检查 JVM 参数（堆大小、GC 策略）
6. 代码层面排查（静态集合无限增长、ThreadLocal 泄漏等）
```

### Q4: 接口性能优化手段？

```
1. 数据库优化：索引、SQL 优化、分库分表
2. 缓存：Redis 缓存热点数据
3. 异步：消息队列解耦耗时操作
4. 并发：多线程 / 协程并行处理
5. 批量：批量查询代替循环单条查询
6. 连接池：合理配置数据库/线程连接池
7. CDN / 静态化：减少动态计算
```
