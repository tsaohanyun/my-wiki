# 产品需求文档(PRD)模板

## 概述

产品需求文档（Product Requirements Document, PRD）是产品开发过程中最核心的文档之一，它清晰地定义了产品要做什么、为谁做、为什么做以及如何验证。

---

## 一、PRD 标准结构

### 1.1 文档信息

| 字段 | 内容 |
|------|------|
| 产品名称 | [产品名] |
| 版本号 | v1.0.0 |
| 文档版本 | 1.0 |
| 作者 | [姓名] |
| 最后更新 | YYYY-MM-DD |
| 审核人 | [姓名] |
| 状态 | 草稿 / 评审中 / 已确认 / 已变更 |

### 1.2 修订历史

| 版本 | 日期 | 修改人 | 修改内容 | 审核人 |
|------|------|--------|----------|--------|
| 1.0 | YYYY-MM-DD | 张三 | 初始版本 | 李四 |

## 二、背景与目标

### 2.1 项目背景

描述产品/功能的业务背景，回答以下问题：
- 当前面临什么问题或机会？
- 为什么现在需要解决？
- 与公司战略的关系是什么？

> **示例：** 用户反馈在移动端完成订单支付的成功率仅为 62%，远低于桌面端的 89%。移动端支付流程步骤过多（共 7 步），是导致用户流失的主要原因。

### 2.2 产品目标

使用 **SMART 原则** 定义目标：

| 维度 | 描述 | 示例 |
|------|------|------|
| Specific | 具体的 | 优化移动端支付流程 |
| Measurable | 可衡量的 | 支付成功率提升至 80% |
| Achievable | 可实现的 | 减少支付步骤至 3 步 |
| Relevant | 相关的 | 提升 GMV 和用户满意度 |
| Time-bound | 有时限的 | 2025 Q3 上线 |

### 2.3 成功指标（KPI）

| 指标 | 当前值 | 目标值 | 衡量方式 |
|------|--------|--------|----------|
| 移动端支付成功率 | 62% | 80% | 埋点数据 |
| 平均支付时长 | 45s | 20s | 埋点数据 |
| 支付页面跳出率 | 38% | 15% | Google Analytics |

## 三、用户故事

### 3.1 用户角色定义

| 角色 | 描述 | 核心需求 |
|------|------|----------|
| 新用户 | 首次使用产品的用户 | 快速完成注册和首单 |
| 老用户 | 有购买历史的用户 | 便捷复购，查看订单 |
| 商家 | 入驻的商户 | 管理商品、处理订单 |

### 3.2 用户故事编写规范

采用标准格式：

```
作为 [角色]，
我希望 [功能/行为]，
以便 [目的/价值]。
```

**验收标准（Acceptance Criteria）** 采用 Given-When-Then 格式：

```
Given [前置条件]
When [触发动作]
Then [预期结果]
```

### 3.3 用户故事示例

#### Epic：移动端快捷支付

**用户故事 1：一键支付**
```
作为 已绑定支付方式的老用户，
我希望 在确认订单后直接点击"一键支付"完成付款，
以便 无需重复选择支付方式，缩短支付时间。
```

**验收标准：**
```
AC1:
Given 用户已绑定至少一种支付方式且设置了默认支付
When 用户在订单确认页点击"一键支付"
Then 系统使用默认支付方式完成扣款，3秒内返回支付结果

AC2:
Given 用户未绑定任何支付方式
When 用户进入订单确认页
Then 不展示"一键支付"按钮，展示"添加支付方式"引导

AC3:
Given 用户的默认支付方式余额不足
When 用户点击"一键支付"
Then 提示余额不足，并推荐其他可用支付方式
```

**用户故事 2：支付方式管理**
```
作为 有多种支付方式的用户，
我希望 能设置和切换默认支付方式，
以便 在支付时自动使用我偏好的方式。
```

**验收标准：**
```
AC1:
Given 用户已绑定 2 种以上支付方式
When 用户在设置中长按某支付方式并选择"设为默认"
Then 该支付方式标记为默认，原默认方式取消标记

AC2:
Given 用户只绑定 1 种支付方式
When 用户尝试取消默认
Then 提示"至少保留一种默认支付方式"
```

## 四、功能需求

### 4.1 功能列表

| ID | 功能名称 | 优先级 | 描述 | 依赖 |
|----|----------|--------|------|------|
| F001 | 一键支付 | P0 | 使用默认支付方式快速支付 | F003 |
| F002 | 支付密码验证 | P0 | 大额支付需要密码确认 | 无 |
| F003 | 默认支付设置 | P1 | 设置/修改默认支付方式 | 无 |
| F004 | 支付结果页优化 | P1 | 展示支付状态和推荐内容 | F001 |
| F005 | 支付方式排序 | P2 | 按使用频率自动排序 | F003 |

### 4.2 详细功能说明

#### F001：一键支付

**功能描述：**
用户在订单确认页，如果已设置默认支付方式，展示"一键支付"按钮。点击后使用默认方式完成支付。

**业务规则：**
1. 单笔金额 ≤ 200 元：免密支付
2. 单笔金额 > 200 元：需要输入支付密码
3. 单日累计 > 1000 元：需要短信验证
4. 连续支付间隔 < 30s：需要二次确认

**异常处理：**

| 异常场景 | 处理方式 |
|----------|----------|
| 网络超时 | 展示重试按钮，最多重试 3 次 |
| 支付渠道异常 | 自动切换备用渠道 |
| 重复提交 | 幂等处理，返回已有结果 |
| 余额不足 | 引导切换支付方式或充值 |

## 五、优先级排序框架

### 5.1 RICE 评分模型

| 维度 | 说明 | 评分范围 |
|------|------|----------|
| **R**each（覆盖度） | 影响的用户数量/比例 | 1-10 |
| **I**mpact（影响力） | 对用户/业务的影响程度 | 0.25/0.5/1/2/3 |
| **C**onfidence（信心度） | 对估算的把握程度 | 50%/80%/100% |
| **E**ffort（工作量） | 需要的人·周数 | 1-10 |

**RICE 分数 = (Reach × Impact × Confidence) / Effort**

### 5.2 RICE 评分示例

| 功能 | Reach | Impact | Confidence | Effort | RICE 分数 | 优先级 |
|------|-------|--------|------------|--------|-----------|--------|
| 一键支付 | 8 | 3 | 80% | 3 | 6.4 | P0 |
| 支付密码 | 9 | 2 | 100% | 2 | 9.0 | P0 |
| 默认支付设置 | 6 | 2 | 80% | 2 | 4.8 | P1 |
| 结果页优化 | 7 | 1 | 80% | 1 | 5.6 | P1 |
| 支付方式排序 | 5 | 0.5 | 50% | 1 | 1.25 | P2 |

### 5.3 MoSCoW 方法

| 类别 | 含义 | 示例 |
|------|------|------|
| **Must have** | 必须有，核心功能 | 一键支付、支付安全 |
| **Should have** | 应该有，重要但非核心 | 默认支付设置 |
| **Could have** | 可以有，锦上添花 | 支付方式排序 |
| **Won't have** | 本期不做 | 社交支付（暂不考虑） |

## 六、非功能需求

### 6.1 性能要求

| 指标 | 要求 |
|------|------|
| 页面加载时间 | ≤ 2s（首屏） |
| 支付接口响应 | ≤ 500ms（P99） |
| 并发支持 | 10,000 QPS |
| 可用性 | 99.95% |

### 6.2 安全要求

- 支付数据全程加密传输（TLS 1.3）
- 敏感信息脱敏展示（卡号显示后四位）
- 支付操作日志全量记录
- 符合 PCI DSS 合规要求

### 6.3 兼容性要求

| 平台 | 最低版本 |
|------|----------|
| iOS | 14.0+ |
| Android | 8.0+ |
| 微信小程序 | 基础库 2.20.0+ |

## 九、风险与依赖

| 风险/依赖 | 影响 | 概率 | 应对措施 |
|-----------|------|------|----------|
| 支付渠道接口变更 | 支付失败率上升 | 中 | 抽象支付层，支持快速切换 |
| 安全审计未通过 | 上线延迟 | 低 | 提前进行安全评审 |
| 大促期间流量激增 | 系统不稳定 | 高 | 提前压测，准备降级方案 |
