# 数据湖架构

> 数据湖（Data Lake）解决了传统数据仓库**无法存储非结构化数据**和**扩展成本高**的问题。现代数据湖在对象存储/HDFS 之上增加了 ACID 事务、Schema 演进、时间旅行等能力，形成了**湖仓一体（Lakehouse）** 架构。三大主流方案为 **Delta Lake、Apache Iceberg、Apache Hudi**。

---

## 一、为什么需要数据湖？

### 1.1 传统数仓的痛点

| 痛点 | 说明 |
|------|------|
| 存储成本高 | MPP 数据库使用专用存储，扩展到 PB 级成本极高 |
| 无法存半结构化/非结构化数据 | JSON、Avro、图片、视频等 |
| Schema 刚性 | 表结构变更需 DDL 操作，影响大 |
| 批处理延迟高 | 传统 ETL 日级延迟 |

### 1.2 湖仓一体（Lakehouse）的核心能力

```
┌─────────────────────────────────────────┐
│  查询引擎层（Spark / Flink / Trino / Presto）  │
├─────────────────────────────────────────┤
│  表格式层（Delta / Iceberg / Hudi）          │
│  ACID | 时间旅行 | Schema演进 | 并发控制      │
├─────────────────────────────────────────┤
│  存储层（S3 / HDFS / OSS / Azure Blob）     │
└─────────────────────────────────────────┘
```

| 能力 | 说明 |
|------|------|
| **ACID 事务** | 保证并发读写的正确性 |
| **时间旅行** | 查询历史版本，支持数据回滚 |
| **Schema 演进** | 加列、删列、改名无需重写数据 |
| **Upsert/Delete** | 支持 CRUD 操作（传统数据湖只支持 Append） |
| **增量消费** | 流式读取新增数据，构建实时管道 |
| **统一批流** | 同一份数据同时服务批处理和流处理 |

## 二、Delta Lake

> 由 Databricks 开源，深度集成 Spark 生态，与 Parquet 文件格式兼容。

### 2.1 架构

```
Delta Lake Table
├── _delta_log/
│   ├── 00000000000000000000.json    ← 提交日志（Commit Log）
│   ├── 00000000000000000001.json
│   └── ...
├── part-00000-xxxx.snappy.parquet   ← 数据文件（标准 Parquet）
├── part-00001-xxxx.snappy.parquet
└── ...
```

### 2.2 核心操作（PySpark）

```python
from pyspark.sql import SparkSession

spark = SparkSession.builder \
    .appName("Delta Lake Demo") \
    .config("spark.sql.extensions", "io.delta.sql.DeltaSparkSessionExtension") \
    .config("spark.sql.catalog.spark_catalog",
            "org.apache.spark.sql.delta.catalog.DeltaCatalog") \
    .getOrCreate()

# ========== 创建 Delta 表 ==========
df = spark.read.json("data/raw/events.json")

df.write.format("delta") \
    .mode("overwrite") \
    .option("overwriteSchema", "true") \
    .save("/delta/events")

# 写入 Hive Metastore 托管表
spark.sql("""
    CREATE TABLE IF NOT EXISTS analytics.events (
        event_id STRING,
        user_id BIGINT,
        event_type STRING,
        event_time TIMESTAMP,
        amount DOUBLE
    )
    USING DELTA
    LOCATION '/delta/events'
""")

# ========== Upsert（MERGE） ==========
new_data = spark.read.json("data/incremental/new_events.json")

new_data.alias("new").merge(
    spark.table("analytics.events").alias("old"),
    "new.event_id = old.event_id"
).whenMatchedUpdateAll() \
 .whenNotMatchedInsertAll() \
 .execute()

# ========== 时间旅行 ==========
# 查看历史版本
history = spark.sql("DESCRIBE HISTORY analytics.events")
history.select("version", "timestamp", "operation", "operationParameters").show()

# 读取指定版本
old_version = spark.read.format("delta") \
    .option("versionAsOf", 5) \
    .load("/delta/events")

# 读取指定时间点
snapshot = spark.read.format("delta") \
    .option("timestampAsOf", "2026-06-27 10:00:00") \
    .load("/delta/events")

# ========== 回滚 ==========
spark.sql("RESTORE TABLE analytics.events TO VERSION AS OF 5")
spark.sql("RESTORE TABLE analytics.events TO TIMESTAMP AS OF '2026-06-27 10:00:00'")

# ========== Schema 演进 ==========
# 自动合并 Schema（新增列）
new_data.write.format("delta") \
    .mode("append") \
    .option("mergeSchema", "true") \
    .save("/delta/events")

# ========== Vacuum 清理旧文件 ==========
# 删除超过7天且不引用的旧数据文件
spark.sql("VACUUM analytics.events RETAIN 168 HOURS")
# 强制查看将被删除的文件（不实际删除）
spark.sql("VACUUM analytics.events RETAIN 168 HOURS DRY RUN")
```

### 2.3 Delta Lake 特性

| 特性 | 说明 |
|------|------|
| **Optimize（Z-Order）** | 多维聚簇，加速范围查询 |
| **Auto Compaction** | 自动合并小文件 |
| **Liquid Clustering** | 替代 Z-Order，自适应聚簇（Delta 3.0+） |
| **Delta Live Tables** | 声明式数据管道 |

```sql
-- 优化（合并小文件 + Z-Order 聚簇）
OPTIMIZE analytics.events ZORDER BY (user_id, event_type);

-- 自动压缩配置
SET spark.databricks.delta.autoOptimize.optimizeWrite = true;
SET spark.databricks.delta.autoOptimize.autoCompact = true;
```

## 三、Apache Iceberg

> 由 Netflix 开发，支持多引擎（Spark、Flink、Trino、Doris），**引擎解耦**是最大优势。

### 3.1 架构

```
Iceberg Table
├── metadata/
│   ├── v1.metadata.json          ← 表元数据快照
│   ├── v2.metadata.json
│   ├── snap-xxx-1.avro           ← 快照 Manifest List
│   ├── manifest-xxx-1.avro       ← Manifest 文件（指向数据文件）
│   └── manifest-xxx-2.avro
├── data/
│   ├── 00001-xxxx.parquet        ← 数据文件
│   └── 00002-xxxx.parquet
└── partition-data/
    ├── event_date=2026-06-27/
    └── event_date=2026-06-28/
```

**三层元数据架构**：`Metadata File → Manifest List → Manifest File → Data File`

### 3.2 核心操作（Spark）

```python
spark = SparkSession.builder \
    .config("spark.sql.extensions",
            "org.apache.iceberg.spark.extensions.IcebergSparkSessionExtensions") \
    .config("spark.sql.catalog.glue",
            "org.apache.iceberg.spark.SparkCatalog") \
    .config("spark.sql.catalog.glue.type", "glue") \
    .config("spark.sql.catalog.glue.warehouse", "s3://my-bucket/warehouse") \
    .getOrCreate()

# ========== 创建表 ==========
spark.sql("""
    CREATE TABLE glue.analytics.events (
        event_id    STRING,
        user_id     BIGINT,
        event_type  STRING,
        event_time  TIMESTAMP,
        amount      DOUBLE
    )
    USING ICEBERG
    PARTITIONED BY (days(event_time))
    TBLPROPERTIES (
        'write.format.default' = 'parquet',
        'write.parquet.compression-codec' = 'zstd',
        'format-version' = '2'
    )
""")

# ========== 写入 ==========
df.writeTo("glue.analytics.events").createOrReplace()

# 增量写入
df.writeTo("glue.analytics.events").append()

# ========== Upsert ==========
spark.sql("""
    MERGE INTO glue.analytics.events AS target
    USING updates AS source
    ON target.event_id = source.event_id
    WHEN MATCHED THEN UPDATE SET *
    WHEN NOT MATCHED THEN INSERT *
""")

# ========== 隐藏分区（Hidden Partitioning） ==========
# Iceberg 自动管理分区，无需手动指定分区列
# 查询时直接用 event_time 过滤，Iceberg 自动做分区裁剪
spark.sql("""
    SELECT * FROM glue.analytics.events
    WHERE event_time BETWEEN '2026-06-27' AND '2026-06-28'
""")

# ========== 快照管理 ==========
# 查看快照历史
spark.sql("SELECT * FROM glue.analytics.events.history").show()
spark.sql("SELECT * FROM glue.analytics.events.snapshots").show()

# 时间旅行
spark.sql("""
    SELECT * FROM glue.analytics.events VERSION AS OF 5
""")

# ========== 维护操作 ==========
# 合并小文件
spark.sql("CALL glue.system.rewrite_data_files('analytics.events')")

# 过期快照（保留最近7天）
spark.sql("""
    CALL glue.system.expire_snapshots('analytics.events',
        TIMESTAMP '2026-06-21 00:00:00')
""")

# 移除孤立文件
spark.sql("""
    CALL glue.system.remove_orphan_files('analytics.events')
""")
```

### 3.3 Flink + Iceberg（流批一体）

```java
// Flink 写入 Iceberg（CDC 实时同步）
StreamExecutionEnvironment env =
    StreamExecutionEnvironment.getExecutionEnvironment();

// 从 MySQL CDC 读取
DataStream<RowData> cdcStream = env.fromSource(
    MySqlSource.<RowData>builder()
        .hostname("mysql-host")
        .port(3306)
        .databaseList("analytics")
        .tableList("analytics.users")
        .username("root")
        .password("secret")
        .deserializer(new JsonDebeziumDeserializationSchema())
        .build(),
    WatermarkStrategy.noWatermarks(),
    "mysql-cdc-source"
);

// 写入 Iceberg
tableEnv.executeSql("""
    CREATE CATALOG iceberg_catalog WITH (
        'type' = 'iceberg',
        'catalog-impl' = 'org.apache.iceberg.aws.glue.GlueCatalog',
        'warehouse' = 's3://my-bucket/warehouse'
    )
""");

tableEnv.executeSql("""
    INSERT INTO iceberg_catalog.analytics.users
    /*+ OPTIONS('upsert-enabled' = 'true') */
    SELECT * FROM cdc_source
""");
```

## 四、Apache Hudi

> 由 Uber 开发，**强调 Upsert/Delete 能力**，适合 **CDC 数据同步** 和 **近实时增量处理**。

### 4.1 两种表类型

| 表类型 | 说明 | 写入方式 | 适用场景 |
|--------|------|----------|----------|
| **CoW（Copy-On-Write）** | 写时复制 | 每次更新重写整个 Parquet 文件 | 读多写少、批处理 |
| **MoR（Merge-On-Read）** | 读时合并 | 更新写入 Avro 日志文件，读时合并 | 写多读多、近实时 |

### 4.2 核心操作（Spark）

```python
from pyspark.sql import SparkSession

spark = SparkSession.builder \
    .config("spark.jars.packages",
            "org.apache.hudi:hudi-spark3-bundle_2.12:0.14.0") \
    .config("spark.sql.extensions",
            "org.apache.spark.sql.hudi.HoodieSparkSessionExtension") \
    .config("spark.sql.catalog.hudi_catalog",
            "org.apache.spark.sql.hudi.catalog.HoodieCatalog") \
    .getOrCreate()

# ========== 创建 MoR 表（CDC 同步） ==========
common_opts = {
    "hoodie.table.name": "user_events",
    "hoodie.table.type": "MERGE_ON_READ",
    "hoodie.datasource.write.recordkey.field": "event_id",
    "hoodie.datasource.write.precombine.field": "event_time",
    "hoodie.datasource.write.partitionpath.field": "dt",
    "hoodie.datasource.write.hive_style_partitioning": "true",
    "hoodie.upsert.shuffle.parallelism": "200",
    "hoodie.cleaner.policy": "KEEP_LATEST_COMMITS",
    "hoodie.cleaner.commits.retained": "10",
    "hoodie.compact.inline": "true",
    "hoodie.compact.inline.max.delta.commits": "5",  # 每5次提交触发压缩
}

# 写入（Upsert 模式）
df.write.format("hudi") \
    .options(**common_opts) \
    .option("hoodie.datasource.write.operation", "upsert") \
    .mode("append") \
    .save("s3://my-bucket/hudi/user_events")

# ========== 增量查询 ==========
# 获取最新 Instant Time
commit_time = spark.read.format("hudi") \
    .load("s3://my-bucket/hudi/user_events") \
    .select("_hoodie_commit_time") \
    .distinct() \
    .orderBy("_hoodie_commit_time", ascending=False) \
    .first()[0]

# 增量读取（只读取上次提交后的新数据）
incremental_df = spark.read.format("hudi") \
    .option("hoodie.datasource.query.type", "incremental") \
    .option("hoodie.datasource.read.begin.instanttime", commit_time) \
    .load("s3://my-bucket/hudi/user_events")

# ========== 时间旅行 ==========
point_in_time_df = spark.read.format("hudi") \
    .option("as.of.instant", "20260627100000") \
    .load("s3://my-bucket/hudi/user_events")

# ========== Flink + Hudi（实时 CDC 入湖） ==========
tableEnv.executeSql("""
    CREATE TABLE hudi_events (
        event_id STRING PRIMARY KEY NOT ENFORCED,
        user_id BIGINT,
        event_type STRING,
        event_time TIMESTAMP(3),
        amount DOUBLE,
        dt STRING
    )
    WITH (
        'connector' = 'hudi',
        'path' = 's3://my-bucket/hudi/events',
        'table.type' = 'MERGE_ON_READ',
        'hoodie.datasource.write.precombine.field' = 'event_time',
        'hoodie.datasource.write.partitionpath.field' = 'dt',
        'compaction.tasks' = '1',
        'compaction.delta_commits' = '5'
    )
""");

tableEnv.executeSql("""
    INSERT INTO hudi_events
    SELECT * FROM mysql_cdc_source
""");
```

### 4.3 Hudi 索引类型

| 索引 | 说明 | 适用场景 |
|------|------|----------|
| **Bloom Index** | 基于 Bloom Filter | 默认，通用 |
| **Simple Index** | 全量比对 | 小表 |
| **Bucket Index** | 哈希分桶 | 固定数量分桶 |
| **HBase Index** | 外部 HBase 索引 | 超大表，低延迟查找 |

## 五、三大方案对比与选型

### 5.1 功能对比矩阵

| 维度 | Delta Lake | Iceberg | Hudi |
|------|-----------|---------|------|
| **起源** | Databricks | Netflix | Uber |
| **引擎生态** | Spark 深度绑定 | 多引擎（Spark/Flink/Trino/Doris） | Spark + Flink |
| **ACID 事务** | ✅ | ✅ | ✅ |
| **时间旅行** | ✅ | ✅ | ✅ |
| **Schema 演进** | ✅ | ✅（最强） | ✅ |
| **Upsert 性能** | 中 | 中 | ✅ 最强 |
| **分区演进** | ❌（固定） | ✅（隐藏分区，可演进） | ❌（固定） |
| **CDC 友好度** | 中 | 中 | ✅ 专门优化 |
| **社区活跃度** | 高（Databricks 背书） | 高（ASF 顶级项目） | 高 |
| **云原生** | ✅ | ✅（AWS 深度合作） | ✅ |
| **压缩策略** | Optimize/Z-Order | rewrite_data_files | Compaction（MoR） |
| **流式写入** | Structured Streaming | ✅ Flink Sink | ✅ Flink Sink（最强） |

### 5.2 选型建议

```
你的场景是什么？
│
├─ 已使用 Databricks？
│   └─ ✅ Delta Lake（原生支持，零成本）
│
├─ 需要多引擎协同（Spark + Trino + Flink）？
│   └─ ✅ Iceberg（引擎解耦最佳）
│
├─ CDC 实时同步（MySQL → 湖）？
│   └─ ✅ Hudi（Upsert/增量消费最佳）
│
├─ 纯追加日志分析？
│   └─ ✅ Iceberg 或 Delta（无需频繁更新）
│
├─ 需要频繁 Upsert + 近实时查询？
│   └─ ✅ Hudi MoR（专为该场景设计）
│
└─ 云上全托管道（AWS）？
    └─ ✅ Iceberg（AWS 全面支持，Lake Formation 集成）
```

### 5.3 性能基准参考

> 以下为公开基准测试的**趋势性结论**（实际数据因场景而异）：

| 场景 | 最优 |
|------|------|
| 批量写入吞吐 | Delta ≈ Iceberg > Hudi |
| 小文件合并效率 | Iceberg > Delta > Hudi |
| Upsert 延迟 | Hudi MoR > Iceberg > Delta |
| 查询性能（无优化） | 三者接近 |
| 查询性能（Z-Order/聚簇后） | Delta ≈ Iceberg > Hudi |

## 六、架构最佳实践

### 6.1 分层存储设计

```
┌─────────────────────────────────────────────────┐
│  ODS 层（原始数据）                                 │
│  格式: Iceberg/Delta Append-Only                 │
│  分区: 天 + 小时                                    │
│  压缩: ZSTD                                        │
├─────────────────────────────────────────────────┤
│  DWD 层（明细数据）                                 │
│  格式: Iceberg/Delta with Upsert                  │
│  分区: 天 + 业务域                                  │
│  聚簇: Z-Order on (user_id, event_type)          │
├─────────────────────────────────────────────────┤
│  DWS 层（汇总数据）                                 │
│  格式: Iceberg/Delta                               │
│  分区: 天                                          │
│  预聚合: 日/周/月粒度                               │
├─────────────────────────────────────────────────┤
│  ADS 层（应用数据）                                 │
│  格式: Parquet / Presto / StarRocks             │
│  物化视图: 高频查询预计算                            │
└─────────────────────────────────────────────────┘
```

### 6.2 文件管理

```sql
-- 定期合并小文件（建议每日）
-- Iceberg
CALL system.rewrite_data_files('analytics.events',
    WHERE event_time >= TIMESTAMP '2026-06-27');

-- Delta
OPTIMIZE analytics.events ZORDER BY (user_id);

-- Hudi
CALL run_compaction('analytics.events');
```

### 6.3 数据生命周期

| 数据年龄 | 存储层 | 操作 |
|----------|--------|------|
| 0-30 天 | 标准存储 | 热查询 |
| 30-90 天 | 标准存储 | 冷查询 |
| 90-365 天 | 归档存储 | 迁移到 S3 Glacier |
| > 365 天 | 归档/删除 | 过期清理 |
