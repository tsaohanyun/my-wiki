# 项目组合管理

## 概述

**项目组合管理（Project Portfolio Management, PPM）** 是对组织内所有项目和项目群进行集中管理的方法论，目标是确保每个项目都能为组织战略目标做出贡献，同时实现**资源的最优配置**和**风险与收益的平衡**。

```
                    ┌─────────────────────┐
                    │   组织战略目标       │
                    └──────────┬──────────┘
                               │
                    ┌──────────▼──────────┐
                    │   项目组合           │
                    │  (Portfolio)         │
                    │                     │
                    │  • 战略对齐评估      │
                    │  • 优先级排序        │
                    │  • 资源平衡          │
                    │  • 风险管理          │
                    └──┬───────┬───────┬──┘
                       │       │       │
                ┌──────▼┐ ┌────▼───┐ ┌▼──────┐
                │项目群A │ │项目群B  │ │项目群C │
                └──┬──┬─┘ └──┬──┬─┘ └─┬──┬──┘
                   │  │      │  │     │  │
                ┌──▼┐│┌─────▼┐│┌─────▼┐│
                │P1│││ P3   │││ P5   ││
                └──┘││└─────┘││└─────┘│
                ┌──┐ ││┌─────┐││┌─────┐│
                │P2│ ││ P4   │││ P6   ││
                └──┘ ││└─────┘││└─────┘│
```

### 项目 vs 项目群 vs 项目组合

| 层级 | 定义 | 关注点 | 时间维度 |
|------|------|--------|----------|
| **项目（Project）** | 为创造独特产品/服务而进行的临时性活动 | 交付结果 | 有明确起止时间 |
| **项目群（Program）** | 一组相互关联的项目，协同管理以获得单独管理无法获得的收益 | 收益实现 | 中长期 |
| **项目组合（Portfolio）** | 为实现战略目标而集中管理的项目、项目群和运营的组合 | 战略对齐与投资回报 | 持续运营 |

## 一、项目分类与战略对齐

### 1.1 项目分类框架

#### 方法一：战略价值矩阵

```
                        高战略价值
                            │
                ┌───────────┼───────────┐
                │           │           │
         战略型 │  ⭐ 转型   │  🚀 突破   │ 高确定性
        项目    │   项目     │   项目     │
                │(高投入)    │(高投入)    │
                │           │           │
    ────────────┼───────────┼───────────┤
                │           │           │
         运营型 │  🛠️ 优化   │  📊 常规   │ 低确定性
        项目    │   项目     │   项目     │
                │(低投入)    │(低投入)    │
                │           │           │
                └───────────┼───────────┘
                            │
                        低战略价值
              低确定性                  高确定性
```

| 类别 | 特征 | 管理重点 | 示例 |
|------|------|---------|------|
| 🚀 **突破型** | 高价值高确定性 | 确保资源到位，快速交付 | 核心产品3.0发布 |
| ⭐ **转型型** | 高价值低确定性 | 分阶段投入，设置里程碑闸口 | 全面上云迁移 |
| 📊 **常规型** | 低价值高确定性 | 标准化流程，降低成本 | 季度安全补丁更新 |
| 🛠️ **优化型** | 低价值低确定性 | 严格控制范围和预算 | 内部工具优化 |

#### 方法二：运行-增长-转型（Run-Grow-Transform）

| 类别 | 资源占比建议 | 目标 | 典型项目 |
|------|-------------|------|---------|
| **运行（Run）** | 50-70% | 维持当前业务运营 | 系统维护、合规改造、安全更新 |
| **增长（Grow）** | 20-30% | 提升现有业务效率/规模 | 性能优化、新功能开发、市场扩展 |
| **转型（Transform）** | 5-15% | 创造全新业务能力 | 新产品线、新技术平台、业务模式创新 |

## 二、优先级排序

### 2.1 优先级排序方法

#### 方法一：WSJF（加权最短作业优先）

来自SAFe框架，基于**延迟成本（Cost of Delay, CoD）**进行排序：

```
                延迟成本（CoD）
WSJF = ────────────────────────────
                作业时长（Job Size）

延迟成本（CoD）= 用户/业务价值 + 时间紧迫性 + 风险降低/机会促进
```

```markdown
## WSJF评分模板

| 项目 | 用户/业务价值 | 时间紧迫性 | 风险降低 | CoD总分 | 作业时长 | WSJF |
|------|-------------|-----------|---------|---------|---------|------|
| 项目A | 8 | 5 | 3 | 16 | 5 | 3.2 |
| 项目B | 5 | 8 | 5 | 18 | 3 | 6.0 |
| 项目C | 3 | 3 | 8 | 14 | 8 | 1.75 |

→ 优先级排序：项目B > 项目A > 项目C
```

#### 方法二：评分矩阵法

| 项目 | 战略对齐(25%) | 财务回报(25%) | 风险(20%) | 紧迫性(15%) | 资源可用(15%) | 总分 |
|------|-------------|-------------|----------|------------|-------------|------|
| 项目A | 4×0.25=1.0 | 5×0.25=1.25 | 4×0.20=0.8 | 3×0.15=0.45 | 4×0.15=0.6 | **4.10** |
| 项目B | 5×0.25=1.25 | 3×0.25=0.75 | 5×0.20=1.0 | 5×0.15=0.75 | 3×0.15=0.45 | **4.20** |
| 项目C | 3×0.25=0.75 | 4×0.25=1.0 | 3×0.20=0.6 | 4×0.15=0.6 | 5×0.15=0.75 | **3.70** |

> 评分采用1-5分制，5分为最优

#### 方法三：气泡图可视化

```
   业务价值
   高 ↑
      │     ●D     ●A
      │
      │  ●C         ●B
      │
      │     ●E
      │
   低 └──────────────────→ 高
      低          实现难度     高

   气泡大小 = 项目投资规模
   理想区域 = 高价值 + 低难度（左上角）
```

## 三、资源分配

### 3.1 资源分配原则

| 原则 | 说明 |
|------|------|
| **战略优先** | 高战略价值项目优先获得资源 |
| **能力匹配** | 确保分配的人员具备所需技能 |
| **不过载** | 任何人的利用率不超过85% |
| **缓冲保留** | 预留10-15%资源应对意外 |
| **动态调整** | 每季度至少重新评估一次资源分配 |

### 3.3 资源平衡与平滑

| 技术 | 说明 | 适用场景 |
|------|------|---------|
| **资源平衡（Resource Leveling）** | 调整项目进度以不超过资源上限 | 资源严格受限 |
| **资源平滑（Resource Smoothing）** | 在不延长关键路径的前提下优化分配 | 进度不可延后 |
| **资源池化** | 建立共享资源池按需分配 | 技能可跨项目使用 |
| **外包/招聘** | 通过外部资源补充内部不足 | 内部技能不足或临时高峰 |

## 四、投资回报分析

### 4.1 财务评估方法

#### 方法一：NPV（净现值）

```
              n   现金流t
   NPV = Σ  ─────────── - 初始投资
             t=1  (1+r)^t

   r = 折现率（通常取10-15%）
```

| 项目 | 初始投资 | 第1年 | 第2年 | 第3年 | NPV(12%) | 决策 |
|------|---------|-------|-------|-------|----------|------|
| 项目A | -100万 | 40万 | 50万 | 60万 | **+18.5万** | ✅ 接受 |
| 项目B | -100万 | 20万 | 40万 | 80万 | **+5.2万** | ✅ 接受 |
| 项目C | -100万 | 10万 | 30万 | 40万 | **-35.8万** | ❌ 拒绝 |

> NPV > 0 表示项目创造价值，NPV越大越好。

#### 方法二：IRR（内部收益率）

```
   IRR = 使NPV = 0的折现率

   如果 IRR > 资本成本（如12%），则项目可接受
```

#### 方法三：投资回收期（Payback Period）

```
   初始投资 100万
   年均现金流 30万
   回收期 = 100/30 = 3.3年
```

#### 方法四：ROI（投资回报率）

```
              总收益 - 总投资
   ROI = ────────────────── × 100%
                 总投资
```

### 4.2 综合评估模型

#### 平衡计分卡式项目评估

| 维度 | 指标 | 权重 | 项目A | 项目B |
|------|------|------|-------|-------|
| **财务** | NPV、ROI | 30% | 4 | 3 |
| **客户** | 客户满意度提升 | 25% | 5 | 4 |
| **内部流程** | 效率提升 | 20% | 3 | 5 |
| **学习成长** | 能力建设 | 15% | 4 | 3 |
| **战略** | 战略目标支撑 | 10% | 5 | 4 |
| **总计** | | 100% | **4.15** | **3.70** |

### 4.3 项目组合的财务优化

```
   投资回报
   高 ↑
      │     ●C
      │        ●A
      │  ●B
      │           ●D
      │    ●E
   低 └──────────────────→ 投资
      低                   高

   高效前沿（Efficient Frontier）：
   在给定投资预算下，最大化整体回报的项目组合

   ─── 高效前沿线 ───
   线上项目：在当前投资水平下回报最优
   线下项目：可被更优组合替代
```

### 5.2 阶段闸门（Stage-Gate）审查

| 闸门 | 时间点 | 审查内容 | 决策选项 |
|------|--------|---------|---------|
| **Gate 0: 发现** | 概念阶段 | 是否值得进一步研究 | Go / Kill / Hold |
| **Gate 1: 评估** | 可行性分析后 | 商业案例是否成立 | Go / Kill / Hold / Recycle |
| **Gate 2: 开发** | 开发计划后 | 计划是否可行 | Go / Kill / Hold |
| **Gate 3: 测试** | 开发完成后 | 产品是否达标 | Go / Kill / Hold |
| **Gate 4: 发布** | 测试通过后 | 是否准备好上线 | Go / Kill / Conditional Go |
| **Gate 5: 上线后** | 上线3-6个月 | 是否达到预期收益 | Continue / Improve / Kill |

## 七、最佳实践

### 7.1 PPM的"DO"

- ✅ **建立统一的项目入口：** 所有项目必须经过标准化的评估和审批
- ✅ **定期重新平衡：** 每季度审查组合的健康度和优先级
- ✅ **敢于终止：** 杀掉不再有价值的项目是释放资源的最快方式
- ✅ **透明化决策：** 让团队了解为什么某些项目被优先
- ✅ **量力而行：** 并行项目数不超过组织管理能力
- ✅ **数据驱动：** 用NPV、IRR等量化指标辅助决策

### 7.2 PPM的"DON'T"

- ❌ **不要来者不拒：** 没有优先级的"什么都做"等于什么都做不好
- ❌ **不要忽视沉没成本：** 过去投入不应影响是否继续的决策
- ❌ **不要只看单个项目：** 项目间的依赖和冲突需要在组合层面管理
- ❌ **不要过度官僚化：** 评估流程不应比项目本身更耗时
- ❌ **不要忽视运营需求：** Run类项目虽不显眼但不可或缺
