# 设计思维

## 概述

**设计思维（Design Thinking）** 是一种以人为本的创新问题解决方法论，由IDEO设计公司和斯坦福大学d.school联合推广。它强调通过**深度理解用户**来发现真正的需求，并以**创意构思和快速原型**来探索解决方案。

设计思维不是线性的流程，而是一个**迭代循环**——团队可以在不同阶段之间来回穿梭。

```
    同理心（Empathize）         测试（Test）
         │                        │
         ▼                        │
    定义（Define）           原型（Prototype）
         │                        │
         ▼                        │
    构思（Ideate）────────────────┘
```

> 💡 **核心信念：** 创新不是天才的灵光一现，而是一个可以通过系统方法驱动的协作过程。

## 一、五阶段方法论

### 第一阶段：同理心（Empathize）

> **目标：** 深入理解用户的真实需求、痛点、情感和行为模式。

#### 1.1 同理心 vs 同情心

| 维度 | 同情心（Sympathy） | 同理心（Empathy） |
|------|-------------------|------------------|
| 视角 | "我为他们感到难过" | "我理解他们的感受" |
| 关系 | 旁观者 | 共同体验者 |
| 结果 | 表面理解 | 深层洞察 |
| 方法 | 假设/推测 | 观察/访谈/沉浸 |

#### 1.2 同理心研究方法

**方法一：沉浸式观察（Shadowing）**
```
   ┌─────────────────────────────────────────────┐
   │  观察用户的真实工作/生活场景                  │
   │  • 不打断，不引导                            │
   │  • 记录关键行为、犹豫点、变通方法             │
   │  • 关注"说"与"做"之间的差距                  │
   └─────────────────────────────────────────────┘
```

**方法二：深度访谈（In-depth Interview）**
- 使用开放式问题（"请告诉我你上次……的经历"）
- 追问"为什么"（但要自然，不要像审讯）
- 关注情感和故事，而非数据和结论
- **关键提示：** 用户说的、用户做的、用户想的，三者往往不同

**方法三：用户体验地图（Journey Map）**

```
   阶段    发现      评估      购买      使用      推荐
            │        │        │        │        │
   行为    搜索     比较     下单     安装     分享
            │        │        │        │        │
   情绪    😕困惑   😟焦虑   😊满意   😠挫败   😄惊喜
            │        │        │        │        │
   痛点    信息少   评价杂   步骤多   难上手   没渠道
```

#### 1.3 同理心研究模板

```markdown
## 用户同理心研究记录

### 基本信息
- 受访者编号：U001
- 角色/画像：[描述]
- 访谈日期：[日期]
- 访谈时长：[分钟]

### 关键观察
| 观察（用户做了什么） | 洞察（意味着什么） | 需求（真正需要什么） |
|---------------------|-------------------|---------------------|
| [行为1] | [洞察1] | [需求1] |
| [行为2] | [洞察2] | [需求2] |

### 用户语录
- "[原话引用1]"
- "[原话引用2]"

### 情感分析
- 😊 正面时刻：[描述]
- 😟 负面时刻：[描述]
- 😐 中性时刻：[描述]
```

### 第二阶段：定义（Define）

> **目标：** 将同理心阶段的发现综合提炼为清晰、可执行的问题陈述。

#### 2.1 问题陈述模板（POV Statement）

使用 **"我们如何可能……"（How Might We, HMW）** 句式：

```
[用户画像] 需要 [用户需求] 因为 [洞察/原因]

示例：
"忙碌的新手妈妈 需要一种快速获取可靠育儿信息的方式，
 因为她害怕做出错误的决定但没时间验证每个建议。"

→ HMW："我们如何可能让新手妈妈在3分钟内获得值得信赖的育儿建议？"
```

#### 2.2 HMW的提炼技巧

| 技巧 | 示例 |
|------|------|
| **放大视角** | HMW重新设计用户的生活体验？ |
| **缩小视角** | HMW优化支付环节的第三步？ |
| **转变视角** | HMW从孩子角度让妈妈更安心？ |
| **去除假设** | HMW不需要文字就能传递信息？ |
| **逆向思考** | HMW让用户主动求助而不是被动等待？ |

#### 2.3 用户画像（Persona）

```markdown
## 用户画像模板

| 维度 | 内容 |
|------|------|
| **姓名/头像** | 小李 👤 |
| **年龄/职业** | 32岁 / 产品经理 |
| **核心目标** | 高效完成产品规划与协作 |
| **主要痛点** | 信息散落各处、跨团队沟通低效 |
| **行为习惯** | 每天使用5+工具、习惯移动端办公 |
| **技术熟练度** | 高 |
| **关键场景** | 周一规划会、临时需求变更 |
| **一句话描述** | "我需要一站式掌握产品全貌" |
```

### 第三阶段：构思（Ideate）

> **目标：** 在不受限制的环境中产生尽可能多的解决方案。

#### 3.1 构思的核心原则

```
   ┌────────────────────────────────────────┐
   │                                        │
   │    数量 > 质量（先发散）                │
   │    延迟评判（不批评任何想法）           │
   │    鼓励疯狂的想法                       │
   │    在别人想法上叠加改进                 │
   │                                        │
   │         先发散 ──────→ 再收敛           │
   │    (Divergent)         (Convergent)    │
   └────────────────────────────────────────┘
```

#### 3.2 构思方法库

**方法一：头脑风暴（Brainstorming）**

| 规则 | 说明 |
|------|------|
| 不评判 | 任何想法都先记录 |
| 追求数量 | 哪怕天马行空 |
| 欢迎疯狂 | 打破常规思维 |
| 叠加改进 | "是的，而且……" |

**方法二：SCAMPER技巧**

| 字母 | 技巧 | 引导问题 |
|------|------|---------|
| **S** | 替代（Substitute） | 能用什么替代当前组件？ |
| **C** | 合并（Combine） | 能将哪些功能合并？ |
| **A** | 适配（Adapt） | 能借鉴其他领域的什么？ |
| **M** | 修改（Modify） | 能改变什么？大小、颜色、形态？ |
| **P** | 用途（Put to other use） | 还能用在哪里？ |
| **E** | 消除（Eliminate） | 能去掉什么？ |
| **R** | 重排（Rearrange） | 能重新排列什么？ |

**方法三：头脑书写（Brainwriting / 6-3-5法）**

```
   6人 × 每人写3个想法 × 5轮 = 108个想法

   时间线：
   第0分钟：每人写3个想法（5分钟）
   第5分钟：向右传递，在别人想法上拓展（5分钟）
   第10分钟：再传递，继续拓展（5分钟）
   ...重复5轮
   第25分钟：汇总所有想法，分类聚类
```

#### 3.3 方案收敛方法

**点子投票法（Dot Voting）：**
- 每人获得3-5个投票点
- 在所有方案上贴点投票
- 得票最高的2-3个方案进入原型阶段

**方案评估矩阵：**
```
                        高可行性
                           │
              ② 快赢方案   │  ① 明星方案
              (低投入高价值)│  (高投入高价值)
                           │
    ───────────────────────┼──────────────────
                           │
              ④ 避免方案   │  ③ 战略方案
              (低投入低价值)│  (高投入低价值)
                           │
                        低可行性
    低价值                              高价值
```

### 第四阶段：原型（Prototype）

> **目标：** 快速构建低成本的概念模型，用于验证想法和收集反馈。

#### 4.1 原型的保真度光谱

```
   低保真                                          高保真
   快速/廉价                                       耗时/昂贵
   │                                               │
   ├────────┬────────┬────────┬────────┬───────────┤
   │        │        │        │        │           │
   纸质     线框图   可交互   低保真   高保真      代码
   草图     Sketch  原型     MVP     原型        原型
                    (Figma)                    (实际技术)
   │        │        │        │        │           │
   概念阶段 早期设计 功能验证 用户测试 投资演示     上线前
```

#### 4.2 不同阶段的原型选择

| 阶段 | 推荐原型类型 | 目的 | 时间投入 |
|------|-------------|------|----------|
| 概念验证 | 纸质原型/草图 | 验证想法是否合理 | 数小时 |
| 早期设计 | 线框图/Wireframe | 验证信息架构和流程 | 1-3天 |
| 功能验证 | 可交互原型 | 验证用户流程是否流畅 | 3-7天 |
| 用户测试 | 低保真功能原型 | 获取真实用户反馈 | 1-2周 |
| 最终验证 | 高保真原型 | 投资者/管理层演示 | 2-4周 |

#### 4.3 原型构建模板

```markdown
## 原型定义模板

### 目标
- 核心假设：[这个原型要验证什么假设]
- 目标用户：[谁会使用这个原型]

### 范围
- 包含的界面/流程：[列出]
- 排除的界面/流程：[列出]
- 交互程度：[静态/可点击/可输入]

### 工具选择
- [ ] 纸质
- [ ] Figma/Sketch
- [ ] 原型工具（Axure/Framer）
- [ ] 无代码平台（Webflow/Bubble）
- [ ] 代码实现

### 验收标准
- 用户能完成核心任务：[描述任务]
- 完成时间 < [X]分钟
- 用户反馈净推荐值 > [X]
```

### 第五阶段：测试（Test）

> **目标：** 将原型放到真实用户面前，收集反馈并决定下一步方向。

#### 4.4 可用性测试方法

**方法一：有声思维法（Think-Aloud）**
```
   用户在使用原型时不断口头描述自己的思考过程
   "我现在在看这个按钮，不知道点击后会发生什么……"
   "这个信息我看不到，我在找……"
```

**方法二：任务测试**
```markdown
## 可用性测试脚本模板

### 热身问题（5分钟）
- "请介绍一下你平时如何[相关场景]？"
- "你上一次遇到[相关问题]是什么情况？"

### 任务测试（20分钟）
| 任务编号 | 任务描述 | 成功标准 | 完成时间 | 困难度(1-7) |
|----------|---------|---------|---------|------------|
| T1 | [任务描述] | [标准] | ___ | ___ |
| T2 | [任务描述] | [标准] | ___ | ___ |

### 总结问题（5分钟）
- "整体感受如何？"
- "有什么特别喜欢的？"
- "有什么让你困惑的？"
- "如果可以改一件事，你想改什么？"
```

#### 4.5 测试结果分析

```
   ┌─────────────────────────────────────────────┐
   │              测试结果分类                     │
   │                                             │
   │  ┌──────────┐    ┌──────────┐               │
   │  │ 需要修复  │    │ 可以优化  │               │
   │  │ (Critical)│    │ (Nice to │               │
   │  │           │    │  have)   │               │
   │  └──────────┘    └──────────┘               │
   │                                             │
   │  ┌──────────┐    ┌──────────┐               │
   │  │ 已验证   │    │ 新洞察   │               │
   │  │(Validated)│   │(New insight)│            │
   │  └──────────┘    └──────────┘               │
   └─────────────────────────────────────────────┘
```

#### 4.6 迭代决策

测试后的可能决策：

| 结果 | 决策 | 下一步 |
|------|------|--------|
| 原型基本被接受 | ✅ **进化** | 优化细节，提升保真度 |
| 部分有效 | 🔄 **迭代** | 修改问题部分，再次测试 |
| 基本无效 | 🔁 **回退** | 回到构思或定义阶段 |
| 发现新问题 | 🔍 **重新同理** | 补充用户研究 |

## 二、用户中心设计（UCD）原则

### 2.1 UCD核心原则

| 原则 | 说明 | 实践方法 |
|------|------|---------|
| **以用户为设计起点** | 从用户需求出发，而非技术或业务限制 | 用户研究先于产品设计 |
| **持续用户参与** | 用户参与设计全流程，不只是最终测试 | 每个阶段都纳入用户反馈 |
| **迭代设计** | 设计是不断修正的过程 | 快速原型 + 快速测试 |
| **多学科协作** | 设计、技术、业务共同参与 | 跨职能设计工作坊 |
| **可及性设计** | 考虑所有用户，包括有障碍的人群 | WCAG标准遵循 |

### 2.2 双钻模型（Double Diamond）

由英国设计委员会提出，将设计思维分为两个"发散-收敛"循环：

```
   发散           收敛         发散           收敛
   ╲   ╱         ╲   ╱        ╲   ╱         ╲   ╱
    ╲ ╱           ╲ ╱          ╲ ╱           ╲ ╱
     ╳             ╳            ╳             ╳
    ╱ ╲           ╱ ╲          ╱ ╲           ╱ ╲
   ╱   ╲         ╱   ╲        ╱   ╲         ╱   ╲

   发现          定义          构思          交付
  (Discover)   (Define)     (Develop)     (Deliver)

   问题空间                    解决方案空间
```

## 三、设计思维与敏捷开发的结合

### 3.1 融合框架

```
   设计思维（探索"正确的事"）
         │
         │  Discover → Define → Ideate → Prototype → Test
         │
         ▼ 输出：经过验证的用户需求与解决方案方向
         │
   敏捷开发（"正确地做事"）
         │
         │  Sprint Planning → Sprint → Review → Retrospective
         │
         ▼ 输出：可交付的软件产品
         │
   精益运营（持续优化）
         │
         ▼ 输出：数据驱动的持续改进
```

### 3.2 实践建议

| 设计思维阶段 | 敏捷中的对应 | 实践 |
|-------------|-------------|------|
| 同理心 + 定义 | Sprint前的Discovery | 每个迭代前进行1-2周用户研究 |
| 构思 + 原型 | Sprint内的设计工作 | 设计师与开发者紧密协作 |
| 测试 | Sprint Review | 每次评审纳入用户测试结果 |
| 迭代 | Sprint Retrospective | 回顾设计假设的验证情况 |

## 四、最佳实践

### 4.1 设计思维的"DO"

- ✅ **展示，不要告诉：** 用原型和故事板展示想法
- ✅ **拥抱模糊性：** 接受前期的不确定性
- ✅ **走到用户中间：** 去用户的真实环境观察
- ✅ **快速失败：** 在原型阶段发现问题成本最低
- ✅ **跨职能团队：** 汇聚不同背景的人
- ✅ **保持好奇：** 问"为什么"直到找到根本需求

### 4.2 设计思维的"DON'T"

- ❌ **不要跳过同理心：** 没有用户洞察的设计是空中楼阁
- ❌ **不要过早收敛：** 构思阶段要充分发散
- ❌ **不要追求完美原型：** 原型是工具不是产品
- ❌ **不要只问不用看：** 用户说的和做的不一样
- ❌ **不要线性执行：** 允许在不同阶段间回退
