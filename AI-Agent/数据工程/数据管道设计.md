## 1. 架构概览

```
                    ┌─────────────────────────────────────────┐
                    │            Data Sources                  │
                    │  Databases | APIs | Logs | IoT | Files   │
                    └───────────┬──────────────┬──────────────┘
                                │              │
                     ┌──────────▼──┐    ┌──────▼────────┐
                     │ Batch Path  │    │  Stream Path   │
                     │ (Scheduled) │    │  (Real-time)   │
                     └──────┬──────┘    └───────┬───────┘
                            │                   │
                     ┌──────▼───────────────────▼──────┐
                     │        Serving Layer             │
                     │  Data Warehouse | ML | Dashboard │
                     └─────────────────────────────────┘
```

| 维度       | 批处理           | 流处理             |
| ---------- | ---------------- | ------------------ |
| 延迟       | 分钟 ~ 小时      | 毫秒 ~ 秒          |
| 数据量     | TB 级            | 持续流             |
| 吞吐量     | 高               | 中~高              |
| 复杂度     | 较低             | 较高（状态管理）   |
| 成本       | 低（按需）       | 高（持续运行）     |

## 2. 批处理管道（Batch Processing）

### 2.1 Spark 批处理示例

```python
from pyspark.sql import SparkSession
from pyspark.sql import functions as F
from pyspark.sql.window import Window

spark = SparkSession.builder \
    .appName("batch_aggregation") \
    .config("spark.sql.shuffle.partitions", "200") \
    .config("spark.sql.adaptive.enabled", "true") \
    .config("spark.sql.adaptive.skewJoin.enabled", "true") \
    .getOrCreate()

# ── 读取 ──
orders = spark.read.parquet("s3://data-lake/raw/orders/dt=2026-06-28/")
users = spark.read.parquet("s3://data-lake/raw/users/")

# ── 转换 ──
enriched = orders.join(users, on="user_id", how="left")

# 窗口函数：计算用户消费排名
user_window = Window.partitionBy("user_id").orderBy(F.desc("amount"))
enriched = enriched.withColumn("rank_in_user", F.row_number().over(user_window))

# 每日汇总
daily_summary = (
    enriched
    .groupBy(F.col("order_date"), F.col("country"))
    .agg(
        F.sum("amount").alias("total_revenue"),
        F.count("order_id").alias("order_count"),
        F.avg("amount").alias("avg_order_value"),
        F.collect_set("user_id").alias("unique_users"),
    )
    .orderBy(F.desc("total_revenue"))
)

# ── 写入 ──
daily_summary.write \
    .mode("overwrite") \
    .partitionBy("order_date") \
    .parquet("s3://data-lake/curated/daily_revenue/")

spark.stop()
```

### 2.2 基于时间分区的水位线

```python
from datetime import datetime, timedelta

def get_batch_watermark(table_name: str) -> str:
    """获取上次成功处理的时间点"""
    # 从元数据表读取水位线
    result = spark.sql(f"""
        SELECT MAX(processed_at) as watermark
        FROM meta.pipeline_checkpoint
        WHERE table_name = '{table_name}'
    """).collect()
    return result[0]["watermark"] if result else "1970-01-01"

def process_batch():
    watermark = get_batch_watermark("orders")
    current_time = datetime.utcnow()

    new_data = spark.read.parquet("s3://data-lake/raw/orders/") \
        .where(f"updated_at > '{watermark}' AND updated_at <= '{current_time}'")

    # ... 处理逻辑 ...

    # 更新水位线
    spark.sql(f"""
        INSERT INTO meta.pipeline_checkpoint
        VALUES ('orders', '{current_time}')
    """)
```

## 3. 流处理管道（Stream Processing）

### 3.1 Apache Flink 流处理

```java
// FlinkKafkaPipeline.java
import org.apache.flink.streaming.api.environment.StreamExecutionEnvironment;
import org.apache.flink.streaming.api.datastream.DataStream;
import org.apache.flink.streaming.connectors.kafka.FlinkKafkaConsumer;
import org.apache.flink.api.common.serialization.SimpleStringSchema;
import org.apache.flink.streaming.api.windowing.time.Time;

public class OrderStreamPipeline {

    public static void main(String[] args) throws Exception {
        final StreamExecutionEnvironment env =
            StreamExecutionEnvironment.getExecutionEnvironment();

        // Kafka Source
        Properties props = new Properties();
        props.setProperty("bootstrap.servers", "kafka:9092");
        props.setProperty("group.id", "flink-order-consumer");

        DataStream<String> stream = env.addSource(
            new FlinkKafkaConsumer<>("orders", new SimpleStringSchema(), props)
        );

        // 处理逻辑
        DataStream<OrderEvent> parsed = stream
            .map(new OrderEventParser())
            .filter(event -> event.amount > 0)
            .keyBy(OrderEvent::getUserId)
            .timeWindow(Time.minutes(5))
            .aggregate(new OrderAggregator());

        // Sink：写入 ClickHouse
        parsed.addSink(new ClickHouseSink());

        env.execute("Real-time Order Aggregation");
    }
}
```

### 3.2 Spark Structured Streaming

```python
from pyspark.sql import SparkSession
from pyspark.sql import functions as F
from pyspark.sql.types import StructType, StringType, DoubleType, TimestampType

spark = SparkSession.builder \
    .appName("streaming_orders") \
    .config("spark.sql.streaming.checkpointLocation", "/checkpoint/orders") \
    .getOrCreate()

schema = StructType() \
    .add("order_id", StringType()) \
    .add("user_id", StringType()) \
    .add("amount", DoubleType()) \
    .add("event_time", TimestampType())

# Kafka Source
stream = spark \
    .readStream \
    .format("kafka") \
    .option("kafka.bootstrap.servers", "kafka:9092") \
    .option("subscribe", "orders") \
    .option("startingOffsets", "latest") \
    .load()

parsed = stream.selectExpr("CAST(value AS STRING) as json_str") \
    .select(F.from_json("json_str", schema).alias("data")) \
    .select("data.*")

# 窗口聚合：每5分钟滚动窗口
windowed_agg = parsed \
    .withWatermark("event_time", "10 minutes") \
    .groupBy(
        F.window("event_time", "5 minutes"),
        F.col("user_id")
    ) \
    .agg(
        F.sum("amount").alias("total_spent"),
        F.count("order_id").alias("order_count")
    )

# 写入控制台（测试）或 Sink（生产）
query = windowed_agg \
    .writeStream \
    .outputMode("update") \
    .format("console") \
    .option("truncate", "false") \
    .trigger(processingTime="1 minute") \
    .start()

query.awaitTermination()
```

### 3.3 Kafka Streams（Python via Faust）

```python
import faust

app = faust.App(
    "order-processor",
    broker="kafka://localhost:9092",
    store="memory://"
)

class Order(faust.Record, serializer="json"):
    order_id: str
    user_id: str
    amount: float
    currency: str

orders_topic = app.topic("orders", value_type=Order)
high_value_topic = app.topic("high_value_orders", value_type=Order)

@app.agent(orders_topic)
async def process_orders(orders):
    async for order in orders:
        # 过滤高价值订单
        if order.amount > 10000:
            await high_value_topic.send(value=order)

        # 异常检测
        if order.amount < 0:
            print(f"⚠️ 异常订单: {order.order_id}, 金额: {order.amount}")

if __name__ == "__main__":
    app.main()
```

## 4. Lambda 架构

> **Lambda 架构** = 批处理层 + 速度层 + 服务层

```
                ┌──────────────┐
                │  Data Source  │
                └──────┬───────┘
                       │
            ┌──────────┴──────────┐
            │                     │
   ┌────────▼────────┐  ┌────────▼────────┐
   │  Batch Layer    │  │  Speed Layer    │
   │  (HDFS/S3+Spark)│  │  (Kafka+Storm)  │
   │  全量，精确      │  │  增量，近似      │
   └────────┬────────┘  └────────┬────────┘
            │                     │
            └──────────┬──────────┘
                       │
              ┌────────▼────────┐
              │  Serving Layer   │
              │  (HBase/Cassandra)│
              │  Batch View +    │
              │  Real-time View   │
              └─────────────────┘
```

### Lambda 架构优缺点

| 优点                       | 缺点                                |
| -------------------------- | ----------------------------------- |
| 兼顾延迟与准确性           | 两条代码路径，维护成本高            |
| 批处理层可重算历史数据     | 需要维护两套引擎                    |
| 速度层故障不影响批处理     | 一致性合并逻辑复杂                  |

### Lambda 架构实现伪代码

```python
class LambdaPipeline:

    def batch_view(self, query):
        """批处理层：预计算全量视图"""
        all_data = spark.read.parquet("s3://lake/all_orders/")
        result = all_data.transform(query)
        result.write \
            .format("hbase") \
            .option("hbase.columns.mapping", "cf:revenue") \
            .save()

    def speed_view(self, event):
        """速度层：实时更新增量视图"""
        redis_client.incrbyfloat(
            f"daily_revenue:{event.date}",
            event.amount
        )

    def query(self, params):
        """服务层：合并批处理+实时视图"""
        batch_result = hbase.get(params)        # 来自批处理层
        realtime_delta = redis.get(params)      # 来自速度层
        return batch_result + realtime_delta    # 合并结果
```

## 5. Kappa 架构

> **Kappa 架构** = 统一以流为核心，移除批处理层

```
                ┌──────────────┐
                │  Data Source  │
                └──────┬───────┘
                       │
              ┌────────▼────────┐
              │   Kafka (Log)   │  ← 不可变日志，全量存储
              └────────┬────────┘
                       │
              ┌────────▼────────┐
              │  Stream Engine   │
              │  (Flink/Spark)   │
              │  统一批+流处理    │
              └────────┬────────┘
                       │
              ┌────────▼────────┐
              │  Serving Layer   │
              │  (Druid/ClickHouse)│
              └─────────────────┘
```

### Kappa 核心思想

1. **一切皆流**：所有数据通过 Kafka 进入，以事件日志形式保存
2. **单一引擎**：使用同一个流处理引擎同时处理实时和回溯
3. **重放即批处理**：需要重算时，从 Kafka 起始位置重新消费

```python
# Kappa 架构：回溯重算
def recompute_from_beginning():
    """从 Kafka 开头重新消费，实现"批处理""""
    consumer = KafkaConsumer(
        "orders",
        bootstrap_servers="kafka:9092",
        auto_offset_reset="earliest",        # 从最早开始
        enable_auto_commit=False,
        consumer_timeout_ms=300000            # 无数据后超时退出
    )

    for message in consumer:
        event = json.loads(message.value)
        process_event(event)                   # 与实时处理同一逻辑

    consumer.close()
    print("✅ 回溯重算完成")
```

### Kappa vs Lambda

| 特性          | Lambda               | Kappa                  |
| ------------- | -------------------- | ---------------------- |
| 处理引擎      | 两套（批 + 流）       | 一套（流）              |
| 代码路径      | 双份                  | 单份                    |
| 历史重算      | 批处理层直接重跑       | Kafka 回放              |
| 实现复杂度    | 高                    | 中                      |
| 延迟保证      | 批处理有延迟窗口       | 统一低延迟              |
| 适用场景      | 需要超高精度           | 大多数实时场景          |

## 6. 技术选型对比

### 6.1 流处理引擎对比

| 引擎        | 延迟     | 吞吐量  | 状态管理 | Exactly-Once | 推荐场景           |
| ----------- | -------- | ------- | -------- | ------------ | ------------------ |
| **Flink**   | 毫秒级   | 极高    | 强       | ✅           | 复杂流处理、CEP    |
| **Spark SS**| 秒级     | 高      | 中       | ✅           | 批流统一、团队熟悉  |
| **Kafka Streams** | 毫秒级 | 高    | 中       | ✅           | 轻量级、Kafka生态   |
| **Faust**   | 秒级     | 中      | 弱       | ❌           | Python快速原型     |

### 6.2 存储层选型

| 场景                | 推荐存储              | 理由                       |
| ------------------- | --------------------- | -------------------------- |
| 流式聚合查询         | ClickHouse / Druid    | 预聚合 + 低延迟 OLAP       |
| 实时点查询           | Redis / HBase         | KV存储，毫秒响应           |
| 数据湖               | S3 + Iceberg/Hudi     | 低成本 + ACID + 时间旅行    |
| 分析型仓库           | Snowflake / BigQuery  | 弹性扩展 + 标准SQL          |

## 7. 最佳实践

### 7.1 Kafka 配置优化

```python
# Producer 配置
producer_config = {
    "bootstrap.servers": "kafka:9092",
    "acks": "all",                        # 确保所有副本收到
    "retries": 2147483647,                # 无限重试
    "max.in.flight.requests.per.connection": 5,
    "enable.idempotence": True,           # 幂等生产者
    "compression.type": "lz4",            # 压缩
    "batch.size": 32768,                  # 批量大小
    "linger.ms": 10,                      # 延迟聚合
}

# Consumer 配置
consumer_config = {
    "bootstrap.servers": "kafka:9092",
    "group.id": "pipeline-consumer-group",
    "enable.auto.commit": False,           # 手动提交
    "auto.offset.reset": "earliest",
    "max.poll.records": 500,
    "isolation.level": "read_committed",   # 只读已提交消息
}
```

### 7.2 状态管理与 Checkpoint

```python
# Flink Checkpoint 配置
env.enableCheckpointing(60000)  # 每60秒一次Checkpoint
```

```python
# Spark Structured Streaming 反压
```

### 7.4 关键实践清单

- ✅ **Schema Registry**：统一管理 Kafka 消息的 schema 演进
- ✅ **Dead Letter Queue**：异常消息进入死信队列，不阻塞管道
- ✅ **水位线语义**：明确处理迟到数据策略
- ✅ **Exactly-Once 语义**：端到端保证幂等写入
- ✅ **监控指标**：消费延迟（lag）、吞吐量、错误率
- ✅ **分区策略**：按业务键分区，保证同键有序
