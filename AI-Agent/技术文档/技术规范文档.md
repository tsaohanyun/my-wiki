## 1. 概述

### 1.1 三类文档对比

| 维度 | RFC | ADR | 设计文档 |
|------|-----|-----|---------|
| 目的 | 征求意见、讨论方案 | 记录已做出的决策 | 详细设计实现方案 |
| 时机 | 方案探索阶段 | 决策做出时 | 方案确定后 |
| 状态 | Draft → Accepted/Rejected | Accepted（不可变） | Living（持续更新） |
| 粒度 | 系统级/跨团队 | 单一决策点 | 功能模块级 |
| 读者 | 所有相关方 | 未来开发者 | 实现团队 |

### 1.2 何时需要写

```
面临技术决策？
├── 需要团队讨论和共识？ → RFC
├── 已做出决策需记录？ → ADR
└── 需要详细的实现方案？ → 设计文档
```

# RFC-001: 统一身份认证服务设计

## 摘要

用 3-5 句话概括本 RFC 的核心内容和提议。

**示例**：本 RFC 提议构建一个统一的身份认证服务（Auth Service），
支持 OAuth2、SAML、JWT 三种认证方式，为所有内部系统提供集中化的
认证与授权能力，替代各系统独立的认证模块。

## 动机

### 问题描述

当前系统存在的痛点：

1. **认证逻辑分散**：8 个子系统各自实现认证，维护成本高
2. **安全风险**：部分系统使用过时的认证方案（如 MD5 密码哈希）
3. **用户体验差**：用户需要在各系统间重复登录
4. **审计困难**：无法统一追踪用户行为

### 目标

- ✅ 统一认证入口，降低维护成本
- ✅ 支持单点登录（SSO）
- ✅ 提供 OAuth2 标准接口
- ✅ 支持 RBAC + ABAC 权限模型

### 非目标

- ❌ 不负责用户管理（用户 CRUD 由现有 HR 系统负责）
- ❌ 不替代外部 IdP（如 Google、GitHub 登录）
- ❌ 不实现细粒度的 API 网关功能

## 详细设计

### 架构概述

```
┌─────────┐     ┌──────────────────┐     ┌───────────┐
│  Client  │────▶│  Auth Service    │────▶│  Redis    │
│          │     │  (Go + gRPC)     │     │  (Token)  │
└─────────┘     └────────┬─────────┘     └───────────┘
                         │
                ┌────────┴─────────┐
                │                  │
         ┌──────▼──────┐   ┌──────▼──────┐
         │ PostgreSQL   │   │  Audit Log  │
         │ (User Data)  │   │  (Kafka)    │
         └─────────────┘   └─────────────┘
```

### 核心组件

#### 1. Token Service

负责 JWT Token 的签发、验证、刷新。

- **签发**：登录成功后签发 Access Token（15min）+ Refresh Token（7d）
- **验证**：验证签名、过期时间、黑名单
- **刷新**：使用 Refresh Token 获取新的 Access Token
- **撤销**：支持主动登出，加入 Token 黑名单

#### 2. Permission Engine

基于 RBAC + ABAC 混合模型：

- **RBAC**：角色 → 权限映射（粗粒度）
- **ABAC**：属性条件判断（细粒度，如「只能访问自己部门的数据」）

### API 设计

#### 登录接口

```
POST /api/v1/auth/login
Content-Type: application/json

Request:
{
  "username": "string",
  "password": "string",
  "captcha": "string"
}

Response 200:
{
  "accessToken": "string",
  "refreshToken": "string",
  "expiresIn": 900,
  "user": {
    "id": "string",
    "name": "string",
    "roles": ["string"]
  }
}

Response 401:
{
  "error": "INVALID_CREDENTIALS",
  "message": "用户名或密码错误"
}
```

### 数据模型

```sql
CREATE TABLE users (
    id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    username    VARCHAR(50) UNIQUE NOT NULL,
    email       VARCHAR(255) UNIQUE NOT NULL,
    password    VARCHAR(255) NOT NULL,  -- bcrypt hashed
    status      VARCHAR(20) DEFAULT 'active',
    created_at  TIMESTAMPTZ DEFAULT NOW(),
    updated_at  TIMESTAMPTZ DEFAULT NOW()
);

CREATE TABLE roles (
    id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    name        VARCHAR(50) UNIQUE NOT NULL,
    description TEXT
);

CREATE TABLE permissions (
    id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    resource    VARCHAR(100) NOT NULL,
    action      VARCHAR(50) NOT NULL,
    UNIQUE(resource, action)
);

CREATE TABLE user_roles (
    user_id     UUID REFERENCES users(id),
    role_id     UUID REFERENCES roles(id),
    PRIMARY KEY (user_id, role_id)
);

CREATE TABLE role_permissions (
    role_id       UUID REFERENCES roles(id),
    permission_id UUID REFERENCES permissions(id),
    PRIMARY KEY (role_id, permission_id)
);
```

### 安全设计

- 密码使用 **bcrypt**（cost factor = 12）
- Token 使用 **RS256** 签名（非对称加密）
- 所有接口启用 **rate limiting**（100 req/min）
- 支持 **IP 白名单**（管理接口）
- 审计日志写入 **Kafka** + **Elasticsearch**

## 备选方案

### 方案 A：使用 Keycloak（被否决）

- ✅ 开箱即用，功能完善
- ❌ Java 技术栈，团队不熟悉
- ❌ 运维复杂，资源占用大
- ❌ 定制化困难

### 方案 B：自研轻量级服务（采纳）

- ✅ Go 技术栈，团队熟悉
- ✅ 轻量级，资源占用低
- ✅ 完全可控，易于定制
- ❌ 需要自研，初期投入较大

## 风险与缓解

| 风险 | 影响 | 缓解措施 |
|------|------|---------|
| 认证服务宕机 | 所有系统无法登录 | 多实例部署 + 健康检查 + Token 本地缓存 |
| Token 泄露 | 账户被盗用 | 短有效期 + 黑名单机制 + IP 绑定 |
| 性能瓶颈 | 登录延迟高 | Redis 缓存 + 连接池 + 水平扩展 |

## 迁移计划

### Phase 1: MVP（2 周）
- 实现基础登录/注册
- JWT Token 签发与验证

### Phase 2: SSO（2 周）
- 实现 OAuth2 授权码流程
- 对接第一个子系统

### Phase 3: 全量迁移（4 周）
- 逐步迁移所有子系统
- 下线旧认证模块

```
Draft → Review → Accepted
              ↘ Rejected
              ↘ Deferred
```

| 状态 | 含义 |
|------|------|
| Draft | 起草中，尚未提交评审 |

| Accepted | 评审通过，进入实现 |
| Rejected | 评审未通过 |

| Implemented | 已实现 |
| Superseded | 被新的 RFC 替代 |

## 3. ADR（架构决策记录）

### 3.1 什么是 ADR

> **ADR（Architecture Decision Record）** 是一份简短的文本文档，记录一个重要的架构决策、决策的背景以及决策的结果。

### 3.2 ADR 核心原则

- **不可变**：一旦 Accept，内容不再修改（如需变更，创建新 ADR 并标记旧 ADR 为 Superseded）
- **简洁**：一个 ADR 只记录一个决策
- **上下文驱动**：记录「为什么」而非「怎么做」
- **可追溯**：编号管理，形成决策时间线

### 3.3 ADR 模板

```markdown
# ADR-007: 选择 PostgreSQL 作为主数据库

## 状态

Accepted (2026-06-28)

## 背景

在统一身份认证服务（RFC-001）的设计中，需要选择主数据库存储用户数据和权限信息。

### 当前情况
- 团队有 PostgreSQL 和 MySQL 经验
- 现有系统主要使用 MySQL 5.7
- 数据量预估：用户 50 万，日查询量 100 万

### 约束条件
- 必须支持 JSON 字段（存储灵活的权限配置）
- 必须支持行级安全（RLS）
- 需要强大的全文搜索能力

## 决策

选择 **PostgreSQL 16** 作为主数据库。

## 候选方案

### 方案 A: MySQL 8.0

- ✅ 团队更熟悉
- ✅ 现有运维工具链完善
- ❌ JSON 查询性能不如 PostgreSQL
- ❌ 不支持行级安全（RLS）
- ❌ 全文搜索功能较弱

### 方案 B: PostgreSQL 16（采纳）

- ✅ 原生 JSONB 支持，查询性能优秀
- ✅ 支持行级安全（RLS），天然适合多租户
- ✅ 强大的全文搜索和正则匹配
- ✅ 更严格的 SQL 标准合规性
- ❌ 团队需要学习新知识
- ❌ 运维工具链需要调整

### 方案 C: MongoDB

- ✅ Schema-free，灵活
- ❌ 事务支持较弱
- ❌ 关系型查询不友好
- ❌ 团队经验不足

## 理由

1. JSONB 支持：权限配置需要灵活的 JSON 结构，PostgreSQL 的 JSONB 在查询性能上明显优于 MySQL
2. 行级安全（RLS）：认证服务需要确保用户只能访问自己的数据，RLS 是最优雅的方案
3. 长期价值：PostgreSQL 在功能丰富度上持续领先，投资回报率高

## 影响

### 正面影响
- 更灵活的数据建模能力
- 更强的数据安全保证
- 更好的查询性能（JSON 场景）

### 负面影响
- 需要培训团队（预计 1 周）
- 需要搭建新的监控和备份体系
- 部分工具（如 ORM）需要适配

### 后续行动
- [ ] 团队 PostgreSQL 培训（1 周）
- [ ] 搭建 PostgreSQL 监控（Prometheus + Grafana）
- [ ] 设计数据迁移方案（如有历史数据）
```
