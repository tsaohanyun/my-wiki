# 数据安全与隐私

> 数据安全与隐私保护是数据治理的核心要求，通过数据分类分级、访问控制、脱敏处理和合规管理，确保数据在可用性与安全性之间取得平衡。

## 概述

数据安全的目标是保护数据的**机密性（Confidentiality）**、**完整性（Integrity）**和**可用性（Availability）**（CIA三要素）。数据隐私则侧重于保护个人信息主体的合法权益。

核心原则：
- **最小权限**：只授予完成工作所需的最低权限
- **数据最小化**：只收集和使用必要的数据
- **目的限制**：数据仅用于声明的目的
- **默认保护**：默认采用最高安全级别

## 数据分类

### 分类体系

```
数据分类框架
├── 按业务域分类
│   ├── 客户数据：个人信息、交易记录、行为数据
│   ├── 财务数据：收入、成本、利润、预算
│   ├── 运营数据：供应链、库存、物流
│   ├── 人力资源：员工信息、薪酬、绩效
│   └── 研发数据：代码、设计文档、专利
├── 按数据类型分类
│   ├── 结构化数据：数据库表、CSV 文件
│   ├── 半结构化数据：JSON、XML、日志
│   └── 非结构化数据：文档、图片、音视频
└── 按敏感程度分类
    ├── 公开数据（L1）
    ├── 内部数据（L2）
    ├── 敏感数据（L3）
    ├── 机密数据（L4）
    └── 绝密数据（L5）
```

### 敏感数据识别

| 数据类别 | 典型字段 | 敏感级别 | 法规依据 |
|----------|----------|----------|----------|
| **个人身份信息** | 身份证号、护照号 | L4 | 个人信息保护法 |
| **个人敏感信息** | 生物特征、健康、金融 | L5 | 个人信息保护法 |
| **财务数据** | 银行账号、交易金额 | L4 | 金融监管要求 |
| **商业机密** | 定价策略、客户名单 | L4 | 反不正当竞争法 |
| **技术机密** | 源代码、算法模型 | L4 | 知识产权保护 |
| **员工数据** | 薪酬、绩效评估 | L3 | 劳动法 |

### 自动分类技术

1. **正则表达式**：匹配身份证号、手机号、邮箱等模式
2. **NLP 模型**：基于自然语言处理识别敏感内容
3. **机器学习**：训练分类模型自动识别敏感数据
4. **元数据标记**：基于字段名称和描述进行分类

## 访问控制

### 访问控制模型

| 模型 | 说明 | 适用场景 |
|------|------|----------|
| **DAC（自主访问控制）** | 数据所有者自行授权 | 小型团队、灵活场景 |
| **MAC（强制访问控制）** | 基于安全标签的强制策略 | 军事、政府等高安全场景 |
| **RBAC（基于角色的访问控制）** | 按角色分配权限 | 企业通用场景 |
| **ABAC（基于属性的访问控制）** | 基于用户/资源/环境属性 | 复杂细粒度场景 |

### 权限模型设计

```
RBAC 权限模型：
┌─────────┐    ┌─────────┐    ┌─────────┐
│  用户    │───→│  角色    │───→│  权限    │
└─────────┘    └─────────┘    └─────────┘
                    │               │
                    ▼               ▼
              ┌─────────┐    ┌─────────┐
              │ 角色继承 │    │ 资源+操作│
              └─────────┘    └─────────┘

示例角色：
├── 数据管理员：全部权限（读写删+授权）
├── 数据分析师：读取权限 + 部分脱敏数据
├── 业务用户：读取本部门数据
├── 外部用户：仅读取公开数据
└── 审计员：只读 + 审计日志访问
```

### 权限管理流程

1. **权限申请**：用户提交权限申请，说明用途
2. **审批流程**：直属上级 → 数据所有者 → 安全管理员
3. **权限生效**：系统自动配置权限
4. **定期审查**：季度权限审查，回收不再需要的权限
5. **离职回收**：员工离职时自动回收所有权限

## 脱敏策略

### 脱敏方式

| 方式 | 说明 | 示例 | 适用场景 |
|------|------|------|----------|
| **遮蔽** | 部分字符替换为* | 138****5678 | 显示展示 |
| **替换** | 用假数据替换真数据 | 张三 → 用户A | 测试环境 |
| **加密** | 可逆加密 | AES加密 | 需要还原时 |
| **哈希** | 不可逆哈希 | SHA256 | 关联分析 |
| **截断** | 保留部分精度 | 2026-06 → 2026 | 统计分析 |
| **偏移** | 数值加随机偏移 | 100 → 95~105 | 数值型数据 |
| **置乱** | 打乱数据顺序 | 列值随机重排 | 保持分布特征 |

### 脱敏规则配置

```yaml
# 脱敏策略配置示例
masking_rules:
  - name: "手机号脱敏"
    field: phone
    strategy: mask
    pattern: "138****5678"
    preserve_first: 3
    preserve_last: 4

  - name: "身份证号脱敏"
    field: id_card
    strategy: mask
    pattern: "110***********1234"
    preserve_first: 3
    preserve_last: 4

  - name: "姓名脱敏"
    field: name
    strategy: mask
    pattern: "张*"
    preserve_first: 1

  - name: "邮箱脱敏"
    field: email
    strategy: hash
    algorithm: sha256

  - name: "金额脱敏"
    field: amount
    strategy: range
    bucket_size: 1000
```

### 脱敏场景

| 场景 | 脱敏方式 | 说明 |
|------|----------|------|
| **生产环境展示** | 遮蔽 | 用户看到的部分脱敏数据 |
| **开发测试** | 替换 | 使用假数据替代真实数据 |
| **数据分析** | 统计脱敏 | 保持统计特征，隐藏个体 |
| **数据共享** | 加密/哈希 | 可控的数据共享 |
| **日志记录** | 遮蔽 | 日志中的敏感信息脱敏 |

## 合规要求

### 主要法规

| 法规 | 地域 | 核心要求 |
|------|------|----------|
| **《个人信息保护法》** | 中国 | 个人信息处理规则、跨境传输限制 |
| **《数据安全法》** | 中国 | 数据分类分级、重要数据保护 |
| **《网络安全法》** | 中国 | 网络运营者义务、数据本地化 |
| **GDPR** | 欧盟 | 数据主体权利、数据保护官 |
| **CCPA/CPRA** | 美国加州 | 消费者隐私权、数据销售限制 |
| **HIPAA** | 美国 | 医疗健康信息保护 |
| **PCI DSS** | 全球 | 支付卡数据安全标准 |

### 合规检查清单

```
□ 数据分类分级是否完成
□ 隐私政策是否更新并公示
□ 用户同意机制是否合规
□ 数据主体权利响应机制是否建立
  □ 知情权
  □ 访问权
  □ 更正权
  □ 删除权
  □ 可携带权
  □ 拒绝自动化决策权
□ 数据跨境传输是否合规
□ 数据安全影响评估是否完成
□ 数据泄露应急响应计划是否就绪
□ 数据保护官（DPO）是否指定
□ 第三方数据处理协议是否签署
□ 数据保留期限是否明确
```

### 数据安全影响评估（DSIA）

1. **识别数据处理活动**：梳理数据收集、存储、使用、共享、删除全流程
2. **评估风险**：识别数据泄露、滥用、丢失等风险
3. **确定保护措施**：制定技术措施和管理措施
4. **持续监控**：定期评估措施有效性

## 实施方法论

### 阶段一：安全评估（2-4周）

1. 梳理数据资产，识别敏感数据
2. 评估现有安全措施的有效性
3. 识别合规差距和风险点
4. 制定安全治理路线图

### 阶段二：制度建设（4-6周）

1. 制定数据安全管理制度
2. 建立数据分类分级标准
3. 设计访问控制策略
4. 制定脱敏规则

### 阶段三：技术实施（8-12周）

1. 部署数据安全平台
2. 实施访问控制系统
3. 配置脱敏规则
4. 建立审计日志系统

### 阶段四：运营监控（持续）

1. 安全事件监控和响应
2. 定期安全审计
3. 合规性持续评估
4. 安全意识培训
