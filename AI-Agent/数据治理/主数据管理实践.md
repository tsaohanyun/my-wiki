# 主数据管理实践

> 主数据管理（Master Data Management, MDM）是确保企业核心业务实体数据的一致性、准确性、完整性和权威性的系统性方法论。

## 概述

主数据是企业跨系统共享的核心业务实体数据，包括客户、产品、供应商、员工、组织架构等。MDM 的目标是建立**单一可信数据源（Single Source of Truth）**，消除数据孤岛，实现数据的一致性治理。

## MDM 架构

### 架构模式

| 模式 | 描述 | 适用场景 |
|------|------|----------|
| **注册模式（Registry）** | 各系统保留数据所有权，MDM 仅建立索引和交叉引用 | 数据源多、改造成本敏感 |
| **共存模式（Coexistence）** | 各系统保留数据，MDM 维护一份统一视图 | 渐进式治理、过渡阶段 |
| **集中模式（Consolidation）** | 将各系统数据汇聚到中央 MDM 存储 | 需要统一报表和分析 |
| **交易模式（Transactional）** | MDM 作为主数据的唯一写入点，分发到下游系统 | 严格数据管控、新建系统 |

### 技术架构层次

```
┌─────────────────────────────────────────────────┐
│               数据消费层                         │
│   BI报表 │ 业务系统 │ 数据分析 │ API服务        │
├─────────────────────────────────────────────────┤
│               数据分发层                         │
│   事件驱动 │ 批量同步 │ API接口 │ 消息队列       │
├─────────────────────────────────────────────────┤
│               MDM 核心平台                       │
│   数据模型 │ 匹配合并 │ 数据质量 │ 工作流审批    │
├─────────────────────────────────────────────────┤
│               数据集成层                         │
│   ETL │ CDC │ API采集 │ 文件导入                 │
├─────────────────────────────────────────────────┤
│               数据源系统                         │
│   ERP │ CRM │ SCM │ HR系统 │ 自研系统           │
└─────────────────────────────────────────────────┘
```

### 关键组件

1. **数据模型管理**：定义主数据实体的标准模型和扩展属性
2. **匹配与合并引擎**：基于规则和机器学习的去重、合并
3. **数据质量引擎**：实时校验和批量质量评估
4. **工作流引擎**：数据变更审批、异常处理流程
5. **数据分发服务**：将黄金记录同步到下游系统

## 数据标准

### 标准制定流程

1. **识别主数据域**：梳理业务实体，确定需要治理的主数据类型
2. **定义数据标准**：
   - 属性命名规范（中英文命名、编码规则）
   - 数据类型与格式（日期格式、编码长度）
   - 值域范围（枚举值、参考数据）
   - 唯一性约束（主键、业务键）
3. **建立数据字典**：记录每个属性的业务含义、技术定义和负责人
4. **版本管理**：标准变更的版本控制和变更记录

### 命名规范示例

```
客户主数据属性标准：
├── 客户编码 (customer_code)    - VARCHAR(20) - 格式: C+YYYYMMDD+6位序号
├── 客户名称 (customer_name)    - VARCHAR(200) - 必填
├── 客户类型 (customer_type)    - ENUM(企业/个人/政府) - 必填
├── 统一社会信用代码 (uscc)     - CHAR(18) - 企业客户必填
├── 联系电话 (phone)            - VARCHAR(20) - 格式校验
├── 地址 (address)              - VARCHAR(500)
└── 状态 (status)               - ENUM(活跃/暂停/注销)
```

## 数据质量

### 主数据质量维度

| 维度 | 说明 | 检查方式 |
|------|------|----------|
| **完整性** | 必填字段是否有值 | 空值检查 |
| **准确性** | 数据是否反映真实情况 | 外部数据比对、人工抽检 |
| **一致性** | 跨系统数据是否一致 | 交叉比对 |
| **唯一性** | 是否存在重复记录 | 匹配规则检测 |
| **时效性** | 数据是否及时更新 | 更新频率监控 |
| **合规性** | 是否符合标准规范 | 规则引擎校验 |

### 质量评分模型

```
质量得分 = Σ(维度权重 × 维度得分)

示例：
完整性(30%) × 95 + 准确性(25%) × 90 + 一致性(20%) × 88
+ 唯一性(15%) × 92 + 时效性(10%) × 85 = 91.35
```

## 数据分发

### 分发策略

- **实时分发**：基于事件驱动（CDC + Kafka），适用于时效性要求高的场景
- **准实时分发**：基于消息队列的异步同步，延迟控制在分钟级
- **批量分发**：定时批量推送，适用于非实时场景
- **按需分发**：通过 API 按需查询，适用于低频场景

### 分发架构

```
MDM 平台
  │
  ├── CDC 捕获变更 ──→ Kafka ──→ 下游消费组
  │                                ├── 数据仓库
  │                                ├── 业务系统A
  │                                └── 业务系统B
  │
  ├── REST API ──→ 按需查询服务
  │
  └── 批量导出 ──→ SFTP/文件共享 ──→ 遗留系统
```

## 实施方法论

### 阶段一：规划（4-6周）

1. 评估现状，识别痛点和优先级
2. 确定主数据域范围和治理目标
3. 建立数据治理组织（数据所有者、数据管家）
4. 选择试点业务域（建议从客户或产品开始）

### 阶段二：设计（6-8周）

1. 设计主数据模型和数据标准
2. 制定匹配合并规则
3. 设计数据质量规则集
4. 规划系统集成方案

### 阶段三：建设（12-16周）

1. 部署 MDM 平台
2. 开发数据集成管道
3. 配置匹配合并规则
4. 实现数据质量校验

### 阶段四：推广（持续）

1. 试点上线并验证效果
2. 推广到更多主数据域
3. 持续优化规则和流程
4. 建立运营度量体系
