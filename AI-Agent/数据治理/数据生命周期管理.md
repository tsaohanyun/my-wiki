# 数据生命周期管理

> 数据生命周期管理是对数据从创建到销毁全过程的系统性管理，确保数据在各阶段得到适当的安全保护、质量保障和成本优化。

## 概述

数据生命周期管理（Data Lifecycle Management, DLM）涵盖数据的完整生命历程，目标是：

- **合规性**：满足法规对数据保留和销毁的要求
- **成本优化**：通过分层存储降低存储成本
- **性能优化**：热数据快速访问，冷数据不占用资源
- **风险管理**：及时销毁不再需要的数据，降低泄露风险

## 数据生命周期阶段

### 全景视图

```
创建 → 采集 → 存储 → 处理 → 分析 → 共享 → 归档 → 销毁
 │      │      │      │      │      │      │      │
 ▼      ▼      ▼      ▼      ▼      ▼      ▼      ▼
质量   校验   分层   ETL    BI    API   冷热   擦除
标准   清洗   策略   计算   报表   共享   分离   审计
```

### 各阶段详细说明

| 阶段 | 关键活动 | 管理要点 |
|------|----------|----------|
| **创建** | 数据录入、系统生成 | 数据标准、质量校验 |
| **采集** | 数据抽取、导入 | 数据清洗、格式转换 |
| **存储** | 持久化存储 | 存储分层、备份策略 |
| **处理** | ETL、数据加工 | 数据质量、血缘追踪 |
| **分析** | 报表、数据挖掘 | 访问控制、脱敏策略 |
| **共享** | 数据交换、API | 权限管理、审计日志 |
| **归档** | 冷数据归档 | 保留策略、恢复能力 |
| **销毁** | 数据删除、擦除 | 审批流程、不可恢复 |

## 数据创建

### 数据来源分类

```
数据来源
├── 内部生成
│   ├── 业务系统：交易记录、操作日志
│   ├── IoT 设备：传感器数据、设备日志
│   ├── 用户行为：点击流、操作轨迹
│   └── 内部文档：报告、方案、制度
├── 外部采集
│   ├── 第三方数据：市场数据、行业数据
│   ├── 公开数据：政府公开数据、开放数据集
│   └── 合作伙伴：供应链数据、渠道数据
└── 数据加工
    ├── ETL 转换：清洗、聚合、关联
    ├── 模型输出：预测结果、推荐列表
    └── 分析结果：报表、指标、洞察
```

### 创建阶段管理要求

1. **数据标准执行**：确保数据符合命名、格式、编码标准
2. **质量门禁**：创建时即进行质量校验
3. **元数据登记**：自动采集技术元数据，补充业务元数据
4. **分类分级标记**：自动或手动标记数据敏感级别
5. **所有权确认**：明确数据所有者和管理者

## 数据存储

### 存储分层策略

| 层级 | 存储介质 | 访问频率 | 保留时间 | 成本 | 适用数据 |
|------|----------|----------|----------|------|----------|
| **热存储** | SSD/内存 | 频繁 | 0-3个月 | 高 | 当前业务数据 |
| **温存储** | SAS/SSD | 定期 | 3-12个月 | 中 | 近期历史数据 |
| **冷存储** | SATA/对象存储 | 偶尔 | 1-5年 | 低 | 历史归档数据 |
| **冰存储** | 磁带/冰川存储 | 极少 | 5年以上 | 极低 | 法规要求保留数据 |

### 存储架构

```
┌─────────────────────────────────────────────────┐
│                数据访问层                        │
│   SQL查询 │ API │ BI工具 │ 文件访问             │
├─────────────────────────────────────────────────┤
│                数据虚拟化层                      │
│   统一视图 │ 数据联邦 │ 查询路由                │
├──────────┬──────────┬──────────┬────────────────┤
│  热存储   │  温存储   │  冷存储   │  冰存储        │
│  SSD/内存 │  SAS/SSD  │  对象存储 │  磁带/归档     │
│  0-3月    │  3-12月   │  1-5年    │  5年+          │
│  高性能   │  均衡     │  低成本   │  最低成本      │
└──────────┴──────────┴──────────┴────────────────┘
           ← 自动分层迁移 →
```

### 存储管理策略

1. **容量规划**：基于数据增长预测规划存储容量
2. **自动分层**：基于访问频率和数据年龄自动迁移
3. **数据压缩**：冷数据压缩存储，节省空间
4. **去重**：消除重复数据，减少存储占用
5. **备份策略**：3-2-1 备份原则（3份副本、2种介质、1份异地）

## 数据归档

### 归档策略

```yaml
# 归档策略配置示例
archive_policies:
  - name: "订单数据归档"
    source_table: orders
    conditions:
      - field: order_date
        operator: older_than
        value: "2 years"
      - field: status
        operator: equals
        value: "completed"
    destination: cold_storage
    retention: "7 years"
    compression: gzip
    encryption: aes256

  - name: "日志数据归档"
    source_table: system_logs
    conditions:
      - field: log_date
        operator: older_than
        value: "6 months"
    destination: ice_storage
    retention: "3 years"
    compression: lz4

  - name: "用户行为数据归档"
    source_table: user_events
    conditions:
      - field: event_date
        operator: older_than
        value: "1 year"
    destination: object_storage
    retention: "5 years"
    compression: snappy
```

### 归档流程

```
1. 识别归档候选数据
   ↓
2. 验证保留策略合规性
   ↓
3. 创建归档快照
   ↓
4. 执行数据迁移
   ↓
5. 验证归档完整性
   ↓
6. 更新元数据和血缘
   ↓
7. 删除源数据（可选）
   ↓
8. 记录归档日志
```

## 数据销毁

### 销毁方式

| 方式 | 安全级别 | 说明 | 适用场景 |
|------|----------|------|----------|
| **逻辑删除** | 低 | 标记为已删除，数据仍存在 | 一般业务数据 |
| **覆写擦除** | 中 | 用随机数据多次覆写 | 磁盘数据 |
| **消磁** | 高 | 磁性介质消磁处理 | 磁带、硬盘 |
| **物理销毁** | 最高 | 物理粉碎、焚烧 | 高密级介质 |

### 销毁流程

```
销毁申请 → 审批确认 → 保留期验证 → 执行销毁 → 销毁验证 → 审计记录
    │          │          │           │          │          │
    ▼          ▼          ▼           ▼          ▼          ▼
 数据管家    数据所有者   法务确认    技术团队    审计团队    合规团队
```

### 销毁审批要求

1. **法律合规确认**：法务确认无保留义务
2. **业务影响评估**：确认无业务依赖
3. **备份清理**：同步清理备份副本
4. **元数据更新**：标记数据状态为已销毁
5. **审计记录**：保留销毁操作的审计日志

## 保留策略

### 保留期限参考

| 数据类型 | 保留期限 | 法规依据 |
|----------|----------|----------|
| 财务凭证 | 10-30年 | 会计法、税法 |
| 合同文件 | 合同终止后10年 | 合同法 |
| 员工档案 | 离职后2年 | 劳动法 |
| 客户交易记录 | 5-10年 | 反洗钱法 |
| 系统日志 | 6个月-3年 | 网络安全法 |
| 个人信息 | 实现目的所需最短时间 | 个人信息保护法 |
| 视频监控 | 30天-90天 | 公共安全管理 |

### 保留策略配置

```yaml
retention_policies:
  - name: "财务数据保留"
    data_category: financial
    retention_period: 10 years
    legal_basis: "会计档案管理办法"
    archive_before_delete: true
    approval_required: true

  - name: "个人信息保留"
    data_category: personal_info
    retention_period: purpose_based
    legal_basis: "个人信息保护法"
    conditions:
      - "用户注销后30天删除"
      - "业务完成后1年删除"
    exceptions:
      - "法律要求的保留"

  - name: "日志数据保留"
    data_category: system_logs
    retention_period: 1 year
    archive_after: 6 months
    auto_delete: true
```

## 实施方法论

### 阶段一：现状评估（2-4周）

1. 梳理数据资产清单和存储现状
2. 评估数据增长趋势和存储成本
3. 识别合规要求和保留义务
4. 评估现有生命周期管理能力

### 阶段二：策略制定（4-6周）

1. 制定数据分类分级标准
2. 设计存储分层策略
3. 制定保留和归档策略
4. 制定销毁流程和审批机制

### 阶段三：技术实施（8-12周）

1. 部署存储分层和自动迁移系统
2. 实现归档自动化
3. 配置保留策略引擎
4. 建立销毁审批和审计系统

### 阶段四：运营优化（持续）

1. 监控存储使用和成本
2. 优化分层和归档策略
3. 定期合规审查
4. 持续改进流程
