# 数据质量管理框架

> 数据质量管理是通过建立质量标准、执行质量检查、监控质量趋势和持续改进，确保数据满足业务需求的系统性方法。

## 概述

数据质量直接影响业务决策的准确性和运营效率。低质量数据导致：
- 决策失误和业务损失
- 客户体验下降
- 合规风险增加
- 运营成本上升

数据质量管理框架的核心理念：**预防为主，检测为辅，持续改进**。

## 质量维度

### 六大核心维度

| 维度 | 定义 | 度量方式 | 典型问题 |
|------|------|----------|----------|
| **完整性** | 数据是否缺失 | 空值率、缺失率 | 必填字段为空 |
| **准确性** | 数据是否正确 | 与真实值的偏差率 | 地址错误、金额偏差 |
| **一致性** | 跨系统数据是否一致 | 差异率 | 同一客户不同系统名称不同 |
| **唯一性** | 是否存在重复 | 重复率 | 客户重复记录 |
| **时效性** | 数据是否及时更新 | 延迟时间 | T+1 数据未按时产出 |
| **合规性** | 是否符合标准规范 | 违规率 | 编码不符合规范 |

### 扩展维度

- **可用性**：数据是否可被正确理解和使用
- **可追溯性**：数据来源和变更历史是否可追溯
- **稳定性**：数据质量是否保持稳定

## 质量规则

### 规则分类

```
质量规则
├── 基础规则
│   ├── 非空检查 (NOT NULL)
│   ├── 类型检查 (TYPE CHECK)
│   ├── 格式检查 (PATTERN CHECK)
│   └── 长度检查 (LENGTH CHECK)
├── 业务规则
│   ├── 值域检查 (RANGE CHECK)
│   ├── 枚举检查 (ENUM CHECK)
│   ├── 逻辑检查 (LOGIC CHECK)
│   └── 交叉检查 (CROSS CHECK)
├── 统计规则
│   ├── 波动检查 (FLUCTUATION CHECK)
│   ├── 分布检查 (DISTRIBUTION CHECK)
│   ├── 异常值检查 (OUTLIER CHECK)
│   └── 趋势检查 (TREND CHECK)
└── 关系规则
    ├── 引用完整性 (REFERENTIAL INTEGRITY)
    ├── 主键唯一性 (PRIMARY KEY)
    └── 外键约束 (FOREIGN KEY)
```

### 规则定义示例

```yaml
# 订单金额质量规则
rules:
  - name: "订单金额非空"
    type: completeness
    table: orders
    column: amount
    check: "amount IS NOT NULL"
    severity: critical

  - name: "订单金额正数"
    type: accuracy
    table: orders
    column: amount
    check: "amount > 0"
    severity: critical

  - name: "订单金额波动检查"
    type: statistical
    table: orders
    column: total_amount
    check: "日波动率 < 50%"
    severity: warning

  - name: "客户编码格式"
    type: compliance
    table: customers
    column: customer_code
    check: "REGEXP '^[C]\\d{13}$'"
    severity: major
```

### 规则管理

- **规则库**：集中管理所有质量规则，支持版本控制
- **规则模板**：可复用的规则模板，降低配置成本
- **规则优先级**：按业务影响设定 critical/major/minor/warning
- **规则依赖**：规则之间的执行顺序和依赖关系

## 质量监控

### 监控体系

```
┌─────────────────────────────────────────────┐
│              质量仪表盘                       │
│   总体评分 │ 趋势图 │ 问题分布 │ 告警列表    │
├─────────────────────────────────────────────┤
│              告警管理                         │
│   告警规则 │ 通知策略 │ 升级机制 │ 处理跟踪   │
├─────────────────────────────────────────────┤
│              质量检测引擎                     │
│   实时检测 │ 批量检测 │ 抽样检测 │ 探查分析   │
├─────────────────────────────────────────────┤
│              数据源层                         │
│   数据库 │ 数据仓库 │ 数据湖 │ API           │
└─────────────────────────────────────────────┘
```

### 监控指标

| 指标类别 | 具体指标 | 说明 |
|----------|----------|------|
| **质量得分** | 总体质量分、各维度得分 | 综合评估数据质量 |
| **问题统计** | 问题数量、问题分布、严重程度 | 量化质量问题 |
| **趋势分析** | 质量得分趋势、问题趋势 | 识别质量变化 |
| **覆盖率** | 规则覆盖率、数据覆盖率 | 评估监控完整性 |
| **处理效率** | 平均处理时间、解决率 | 评估改进效果 |

### 告警策略

```
告警级别：
├── P0 - 紧急：核心指标数据异常，影响生产
│   → 立即通知 + 电话 + 自动停服
├── P1 - 严重：数据质量严重下降
│   → 即时通知 + 工单创建
├── P2 - 一般：数据质量轻微异常
│   → 日报汇总 + 限期整改
└── P3 - 提示：数据质量预警
    → 周报汇总 + 跟踪观察
```

## 质量改进

### 改进流程（PDCA）

```
Plan（计划）──→ Do（执行）──→ Check（检查）──→ Act（改进）
    │                │               │               │
    ▼                ▼               ▼               ▼
 识别问题         实施修复        验证效果        固化经验
 分析根因         配置规则        监控趋势        优化规则
 制定方案         修复数据        评估收益        推广应用
```

### 根因分析方法

1. **5Why 分析**：连续追问"为什么"，找到根本原因
2. **鱼骨图**：从人、机、料、法、环、测六方面分析
3. **帕累托分析**：识别导致 80% 问题的 20% 关键因素
4. **数据溯源**：通过血缘关系追踪问题数据的源头

### 改进策略

| 策略 | 说明 | 适用场景 |
|------|------|----------|
| **源头治理** | 在数据产生时即保证质量 | 新建系统、重大改造 |
| **过程控制** | 在数据流转过程中拦截问题 | ETL 过程、数据集成 |
| **事后修复** | 发现问题后进行数据修复 | 历史数据、存量问题 |
| **制度保障** | 通过管理制度约束数据质量 | 组织层面的长期治理 |

## 实施方法论

### 阶段一：质量评估（2-4周）

1. 识别关键数据资产和业务场景
2. 进行数据质量现状评估（探查分析）
3. 建立质量基线和度量指标
4. 确定治理优先级

### 阶段二：规则建设（4-6周）

1. 制定数据质量标准
2. 设计质量规则库
3. 配置规则引擎
4. 建立告警和通知机制

### 阶段三：监控运营（持续）

1. 部署质量监控平台
2. 运行质量检测任务
3. 处理质量告警和问题
4. 生成质量报告

### 阶段四：持续改进（持续）

1. 分析质量问题根因
2. 推动源头治理
3. 优化质量规则
4. 度量改进效果
