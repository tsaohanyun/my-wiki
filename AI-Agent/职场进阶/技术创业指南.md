# 技术创业指南

> 技术创业最难的不是写代码，而是从"我能做什么"转向"用户需要什么"。

---

## 📌 技术人创业的核心挑战

技术人创业有天然优势——能快速实现产品原型，但也面临独特挑战：

| 挑战 | 表现 | 应对 |
|------|------|------|
| 技术执念 | 过度追求技术完美，忽视市场验证 | 以MVP思维驱动，先验证再优化 |
| 用户盲区 | 习惯以技术视角而非用户视角思考 | 建立持续的用户访谈和反馈机制 |
| 商业薄弱 | 对定价、获客、商业模式缺乏经验 | 主动学习商业知识，找互补合伙人 |
| 全能幻觉 | 试图一个人搞定所有事 | 尽早组建互补团队，学会分权 |

## 一、技术选型

### 1.1 技术选型的核心原则

> **原则：选择让你最快验证假设的技术，而不是最酷的技术。**

```
技术选型的优先级排序：
1. 团队最熟悉的技术栈（降低学习成本）
2. 生态成熟、文档完善（降低踩坑概率）
3. 社区活跃、人才可获取（降低招人难度）
4. 运维成本低（创业初期没有专职运维）
5. 能支撑 10x 增长（但不为不存在的100x过度设计）
```

### 1.2 不同阶段的技术策略

| 阶段 | 技术策略 | 核心原则 |
|------|---------|---------|
| 验证期（0-100用户） | 单体应用 + 托管服务 | 速度优先，用最简方式上线 |
| 增长期（100-10K用户） | 模块化单体 + 关键路径优化 | 保持简单，按瓶颈优化 |
| 规模期（10K+用户） | 按需拆分微服务 | 按团队规模和瓶颈驱动拆分 |

### 1.3 创业技术栈推荐（2026年视角）

**AI Agent / LLM 应用方向**：

```
前端：
  框架：Next.js（SSR + 前端一体）
  UI：TailwindCSS + shadcn/ui
  状态：Zustand（轻量）
  部署：Vercel / Cloudflare Pages

后端：
  语言：Python（AI生态最佳）或 Go（性能优先）
  框架：FastAPI（Python）/ Gin（Go）
  API：tRPC（类型安全）或 RESTful

AI/ML：
  LLM接入：OpenAI API / Claude API / 本地开源模型
  框架：LangChain / LlamaIndex / Vercel AI SDK
  向量库：Pinecone（托管）/ Milvus（自建）
  推理优化：vLLM / TensorRT-LLM

基础设施：
  托管：AWS / 阿里云 / 腾讯云
  容器：Docker + Serverless（初期）
  数据库：PostgreSQL（关系型）+ Redis（缓存）
  消息队列：托管型（如 AWS SQS）
  监控：Datadog / Grafana Cloud
```

### 1.4 技术选型决策记录模板

```
决策：选择 [方案A] 而非 [方案B]

背景：
  当前阶段：[MVP / 增长 / 规模]
  团队规模：[X] 人
  团队技能：[列出相关技能]

方案A的优势：
  1. 团队已有 [X] 年经验
  2. 社区活跃，遇到问题容易找到解决方案
  3. 招人成本低

方案B的劣势：
  1. 学习曲线陡峭
  2. 生态不够成熟
  3. 可能需要 3-6 个月的摸索期

决策时间窗：可逆的，未来 12 个月内可以迁移
```

## 二、MVP构建

### 2.1 什么是真正的MVP？

> MVP（Minimum Viable Product）不是"功能最少的版本"，而是"能验证核心假设的最小产品"。

```
MVP 的核心问题：
  1. 我们要验证什么假设？
  2. 验证这个假设最少需要什么功能？
  3. 如何度量验证结果？
```

### 2.2 MVP构建方法论

**方法一：假设-实验框架**

```
假设："用户愿意为AI驱动的代码审查工具每月支付 $29"

实验设计：
  - Landing Page + Waitlist（1周）
    → 验证兴趣度（注册转化率 > 5%）

  - 假门测试（Fake Door Test）（1周）
    → 展示"付费"按钮，点击率 > 3%

  - 手动MVP（Wizard of Oz）（2-4周）
    → 前端自动化，后端人工处理
    → 验证用户是否真的需要这个功能

  - 最小可用产品（4-8周）
    → 核心功能自动化
    → 验证付费意愿和留存
```

**方法二：用户旅程优先级**

```
用户旅程：
  发现 → 注册 → 体验核心功能 → 获得价值 → 留存 → 付费

MVP 聚焦：
  ❌ 不要做：精美的官网、复杂的后台、所有边缘场景
  ✅ 必须做：核心功能闭环、关键路径流畅、基础的转化引导
```

### 2.3 MVP 速度优化技巧

| 技巧 | 描述 | 节省时间 |
|------|------|---------|
| 用模板和脚手架 | Next.js/SAAS模板直接改 | 1-2周 |
| 先用托管服务 | 认证用Auth0、支付用Stripe | 1-2周 |
| 暂时人工处理 | 早期订单用Excel管理 | 2-3周 |
| 开源组件复用 | UI组件用shadcn/ui | 1周 |
| 不做后台 | 早期用数据库直接看数据 | 2-4周 |

### 2.4 度量MVP成功的关键指标

```
AARRR 海盗指标：
  Acquisition（获取）：用户从哪来？
  Activation（激活）：用户第一次感受到价值了吗？
  Retention（留存）：用户会回来吗？
  Revenue（收入）：用户付费了吗？
  Referral（推荐）：用户会推荐给别人吗？

关键阈值（参考）：
  - 注册→激活转化率 > 25%
  - 周留存率 > 40%（第一周回来）
  - 付费转化率 > 3%
  - NPS > 30
```

## 三、团队组建

### 3.1 创业团队的黄金三角

```
        产品（Product）
          /         \
         /           \
    技术（Tech）—— 商业（Business）
```

| 角色 | 核心职责 | 为什么需要 |
|------|---------|-----------|
| 技术合伙人 | 产品实现、技术架构 | 没有技术就无法实现产品 |
| 产品合伙人 | 用户研究、产品设计 | 没有产品洞察就会做错方向 |
| 商业合伙人 | 市场拓展、融资、运营 | 没有商业能力就无法变现 |

### 3.2 找合伙人

**标准**：
1. 价值观一致（最重要）
2. 能力互补（不要找和你一样的人）
3. 全职投入（兼职创业成功率极低）
4. 风险承受能力匹配

**渠道**：
- 之前的同事/同学（最靠谱）
- 创业社区（如 Indie Hackers、V2EX、即刻）
- 创业活动（黑客松、创业比赛）
- 创业孵化器/加速器

### 3.3 早期团队架构

**0-5人阶段**：
```
创始人（全栈）
  → 什么都做，速度第一
  → 没有明确的角色分工
```

**5-15人阶段**：
```
创始人（战略 + 产品）
  ├── 技术团队（2-3人）
  │     → 1个技术Lead + 工程师
  ├── 产品/设计（1-2人）
  │     → 产品经理 + UI/UX设计
  └── 运营/商务（1-2人）
        → 市场推广 + 用户运营
```

**15-50人阶段**：
```
CEO（战略 + 融资）
  ├── CTO（技术管理）
  ├── CPO（产品管理）
  ├── COO（运营管理）
  └── HR/Finance（支持职能）
```

### 3.4 股权分配

| 阶段 | 股权分配原则 |
|------|------------|
| 2人创始团队 | 50/50 或 55/45（避免平均主义陷阱） |
| 3人创始团队 | 40/30/30 或 33/33/33 |
| 员工期权池 | 预留 10-20% 的期权池 |
| 顾问 | 0.5-1% |

> ⚠️ **重要提醒**：所有创始人的股权都应该有 **vesting schedule**（通常 4 年 vesting + 1 年 cliff），防止联合创始人早期离开导致股权僵局。

## 四、融资

### 4.1 融资阶段一览

| 轮次 | 资金量（参考） | 核心目标 | 投资人关注 |
|------|--------------|---------|-----------|
| 种子轮（Pre-Seed/Seed） | $50K-$1M | 验证PMF | 团队、想法、早期数据 |
| A轮 | $2M-$10M | 扩大规模 | 增长数据、单位经济模型 |
| B轮 | $10M-$50M | 加速增长 | 收入、市场份额、可复制性 |
| C轮+ | $50M+ | 规模化、IPO准备 | 盈利能力、市场领导力 |

### 4.2 融资Pitch Deck 结构

```
1. 一句话介绍（10秒讲清楚）
   "我们是 [公司名]，用 AI 帮助开发者 [解决什么问题]。"

2. 问题
   → 目标用户的痛点是什么？
   → 现有解决方案为什么不够好？

3. 解决方案
   → 你的产品如何解决问题？
   → 比现有方案好在哪里？

4. 市场
   → TAM（总可用市场）/ SAM / SOM
   → 市场增长趋势

5. 产品（Demo）
   → 展示核心功能
   → 强调差异化

6. 商业模式
   → 怎么赚钱？
   → 单位经济模型（LTV/CAC）

7. 牵引力
   → 用户增长、收入、关键合作
   → 数据趋势向上

8. 竞争分析
   → 竞品矩阵
   → 你的护城河

9. 团队
   → 为什么是你们？
   → 团队的独特优势

10. 融资需求
    → 融多少钱？
    → 钱怎么花？（里程碑）
```

### 4.3 技术人融资的注意事项

**要做的**：
- ✅ 用数据说话（用户数、收入、增长率）
- ✅ 展示技术壁垒和可防御性
- ✅ 找懂技术的投资人
- ✅ 准备好被问"你不懂商业"的回答

**不要做的**：
- ❌ 过度强调技术而忽视商业逻辑
- ❌ 在 Pitch 中展示过多代码细节
- ❌ 因为估值低就拒绝好的投资人
- ❌ 在需要钱的时候才找钱（永远提前 6 个月开始）

## 五、产品市场匹配（PMF）

### 5.1 什么是PMF？

> PMF（Product-Market Fit）= 你的产品恰好满足了一个足够大的市场的真实需求。

**Marc Andreessen 的定义**：
> "Product/market fit means being in a good market with a product that can satisfy that market."

### 5.2 PMF 的信号

**有PMF的感觉**：
```
✅ 用户在主动拉人来用你的产品
✅ 你在忙着扩容而不是找用户
✅ 用户留存曲线趋于平稳（不再持续下降）
✅ 用户在催你加功能，而不是在抱怨
✅ 你开始拒绝一些用户需求（因为太忙了）
```

**没有PMF的感觉**：
```
❌ 产品用起来"还行"，但用户没有激动
❌ 获客成本持续高于用户终身价值
❌ 用户流失率居高不下
❌ 你不知道下一个用户从哪来
❌ 方向频繁变动，总在追风口
```

### 5.3 PMF 量化指标

| 指标 | 达标 | 优秀 |
|------|------|------|
| Sean Ellis 测试（"非常失望如果不能用"的比例） | > 40% | > 60% |
| NPS（净推荐值） | > 30 | > 50 |
| 月留存率 | > 25% | > 40% |
| LTV/CAC | > 3 | > 5 |
| 自然增长占比 | > 30% | > 50% |

### 5.4 如果没有PMF怎么办？

```
策略一：Pivot（转型）
  → 保留核心能力，改变目标市场或产品形态
  → 示例：Slack 从游戏公司内部工具转型为企业通讯

策略二：深耕
  → 深入理解少数核心用户的需求
  → 把一个细分场景做到极致
  → 示例：Superhuman 专注做"最快的企业邮件"

策略三：加速
  → 已经有PMF信号，加大投入
  → 扩大市场、优化产品、加大获客
```
