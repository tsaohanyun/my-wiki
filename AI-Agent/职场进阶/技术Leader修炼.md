# 技术Leader修炼

> 从优秀工程师到技术Leader，最大的转变不是技术能力的提升，而是认知模式的切换——从"自己把事情做好"到"让团队把事情做好"。

---

## 📌 核心命题

技术Leader（Tech Lead）是介于纯技术路线和纯管理路线之间的关键角色。你的首要职责是**确保技术交付质量和团队产出效率**，同时在业务目标与技术愿景之间架起桥梁。

### 技术Leader vs 工程师

| 维度 | 优秀工程师 | 技术Leader |
|------|-----------|-----------|
| 核心产出 | 个人代码贡献 | 团队整体产出 |
| 价值衡量 | 解决了多难的问题 | 团队效能和交付质量 |
| 时间分配 | 深度思考技术问题 | 多线程：技术决策+团队赋能+沟通 |
| 成功标准 | 自己做好 | 让团队做好 |
| 影响半径 | 个人/小组 | 整个团队/多个团队 |

## 一、角色转变

### 1.1 认知升级三阶段

```
阶段一：阵痛期（0-3个月）
  ❌ "为什么我自己做更快？"
  ❌ 习惯性地冲到一线写代码
  ❌ 微观管理，盯着每个人的提交

阶段二：适应期（3-6个月）
  🔄 开始接受"不完美但足够好"的交付
  🔄 学会授权和信任
  🔄 建立团队流程和规范

阶段三：成熟期（6-12个月）
  ✅ 团队自运转，Leader是"清障者"
  ✅ 投入精力在技术规划和人才培养
  ✅ 形成自己的管理风格
```

### 1.2 时间分配框架

| 时间块 | 占比 | 内容 |
|--------|------|------|
| 技术决策与架构 | 25% | 技术方案评审、架构设计、技术债管理 |
| 团队赋能 | 30% | 1on1、代码Review、技术分享、指导成长 |
| 向上沟通 | 15% | 汇报进展、对齐目标、争取资源 |
| 跨团队协作 | 15% | 需求评审、技术对齐、依赖协调 |
| 个人技术成长 | 15% | 保持技术敏感度、学习新领域 |

### 1.3 常见陷阱

- **英雄主义陷阱**：自己成为团队瓶颈，所有关键决策都过自己的手
- **甩手掌柜陷阱**：完全放权，丧失对技术细节的掌控力
- **技术执念陷阱**：仍然沉迷于技术细节，忽视团队管理和业务理解
- **老好陷阱**：不敢对团队成员提出高标准要求

## 二、团队赋能

### 2.1 1on1 体系

**频率**：每周或每两周一次，每次 30-45 分钟

**1on1 标准框架**：

```
前 5 分钟：寒暄 + 情绪感知
中间 25 分钟：
  - 工作进展与卡点（10min）
  - 个人成长与反馈（10min）
  - 开放话题：你的想法/建议（5min）
最后 5 分钟：总结行动项 + 下次关注点
```

**1on1 核心原则**：
- 倾听 > 说话（70/30原则）
- 聚焦个人，而非任务汇报
- 记录每次1on1要点，建立成长档案
- 及时给予正向反馈，批评要具体且对事不对人

### 2.2 反馈技巧 — SBI模型

```
S (Situation)：描述情境
  "在上周三的技术方案评审中..."

B (Behavior)：描述行为
  "你提出了三种备选方案，并从性能、成本、可维护性三个维度做了对比..."

I (Impact)：说明影响
  "这让团队对方案的理解非常深入，最终决策也很高效。继续保持。"
```

### 2.3 技术梯队建设

**能力矩阵示例**：

| 级别 | 技术能力 | 工程能力 | 影响力 | 培养重点 |
|------|---------|---------|--------|---------|
| 初级工程师 | 能完成明确任务 | 基础编码规范 | 个人 | 基础功扎实 |
| 中级工程师 | 能独立解决复杂问题 | 代码质量意识 | 小组 | 模块设计能力 |
| 高级工程师 | 能主导技术方案 | 架构设计能力 | 团队 | 系统思维 |
| 技术专家 | 能跨领域解决难题 | 全局技术视野 | 多团队 | 技术战略 |

### 2.4 技术分享机制

- **双周技术分享**：轮流主讲，15-20分钟，主题自选
- **代码Review文化**：不是挑错，是学习和知识传播
- **技术读书会**：每月精读一本技术书籍，共读讨论
- **Hackathon**：每季度一次，鼓励创新和技术探索

## 三、技术决策

### 3.1 技术决策框架

```
Step 1：明确决策目标
  - 我们要解决什么问题？
  - 成功标准是什么？
  - 有哪些约束条件？（时间、人力、预算、技术栈）

Step 2：收集信息
  - 现有系统的瓶颈在哪？
  - 业界最佳实践是什么？
  - 团队能力是否匹配？

Step 3：生成备选方案
  - 至少提出 2-3 个可行方案
  - 每个方案列出优缺点、风险、成本

Step 4：评估与选择
  - 决策矩阵法：加权评分
  - 考虑短期和长期影响
  - 评估可逆性（可逆决策可以更激进）

Step 5：决策文档化
  - 记录决策背景、方案对比、最终选择及理由
  - 形成 ADR（Architecture Decision Record）
```

### 3.2 技术债管理

**技术债分类**：

| 类型 | 示例 | 处理策略 |
|------|------|---------|
| 故意型技术债 | 为赶进度做的快捷实现 | 预留每迭代的20%时间偿还 |
| 无意型技术债 | 设计不佳导致的债务 | 优先重构核心路径 |
| 腐烂型技术债 | 技术过时（如旧框架） | 制定迁移计划 |
| 环境型技术债 | 工具链不完善 | 持续改进工程效率 |

**技术债看板实践**：
- 每个技术债记录：描述、影响、修复成本、优先级
- 每迭代分配固定比例时间（15-20%）给技术债清理
- 定期复盘：技术债趋势是在增加还是减少？

### 3.3 技术评审 Checklist

- [ ] 方案是否解决了核心问题？
- [ ] 是否考虑了向后兼容性？
- [ ] 可扩展性如何？能否支撑 3-5x 增长？
- [ ] 安全性是否充分考量？
- [ ] 监控和告警是否到位？
- [ ] 回滚方案是否清晰？
- [ ] 依赖的外部服务是否稳定可靠？
- [ ] 性能指标是否达标？

## 四、向上管理

### 4.1 向上管理的核心心法

> 向上管理不是"讨好老板"，而是**让上级对你的工作有信心、有掌控感**。

**三个关键**：
1. **预期管理**：主动对齐期望，不承诺做不到的事
2. **信息透明**：坏消息早说，好消息定量说
3. **方案导向**：带着方案汇报问题，而不是只抛问题

### 4.2 向上沟通的四种场景

**场景一：项目进度汇报**

```
✅ 好的汇报：
"AI Agent 推理优化项目目前完成度 60%。
已完成的部分：模型蒸馏、推理引擎升级，延迟降低 40%。
剩余风险：GPU 资源可能不足，已有备选方案（云端弹性扩容）。
预计上线时间不变，仍有 3 天 buffer。"

❌ 差的汇报：
"项目进展还行，还在搞，应该能按时完成。"
```

**场景二：资源争取**

```
框架：背景 → 目标 → 差距 → 方案 → 需要的支持
"我们需要 2 个额外工程师来支持 Q3 的系统重构项目。
原因是当前团队同时维护 3 个产品线，已超负荷。
如果不增援，预计会有 2 个重要里程碑延期。
建议方案：内部借调 1 人 + 招聘 1 人。"
```

**场景三：坏消息传递**

- **第一时间传递**：不要等到无法挽回
- **携带影响评估**：影响范围、程度、持续时间
- **附带应对方案**：至少提供一个止损方案
- **提出后续改进**：如何避免类似问题再次发生

**场景四：目标对齐**

- 理解上级的 OKR/KPI
- 将团队目标与上级目标对齐
- 定期同步进展，确保方向不偏

### 4.3 与不同风格的上级协作

| 上级风格 | 特征 | 协作策略 |
|----------|------|---------|
| 细节型 | 关注数据、过程 | 准备充分的细节和数据支撑 |
| 结果型 | 只看结果 | 汇报精炼，先说结论 |
| 关系型 | 重视团队氛围 | 多进行非正式沟通，建立信任 |
| 创新型 | 追求突破 | 展示创新想法和长期规划 |

## 五、向下负责

### 5.1 什么是向下负责？

> 向上管理是对上级负责，向下负责是对团队成员负责。技术Leader不仅要对产出负责，更要对**人的成长**负责。

### 5.2 向下负责的五个维度

**1. 清晰的方向**
- 确保每个成员理解团队目标和自己的角色
- 定期同步战略变化和优先级调整
- 帮助成员看到自己工作的价值

**2. 成长的土壤**
- 给有潜力的成员更有挑战的任务
- 容许犯错，建立安全失败的文化
- 为团队成员的成长创造机会和资源

**3. 信任与尊重**
- 授权而不放权（放权=不管，授权=给目标+支持+信任）
- 公开表扬，私下批评
- 尊重技术意见，即使不完全认同也认真倾听

**4. 关怀与支持**
- 关注成员的工作状态和心理健康
- 在成员遇到困难时提供帮助
- 为团队挡住不必要的压力和干扰

**5. 公正的评价**
- 建立透明、可量化的评价体系
- 不搞"论资排辈"，以贡献和能力为准
- 考核反馈要及时、具体、可执行

### 5.3 团队文化建设

**优秀技术团队文化的特征**：
- 🔍 **透明**：决策过程、技术方案、团队目标公开透明
- 🤝 **协作**：互相帮助，而不是"这不是我的事"
- 📈 **成长**：持续学习是常态，停滞不前才令人不安
- 🛡️ **安全**：可以大胆试错，不必害怕"背锅"
- 🎯 **聚焦**：做重要的事，不被低价值工作稀释精力

## 🛠️ 工具推荐

### 管理工具

| 类别 | 工具 | 用途 |
|------|------|------|
| 项目管理 | Jira / Linear / 飞书项目 | 任务跟踪、迭代管理 |
| 文档协作 | Notion / Confluence / 飞书文档 | ADR、技术文档、知识库 |
| 1on1管理 | Lattice / 内部表格 | 跟踪1on1记录和成长计划 |
| 代码Review | GitHub PR / GitLab MR | 代码质量保障 |
| 效能度量 | 敏捷看板 / 效能平台 | DORA指标、交付效率 |

### 效能度量指标（DORA Four Keys）

1. **部署频率** — 多久发布一次？
2. **变更前置时间** — 从提交到部署多久？
3. **变更失败率** — 发布后出故障的比例？
4. **平均恢复时间** — 故障多久恢复？
