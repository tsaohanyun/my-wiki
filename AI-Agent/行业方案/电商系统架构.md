# 电商系统架构

## 概述

电商系统架构围绕高并发、高可用、可扩展三大目标，构建支撑亿级用户、百万级QPS的技术平台，涵盖商品、订单、库存、支付、推荐等核心系统。

## 一、高并发设计

### 1. 多级缓存架构

```
请求流程:
用户请求 → CDN缓存 → Nginx缓存 → 本地缓存 → Redis缓存 → 数据库

缓存策略:
┌──────────────┬────────────────┬─────────────┬────────────┐
│    层级      │    技术方案     │   命中率     │   延迟     │
├──────────────┼────────────────┼─────────────┼────────────┤
│ CDN缓存      │ 阿里云CDN      │ 80%+        │ < 10ms     │
│ Nginx缓存    │ proxy_cache    │ 70%+        │ < 5ms      │
│ 本地缓存     │ Caffeine       │ 90%+        │ < 1ms      │
│ Redis缓存    │ Redis Cluster  │ 95%+        │ < 5ms      │
│ 数据库       │ MySQL/分库分表  │ -           │ < 50ms     │
└──────────────┴────────────────┴─────────────┴────────────┘
```

### 3. 限流与熔断

```yaml
限流策略:
  - 令牌桶算法: 平滑限流，允许突发
  - 滑动窗口: 精确统计请求量
  - 分布式限流: Redis + Lua脚本

熔断策略:
  - 慢调用比例: 响应时间 > 阈值比例触发
  - 异常比例: 异常请求比例触发
  - 异常数: 异常请求数触发

降级策略:
  - 返回默认值
  - 返回缓存数据
  - 功能降级 (关闭非核心功能)
  - 流量切换 (切到备用集群)
```

public class OrderIdGenerator {
    // 时间戳(41bit) + 机器ID(10bit) + 序列号(12bit)
    // 单机每毫秒可生成4096个ID
    
    // 订单号格式: 业务前缀 + 时间 + 机器ID + 序列号
    // 示例: ORD202606271234567890
    
    public String generate() {
        long timestamp = System.currentTimeMillis();
        long sequence = getSequence(timestamp);

## 五、技术选型建议

### 1. 大促容量规划

```yaml
容量评估:
  - 预估峰值QPS: 历史峰值 × 2
  - 服务实例数: 峰值QPS / 单实例QPS × 1.5
  - 数据库连接: 服务实例数 × 连接池大小
  - Redis容量: 热点数据量 × 3

压测策略:
  - 单接口压测: 验证单服务性能
  - 全链路压测: 模拟真实流量
  - 极限压测: 找到系统瓶颈
  - 故障演练: 验证容灾能力

大促保障:
  - 限流降级预案
  - 弹性扩容方案
  - 数据预热方案
  - 应急响应机制
```
