# 自动化测试框架

## 概述

自动化测试框架是质量保障体系的核心基础设施，通过系统化的测试策略和工具链，确保软件产品在快速迭代中保持高质量。

---

## 测试金字塔

测试金字塔是指导测试策略的经典模型，强调不同层级测试的数量和投入比例。

```
        /  E2E  \          ← 少量端到端测试
       /  集成测试  \       ← 适量集成测试
      /   单元测试    \     ← 大量单元测试
     /__________________\
```

### 各层测试说明

| 测试层级 | 占比 | 执行速度 | 维护成本 | 发现缺陷能力 |
|---------|------|---------|---------|-------------|
| 单元测试 | 70% | 极快（毫秒级） | 低 | 逻辑缺陷 |
| 集成测试 | 20% | 中等（秒级） | 中 | 接口/交互缺陷 |
| E2E测试 | 10% | 慢（分钟级） | 高 | 业务流程缺陷 |

### 各层测试示例

#### 单元测试（Python + pytest）

```python
# tests/unit/test_user_service.py
import pytest
from unittest.mock import Mock, patch
from app.services.user_service import UserService

class TestUserService:
    """用户服务单元测试"""

    @pytest.fixture
    def mock_user_repo(self):
        """模拟用户仓库"""
        return Mock()

    @pytest.fixture
    def user_service(self, mock_user_repo):
        return UserService(repository=mock_user_repo)

    def test_create_user_success(self, user_service, mock_user_repo):
        """测试成功创建用户"""
        user_data = {"name": "张三", "email": "zhangsan@example.com"}
        mock_user_repo.save.return_value = {**user_data, "id": 1}

        result = user_service.create_user(user_data)

        assert result["id"] == 1
        assert result["name"] == "张三"
        mock_user_repo.save.assert_called_once()

    def test_create_user_duplicate_email(self, user_service, mock_user_repo):
        """测试重复邮箱创建用户"""
        mock_user_repo.exists_by_email.return_value = True

        with pytest.raises(ValueError, match="邮箱已存在"):
            user_service.create_user({"name": "张三", "email": "dup@example.com"})

    @pytest.mark.parametrize("email,expected", [
        ("valid@example.com", True),
        ("invalid-email", False),
        ("", False),
        ("test@test.co", True),
    ])
    def test_email_validation(self, email, expected):
        """测试邮箱格式验证"""
        from app.utils.validators import is_valid_email
        assert is_valid_email(email) == expected
```

#### 集成测试（API + 数据库）

```python
# tests/integration/test_user_api.py
import pytest
from fastapi.testclient import TestClient
from app.main import app
from app.database import get_db, Base, engine

@pytest.fixture(scope="module")
def test_client():
    """创建测试客户端"""
    Base.metadata.create_all(bind=engine)
    with TestClient(app) as client:
        yield client
    Base.metadata.drop_all(bind=engine)

class TestUserAPI:
    """用户API集成测试"""

    def test_create_and_get_user(self, test_client):
        """测试创建并获取用户"""
        # 创建用户
        create_resp = test_client.post("/api/v1/users", json={
            "name": "李四",
            "email": "lisi@example.com",
            "password": "SecurePass123!"
        })
        assert create_resp.status_code == 201
        user_id = create_resp.json()["id"]

        # 获取用户
        get_resp = test_client.get(f"/api/v1/users/{user_id}")
        assert get_resp.status_code == 200
        assert get_resp.json()["name"] == "李四"
        assert "password" not in get_resp.json()

    def test_list_users_pagination(self, test_client):
        """测试用户列表分页"""
        response = test_client.get("/api/v1/users?page=1&size=10")
        assert response.status_code == 200
        data = response.json()
        assert "items" in data
        assert "total" in data
        assert len(data["items"]) <= 10
```

#### E2E测试（Playwright）

```python
# tests/e2e/test_user_workflow.py
import pytest
from playwright.async_api import async_playwright, expect

@pytest.mark.e2e
class TestUserWorkflow:
    """用户工作流端到端测试"""

    @pytest.fixture(autouse=True)
    async def setup(self):
        async with async_playwright() as p:
            self.browser = await p.chromium.launch(headless=True)
            self.page = await self.browser.new_page()
            yield
            await self.browser.close()

    async def test_user_registration_flow(self):
        """测试用户注册完整流程"""
        await self.page.goto("http://localhost:3000/register")

        # 填写注册表单
        await self.page.fill('[data-testid="name-input"]', "王五")
        await self.page.fill('[data-testid="email-input"]', "wangwu@example.com")
        await self.page.fill('[data-testid="password-input"]', "SecurePass123!")
        await self.page.fill('[data-testid="confirm-password-input"]', "SecurePass123!")

        # 提交
        await self.page.click('[data-testid="register-button"]')

        # 验证跳转到登录页面
        await expect(self.page).to_have_url("/login")
        await expect(self.page.locator(".success-message")).to_contain_text("注册成功")
```

## 测试策略

### 测试类型矩阵

| 测试类型 | 目的 | 时机 | 工具 | 执行频率 |
|---------|------|------|------|---------|
| 单元测试 | 验证函数/方法逻辑 | 开发阶段 | pytest, JUnit | 每次提交 |
| 集成测试 | 验证模块间交互 | 合并前 | pytest, TestContainers | 每次PR |
| API测试 | 验证接口契约 | CI阶段 | Postman, RestAssured | 每次构建 |
| E2E测试 | 验证业务流程 | 部署前 | Playwright, Cypress | 每日/发版前 |
| 性能测试 | 验证系统性能 | 发版前 | JMeter, Locust | 里程碑/发版前 |
| 安全测试 | 验证安全性 | 定期 | OWASP ZAP, Burp Suite | 每周/每月 |
| 回归测试 | 验证无副作用 | 每次变更 | 全套测试套件 | 每次部署 |

### 测试分层策略

```yaml
# pytest.ini - 测试配置
[tool.pytest.ini_options]
testpaths = ["tests"]
python_files = ["test_*.py"]
python_functions = ["test_*"]
markers =
    unit: 单元测试
    integration: 集成测试
    e2e: 端到端测试
    slow: 慢速测试
    smoke: 冒烟测试

# 按层级执行
# pytest -m unit          # 只运行单元测试
# pytest -m integration   # 只运行集成测试
# pytest -m "not slow"    # 排除慢速测试
```

### Allure测试报告配置

```python
# conftest.py
import allure
import pytest
from datetime import datetime

@pytest.hookimpl(hookwrapper=True)
def pytest_runtest_makereport(item, call):
    outcome = yield
    report = outcome.get_result()

    if report.when == "call" and report.failed:
        # 失败时自动截图
        if hasattr(item, "funcargs") and "page" in item.funcargs:
            page = item.funcargs["page"]
            screenshot = page.screenshot()
            allure.attach(
                screenshot,
                name=f"failure_{datetime.now().strftime('%Y%m%d_%H%M%S')}",
                attachment_type=allure.attachment_type.PNG
            )
```

### TestContainers配置

```python
# tests/conftest.py
import pytest
from testcontainers.postgres import PostgresContainer
from testcontainers.redis import RedisContainer

@pytest.fixture(scope="session")
def postgres():
    """启动PostgreSQL测试容器"""
    with PostgresContainer("postgres:15-alpine") as pg:
        yield pg

@pytest.fixture(scope="session")
def redis():
    """启动Redis测试容器"""
    with RedisContainer("redis:7-alpine") as r:
        yield r

@pytest.fixture(scope="function")
def db_session(postgres):
    """创建数据库会话"""
    from sqlalchemy import create_engine
    from sqlalchemy.orm import sessionmaker

    engine = create_engine(postgres.get_connection_url())
    Session = sessionmaker(bind=engine)
    session = Session()

    yield session

    session.rollback()
    session.close()
```

## 测试数据管理

### 测试数据策略

#### 1. Factory模式（推荐）

```python
# tests/factories.py
import factory
from faker import Faker
from app.models.user import User
from app.database import SessionLocal

fake = Faker("zh_CN")

class UserFactory(factory.alchemy.SQLAlchemyModelFactory):
    """用户工厂"""
    class Meta:
        model = User
        sqlalchemy_session = SessionLocal()
        sqlalchemy_session_persistence = "commit"

    id = factory.Sequence(lambda n: n + 1)
    name = factory.LazyFunction(fake.name)
    email = factory.LazyFunction(fake.email)
    age = factory.LazyFunction(lambda: fake.random_int(min=18, max=65))
    is_active = True

class AdminFactory(UserFactory):
    """管理员工厂"""
    role = "admin"
    is_active = True

# 使用方式
# user = UserFactory()                    # 创建普通用户
# admin = AdminFactory(name="超级管理员")  # 创建管理员
# users = UserFactory.create_batch(10)    # 批量创建
```

#### 2. Fixture模式

```python
# tests/fixtures/users.py
import json
from pathlib import Path

FIXTURES_DIR = Path(__file__).parent / "data"

def load_fixture(name: str) -> dict:
    """加载JSON测试数据"""
    with open(FIXTURES_DIR / f"{name}.json") as f:
        return json.load(f)

@pytest.fixture
def sample_users():
    """加载样例用户数据"""
    return load_fixture("sample_users")

# tests/data/sample_users.json
# [
#   {"name": "张三", "email": "zhangsan@test.com", "age": 25},
#   {"name": "李四", "email": "lisi@test.com", "age": 30}
# ]
```

## CI/CD集成

### GitHub Actions配置

```yaml
# .github/workflows/test.yml
name: 自动化测试

on:
  push:
    branches: [main, develop]
  pull_request:
    branches: [main]

jobs:
  unit-tests:
    name: 单元测试
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: 设置Python
        uses: actions/setup-python@v5
        with:
          python-version: "3.11"

      - name: 安装依赖
        run: pip install -r requirements-test.txt

      - name: 运行单元测试
        run: |
          pytest tests/unit \
            --cov=app \
            --cov-report=xml \
            --cov-report=html \
            -v --tb=short

      - name: 上传覆盖率报告
        uses: codecov/codecov-action@v3
        with:
          file: ./coverage.xml

  integration-tests:
    name: 集成测试
    needs: unit-tests
    runs-on: ubuntu-latest
    services:
      postgres:
        image: postgres:15
        env:
          POSTGRES_DB: testdb
          POSTGRES_PASSWORD: testpass
        ports:
          - 5432:5432
      redis:
        image: redis:7
        ports:
          - 6379:6379
    steps:
      - uses: actions/checkout@v4
      - name: 运行集成测试
        run: pytest tests/integration -v --tb=short
        env:
          DATABASE_URL: postgresql://postgres:testpass@localhost:5432/testdb
          REDIS_URL: redis://localhost:6379/0

  e2e-tests:
    name: E2E测试
    needs: integration-tests
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: 安装Playwright
        run: |
          pip install playwright pytest-playwright
          playwright install --with-deps chromium

      - name: 运行E2E测试
        run: pytest tests/e2e -v --tb=short --alluredir=allure-results

      - name: 生成Allure报告
        uses: simple-elf/allure-report-action@master
        with:
          allure_results: allure-results
```

## 最佳实践

### 1. 测试编写原则
- **FIRST原则**：Fast（快速）、Independent（独立）、Repeatable（可重复）、Self-validating（自验证）、Timely（及时）
- **AAA模式**：Arrange（准备）→ Act（执行）→ Assert（断言）
- **一个测试一个断言焦点**：每个测试只验证一个行为

### 2. 测试维护策略
- 使用Page Object模式管理E2E测试
- 测试数据与测试逻辑分离
- 定期清理过时测试用例
- 测试代码同样需要Code Review

### 3. 覆盖率目标
- 单元测试覆盖率 ≥ 80%
- 关键业务逻辑覆盖率 ≥ 90%
- 不存在未测试的公开API

### 4. 常见陷阱
- ❌ 测试依赖外部服务（应Mock）
- ❌ 测试之间存在顺序依赖
- ❌ 过度测试实现细节而非行为
- ❌ 忽视测试的可读性
