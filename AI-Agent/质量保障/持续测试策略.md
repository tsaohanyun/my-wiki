# 持续测试策略

## 概述

持续测试是在软件交付生命周期的每个阶段持续执行自动化测试的实践，通过测试左移（Shift-Left）和测试右移（Shift-Right）实现质量内建，确保快速交付的同时保持高质量。

---

## 测试左移（Shift-Left）

### 左移核心理念

```
传统流程：  需求 → 设计 → 开发 → 测试 → 部署
                        ↑ 质量问题发现太晚，修复成本高

左移流程：  需求→测试  设计→测试  开发→测试  集成→测试  部署→监控
           ↑ 越早发现问题，修复成本越低
```

### 左移实践清单

| 阶段 | 实践 | 工具 | 负责人 |
|------|------|------|--------|
| 需求阶段 | 验收标准定义、需求评审 | BDD工具 | 产品+测试 |
| 设计阶段 | 架构评审、威胁建模 | 威胁建模工具 | 架构+安全 |
| 编码阶段 | TDD、静态分析、代码审查 | pytest, SonarQube | 开发 |
| 提交阶段 | Pre-commit检查、单元测试 | Git Hooks, CI | 开发 |
| 构建阶段 | 集成测试、安全扫描 | GitHub Actions | CI/CD |
| 部署前 | E2E测试、性能测试 | Playwright, Locust | QA |

```gherkin
# language: zh-CN
功能: 用户注册
  作为一个新用户

    假定 邮箱 "existing@example.com" 已注册
    当 用户输入邮箱 "existing@example.com"
```

```python
# tests/bdd/steps/test_user_registration.py
from pytest_bdd import scenarios, given, when, then, parsers
from playwright.sync_api import Page

scenarios("../features/user_registration.feature")

@given("用户访问注册页面")
def visit_register_page(page: Page):
    page.goto("http://localhost:3000/register")

@when(parsers.parse('用户输入用户名 "{name}"'))
def input_name(page: Page, name: str):
    page.fill('[data-testid="name-input"]', name)

@when(parsers.parse('用户输入邮箱 "{email}"'))
def input_email(page: Page, email: str):
    page.fill('[data-testid="email-input"]', email)

@when(parsers.parse('用户输入密码 "{password}"'))
def input_password(page: Page, password: str):
    page.fill('[data-testid="password-input"]', password)

@when("用户点击注册按钮")
def click_register(page: Page):
    page.click('[data-testid="register-button"]')

@then(parsers.parse('系统显示 "{message}"'))
def check_message(page: Page, message: str):
    expect(page.locator(".message")).to_contain_text(message)

@then("用户被重定向到登录页面")
def check_redirect(page: Page):
    expect(page).to_have_url("/login")
```

## 测试右移（Shift-Right）

### 右移核心理念

```
部署后持续验证：
  部署 → 生产监控 → 特性验证 → 用户反馈 → 持续改进
         ↑ 生产环境也是测试环境
```

#### 2. 特性标志（Feature Flags）

```python
# app/feature_flags.py
from typing import Optional
import redis

class FeatureFlag:
    """特性标志管理"""

    def __init__(self, redis_client: redis.Redis):
        self.redis = redis_client

    def is_enabled(self, flag_name: str, user_id: Optional[int] = None) -> bool:
        """检查特性是否启用"""
        flag_data = self.redis.hgetall(f"flag:{flag_name}")

        if not flag_data:
            return False

        flag_type = flag_data.get(b"type", b"boolean")

        if flag_type == b"boolean":
            return flag_data.get(b"enabled", b"false") == b"true"

        elif flag_type == b"percentage":
            percentage = int(flag_data.get(b"percentage", 0))
            if user_id:
                return (user_id % 100) < percentage
            return False

        elif flag_type == b"whitelist":
            allowed_users = flag_data.get(b"users", b"").decode().split(",")
            return str(user_id) in allowed_users

        return False

# 使用方式
@app.post("/api/v1/orders")
async def create_order(order_data: OrderCreate, user: User = Depends(get_current_user)):
    # 使用特性标志控制新订单流程
    if feature_flags.is_enabled("new_order_flow", user.id):
        return await create_order_v2(order_data, user)
    else:
        return await create_order_v1(order_data, user)
```

```python
# tests/production/smoke_tests.py
"""
生产环境冒烟测试 - 每次部署后自动执行
"""
import httpx
import pytest

PROD_URL = "https://api.example.com"

@pytest.fixture
def client():
    with httpx.Client(base_url=PROD_URL, timeout=30) as c:
        yield c

class TestProductionSmoke:
    """生产环境冒烟测试套件"""

    def test_homepage_accessible(self, client):
        """验证首页可访问"""
        response = client.get("/")
        assert response.status_code == 200

    def test_health_endpoint(self, client):
        """验证健康检查端点"""
        response = client.get("/health")
        assert response.status_code == 200
        data = response.json()
        assert data["status"] == "healthy"

    def test_api_version(self, client):
        """验证API版本"""
        response = client.get("/api/v1/version")
        assert response.status_code == 200
        assert "version" in response.json()

    def test_core_api_endpoints(self, client):
        """验证核心API端点"""
        endpoints = [
            "/api/v1/products",
            "/api/v1/categories",
            "/api/v1/health/ready",
        ]
        for endpoint in endpoints:
            response = client.get(endpoint)
            assert response.status_code in [200, 401], \
                f"端点 {endpoint} 返回异常状态码 {response.status_code}"
```

## 测试自动化

### CI/CD流水线中的测试分层

```yaml
# .github/workflows/ci.yml
name: CI Pipeline

on:
  push:
    branches: [main, develop, "feature/*"]
  pull_request:
    branches: [main]

env:
  PYTHON_VERSION: "3.11"

jobs:
  # 第1阶段：快速检查（< 2分钟）
  lint:
    name: "代码检查"
    runs-on: ubuntu-latest
    timeout-minutes: 2
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with:
          python-version: ${{ env.PYTHON_VERSION }}

      - name: 安装工具
        run: pip install ruff mypy bandit

      - name: Ruff Lint
        run: ruff check . --output-format=github

      - name: Ruff Format Check
        run: ruff format --check .

      - name: MyPy类型检查
        run: mypy app/ --ignore-missing-imports

      - name: Bandit安全扫描
        run: bandit -r app/ -c pyproject.toml -f github

  # 第2阶段：单元测试（< 5分钟）
  unit-tests:
    name: "单元测试"
    needs: lint
    runs-on: ubuntu-latest
    timeout-minutes: 5
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with:
          python-version: ${{ env.PYTHON_VERSION }}

      - name: 安装依赖
        run: pip install -r requirements-test.txt

      - name: 运行单元测试
        run: |
          pytest tests/unit \
            -v --tb=short \
            --cov=app \
            --cov-report=xml \
            --junitxml=test-results.xml

      - name: 检查覆盖率
        run: |
          coverage report --fail-under=80

      - name: 上传测试报告
        if: always()
        uses: actions/upload-artifact@v4
        with:
          name: unit-test-results
          path: |
            test-results.xml
            coverage.xml

  # 第3阶段：集成测试（< 10分钟）
  integration-tests:
    name: "集成测试"
    needs: unit-tests
    runs-on: ubuntu-latest
    timeout-minutes: 10
    services:
      postgres:
        image: postgres:15-alpine
        env:
          POSTGRES_DB: testdb
          POSTGRES_USER: test
          POSTGRES_PASSWORD: test
        ports: ["5432:5432"]
        options: >-
          --health-cmd pg_isready
          --health-interval 10s
          --health-timeout 5s
          --health-retries 5
      redis:
        image: redis:7-alpine
        ports: ["6379:6379"]
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with:
          python-version: ${{ env.PYTHON_VERSION }}

      - name: 安装依赖
        run: pip install -r requirements-test.txt

      - name: 运行集成测试
        run: pytest tests/integration -v --tb=short
        env:
          DATABASE_URL: postgresql://test:test@localhost:5432/testdb
          REDIS_URL: redis://localhost:6379

  # 第4阶段：E2E测试（< 15分钟，仅PR合并前）
  e2e-tests:
    name: "E2E测试"
    needs: integration-tests
    if: github.event_name == 'pull_request'
    runs-on: ubuntu-latest
    timeout-minutes: 15
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with:
          python-version: ${{ env.PYTHON_VERSION }}

      - name: 安装依赖
        run: |
          pip install -r requirements-test.txt
          playwright install --with-deps chromium

      - name: 启动应用
        run: docker-compose up -d

      - name: 等待服务就绪
        run: |
          for i in $(seq 1 30); do
            curl -s http://localhost:8000/health && break
            sleep 2
          done

      - name: 运行E2E测试
        run: |
          pytest tests/e2e -v --tb=short \
            --alluredir=allure-results

      - name: 生成Allure报告
        uses: simple-elf/allure-report-action@master
        if: always()
        with:
          allure_results: allure-results

      - name: 上传报告
        uses: actions/upload-artifact@v4
        if: always()
        with:
          name: allure-report
          path: allure-report
```

|------|------|------|---------|---------|

### 环境一致性保障

```python
# scripts/env_check.py
"""环境一致性检查脚本"""
import sys
import httpx

def check_environment(env_url: str, checks: dict) -> bool:
    """检查环境状态"""
    all_pass = True

    for service, expected in checks.items():
        try:
            response = httpx.get(f"{env_url}/health/{service}", timeout=10)
            actual = response.json()

            if actual["status"] != expected["status"]:
                print(f"❌ {service}: 状态不匹配 (expected: {expected['status']}, actual: {actual['status']})")
                all_pass = False
            else:
                print(f"✅ {service}: 正常")
        except Exception as e:
            print(f"❌ {service}: 连接失败 - {e}")
            all_pass = False

    return all_pass

if __name__ == "__main__":
    env = sys.argv[1] if len(sys.argv) > 1 else "staging"
    checks = {
        "database": {"status": "up"},
        "redis": {"status": "up"},
        "elasticsearch": {"status": "up"},
    }

    urls = {
        "local": "http://localhost:8000",
        "staging": "https://staging.example.com",
        "production": "https://api.example.com",
    }

    result = check_environment(urls[env], checks)
    sys.exit(0 if result else 1)
```

## 最佳实践

### 1. 测试金字塔维护
- 保持70%单元测试、20%集成测试、10%E2E测试的比例
- E2E测试仅覆盖关键业务流程
- 定期审查和清理无效测试

### 2. 快速反馈
- PR检查 < 10分钟完成
- 关键测试失败时立即通知
- 测试结果可视化展示

### 3. 测试数据管理
- 使用Factory模式生成测试数据
- 测试数据与代码同步版本化
- 敏感数据脱敏处理

### 4. 持续改进
- 跟踪测试通过率趋势
- 分析测试失败的根本原因
- 定期优化慢速测试
- 测试覆盖率稳步提升
