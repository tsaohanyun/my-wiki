# 高并发架构调优

## 概述

高并发架构调优涉及从操作系统内核到应用层的全栈优化。核心目标是在有限硬件资源下最大化**并发连接数**、**请求吞吐量（QPS/TPS）**，同时保证**低延迟**和**高可用性**。

---

## 一、Nginx深度调优

### 1.1 worker进程配置

```nginx
# /etc/nginx/nginx.conf

worker_processes auto;                    # 自动匹配CPU核心数
worker_cpu_affinity auto;                 # CPU亲和性自动绑定
worker_rlimit_nofile 1048576;             # worker最大文件描述符数

events {
    worker_connections 65535;             # 每个worker最大连接数
    use epoll;                            # 使用epoll事件模型
    multi_accept on;                      # 一个worker同时接收多个新连接
    accept_mutex on;                      # 惊群控制(低并发关闭, 高并发开启)
    accept_mutex_delay 500ms;             # 获取互斥锁的等待延迟
}
```

### 1.2 HTTP层优化

```nginx
http {
    # === 基础优化 ===
    sendfile on;                          # 使用sendfile系统调用(零拷贝)
    tcp_nopush on;                        # 数据包聚合(CORK)
    tcp_nodelay on;                       # 禁用Nagle算法(小包立即发送)
    keepalive_timeout 65;                 # keepalive超时
    keepalive_requests 10000;             # 单个keepalive最大请求数
    reset_timedout_connection on;         # 超时连接立即重置

    # === 缓冲区优化 ===
    client_body_buffer_size 16k;          # 请求体缓冲区
    client_max_body_size 50m;             # 最大请求体
    client_header_buffer_size 4k;         # 请求头缓冲区
    large_client_header_buffers 4 16k;    # 大请求头缓冲区

    # 响应缓冲区(对FastCGI/Proxy有效)
    proxy_buffer_size 16k;                # 代理响应头缓冲区
    proxy_buffers 8 32k;                  # 代理响应内容缓冲区
    proxy_busy_buffers_size 64k;          # 繁忙缓冲区大小

    # === 超时控制 ===
    client_body_timeout 30s;
    client_header_timeout 30s;
    send_timeout 30s;

    # === 文件缓存 ===
    open_file_cache max=10000 inactive=60s;
    open_file_cache_valid 30s;
    open_file_cache_min_uses 2;
    open_file_cache_errors on;

    # === Gzip压缩 ===
    gzip on;
    gzip_comp_level 4;                    # 压缩级别(1-9, 4是性能与压缩比平衡点)
    gzip_min_length 1024;                 # 最小压缩大小
    gzip_types text/plain text/css application/json application/javascript
               text/xml application/xml application/xml+rss text/javascript;
    gzip_vary on;
    gzip_proxied any;

    # === 日志优化(高并发下减少日志IO) ===
    access_log off;                       # 或使用缓冲
    # access_log /var/log/nginx/access.log buffer=32k flush=5s;
    error_log /var/log/nginx/error.log warn;
}
```

### 1.3 upstream与负载均衡

```nginx
http {
    upstream backend {
        # 负载均衡策略
        # least_conn;                     # 最少连接数
        # ip_hash;                        # IP哈希(会话保持)
        # hash $request_uri consistent;   # 一致性哈希

        # 后端服务器池
        server 10.0.0.1:8080 weight=5 max_fails=3 fail_timeout=30s;
        server 10.0.0.2:8080 weight=5 max_fails=3 fail_timeout=30s;
        server 10.0.0.3:8080 weight=3 max_fails=3 fail_timeout=30s;

        # 备用服务器
        server 10.0.0.99:8080 backup;

        # 长连接池
        keepalive 128;                    # 到后端的长连接池大小
        keepalive_requests 1000;
        keepalive_timeout 60s;

        # 慢启动(新加入的服务器渐进接收流量)
        server 10.0.0.4:8080 slow_start=30s;
    }

    server {
        listen 80 reuseport backlog=65535;  # reuseport + 队列
        server_name example.com;

        location / {
            proxy_pass http://backend;

            # 代理头设置
            proxy_set_header Host $host;
            proxy_set_header X-Real-IP $remote_addr;
            proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
            proxy_set_header X-Forwarded-Proto $scheme;

            # 长连接到后端
            proxy_http_version 1.1;
            proxy_set_header Connection "";

            # 超时设置
            proxy_connect_timeout 5s;
            proxy_send_timeout 30s;
            proxy_read_timeout 30s;

            # 限速
            # limit_rate 100k;             # 限制每连接带宽
        }
    }
}
```

### 1.4 限流与安全

```nginx
http {
    # 请求限流(基于IP, 每秒10个请求)
    limit_req_zone $binary_remote_addr zone=req_limit:10m rate=10r/s;

    # 连接数限流
    limit_conn_zone $binary_remote_addr zone=conn_limit:10m;

    server {
        # 应用请求限流(突发允许20个排队)
        limit_req zone=req_limit burst=20 nodelay;

        # 应用连接数限流(每IP最大50并发连接)
        limit_conn conn_limit 50;

        # 限流返回状态码
        limit_req_status 429;
        limit_conn_status 429;

        # 防CC攻击
        limit_req zone=req_limit burst=100 nodelay;

        location / {
            proxy_pass http://backend;
        }
    }
}
```

### 1.5 stream模块(TCP/UDP代理)

```nginx
# /etc/nginx/nginx.conf 或 /etc/nginx/stream.conf

stream {
    upstream mysql_backend {
        server 10.0.0.1:3306;
        server 10.0.0.2:3306;
    }

    server {
        listen 3306;
        proxy_pass mysql_backend;
        proxy_connect_timeout 5s;
        proxy_timeout 300s;
    }

    # SSL终端
    server {
        listen 443 ssl;
        ssl_certificate     /etc/nginx/ssl/server.crt;
        ssl_certificate_key /etc/nginx/ssl/server.key;
        proxy_pass backend:8443;
    }
}
```

## 二、内核网络参数深度调优

### 2.1 文件描述符

```bash
# 系统级最大文件描述符
sysctl -w fs.file-max = 2097152

# 查看当前使用
cat /proc/sys/fs/file-nr

# 用户级限制(ulimit)
# 临时
ulimit -n 1048576

# 永久 (/etc/security/limits.conf)
cat >> /etc/security/limits.conf << 'EOF'
*    soft    nofile    1048576
*    hard    nofile    1048576
nginx soft    nproc     65535
nginx hard    nproc     65535
EOF

# systemd服务(如果Nginx通过systemd管理)
# /etc/systemd/system/nginx.service
[Service]
LimitNOFILE=1048576
LimitNPROC=65535
```

### 2.2 完整sysctl网络配置

```bash
# /etc/sysctl.d/99-high-concurrency.conf

# === 连接队列 ===
net.core.somaxconn = 65535
net.core.netdev_max_backlog = 65535
net.ipv4.tcp_max_syn_backlog = 65535

# === TIME_WAIT ===
net.ipv4.tcp_max_tw_buckets = 1048576
net.ipv4.tcp_tw_reuse = 1
net.ipv4.tcp_fin_timeout = 15

# === TCP缓冲区 ===
net.core.rmem_max = 33554432
net.core.wmem_max = 33554432
net.core.rmem_default = 8388608
net.core.wmem_default = 8388608
net.ipv4.tcp_rmem = 4096 87380 33554432
net.ipv4.tcp_wmem = 4096 65536 33554432

# === TCP行为 ===
net.ipv4.tcp_syncookies = 1
net.ipv4.tcp_fastopen = 3
net.ipv4.tcp_slow_start_after_idle = 0
net.ipv4.tcp_no_metrics_save = 1

# === Keepalive ===
net.ipv4.tcp_keepalive_time = 300
net.ipv4.tcp_keepalive_intvl = 30
net.ipv4.tcp_keepalive_probes = 3

# === 拥塞控制 ===
net.ipv4.tcp_congestion_control = bbr
net.core.default_qdisc = fq

# === 端口范围 ===
net.ipv4.ip_local_port_range = 10000 65535

# === 连接跟踪 ===
net.netfilter.nf_conntrack_max = 1048576
net.netfilter.nf_conntrack_tcp_timeout_established = 3600
net.netfilter.nf_conntrack_tcp_timeout_time_wait = 30

# 应用配置
sysctl -p /etc/sysctl.d/99-high-concurrency.conf
```

### 2.3 SYN Flood防护

```bash
# SYN Cookie
sysctl -w net.ipv4.tcp_syncookies = 1

# SYN ACK重试次数(默认5次, 减少可降低SYN Flood影响)
sysctl -w net.ipv4.tcp_synack_retries = 2

# SYN重试次数
sysctl -w net.ipv4.tcp_syn_retries = 3

# 使用iptables限速
iptables -A INPUT -p tcp --syn -m limit --limit 100/s --limit-burst 200 -j ACCEPT
iptables -A INPUT -p tcp --syn -j DROP
```

## 三、连接池优化

### 3.1 Nginx到后端的长连接池

```nginx
upstream backend {
    server 10.0.0.1:8080;
    keepalive 128;              # 保持128个空闲长连接
    keepalive_requests 1000;    # 每个连接最多1000个请求后关闭
    keepalive_timeout 60s;      # 空闲连接保持60秒
}

server {
    location / {
        proxy_pass http://backend;
        proxy_http_version 1.1;              # 必须1.1才能用长连接
        proxy_set_header Connection "";      # 清除Connection头
    }
}
```

### 3.2 应用层连接池配置示例

#### Go连接池

```go
// HTTP客户端连接池
transport := &http.Transport{

    MaxIdleConnsPerHost: 100,           // 每个host最大空闲连接

    IdleConnTimeout:     90 * time.Second,
    DisableCompression:  false,
    // 连接复用
    ForceAttemptHTTP2:   true,
}

client := &http.Client{
    Transport: transport,

```

#### Java (HikariCP数据库连接池)

```java
HikariConfig config = new HikariConfig();
config.setMaximumPoolSize(50);              // 最大连接数(CPU核心数 * 2 + 磁盘数)

```

#### Python连接池

```python
import requests
from requests.adapters import HTTPAdapter
from urllib3.util.retry import Retry

# HTTP连接池
session = requests.Session()
adapter = HTTPAdapter(
    pool_connections=100,     # 连接池大小
    pool_maxsize=100,         # 最大连接数
    max_retries=Retry(
        total=3,
        backoff_factor=0.5,
        status_forcelist=[502, 503, 504]
    )
)
session.mount('http://', adapter)
session.mount('https://', adapter)

# 异步连接池 (aiohttp)
import aiohttp

async def create_connector():
    connector = aiohttp.TCPConnector(
        limit=200,              # 总连接数限制
        limit_per_host=50,      # 每host限制
        ttl_dns_cache=300,      # DNS缓存300秒
        enable_cleanup_closed=True
    )
    timeout = aiohttp.ClientTimeout(total=30, connect=5)
    return aiohttp.ClientSession(connector=connector, timeout=timeout)
```

### 3.3 连接池大小计算公式

```
# 数据库连接池(PostgreSQL公式)
connections = ((core_count * 2) + effective_spindle_count)

# 经验法则
- CPU密集型:  pool_size ≈ CPU核心数
- IO密集型:   pool_size ≈ CPU核心数 * (1 + IO等待/CPU时间)
- 通用推荐:   pool_size = 20 ~ 50 (起步)

# 过大的连接池反而降低性能:
# - 锁竞争增加
# - 上下文切换增多
# - 内存消耗增大
```

## 四、异步IO模型

### 4.1 IO模型对比

| 模型 | 特点 | 适用场景 | 代表 |
|------|------|----------|------|
| **阻塞IO** | 简单，每连接一个线程 | 连接数少 | 传统Tomcat |
| **非阻塞IO** | 轮询，CPU浪费 | 不推荐 | - |
| **IO多路复用** | select/poll/epoll | 高并发首选 | Nginx, Redis |
| **信号驱动IO** | 内核通知 | 特殊场景 | - |
| **异步IO** | 内核完成全部操作 | 大文件IO | io_uring |

### 4.2 epoll详解

```bash
# 查看系统epoll限制
cat /proc/sys/fs/epoll/max_user_watches
# 默认值通常为 内存/25KB，4GB内存约16万

# 调整epoll监控上限
sysctl -w fs.epoll.max_user_watches = 1048576
```

```c
// epoll使用示例(简化)
#include <sys/epoll.h>

int epoll_fd = epoll_create1(0);

struct epoll_event ev;
ev.events = EPOLLIN | EPOLLET;    // 边沿触发(ET模式)
ev.data.fd = listen_fd;
epoll_ctl(epoll_fd, EPOLL_CTL_ADD, listen_fd, &ev);

// ET模式 vs LT模式:
// ET (Edge Trigger): 事件只通知一次, 必须读完所有数据
// LT (Level Trigger): 只要数据未读完, 持续通知
// 高性能推荐ET(Nginx使用ET)
```

### 4.3 Nginx事件模型选择

```nginx
events {
    # Linux: epoll (默认)
    use epoll;

    # ET模式已在Nginx内部实现
    # Nginx默认使用epoll的ET模式(非阻塞)

    worker_connections 65535;
}
```

### 4.4 io_uring (Linux 5.1+)

### 4.5 应用层异步框架推荐

| 语言 | 框架 | IO模型 | 适用场景 |
|------|------|--------|----------|
| Go | net/http (goroutine) | M:N调度 + epoll | 高并发API服务 |

### 6.1 多级缓存

```
用户请求 → CDN → Nginx缓存 → 应用缓存 → 数据库缓存 → 数据库
  ↑                                                    ↑
  L1(边缘)   L2(反向代理)  L3(Redis)    L4(DB)        源
```

### 6.2 异步化

```python
# 消息队列异步处理(解耦削峰)
# 使用Redis/RabbitMQ/Kafka

import redis
import json

# 生产者: 将耗时操作放入队列
r = redis.Redis()
r.lpush('task_queue', json.dumps({
    'task': 'send_email',
    'to': 'user@example.com',
    'subject': 'Welcome'
}))

# 消费者: 异步处理
while True:
    task = r.brpop('task_queue', timeout=30)
    if task:
        process_task(json.loads(task[1]))
```

## 最佳实践

### 调优检查清单

| 层级 | 检查项 | 推荐值 |
|------|--------|--------|
| **OS** | fs.file-max | ≥ 2097152 |
| **OS** | somaxconn | ≥ 65535 |
| **OS** | tcp_max_tw_buckets | ≥ 1048576 |
| **OS** | ulimit -n | ≥ 1048576 |
| **OS** | tcp_congestion_control | bbr |
| **Nginx** | worker_processes | auto |
| **Nginx** | worker_connections | 65535 |
| **Nginx** | keepalive(upstream) | 128+ |
| **Nginx** | sendfile/tcp_nopush | on |
| **Nginx** | access_log | off/buffered |
| **App** | 连接池大小 | 20-100 |
| **App** | HTTP/2 | enabled |
| **App** | 异步框架 | 推荐 |

### 常见陷阱

1. **worker_connections过小**：连接数 = worker_processes × worker_connections，需留足余量
2. **日志IO瓶颈**：高QPS下日志写入是最大瓶颈，考虑缓冲或关闭access_log
3. **DNS解析阻塞**：使用 `resolver` 缓存或 `upstream` 静态配置
4. **keepalive未生效**：必须设置 `proxy_http_version 1.1` 和 `Connection ""`
5. **连接池过大**：连接数并非越多越好，遵循公式计算
6. **忽略TIME_WAIT**：高并发短连接场景必须优化TIME_WAIT参数
