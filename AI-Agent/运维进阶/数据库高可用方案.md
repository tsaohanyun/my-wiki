# 数据库高可用方案

## 概述

数据库高可用（HA）是运维进阶的关键领域。本文系统介绍MySQL高可用的主流方案：**MHA**、**MGR (Group Replication)**、**读写分离**和**分库分表**，涵盖架构设计、部署配置和故障切换。

## 一、MySQL复制基础

### 1.1 主从复制架构

```
                    ┌──────────────────────────────────┐
                    │           Master (写)              │
                    │     binlog → 中继日志              │
                    └───────┬──────────┬───────────────┘
                            │          │
                    ┌───────▼──┐  ┌───▼──────────┐
                    │ Slave 1  │  │   Slave 2     │
                    │  (读)    │  │   (读/备份)    │
                    └──────────┘  └──────────────┘
```

### 1.2 基于GTID的复制配置

```ini
# /etc/my.cnf - Master配置
[mysqld]
server-id = 1
log-bin = mysql-bin
binlog_format = ROW
gtid_mode = ON
enforce_gtid_consistency = ON

# 复制安全
sync_binlog = 1                    # 每次提交刷盘
innodb_flush_log_at_trx_commit = 1 # 每次事务刷redo log

# 半同步复制(减少数据丢失)
# plugin-load = "rpl_semi_sync_master=semisync_master.so"
# rpl_semi_sync_master_enabled = 1
# rpl_semi_sync_master_timeout = 3000  # 3秒超时降级为异步
```

```ini
# /etc/my.cnf - Slave配置
[mysqld]
server-id = 2
log-bin = mysql-bin
binlog_format = ROW
gtid_mode = ON
enforce_gtid_consistency = ON
read_only = ON
super_read_only = ON

# 并行复制(MySQL 8.0)
slave_parallel_type = LOGICAL_CLOCK
slave_parallel_workers = 8
slave_preserve_commit_order = ON
binlog_transaction_dependency_tracking = WRITESET

# 复制延迟监控
slave_net_timeout = 30
```

### 1.3 建立复制

```sql
-- Master: 创建复制用户
CREATE USER 'repl'@'10.0.0.%' IDENTIFIED BY 'StrongPassword!';
GRANT REPLICATION SLAVE ON *.* TO 'repl'@'10.0.0.%';

-- Slave: 配置GTID复制
CHANGE REPLICATION SOURCE TO
    SOURCE_HOST = '10.0.0.1',
    SOURCE_PORT = 3306,
    SOURCE_USER = 'repl',
    SOURCE_PASSWORD = 'StrongPassword!',
    SOURCE_AUTO_POSITION = 1,
    SOURCE_CONNECT_RETRY = 10;

START REPLICA;
SHOW REPLICA STATUS\G
```

## 二、MHA (Master High Availability)

### 2.1 MHA架构

```
              ┌──────────────────────────────┐
              │       MHA Manager             │
              │   (监控 + 故障切换 + VIP)     │
              └──────┬───────────────────────┘
                     │ 监控
         ┌───────────┼───────────┐
         │           │           │
    ┌────▼────┐ ┌────▼────┐ ┌────▼────┐
    │ Master  │ │ Slave 1 │ │ Slave 2 │
    │ (VIP)   │ │         │ │         │
    └─────────┘ └─────────┘ └─────────┘

故障切换流程:
1. Manager检测Master不可达
2. 从最新Slave保存binlog
3. 应用差异binlog到候选Master
4. 提升候选Master为新的Master
5. 其他Slave指向新Master
6. VIP漂移到新Master
```

### 2.2 MHA安装

```bash
# 所有节点安装MHA Node
yum install -y perl-DBD-MySQL perl-Config-Tiny perl-Log-Dispatch \
    perl-Parallel-ForkManager

# Manager节点安装MHA Manager
rpm -ivh mha4mysql-manager-0.58-0.el7.noarch.rpm

# SSH免密互信(所有节点)
ssh-keygen -t rsa -P ''
for host in 10.0.0.1 10.0.0.2 10.0.0.3; do
    ssh-copy-id -i ~/.ssh/id_rsa.pub root@$host
done

# 创建工作目录
mkdir -p /var/log/mha/app1
mkdir -p /etc/mha
```

```perl
[server2]
hostname = 10.0.0.2
port = 3306
candidate_master = 1

[server3]
hostname = 10.0.0.3
port = 3306
no_master = 1               # 不作为候选Master(如备份节点)
```

### 2.4 VIP故障切换脚本

```perl
#!/usr/bin/env perl
# /usr/local/bin/master_ip_failover

use strict;
use warnings FATAL => 'all';
use Getopt::Long;

my (
    $command,          $ssh_user,         $orig_master_host,
    $orig_master_ip,   $orig_master_port, $new_master_host,
    $new_master_ip,    $new_master_port
);

my $vip = '10.0.0.100/24';        # VIP地址
my $key = '0';                     # 网卡别名编号
my $ssh_start_vip = "/sbin/ip addr add $vip dev eth0 label eth0:$key";
my $ssh_stop_vip = "/sbin/ip addr del $vip dev eth0";
my $exit_code = 0;

GetOptions(
    'command=s'          => \$command,
    'ssh_user=s'         => \$ssh_user,
    'orig_master_host=s' => \$orig_master_host,
    'orig_master_ip=s'   => \$orig_master_ip,
    'orig_master_port=i' => \$orig_master_port,
    'new_master_host=s'  => \$new_master_host,
    'new_master_ip=s'    => \$new_master_ip,
    'new_master_port=i'  => \$new_master_port,
);

exit &main();

sub main {
    if ( $command eq "stop" || $command eq "stopssh" ) {
        # 旧Master下线VIP
        &stop_vip();
        $exit_code = 0;
    }
    elsif ( $command eq "start" ) {
        # 新Master上线VIP
        &start_vip();
        $exit_code = 0;
    }
    elsif ( $command eq "status" ) {
        print "Checking VIP status...\n";
        # 检查VIP是否存在
        if ( defined $orig_master_ip ) {
            &status_vip($orig_master_ip);
        }
        $exit_code = 0;
    }
    else {
        &usage();
        exit 1;
    }
    exit $exit_code;
}

sub start_vip() {
    `ssh $ssh_user\@$new_master_host \" $ssh_start_vip \"`;
}

sub stop_vip() {
    `ssh $ssh_user\@$orig_master_host \" $ssh_stop_vip \"`;
}

sub status_vip {
    my ($ip) = @_;
    my $output = `ssh $ssh_user\@$ip \" ip addr show eth0 | grep '$vip' \"`;
    if ($output) {
        print "VIP $vip is on $ip\n";
    } else {
        print "VIP $vip is NOT on $ip\n";
    }
}

sub usage {
    print "Usage: master_ip_failover --command=start|stop|stopssh|status --orig_master_host=host --orig_master_ip=ip --orig_master_port=port --new_master_host=host --new_master_ip=ip --new_master_port=port\n";
}
```


## 三、MGR (MySQL Group Replication)

### 3.1 MGR架构

```
                    ┌─────────────────────────────────┐
                    │      Group Replication Cluster    │
                    │                                   │
                    │  ┌─────────┐  ┌─────────┐        │
                    │  │  Node1  │←→│  Node2  │        │
                    │  │ Primary │  │ Primary │        │
                    │  └────┬────┘  └────┬────┘        │
                    │       │            │              │
                    │       └─────┬──────┘             │
                    │       ┌─────▼──────┐              │
                    │       │   Node3    │              │
                    │       │  Secondary │              │
                    │       └────────────┘              │
                    │                                   │
                    │  Paxos共识 + 自动故障检测         │
                    │  多主/单主模式                    │
                    └───────────────────────────────────┘
```

### 3.2 单主模式配置

```ini
# /etc/my.cnf - 所有节点通用配置(修改server-id和loose-group_replication_local_address)

[mysqld]
server-id = 1
log-bin = mysql-bin
binlog_format = ROW
binlog_checksum = NONE         # MGR要求

gtid_mode = ON
enforce_gtid_consistency = ON

# MGR核心配置
transaction_write_set_extraction = XXHASH64
loose-group_replication_group_name = "aaaaaaaa-aaaa-aaaa-aaaa-aaaaaaaaaaaa"
loose-group_replication_start_on_boot = OFF
loose-group_replication_local_address = "10.0.0.1:33061"
loose-group_replication_group_seeds = "10.0.0.1:33061,10.0.0.2:33061,10.0.0.3:33061"
loose-group_replication_bootstrap_group = OFF

# 单主模式
loose-group_replication_single_primary_mode = ON
loose-group_replication_enforce_update_everywhere_checks = OFF

# 故障检测
loose-group_replication_member_expel_timeout = 5
loose-group_replication_autorejoin_tries = 3   # MySQL 8.0.21+

# 流控
loose-group_replication_flow_control_mode = "QUOTA"
```

### 3.3 启动MGR集群

```sql
-- === Node1 (引导节点) ===

-- 安装插件
INSTALL PLUGIN group_replication SONAME 'group_replication.so';

-- 创建复制用户
SET SQL_LOG_BIN=0;
CREATE USER rpl_user@'%' IDENTIFIED BY 'RplPass123!';
GRANT REPLICATION SLAVE ON *.* TO rpl_user@'%';
FLUSH PRIVILEGES;
SET SQL_LOG_BIN=1;

-- 配置复制通道
CHANGE REPLICATION SOURCE TO SOURCE_USER='rpl_user',
    SOURCE_PASSWORD='RplPass123!' FOR CHANNEL 'group_replication_recovery';

-- 引导启动
SET GLOBAL group_replication_bootstrap_group=ON;
START GROUP_REPLICATION;
SET GLOBAL group_replication_bootstrap_group=OFF;

-- 验证
SELECT * FROM performance_schema.replication_group_members;
```

```sql
-- === Node2/Node3 (加入集群) ===

INSTALL PLUGIN group_replication SONAME 'group_replication.so';

SET SQL_LOG_BIN=0;
CREATE USER rpl_user@'%' IDENTIFIED BY 'RplPass123!';
GRANT REPLICATION SLAVE ON *.* TO rpl_user@'%';
FLUSH PRIVILEGES;
SET SQL_LOG_BIN=1;

CHANGE REPLICATION SOURCE TO SOURCE_USER='rpl_user',
    SOURCE_PASSWORD='RplPass123!' FOR CHANNEL 'group_replication_recovery';

START GROUP_REPLICATION;

-- 验证成员状态
SELECT member_host, member_port, member_state, member_role
FROM performance_schema.replication_group_members;
```

### 3.4 多主模式切换

```sql
-- 停止集群
STOP GROUP_REPLICATION;

-- 切换为多主模式
SET GLOBAL group_replication_single_primary_mode=OFF;
SET GLOBAL group_replication_enforce_update_everywhere_checks=ON;

-- 在引导节点重新启动
SET GLOBAL group_replication_bootstrap_group=ON;
START GROUP_REPLICATION;
SET GLOBAL group_replication_bootstrap_group=OFF;

-- 其他节点加入
START GROUP_REPLICATION;
```

## 四、读写分离

### 4.1 架构设计

```
                              ┌─────────────┐
                        ┌────→│   Master    │── 写入
    应用 ──→ ProxySQL ──┤     │  (10.0.0.1) │
               (路由)    │     └─────────────┘
                        │     ┌─────────────┐
                        ├────→│   Slave 1   │── 读取
                        │     │  (10.0.0.2) │
                        │     └─────────────┘
                        │     ┌─────────────┐
                        └────→│   Slave 2   │── 读取
                              │  (10.0.0.3) │
                              └─────────────┘
```

### 4.2 ProxySQL读写分离配置

```sql
-- 配置主从Hostgroup
-- hg 100 = 写(Primary), hg 1000 = 读(Secondary)

-- 查询规则: 读路由到Slave
INSERT INTO mysql_query_rules
(rule_id, active, match_pattern, destination_hostgroup, apply)
VALUES
(1, 1, '^SELECT.*FOR UPDATE$', 100, 1),    -- SELECT FOR UPDATE走Master
(2, 1, '^SELECT', 1000, 1),                -- 普通SELECT走Slave
(3, 1, '^INSERT|UPDATE|DELETE|CREATE|ALTER|DROP', 100, 1);

LOAD MYSQL QUERY RULES TO RUNTIME;
SAVE MYSQL QUERY RULES TO DISK;

-- 自动检测读写分离
-- ProxySQL 2.0+支持自动识别事务内读写
INSERT INTO mysql_query_rules
(rule_id, active, match_digest, destination_hostgroup, apply)
VALUES
(10, 1, '^SELECT', 1000, 1);

LOAD MYSQL QUERY RULES TO RUNTIME;
SAVE MYSQL QUERY RULES TO DISK;
```

```python
# Python读写分离示例
from sqlalchemy import create_engine
from sqlalchemy.orm import sessionmaker

class RoutingSession:
    """简单的读写分离Session路由"""
    def __init__(self, master_url, slave_urls):
        self.master = create_engine(master_url, pool_size=20)
        self.slaves = [create_engine(url, pool_size=20) for url in slave_urls]
        self.slave_index = 0

    def get_write_engine(self):
        return self.master

    def get_read_engine(self):
        engine = self.slaves[self.slave_index]
        self.slave_index = (self.slave_index + 1) % len(self.slaves)
        return engine

# 使用
router = RoutingSession(
    master_url="mysql://app:pass@10.0.0.1/mydb",
    slave_urls=[
        "mysql://app:pass@10.0.0.2/mydb",
        "mysql://app:pass@10.0.0.3/mydb",
    ]
)

# 读操作路由到Slave
read_session = sessionmaker(bind=router.get_read_engine())()

# 写操作路由到Master
write_session = sessionmaker(bind=router.get_write_engine())()
```

## 五、分库分表

### 5.1 分片策略

```
                    垂直拆分                    水平拆分
                  ┌────────┐              ┌────────────────┐
                  │ DB: Order│              │ DB: user_0      │
                  │ - orders │              │ user(id%4==0)   │
                  │ - items  │              ├────────────────┤
   单库 ────────→ │          │     单表 ──→ │ DB: user_1      │
   (所有表)       │ DB: User │   (大量行)   │ user(id%4==1)   │
                  │ - users  │              ├────────────────┤
                  │ - profile│              │ DB: user_2      │
                  └────────┘              │ DB: user_3      │
                                            └────────────────┘
```

### 5.2 水平分片方案

| 分片策略 | 说明 | 适用场景 | 缺点 |
|----------|------|----------|------|
| **范围分片** | 按ID范围划分 | 时间序列数据、自增ID | 热点问题 |
| **哈希分片** | hash(id) % N | 均匀分布 | 扩容需重新分片 |
| **一致性哈希** | 虚拟节点+哈希环 | 动态扩容 | 实现复杂 |
| **枚举分片** | 按地区/业务类型 | 地理分布数据 | 分布不均 |

### 5.4 分库分表后的挑战与解决方案

| 挑战 | 解决方案 |
|------|----------|
| **跨库JOIN** | 广播表、绑定表、应用层组装 |
| **分布式事务** | XA事务、Saga模式、TCC |
| **全局唯一ID** | Snowflake、号段模式、UUID |
| **分页查询** | 各分片取N条后归并排序 |
| **扩容迁移** | 一致性哈希、双写+灰度迁移 |
| **聚合统计** | 各分片聚合→二次聚合 |

### 5.5 分布式事务示例

```java
// ShardingSphere XA分布式事务
@Configuration
public class XATransactionConfig {

    @Bean
    public ShardingTransactionType xaTransactionType() {
        return new ShardingTransactionType("XATransaction");
    }
}

// 使用
@Service
public class OrderService {

    @Transactional
    @ShardingTransactionType(TransactionType.XA)
    public void createOrder(Order order) {
        // 跨库操作自动使用XA事务
        orderMapper.insert(order);          // 可能路由到ds0
        inventoryMapper.reduce(order);      // 可能路由到ds1
        pointMapper.add(order);             // 可能路由到ds2
    }
}
```

```python
# Saga模式 (Python示例)
# 适合长事务场景, 每个步骤有补偿操作

class OrderSaga:
    def execute(self, order):
        try:
            self.create_order(order)           # 正向操作
            self.deduct_inventory(order)        # 正向操作
            self.add_points(order)              # 正向操作
        except Exception as e:
            # 补偿(逆序执行)
            self.compensate_add_points(order)
            self.compensate_deduct_inventory(order)
            self.compensate_create_order(order)
            raise e
```

## 最佳实践

### 方案选型

| 方案 | 数据一致性 | 自动故障切换 | 运维复杂度 | 适用场景 |
|------|-----------|-------------|-----------|----------|
| **MHA** | 最终一致 | 是(需脚本) | 中 | 传统主从架构 |
| **MGR** | 强一致 | 是(内置) | 中高 | 金融级高可用 |
| **Orchestrator** | 最终一致 | 是 | 中 | 拓扑管理灵活 |
| **ProxySQL** | N/A(路由) | 配合使用 | 低 | 读写分离 |

### 运维原则

1. **GTID优先**：所有复制使用GTID，简化故障恢复
2. **半同步复制**：核心数据使用半同步，减少数据丢失风险
3. **多活架构**：跨机房部署，避免单机房故障
4. **定期演练**：定期进行故障切换演练，验证切换时间
5. **监控覆盖**：Seconds_Behind_Master、连接数、慢查询、磁盘空间
6. **备份验证**：定期恢复备份，验证可用性
7. **连接池**：应用侧必须使用连接池，避免连接风暴
8. **分片谨慎**：不到万不得已不分库分表，优先垂直拆分和读写分离

### 故障切换检查清单

- [ ] VIP是否成功漂移
- [ ] 应用是否自动连接到新Master
- [ ] 复制拓扑是否正确更新
- [ ] 旧Master是否安全下线
- [ ] 监控告警是否恢复
- [ ] 事后是否触发根因分析
