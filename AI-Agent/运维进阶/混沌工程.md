# 混沌工程

## 概述

混沌工程（Chaos Engineering）是一种通过**主动注入故障**来验证系统弹性的实验方法。它不是破坏系统，而是通过受控的故障实验，在真正的大规模故障发生前发现系统弱点，从而建立对系统韧性的信心。

> **核心理念：** 拥抱故障，在故障中学习，而不是等到生产事故才发现问题。

---

## 一、混沌工程原理

### 1.1 起源与发展

```
2010  Netflix 提出 Chaos Monkey
2011  Netflix 发布 Simian Army (猴子军团)
2014  正式定义为 "Chaos Engineering"
2015  Netflix 开源 Chaos Monkey
2019  Chaos Mesh / LitmusChaos 等K8s原生工具兴起
2024  混沌工程成为云原生SRE标准实践
```

### 1.2 四大核心原则（PRINCIPLES OF CHAOS ENGINEERING）

1. **定义系统稳态（Steady State）**：定义可衡量的系统正常行为指标（如延迟P99、错误率、吞吐量）
2. **假设系统保持稳态**：假设在故障条件下系统仍能维持稳态
3. **模拟真实世界事件**：注入服务器宕机、网络延迟、磁盘故障等真实故障
4. **持续自动化实验**：在生产环境中持续运行，自动化执行

### 1.3 实验流程

```
┌──────────────────────────────────────────────────────┐
│                  混沌工程实验流程                      │
│                                                        │
│  1. 定义稳态指标                                       │
│     └→ P99延迟 < 100ms, 错误率 < 0.1%, QPS > 10000   │
│                                                        │
│  2. 形成假设                                           │
│     └→ "当一个节点宕机时，系统仍满足稳态"              │
│                                                        │
│  3. 设计实验                                           │
│     └→ 随机杀死一个Pod，持续5分钟                      │
│                                                        │
│  4. 准备熔断                                           │
│     └→ 设定终止条件：错误率 > 1% 或 P99 > 500ms       │
│                                                        │
│  5. 执行实验                                           │
│     └→ 注入故障，实时监控                              │
│                                                        │
│  6. 分析结果                                           │
│     ├→ 稳态保持 → 加大爆炸半径                         │
│     └→ 稳态打破 → 修复弱点，重新实验                    │
│                                                        │
│  7. 自动化与扩大范围                                   │
│     └→ 扩大到多节点、跨机房、生产环境                  │
└──────────────────────────────────────────────────────┘
```

### 1.4 故障注入类型

| 类型 | 描述 | 示例工具 |
|------|------|----------|
| **网络故障** | 延迟、丢包、分区、带宽限制 | tc, ToxProxy, Chaos Mesh |
| **计算故障** | CPU满载、内存耗尽、进程杀死 | stress-ng, Chaos Monkey |
| **存储故障** | 磁盘满、IO延迟、读写错误 | Chaos Mesh, fiu |
| **时钟漂移** | 时间偏差 | chrony, Chaos Mesh |
| **依赖故障** | 依赖服务不可用 | Toxiproxy, pumba |
| **DNS故障** | 解析失败、解析延迟 | dnschaos |

### 3.1 Chaos Monkey概念

```
Chaos Monkey 随机终止生产环境的实例，
确保工程师在开发和部署服务时总是考虑到弹性。

┌──────────────────────────────┐
│      Chaos Monkey             │
│                                │
│  定时执行(如工作日9-15点)      │
│    └→ 随机选择一个实例         │
│       └→ 终止实例              │
│          └→ 验证自动恢复       │
│                                │
│  目标: 让故障成为常态          │
```


### 4.3 tc / iptables (手动故障注入)

```bash
# === 网络延迟 ===
# 对eth0接口添加100ms延迟+20ms抖动
tc qdisc add dev eth0 root netem delay 100ms 20ms

# 仅影响特定目标IP
tc qdisc add dev eth0 root handle 1: prio
tc qdisc add dev eth0 parent 1:3 handle 30: netem delay 100ms 20ms
tc filter add dev eth0 protocol ip parent 1:0 prio 3 u32 \
    match ip dst 10.0.0.5/32 flowid 1:3

# 恢复
tc qdisc del dev eth0 root

# === 网络丢包 ===
tc qdisc add dev eth0 root netem loss 10%

# === 带宽限制 ===
tc qdisc add dev eth0 root tbf rate 1mbit burst 32kbit latency 400ms

# === 网络分区(完全断开) ===
iptables -A OUTPUT -d 10.0.0.5 -j DROP
iptables -A INPUT -s 10.0.0.5 -j DROP

# 恢复
iptables -D OUTPUT -d 10.0.0.5 -j DROP
iptables -D INPUT -s 10.0.0.5 -j DROP

# === 磁盘故障 ===
# 使用dmsetup模拟IO延迟
dmsetup message my-device 0 "delay_io 100 50"
```

## 五、自动化实验流水线

### 5.1 GitHub Actions集成

```yaml
# .github/workflows/chaos-test.yml
name: Chaos Engineering Test

on:
  schedule:
    - cron: "0 2 * * 1-5"    # 工作日凌晨2点执行

jobs:
  chaos-test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Configure kubectl
        run: |
          echo "${{ secrets.KUBE_CONFIG }}" > kubeconfig
          export KUBECONFIG=kubeconfig

      - name: Run Network Chaos
        run: |
          # 应用网络延迟实验
          kubectl apply -f experiments/network-delay.yaml

          # 等待实验完成
          kubectl wait --for=condition=Complete networkchaos/network-delay \
              -n chaos-testing --timeout=10m || true

      - name: Verify SLO
        run: |
          # 验证SLI指标是否在阈值内
          RESULT=$(curl -s "http://prometheus:9090/api/v1/query?query=histogram_quantile(0.99,sum(rate(http_request_duration_seconds_bucket{job=\"my-service\"}[5m]))by(le))")
          P99=$(echo $RESULT | jq '.data.result[0].value[1]')
          echo "P99 latency: ${P99}s"
          # 如果P99 > 0.5s则失败
          if (( $(echo "$P99 > 0.5" | bc -l) )); then
            echo "::error::P99 latency exceeded SLO threshold"
            exit 1
          fi

      - name: Cleanup
        if: always()
        run: |
          kubectl delete -f experiments/network-delay.yaml --ignore-not-found
```

### 5.2 Python自动化框架

```python
#!/usr/bin/env python3
"""
混沌工程自动化实验框架
"""
import subprocess
import json
import time
import requests
from dataclasses import dataclass
from typing import Optional

@dataclass
class SteadyState:
    """系统稳态定义"""
    name: str
    prometheus_query: str
    threshold: float
    operator: str  # "lt" or "gt"

@dataclass
class ChaosExperiment:
    """混沌实验定义"""
    name: str
    manifest_file: str
    duration_seconds: int
    steady_states: list

class ChaosRunner:
    def __init__(self, prometheus_url: str, namespace: str = "chaos-testing"):
        self.prometheus_url = prometheus_url
        self.namespace = namespace

    def query_metric(self, query: str) -> float:
        """查询Prometheus指标"""
        resp = requests.get(
            f"{self.prometheus_url}/api/v1/query",
            params={"query": query}
        )
        result = resp.json()
        if result["status"] == "success" and result["data"]["result"]:
            return float(result["data"]["result"][0]["value"][1])
        return float('inf')

    def check_steady_state(self, state: SteadyState) -> bool:
        """检查稳态是否满足"""
        value = self.query_metric(state.prometheus_query)
        if state.operator == "lt":
            passed = value < state.threshold
        else:
            passed = value > state.threshold

        status = "✅ PASS" if passed else "❌ FAIL"
        print(f"  {state.name}: {value:.4f} {state.operator} {state.threshold} → {status}")
        return passed

    def apply_manifest(self, file: str):
        """应用混沌实验"""
        subprocess.run(
            ["kubectl", "apply", "-f", file, "-n", self.namespace],
            check=True
        )
        print(f"✅ Applied: {file}")

    def delete_manifest(self, file: str):
        """清理混沌实验"""
        subprocess.run(
            ["kubectl", "delete", "-f", file, "-n", self.namespace,
             "--ignore-not-found"],
            check=True
        )
        print(f"🧹 Cleaned: {file}")

    def run_experiment(self, exp: ChaosExperiment) -> bool:
        """执行混沌实验"""
        print(f"\n{'='*60}")
        print(f"🧪 Experiment: {exp.name}")
        print(f"{'='*60}")

        # 1. 检查初始稳态
        print("\n📊 Pre-experiment steady state:")
        pre_passed = all(self.check_steady_state(s) for s in exp.steady_states)
        if not pre_passed:
            print("⚠️  Steady state not met before experiment. Aborting.")
            return False

        # 2. 注入故障
        print(f"\n💉 Injecting fault: {exp.manifest_file}")
        self.apply_manifest(exp.manifest_file)

        # 3. 持续监控
        passed = True
        interval = 10  # 每10秒检查一次
        checks = exp.duration_seconds // interval

        for i in range(checks):
            time.sleep(interval)
            elapsed = (i + 1) * interval
            print(f"\n⏱️  [{elapsed}s/{exp.duration_seconds}s] Checking steady state:")

            for state in exp.steady_states:
                if not self.check_steady_state(state):
                    print(f"🚨 Steady state violated! Auto-aborting experiment.")
                    passed = False
                    break

            if not passed:
                break

        # 4. 清理
        print("\n🧹 Cleaning up...")
        self.delete_manifest(exp.manifest_file)

        # 5. 等待恢复
        print("⏳ Waiting for recovery (60s)...")
        time.sleep(60)

        # 6. 检查恢复后稳态
        print("\n📊 Post-experiment steady state:")
        post_passed = all(self.check_steady_state(s) for s in exp.steady_states)

        # 7. 结果汇总
        print(f"\n{'='*60}")
        if passed and post_passed:
            print(f"✅ EXPERIMENT PASSED: System remained resilient!")
        else:
            print(f"❌ EXPERIMENT FAILED: System weakness detected!")
        print(f"{'='*60}")

        return passed and post_passed
```

## 六、游戏日（Game Day）

### 6.1 Game Day概念

游戏日是有组织的混沌工程演练，团队在受控环境中模拟故障，验证系统的弹性和团队响应能力。

### 6.2 Game Day流程

```
Phase 1: 准备 (T-7天)
  ├── 确定实验范围和爆炸半径
  ├── 定义稳态指标和终止条件
  ├── 通知相关团队
  └── 准备回滚方案

Phase 2: 执行 (T-0)
  ├── 记录初始状态
  ├── 按计划注入故障
  ├── 团队按On-call流程响应
  ├── 记录所有发现
  └── 达到终止条件时停止

Phase 3: 复盘 (T+1天)
  ├── 实验结果总结
  ├── 发现的弱点分析
  ├── 改进项排期
  └── 更新混沌实验库
```

### 6.3 Game Day实验剧本

```markdown
## Game Day: 数据库主节点故障

### 实验目标
验证数据库主节点故障时，应用是否自动切换到备节点，且零数据丢失。

### 稳态指标
- API P99延迟 < 200ms
- 错误率 < 0.1%
- 数据库写成功率 > 99.9%

### 实验步骤
1. [0:00] 记录稳态指标基线
2. [0:01] 杀死数据库主节点Pod
3. [0:01-0:05] 观察故障切换过程
4. [0:05] 验证新主节点已接管
5. [0:05-0:10] 验证应用自动重连
6. [0:10] 恢复原节点
7. [0:10-0:15] 验证数据一致性

### 终止条件
- 错误率 > 1% 持续超过2分钟
- 数据丢失
- API完全不可用超过1分钟

### 预期结果
- 自动故障切换 < 30秒
- 零数据丢失
- 应用无感知
```

## 最佳实践

### 实施成熟度模型

```
Level 0: 被动响应      → 故障发生才知道问题
Level 1: 手动测试      → 手动注入故障，验证恢复
Level 2: 自动化实验    → CI/CD中自动运行混沌实验
Level 3: 持续混沌      → 生产环境持续运行(如Chaos Monkey)
Level 4: 自适应混沌    → 基于系统状态自动调整实验参数
```

### 黄金法则

| 法则 | 说明 |
|------|------|
| **从小开始** | 先在测试环境，再逐步推广到生产 |
| **爆炸半径递增** | 单Pod → 多Pod → 节点 → 机房 |
| **始终有终止条件** | 设定自动熔断阈值 |
| **团队参与** | Chaos需要开发+运维+QA共同参与 |
| **记录一切** | 每次实验的输入、输出、发现都要归档 |
| **修复优先** | 发现弱点后立即修复，不要积累技术债 |
| **业务时段执行** | Chaos Monkey在工作日工作时间运行，团队在场 |

### 常见反模式

❌ **在无人值守时运行大规模实验**
❌ **没有稳态定义就注入故障**
❌ **没有终止条件的长时间实验**
❌ **只测试简单场景，忽略级联故障**
❌ **发现弱点后不修复，继续运行实验**
❌ **对核心业务直接进行未验证的实验**
