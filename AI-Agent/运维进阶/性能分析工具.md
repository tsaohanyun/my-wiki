# 性能分析工具

## 概述

性能分析是运维进阶的核心能力。本文系统介绍Linux下主流性能分析工具：**perf**、**strace**、**eBPF**、**火焰图**和**系统调用追踪**，帮助快速定位CPU、内存、IO、锁等性能瓶颈。

---

## 一、性能分析方法论

### 1.1 USE方法（资源导向）

对每个资源检查 **Utilization（利用率）**、**Saturation（饱和度）**、**Errors（错误）**：

```
CPU:  利用率 → vmstat us/sy | 饱和度 → run queue | 错误 → 硬件错误
内存: 利用率 → free | 饱和度 → swap/page fault | 错误 → OOM
磁盘: 利用率 → iostat %util | 饱和度 → queue depth | 错误 → dmesg
网络: 利用率 →带宽 | 饱和度 → 队列 | 错误 → 丢包/重传
```

### 1.2 RED方法（服务导向）

对每个服务检查 **Rate（请求速率）**、**Errors（错误率）**、**Duration（延迟）**。

### 1.3 工具选择矩阵

| 分析维度 | 首选工具 | 深入分析 |
|----------|----------|----------|
| CPU热点 | perf top | perf record + 火焰图 |
| 系统调用 | strace | perf trace |
| 内核追踪 | eBPF/BCC | bpftrace |
| 内存分配 | perf mem | pmap / massif |
| 锁竞争 | perf lock | eBPF offcpu |
| 网络延迟 | eBPF tcpdump | tcpreplay |
| 磁盘IO | iostat | biolatency (eBPF) |

## 二、perf - Linux性能分析瑞士军刀

### 2.1 安装与基础

```bash
# 安装perf
yum install -y perf              # RHEL/CentOS
apt install -y linux-tools-common linux-tools-$(uname -r)  # Ubuntu/Debian

# 基础命令
perf --version
perf list                         # 列出可用事件
perf list 'syscalls:*'            # 列出系统调用事件
perf list 'sched:*'               # 列出调度事件
```

### 2.2 实时性能监控

```bash
# CPU热点实时监控(类似top)
perf top

# 监控特定PID
perf top -p <pid>

# 监控特定事件
perf top -e cache-misses,branch-misses
perf top -e 'syscalls:sys_enter_openat'

# 指定刷新频率
perf top -F 100 -d 2              # 100Hz采样, 2秒刷新
```

### 2.3 性能采样与分析

```bash
# CPU采样(默认采样事件cycles)
perf record -F 99 -p <pid> -g -- sleep 30
# -F 99: 99Hz采样频率
# -g: 记录调用栈
# -- sleep 30: 采样30秒

# 采集整个系统
perf record -F 99 -a -g -- sleep 30

# 指定事件采样
perf record -e cache-misses -p <pid> -g -- sleep 30

# 分析结果
perf report                       # 交互式报告
perf report --stdio               # 文本格式输出

# 查看统计汇总
perf report --stdio -n --percent-limit 1

# 导出为脚本分析
perf script > perf_out.txt
```

### 2.4 perf stat - 性能计数器

```bash
# 统计命令的性能计数器
perf stat -e cycles,instructions,cache-misses,branch-misses ./my_program

# 统计PID
perf stat -p <pid> -- sleep 10

# 高级统计 - 复杂事件组合
perf stat -e \
  cycles,instructions,\
  L1-dcache-loads,L1-dcache-load-misses,\
  LLC-loads,LLC-load-misses,\
  branch-instructions,branch-misses \
  ./my_program

# 输出示例分析
# 1. IPC (Instructions Per Cycle) = instructions/cycles
#    IPC < 1 表示CPU等待(内存/缓存未命中)
#    IPC > 1 表示良好的指令级并行
# 2. 缓存命中率 = 1 - cache-misses/cache-loads
```

### 2.5 perf高级功能

```bash
# 追踪上下文切换
perf record -e sched:sched_switch -a -g -- sleep 10
perf script | sed -n '/sched_switch/p' | wc -l   # 统计切换次数

# 追踪锁竞争
perf lock record -p <pid> -- sleep 30
perf lock report

# 内存访问分析
perf mem record -p <pid> -- sleep 30
perf mem report

# 追踪函数调用图(需要调试符号)
perf record --call-graph dwarf -p <pid> -- sleep 30

# 动态追踪函数
perf probe --add='tcp_sendmsg'          # 添加探测点
perf record -e probe:tcp_sendmsg -a -- sleep 10
perf probe --del='tcp_sendmsg'          # 删除探测点
```

### 2.6 perf火焰图生成

```bash
# 安装FlameGraph工具
git clone https://github.com/brendangregg/FlameGraph.git
export PATH=$PATH:$(pwd)/FlameGraph

# CPU火焰图
perf record -F 99 -p <pid> -g -- sleep 30
perf script -i perf.data > out.perf
./flamegraph.pl out.perf > cpu_flame.svg

# 分支预测失败火焰图
perf record -e branch-misses -c 10000 -p <pid> -g -- sleep 30
perf script > branch_miss.perf
./flamegraph.pl branch_miss.perf > branch_flame.svg

# 差分火焰图(对比两次性能)
perf record -F 99 -p <pid> -g -o base.data -- sleep 30
perf record -F 99 -p <pid> -g -o new.data -- sleep 30
perf script -i base.data > base.perf
perf script -i new.data > new.perf
./difffolded.pl base.perf new.perf | ./flamegraph.pl > diff_flame.svg
```

## 三、strace - 系统调用追踪

### 3.1 基础用法

```bash
# 追踪命令的所有系统调用
strace -c ./my_program            # 统计模式
strace -e trace=openat,read,write ./my_program  # 过滤特定调用
strace -p <pid>                   # 追踪运行中的进程
strace -f -p <pid>                # 追踪进程及其子进程

# 常用选项
strace -T -tt -e trace=network ./my_program
# -T: 显示每次调用耗时
# -tt: 微秒级时间戳
# -e trace=network: 仅追踪网络相关调用
```

### 3.2 常见性能分析场景

```bash
# 1. 定位文件打开失败
strace -e trace=open,openat -f ./my_program 2>&1 | grep -i "ENOENT\|EACCES"

# 2. 定位网络连接问题
strace -e trace=connect -p <pid> 2>&1 | grep "ECONNREFUSED\|ETIMEDOUT"

# 3. 统计系统调用开销
strace -c -p <pid>
# 输出示例:
# % time     seconds  usecs/call     calls    errors syscall
# ------ ----------- ----------- --------- --------- ----------------
#   45.00    0.045000          15      3000           futex        <- 锁等待
#   30.00    0.030000          10      3000           read
#   10.00    0.010000           5      2000           write

# 4. 追踪信号
strace -e trace=signal -p <pid>

# 5. 只追踪错误返回
strace -e fault=read -p <pid>     # 注入read错误
```

### 3.3 strace性能注意

> **⚠️ strace对性能影响巨大（2-10倍减速），生产环境慎用！**
> 优先使用 `perf trace` 或 eBPF 替代。

```bash
# perf trace - strace的高性能替代
perf trace -p <pid>
perf trace -e openat -p <pid>     # 仅追踪openat
perf trace --summary -p <pid>     # 统计模式
```

## 四、eBPF - 下一代追踪技术

### 4.1 eBPF概述

eBPF (Extended Berkeley Packet Filter) 允许在内核中安全运行沙盒程序，无需修改内核源码或加载模块。是现代Linux性能分析的核心技术。

### 4.2 BCC工具集安装

```bash
# Ubuntu/Debian
apt install -y bpfcc-tools linux-headers-$(uname -r)

# RHEL/CentOS
yum install -y bcc-tools

# 从源码安装
git clone https://github.com/iovisor/bcc.git
cd bcc
mkdir build && cd build
cmake .. -DCMAKE_INSTALL_PREFIX=/usr
make -j$(nproc)
make install
```

### 4.3 BCC常用工具

```bash
# === CPU分析 ===

# 函数级CPU profiling
profile-bpfcc -p <pid> -d 30     # 采样30秒

# off-CPU分析(线程阻塞在哪)
offcputime-bpfcc -p <pid> 30

# 追踪函数调用延迟
funclatency-bpfcc -p <pid> vfs_read

# === 内存分析 ===

# 页错误追踪
faults-bpfcc

# 内存泄漏检测
memleak-bpfcc -p <pid>

# OOM追踪
oomkill-bpfcc

# === 磁盘IO ===

# 磁盘IO延迟直方图
biolatency-bpfcc -d 60

# 每个进程的IO统计
biotop-bpfcc 10 5               # 每10秒刷新，共5次

# 追踪IO请求大小
biosnoop-bpfcc

# === 网络 ===

# TCP连接追踪
tcplife-bpfcc

# TCP重传追踪
tcpretrans-bpfcc

# TCP RTT测量
tcprtt-bpfcc

# 网络丢包追踪
drops-bpfcc

# === 系统调用 ===

# 追踪execve(进程创建)
execsnoop-bpfcc

# 追踪opensnoop(文件打开)
opensnoop-bpfcc -p <pid>

# 文件打开错误
opensnoop-bpfcc -e

# === 其他 ===

# 追踪runqueue latency(调度延迟)
runqlat-bpfcc 10

# 追踪锁竞争
offcputime-bpfcc --stack-storage-size 4096 30

# 内核函数调用追踪
trace-bpfcc 'tcp_sendmsg'
```

### 4.4 bpftrace - 单行命令追踪

```bash
# 安装
apt install -y bpftrace

# 列出所有探测点
bpftrace -l 'tracepoint:syscalls:sys_enter_*'

# === 常用单行命令 ===

# 统计系统调用
bpftrace -e 'tracepoint:syscalls:sys_enter_* { @[probe] = count(); }' \
    --unsafe interval:s:10 '{ exit(); }'

# 追踪openat调用并显示文件名
bpftrace -e 'tracepoint:syscalls:sys_enter_openat {
    printf("%s open: %s\n", comm, str(args->filename));
}'

# 统计read返回大小的直方图
bpftrace -e 'tracepoint:syscalls:sys_exit_read /comm == "my_app"/ {
    @bytes = hist(args->ret);
}'

# 追踪内核函数延迟
bpftrace -e 'kprobe:vfs_read { @start[tid] = nsecs; }
    kretprobe:vfs_read /@start[tid]/ {
        @us = hist((nsecs - @start[tid]) / 1000);
        delete(@start[tid]);
    }'

# TCP连接追踪
bpftrace -e 'tracepoint:syscalls:sys_enter_connect {
    printf("connect: %s pid=%d\n", comm, pid);
}'

# 统计上下文切换
bpftrace -e 'tracepoint:sched:sched_switch { @[comm] = count(); }' \
    interval:s:10 '{ print(@); clear(@); }'

# 内存分配追踪
bpftrace -e 'kprobe:__kmalloc { @size[comm] = hist(arg1); }'

# CPU on/off 分析
bpftrace -e 'profile:hz:99 {
    @[ustack, kstack, comm] = count();
}'
```

### 4.5 自定义eBPF程序示例

```python
#!/usr/bin/env python3
"""使用BCC追踪TCP连接延迟"""
from bcc import BPF

# eBPF C程序
bpf_text = """
#include <uapi/linux/ptrace.h>
#include <net/sock.h>
#include <bcc/proto.h>

BPF_HASH(connect_start, u32);
BPF_HISTOGRAM(delta);

int trace_connect(struct pt_regs *ctx, struct sock *sk) {
    u32 pid = bpf_get_current_pid_tgid() >> 32;
    u64 ts = bpf_ktime_get_ns();
    connect_start.update(&pid, &ts);
    return 0;
}

int trace_connect_ret(struct pt_regs *ctx) {
    u32 pid = bpf_get_current_pid_tgid() >> 32;
    u64 *tsp = connect_start.lookup(&pid);
    if (tsp == 0) return 0;

    u64 delta_us = (bpf_ktime_get_ns() - *tsp) / 1000;
    delta.atomic_increment(bpf_log2l(delta_us));
    connect_start.delete(&pid);
    return 0;
}
"""

b = BPF(text=bpf_text)
b.attach_kprobe(event="tcp_v4_connect", fn_name="trace_connect")
b.attach_kretprobe(event="tcp_v4_connect", fn_name="trace_connect_ret")

print("Tracing TCP connect latency... Ctrl-C to stop.")
try:
    b.perf_buffer_poll()
except KeyboardInterrupt:
    print("\nTCP connect latency (microseconds):")
    b["delta"].print_log2_hist("usecs")
```

## 五、火焰图深入

### 5.1 火焰图类型

| 类型 | 采集方式 | 分析内容 |
|------|----------|----------|
| **CPU火焰图** | perf record -F 99 -g | CPU时间花在哪 |
| **Off-CPU火焰图** | offcputime (eBPF) | 阻塞/等待时间 |
| **内存火焰图** | perf record -e malloc | 内存分配来源 |
| **差分火焰图** | difffolded.pl | 对比两次性能 |
| **系统调用火焰图** | perf record -e raw_syscalls | 系统调用开销 |

### 5.2 Off-CPU火焰图

```bash
# 使用BCC采集off-CPU数据
offcputime-bpfcc -f -p <pid> 30 > offcpu.stacks

# 生成火焰图
./flamegraph.pl --title="Off-CPU Time" --countname=us \
    --colors=io offcpu.stacks > offcpu_flame.svg
```

### 5.3 内存火焰图

```bash
# 使用perf追踪malloc
perf record -e 'syscalls:sys_enter_brk' -g -p <pid> -- sleep 30
perf script > mem.perf
./stackcollapse-perf.pl mem.perf > mem.folded
./flamegraph.pl --title="Memory Allocation" --colors=mem mem.folded > mem_flame.svg
```

### 5.4 如何阅读火焰图

```
  ┌─main()────────────────────────────────┐
  │  ┌─process_data()──────────────────┐  │
  │  │  ┌─parse_json()─────┐           │  │
  │  │  │  ┌─malloc()─┐    │ ┌─io()──┐ │  │
  │  │  │  │          │    │ │       │ │  │
  │  │  │  └──────────┘    │ └───────┘ │  │
  │  │  └──────────────────┘           │  │
  │  └──────────────────────────────────┘  │
  └───────────────────────────────────────┘
```

- **X轴**：调用栈层级（从下到上是调用关系）
- **Y轴**：采样次数（宽度 = CPU时间占比）
- **规则**：最宽的"火焰"就是优化重点

## 六、综合实战案例

### 案例1：CPU使用率飙升排查

```bash
# Step 1: 定位高CPU进程
top -b -n1 | head -20

# Step 2: 采样CPU热点
perf record -F 99 -p <pid> -g -- sleep 30

# Step 3: 生成火焰图
perf script > out.perf
./stackcollapse-perf.pl out.perf | ./flamegraph.pl > cpu.svg

# Step 4: 分析热点函数
perf report --stdio -n --percent-limit 1
```

### 案例2：请求延迟突然增大

```bash
# Step 1: 使用eBPF追踪on/off CPU
profile-bpfcc -p <pid> -f 30 > oncpu.stacks
offcputime-bpfcc -p <pid> -f 30 > offcpu.stacks

# Step 2: 生成Off-CPU火焰图定位阻塞
./flamegraph.pl --title="Off-CPU" offcpu.stacks > offcpu.svg

# Step 3: 检查是否有锁竞争
offcputime-bpfcc -p <pid> --stack-storage-size 8192 60 | grep -i "futex\|mutex\|lock"

# Step 4: 追踪IO等待
biolatency-bpfcc -d 60
```

### 案例3：内存泄漏排查

```bash
# Step 1: 使用BCC memleak
memleak-bpfcc -p <pid> 60

# Step 2: 使用valgrind(测试环境)
valgrind --leak-check=full --show-leak-kinds=all ./my_program

# Step 3: 使用perf追踪分配
perf record -e 'syscalls:sys_enter_brk,syscalls:sys_enter_mmap' -g -p <pid> -- sleep 30
perf script | awk '{print $NF}' | sort | uniq -c | sort -rn | head
```

## 最佳实践

### 工具选择策略

```
问题: "系统为什么慢？"
  ├── CPU高? → perf top → perf record → 火焰图
  ├── CPU不高但慢? → offcputime (eBPF) → Off-CPU火焰图
  ├── 磁盘慢? → biolatency/biosnoop (eBPF) → iostat
  ├── 网络慢? → tcprtt/tcpretrans (eBPF) → tcpdump
  ├── 锁竞争? → offcputime → grep futex
  └── 不确定? → bpftrace oneliner → 缩小范围
```

### 生产环境注意事项

1. **perf -F 99**：使用99Hz而非100Hz，避免与系统时钟同步导致偏差
2. **strace慎用**：生产环境用 `perf trace` 替代 `strace`
3. **eBPF开销低**：BCC工具适合生产环境（< 1% CPU开销）
4. **采样时间**：至少30秒，覆盖完整请求周期
5. **符号表**：确保安装debuginfo包，否则火焰图只有地址
6. **容器环境**：需要 `--privileged` 或正确的capabilities

### 调试符号安装

```bash
# RHEL/CentOS
debuginfo-install kernel-$(uname -r)
# 或
yum install -y kernel-debuginfo-$(uname -r)

# Ubuntu/Debian
apt install -y linux-image-$(uname -r)-dbgsym

# 应用程序符号
# Go程序: 编译时加 -ldflags='-compressdwarf=false'
# Rust程序: 默认包含
# C/C++程序: 编译时加 -g
```
