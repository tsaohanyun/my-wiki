# 面向对象编程进阶

> 面向对象编程（OOP）进阶涵盖设计原则、设计模式、依赖注入和领域驱动设计等高级主题，帮助构建可维护、可扩展的大型软件系统。

---

## 一、SOLID 原则

SOLID 是面向对象设计的五大基本原则，由 Robert C. Martin 提出。

### S — 单一职责原则（Single Responsibility Principle）

> 一个类应该只有一个引起变化的原因。

```java
// ❌ 违反 SRP：一个类承担多个职责
class User {
    String name;
    String email;

    // 职责1：数据模型
    // 职责2：数据库持久化
    void saveToDatabase() { /* ... */ }
    // 职责3：邮件发送
    void sendWelcomeEmail() { /* ... */ }
    // 职责4：JSON 序列化
    String toJson() { /* ... */ }
}

// ✅ 遵守 SRP：职责分离
class User { String name; String email; }           // 数据模型
class UserRepository { void save(User u) { } }       // 持久化
class EmailService { void sendWelcome(User u) { } }  // 邮件
class UserJsonSerializer { String serialize(User u) { } } // 序列化
```

### O — 开闭原则（Open-Closed Principle）

> 软件实体应该对扩展开放，对修改关闭。

```java
// ❌ 违反 OCP：每次新增类型都要修改 switch
class DiscountCalculator {
    double calculate(Customer c) {
        switch (c.getType()) {
            case REGULAR:  return c.getAmount() * 0.95;
            case VIP:      return c.getAmount() * 0.80;
            case WHOLESALE: return c.getAmount() * 0.70;
            // 新增类型需要修改此类 ❌
        }
    }
}

// ✅ 遵守 OCP：通过多态扩展
interface DiscountStrategy {
    double apply(double amount);
}

class RegularDiscount implements DiscountStrategy {
    public double apply(double amount) { return amount * 0.95; }
}

class VIPDiscount implements DiscountStrategy {
    public double apply(double amount) { return amount * 0.80; }
}

class WholesaleDiscount implements DiscountStrategy {
    public double apply(double amount) { return amount * 0.70; }
}

// 新增折扣策略只需新增类，无需修改现有代码
class DiscountCalculator {
    double calculate(Customer c, DiscountStrategy strategy) {
        return strategy.apply(c.getAmount());
    }
}
```

### L — 里氏替换原则（Liskov Substitution Principle）

> 子类型必须能够替换其基类型而不破坏程序正确性。

```python
from abc import ABC, abstractmethod

# ❌ 违反 LSP：经典的长方形-正方形问题
class Rectangle:
    def __init__(self, width: int, height: int):
        self._width = width
        self._height = height

    def set_width(self, w: int): self._width = w
    def set_height(self, h: int): self._height = h
    def area(self) -> int: return self._width * self._height

class Square(Rectangle):  # 正方形继承长方形
    def set_width(self, w: int):
        self._width = w
        self._height = w  # 副作用！修改宽度也修改了高度

def resize(rect: Rectangle):
    rect.set_width(5)
    rect.set_height(10)
    assert rect.area() == 50  # Square 会失败！

# ✅ 正确做法：使用抽象基类
class Shape(ABC):
    @abstractmethod
    def area(self) -> int: ...

class Rectangle(Shape):
    def __init__(self, w, h): self._w, self._h = w, h
    def area(self) -> int: return self._w * self._h

class Square(Shape):
    def __init__(self, side): self._side = side
    def area(self) -> int: return self._side ** 2
```

### I — 接口隔离原则（Interface Segregation Principle）

> 客户端不应该被迫依赖它不使用的方法。

```java
// ❌ 违反 ISP：胖接口
interface Worker {
    void work();
    void eat();
    void sleep();
}
// Robot 实现这个接口被迫实现 eat() 和 sleep()

// ✅ 遵守 ISP：拆分为小接口
interface Workable  { void work(); }
interface Eatable   { void eat(); }
interface Sleepable { void sleep(); }

class HumanWorker implements Workable, Eatable, Sleepable {
    public void work() { System.out.println("Working..."); }
    public void eat() { System.out.println("Eating..."); }
    public void sleep() { System.out.println("Sleeping..."); }
}

class RobotWorker implements Workable {
    public void work() { System.out.println("Working 24/7..."); }
    // 不需要实现 eat() 和 sleep()
}
```

### D — 依赖倒置原则（Dependency Inversion Principle）

> 高层模块不应该依赖低层模块，二者都应依赖抽象。

```java
// ❌ 违反 DIP：高层直接依赖低层具体实现
class OrderController {
    private MySQLDatabase db = new MySQLDatabase(); // 硬编码依赖
}

// ✅ 遵守 DIP：依赖抽象
interface Database {
    void save(Order order);
    Order findById(String id);
}

class MySQLDatabase implements Database {
    public void save(Order order) { /* MySQL implementation */ }
    public Order findById(String id) { /* ... */ return null; }
}

class OrderController {
    private final Database db;

    // 依赖注入：通过构造器注入
    public OrderController(Database db) {
        this.db = db;
    }
}

// 使用
Database db = new MySQLDatabase();
OrderController controller = new OrderController(db);
```

## 二、设计模式分类

GoF（Gang of Four）23 种设计模式分为三大类：

### 创建型模式（Creational）— 5 种

| 模式 | 意图 | 适用场景 |
|------|------|---------|
| **单例** | 确保一个类只有一个实例 | 配置管理、日志、连接池 |
| **工厂方法** | 定义创建对象的接口，让子类决定实例化哪个类 | 框架扩展点 |
| **抽象工厂** | 创建一系列相关对象 | 跨平台 UI 组件 |
| **建造者** | 分步构建复杂对象 | 复杂配置对象、SQL 构建器 |
| **原型** | 通过克隆已有实例创建新对象 | 创建成本高的对象 |

```python
# 建造者模式示例
class Pizza:
    def __init__(self):
        self.size = None
        self.cheese = False
        self.pepperoni = False
        self.mushrooms = False

class PizzaBuilder:
    def __init__(self):
        self._pizza = Pizza()

    def size(self, size):
        self._pizza.size = size
        return self

    def add_cheese(self):
        self._pizza.cheese = True
        return self

    def add_pepperoni(self):
        self._pizza.pepperoni = True
        return self

    def add_mushrooms(self):
        self._pizza.mushrooms = True
        return self

    def build(self):
        return self._pizza

# 链式调用
pizza = (PizzaBuilder()
    .size("Large")
    .add_cheese()
    .add_pepperoni()
    .add_mushrooms()
    .build())
```

### 结构型模式（Structural）— 7 种

| 模式 | 意图 | 适用场景 |
|------|------|---------|
| **适配器** | 转换接口使不兼容的类协同工作 | 遗留系统适配 |
| **桥接** | 将抽象与实现分离 | 跨平台渲染引擎 |
| **组合** | 统一对待个体和组合对象 | 文件系统、UI 树 |
| **装饰器** | 动态添加职责 | I/O 流包装、日志增强 |
| **外观** | 为复杂子系统提供简化接口 | API 网关 |
| **享元** | 共享细粒度对象 | 游戏中的粒子系统 |
| **代理** | 控制对对象的访问 | 延迟加载、权限控制 |

```python
# 装饰器模式：动态增强 HTTP 客户端
from functools import wraps
from typing import Callable

class HttpClient:
    def get(self, url: str) -> str:
        return f"GET {url}"

def with_retry(max_attempts: int = 3):
    def decorator(func: Callable) -> Callable:
        @wraps(func)
        def wrapper(*args, **kwargs):
            for attempt in range(max_attempts):
                try:
                    return func(*args, **kwargs)
                except Exception as e:
                    if attempt == max_attempts - 1:
                        raise
                    print(f"Retry {attempt + 1}/{max_attempts}: {e}")
        return wrapper
    return decorator

def with_logging(func: Callable) -> Callable:
    @wraps(func)
    def wrapper(*args, **kwargs):
        print(f"Calling {func.__name__}({args}, {kwargs})")
        result = func(*args, **kwargs)
        print(f"Result: {result}")
        return result
    return wrapper

client = HttpClient()
client.get = with_logging(with_retry(3)(client.get))
# client.get("https://api.example.com/data")
```

### 行为型模式（Behavioral）— 11 种

| 模式 | 意图 | 适用场景 |
|------|------|---------|
| **责任链** | 将请求沿处理者链传递 | 中间件、审批流 |
| **命令** | 将请求封装为对象 | 撤销/重做、任务队列 |
| **解释器** | 定义语言的语法树表示 | 规则引擎、表达式求值 |
| **迭代器** | 顺序访问集合元素 | 自定义集合遍历 |
| **中介者** | 集中对象间交互 | 聊天室、事件总线 |
| **备忘录** | 保存和恢复对象状态 | 撤销功能、快照 |
| **观察者** | 一对多依赖通知 | 事件系统、MVVM |
| **状态** | 对象行为随状态变化 | 状态机、工作流 |
| **策略** | 封装可互换的算法 | 排序策略、支付方式 |
| **模板方法** | 定义算法骨架，子类实现步骤 | 框架钩子 |
| **访问者** | 在不修改类的前提下添加操作 | AST 遍历、报表生成 |

```java
// 策略模式示例
interface PaymentStrategy {
    void pay(double amount);
}

class CreditCardPayment implements PaymentStrategy {
    private String cardNumber;
    public CreditCardPayment(String cardNumber) {
        this.cardNumber = cardNumber;
    }
    public void pay(double amount) {
        System.out.printf("Paid $%.2f via Credit Card %s%n", amount, cardNumber);
    }
}

class AlipayPayment implements PaymentStrategy {
    private String accountId;
    public AlipayPayment(String accountId) {
        this.accountId = accountId;
    }
    public void pay(double amount) {
        System.out.printf("Paid $%.2f via Alipay (%s)%n", amount, accountId);
    }
}

class ShoppingCart {
    private PaymentStrategy paymentStrategy;

    public void setPaymentStrategy(PaymentStrategy strategy) {
        this.paymentStrategy = strategy;
    }

    public void checkout(double amount) {
        if (paymentStrategy == null) {
            throw new IllegalStateException("No payment method selected");
        }
        paymentStrategy.pay(amount);
    }
}

// 使用
ShoppingCart cart = new ShoppingCart();
cart.setPaymentStrategy(new CreditCardPayment("1234-5678-9012-3456"));
cart.checkout(99.99);
cart.setPaymentStrategy(new AlipayPayment("user@example.com"));
cart.checkout(49.99);
```

## 三、依赖注入（Dependency Injection, DI）

依赖注入是**控制反转（IoC）** 的一种实现形式，将对象的创建和依赖关系的维护交给容器。

### DI 三种方式

```java
// 1. 构造器注入（推荐）
class OrderService {
    private final OrderRepository repository;
    private final EmailService emailService;

    // 依赖通过构造器注入，可以保证不可变性和可测试性
    @Inject
    public OrderService(OrderRepository repository, EmailService emailService) {
        this.repository = repository;
        this.emailService = emailService;
    }
}

// 2. Setter 注入（可选依赖）
class ReportService {
    private Formatter formatter;

    @Inject
    public void setFormatter(Formatter formatter) {
        this.formatter = formatter;
    }
}

// 3. 字段注入（不推荐，但简洁）
class CacheService {
    @Inject
    private CacheClient client;
}
```

### Spring DI 示例

```java
// 定义 Bean
@Service
public class UserService {

    private final UserRepository userRepository;
    private final PasswordEncoder passwordEncoder;

    // Spring 自动注入依赖
    public UserService(UserRepository userRepository,
                       PasswordEncoder passwordEncoder) {
        this.userRepository = userRepository;
        this.passwordEncoder = passwordEncoder;
    }

    public User register(String username, String rawPassword) {
        String encoded = passwordEncoder.encode(rawPassword);
        User user = new User(username, encoded);
        return userRepository.save(user);
    }
}

// 配置类（Java 配置方式）
@Configuration
public class AppConfig {

    @Bean
    public PasswordEncoder passwordEncoder() {
        return new BCryptPasswordEncoder(12);
    }

    @Bean
    @Profile("dev")
    public UserRepository devUserRepository() {
        return new InMemoryUserRepository();
    }

    @Bean
    @Profile("prod")
    public UserRepository prodUserRepository(DataSource dataSource) {
        return new JdbcUserRepository(dataSource);
    }
}
```

### Python DI（使用 dependency-injector 库）

```python
from dependency_injector import containers, providers
from dataclasses import dataclass

@dataclass
class DatabaseConfig:
    host: str
    port: int

class Database:
    def __init__(self, config: DatabaseConfig):
        self.config = config

class UserService:
    def __init__(self, db: Database):
        self.db = db

# 容器定义
class Container(containers.DeclarativeContainer):
    config = providers.Configuration()

    db_config = providers.Singleton(
        DatabaseConfig,
        host=config.host,
        port=config.port,
    )

    database = providers.Singleton(Database, config=db_config)
    user_service = providers.Factory(UserService, db=database)

# 使用
container = Container()
container.config.from_dict({"host": "localhost", "port": 5432})
service = container.user_service()
```

## 四、领域驱动设计（Domain-Driven Design, DDD）

DDD 由 Eric Evans 提出，核心思想是将软件模型与业务领域紧密对齐。

### 战略设计

```
┌─────────────────────────────────────────────────┐
│                   子域分类                        │
├─────────────┬───────────────┬───────────────────┤
│  核心域      │  支撑域        │  通用域            │
│  (Core)     │  (Supporting) │  (Generic)        │
├─────────────┼───────────────┼───────────────────┤
│  业务核心    │  辅助核心域    │  通用功能          │
│  投入最多    │  适度投入      │  可购买/外包        │
│  最优团队    │  一般团队      │  现成方案          │
└─────────────┴───────────────┴───────────────────┘
```

### 限界上下文（Bounded Context）

```java
// 订单上下文中的 Product
package com.shop.ordering.domain;

public class Product {
    private ProductId id;
    private String name;
    private Money unitPrice;  // 订单视角：关注价格
}

// 库存上下文中的 Product
package com.shop.inventory.domain;

public class Product {
    private SKU sku;
    private int stockLevel;   // 库存视角：关注数量
    private WarehouseLocation location;
}

// 同一个概念在不同上下文中有不同的模型
```

### 战术设计构建块

```java
// 值对象：不可变、无唯一标识
public final class Money {
    private final BigDecimal amount;
    private final Currency currency;

    public Money(BigDecimal amount, Currency currency) {
        this.amount = amount.setScale(2, RoundingMode.HALF_UP);
        this.currency = currency;
    }

    public Money add(Money other) {
        requireSameCurrency(other);
        return new Money(this.amount.add(other.amount), this.currency);
    }

    public Money multiply(int quantity) {
        return new Money(this.amount.multiply(BigDecimal.valueOf(quantity)), this.currency);
    }
}

// 实体：有唯一标识、可变
public class Order {
    private final OrderId id;          // 唯一标识
    private CustomerId customerId;
    private List<OrderLine> lines;
    private OrderStatus status;

    // 领域行为：业务逻辑封装在实体内部
    public void addLine(ProductId productId, int quantity, Money unitPrice) {
        if (status != OrderStatus.DRAFT) {
            throw new IllegalStateException("Can only add lines to draft orders");
        }
        lines.add(new OrderLine(productId, quantity, unitPrice));
    }

    public Money calculateTotal() {
        return lines.stream()
            .map(line -> line.unitPrice().multiply(line.quantity()))
            .reduce(Money.ZERO, Money::add);
    }
}

// 聚合根：保证一致性边界
public class Order {  // Order 是聚合根
    // OrderLine 只能通过 Order 访问
    // 外部不能直接持有 OrderLine 的引用
}

// 领域服务：不属于任何实体的业务逻辑
public class PricingService {
    public Money calculateDiscountedPrice(Order order, DiscountPolicy policy) {
        Money total = order.calculateTotal();
        return policy.apply(total);
    }
}

// 领域事件：通知其他上下文
public class OrderPlacedEvent {
    private final OrderId orderId;
    private final CustomerId customerId;
    private final Money totalAmount;
    private final Instant placedAt;
}

// 仓储：持久化抽象
public interface OrderRepository {
    Order findById(OrderId id);
    void save(Order order);
}
```

### 分层架构

```
┌──────────────────────────────┐
│       用户界面 / API 层       │  ← Controller, DTO
├──────────────────────────────┤
│       应用服务层              │  ← Use Case, Transaction
├──────────────────────────────┤
│       领域层                  │  ← Entity, Value Object, Domain Service
├──────────────────────────────┤
│       基础设施层              │  ← Repository Impl, Messaging, External API
└──────────────────────────────┘
```

```java
// 应用服务层编排领域逻辑
@Service
@Transactional
public class PlaceOrderUseCase {

    private final OrderRepository orderRepo;
    private final CustomerRepository customerRepo;
    private final DomainEventPublisher eventPublisher;

    public OrderId execute(PlaceOrderCommand cmd) {
        Customer customer = customerRepo.findById(cmd.customerId())
            .orElseThrow(() -> new CustomerNotFoundException(cmd.customerId()));

        Order order = new Order(customer.getId());
        for (OrderLineDto line : cmd.lines()) {
            order.addLine(line.productId(), line.quantity(), line.unitPrice());
        }

        orderRepo.save(order);
        eventPublisher.publish(new OrderPlacedEvent(order.getId(), customer.getId(),
            order.calculateTotal(), Instant.now()));

        return order.getId();
    }
}
```

## 最佳实践

### ✅ 推荐做法

1. **SOLID 先行**：在重构前先理解原则，再应用模式
2. **模式不是银弹**：只在真正需要时引入设计模式，避免过度设计
3. **组合优于继承**：优先使用组合/委托而非深层继承链
4. **面向接口编程**：依赖抽象而非具体实现
5. **DDD 分阶段引入**：先从核心域的战术设计开始，再扩展到战略设计

### ❌ 避免

1. **上帝类**：一个类承担过多职责
2. **过度继承**：继承深度超过 3 层就考虑重构
3. **贫血模型**：实体只有 getter/setter，业务逻辑全在 Service 中
4. **模式滥用**：简单逻辑用不上工厂模式、策略模式
