# 并发编程模型

> 并发编程模型是处理多个任务同时执行的设计范式。从经典的线程与锁模型，到 Actor 模型、CSP 模型、Software Transactional Memory，每种模型都为构建高并发系统提供了不同的抽象。

---

## 并发 vs 并行

```
并发                    并行
┌─────────┐            ┌─────────┐ ┌─────────┐
│ Task A  │            │ Task A  │ │ Task B  │
│ Task B  │            │         │ │         │
│ Task C  │            │         │ │ Task C  │
└─────────┘            └─────────┘ └─────────┘
  时间片切换             真正同时执行
  单核也能实现           需要多核
```

## 一、线程与锁模型

最经典的并发模型，基于共享内存 + 互斥锁。

### 基本线程操作

```java
import java.util.concurrent.*;

public class ThreadModel {

    // 方式1：继承 Thread
    class MyThread extends Thread {
        @Override
        public void run() {
            System.out.println("Running in: " + Thread.currentThread().getName());
        }
    }

    // 方式2：实现 Runnable（推荐）
    Runnable task = () -> System.out.println("Task running");

    // 方式3：Callable + Future（有返回值）
    Callable<Integer> callable = () -> {
        Thread.sleep(1000);
        return 42;
    };

    public void basicUsage() throws Exception {
        // 启动线程
        new MyThread().start();
        new Thread(task).start();

        // 使用 Future
        ExecutorService executor = Executors.newFixedThreadPool(4);
        Future<Integer> future = executor.submit(callable);
        Integer result = future.get(5, TimeUnit.SECONDS); // 阻塞等待结果

        executor.shutdown();
    }
}
```

### synchronized 与 Lock

```java
// 1. synchronized（内置锁）
class Counter {
    private int count = 0;

    public synchronized void increment() {
        count++; // 原子操作
    }
}

// 2. ReentrantLock（更灵活）
import java.util.concurrent.locks.*;

class AdvancedCounter {
    private int count = 0;
    private final ReentrantLock lock = new ReentrantLock();
    private final Condition notZero = lock.newCondition();

    public void increment() {
        lock.lock();
        try {
            count++;
            notZero.signalAll(); // 通知等待的线程
        } finally {
            lock.unlock(); // 必须在 finally 中释放
        }
    }

    public void waitForNonZero() throws InterruptedException {
        lock.lock();
        try {
            while (count == 0) {
                notZero.await(); // 释放锁并等待
            }
        } finally {
            lock.unlock();
        }
    }
}

// 3. ReadWriteLock（读写分离）
class ReadWriteCache<K, V> {
    private final Map<K, V> cache = new HashMap<>();
    private final ReadWriteLock rwLock = new ReentrantReadWriteLock();

    public V get(K key) {
        rwLock.readLock().lock();
        try {
            return cache.get(key);
        } finally {
            rwLock.readLock().unlock();
        }
    }

    public void put(K key, V value) {
        rwLock.writeLock().lock();
        try {
            cache.put(key, value);
        } finally {
            rwLock.writeLock().unlock();
        }
    }
}
```

### 并发工具类

```java
// 1. CompletableFuture（Java 8 异步编程）
CompletableFuture<String> future = CompletableFuture
    .supplyAsync(() -> fetchData("url1"), executor)  // 异步获取数据
    .thenApply(this::parseJson)                      // 转换
    .thenCompose(data ->                             // 链式异步

    )
    .exceptionally(ex -> {                           // 错误处理

        return defaultValue;
    })
    .orTimeout(5, TimeUnit.SECONDS);                 // 超时

// 并行组合：所有完成
CompletableFuture<Void> all = CompletableFuture.allOf(future1, future2, future3);

// 并行组合：任一完成
CompletableFuture<Object> any = CompletableFuture.anyOf(future1, future2);

// 2. CountDownLatch（等待 N 个线程完成）
CountDownLatch latch = new CountDownLatch(3);
for (int i = 0; i < 3; i++) {
    new Thread(() -> {
        try { doWork(); }
        finally { latch.countDown(); }
    }).start();
}
latch.await(); // 等待 3 个线程都完成

// 3. CyclicBarrier（线程间同步屏障）

    System.out.println("All threads reached barrier!");

// 4. Semaphore（资源许可）
```

### 无锁并发（CAS / Atomic）

```java
import java.util.concurrent.atomic.*;

// 原子变量
AtomicInteger counter = new AtomicInteger(0);
counter.incrementAndGet();     // ++counter
counter.compareAndSet(0, 1);   // CAS：如果当前值是0则设为1

// LongAdder（高竞争场景优于 AtomicLong）
LongAdder adder = new LongAdder();
adder.increment();

// CAS 原理
public boolean compareAndSwap(int expected, int newValue) {
    // 原子操作：如果当前值 == expected，则设为 newValue 并返回 true
    // 否则返回 false（说明被其他线程修改了）
    return UNSAFE.compareAndSwapInt(this, valueOffset, expected, newValue);
}

// 无锁队列（Michael-Scott 算法）
ConcurrentLinkedQueue<String> queue = new ConcurrentLinkedQueue<>();
queue.offer("item");  // 无锁入队
queue.poll();          // 无锁出队
```

### 常见并发陷阱

```
1. 死锁：线程互相等待对方持有的锁
   Thread1: lock(A) → lock(B)
   Thread2: lock(B) → lock(A)  ← 互相等待

2. 活锁：线程不断重试导致无法推进
3. 饥饿：低优先级线程长期得不到执行
4. 竞态条件：共享状态的非原子访问
5. 可见性问题：CPU 缓存导致的内存不可见
```

## 二、Actor 模型

Actor 模型将并发抽象为独立的 Actor，每个 Actor 有自己的私有状态，通过**异步消息传递**通信，不共享内存。

```
┌──────────┐    msg     ┌──────────┐    msg     ┌──────────┐
│  Actor A │ ────────►  │  Actor B │ ────────►  │  Actor C │
│          │            │          │            │          │
│ state    │            │ state    │            │ state    │
│ mailbox  │◄────────   │ mailbox  │◄────────   │ mailbox  │
└──────────┘    msg     └──────────┘    msg     └──────────┘

特点：
- 每个 Actor 单线程处理消息（无锁）
- 消息按顺序到达
- 无共享状态，天然避免竞态条件
- 可分布在不同的机器上
```

### Akka（JVM）

```scala
import akka.actor._
import akka.pattern.ask
import akka.util.Timeout
import scala.concurrent.duration._
import scala.concurrent.Await

// 定义 Actor
class Counter extends Actor {
  private var count = 0  // 私有状态，外部无法直接访问

  override def receive: Receive = {
    case "increment" =>
      count += 1

    case "get" =>
      sender() ! count  // 异步回复消息

    case "reset" =>
      count = 0
  }
}

// 使用 Actor
object ActorDemo extends App {
  val system = ActorSystem("MySystem")

  val counter = system.actorOf(Props[Counter](), "counter")

  counter ! "increment"
  counter ! "increment"
  counter ! "increment"

  implicit val timeout: Timeout = 5.seconds
  val future = counter ? "get"  // ask 模式，返回 Future
  val result = Await.result(future, 5.seconds)
  println(s"Count: $result")  // Count: 3

  system.terminate()
}
```

```scala
// Actor 层次结构（监督策略）
class Supervisor extends Actor {
  override val supervisorStrategy = OneForOneStrategy(
    maxNrOfRetries = 3,

  ) {
```

### Python（使用 pyactor 或 async actors）

```python
import asyncio
from typing import Any

class Actor:
    """简单的 asyncio Actor 实现"""
    def __init__(self):
        self._inbox: asyncio.Queue = asyncio.Queue()
        self._running = True

    async def send(self, message: Any):
        await self._inbox.put(message)

    async def run(self):
        while self._running:
            message = await self._inbox.get()
            if message == "__stop__":
                self._running = False
                break
            await self.handle(message)

    async def handle(self, message: Any):
        raise NotImplementedError

class CounterActor(Actor):
    def __init__(self):
        super().__init__()
        self._count = 0
        self._response_queue: asyncio.Queue = asyncio.Queue()

    async def handle(self, message: Any):
        if message == "increment":
            self._count += 1
        elif message == "get":
            await self._response_queue.put(self._count)

# 使用
async def main():
    counter = CounterActor()
    task = asyncio.create_task(counter.run())

    await counter.send("increment")
    await counter.send("increment")
    await counter.send("get")

    result = await counter._response_queue.get()
    print(f"Count: {result}")  # Count: 2

    await counter.send("__stop__")
    await task

asyncio.run(main())
```

### Erlang/Elixir（原生 Actor 模型）

```erlang
%% Erlang: 天生基于 Actor 模型
-module(counter).
-export([start/0, loop/1, increment/1, get_count/1]).

start() ->
    spawn(?MODULE, loop, [0]).  % 创建 Actor 进程

loop(Count) ->
    receive
        increment ->
            loop(Count + 1);
        {get, From} ->
            From ! {count, Count},  % 异步回复
            loop(Count);
        stop ->
            ok
    end.

increment(Pid) -> Pid ! increment.
get_count(Pid) ->
    Pid ! {get, self()},
    receive
        {count, Count} -> Count
    after 5000 -> {error, timeout}
    end.
```

## 三、CSP 模型（Communicating Sequential Processes）

CSP 由 Tony Hoare 提出，核心是**通过 Channel 通信**而非共享内存。Go 语言是 CSP 的代表实现。

```
┌──────────┐                          ┌──────────┐
│ Goroutine│ ──► Channel ──►          │ Goroutine│
│    A     │                          │    B     │
└──────────┘                          └──────────┘
                不通过共享内存通信
                而是通过共享通道通信
```

### Go（原生 CSP）

```go
package main

import (
    "fmt"
    "time"
)

func main() {
    // 无缓冲 Channel（同步）
    ch := make(chan string)

    // Goroutine 发送
    go func() {
        ch <- "Hello from goroutine!"  // 阻塞直到有人接收
    }()

    msg := <-ch  // 阻塞直到有数据
    fmt.Println(msg)

    // 有缓冲 Channel（异步）
    buffered := make(chan int, 3)
    buffered <- 1  // 不阻塞（缓冲区未满）
    buffered <- 2
    buffered <- 3
    // buffered <- 4  // 阻塞（缓冲区已满）

    fmt.Println(<-buffered) // 1

    // select 多路复用
    ch1 := make(chan string)
    ch2 := make(chan string)

    go func() {
        time.Sleep(1 * time.Second)
        ch1 <- "from ch1"
    }()
    go func() {
        time.Sleep(2 * time.Second)
        ch2 <- "from ch2"
    }()

    // 等待第一个就绪的 Channel
    select {
    case msg1 := <-ch1:
        fmt.Println(msg1)
    case msg2 := <-ch2:
        fmt.Println(msg2)
    case <-time.After(3 * time.Second):
        fmt.Println("Timeout!")
    }

    // Worker Pool 模式
    jobs := make(chan int, 100)
    results := make(chan int, 100)

    // 启动 3 个 worker
    for w := 0; w < 3; w++ {
        go worker(w, jobs, results)
    }

    // 发送 5 个任务
    for j := 1; j <= 5; j++ {
        jobs <- j
    }
    close(jobs)

    // 收集结果
    for r := 0; r < 5; r++ {
        fmt.Println("Result:", <-results)
    }
}

func worker(id int, jobs <-chan int, results chan<- int) {
    for j := range jobs {
        fmt.Printf("Worker %d processing job %d\n", id, j)
        time.Sleep(time.Second)
        results <- j * j
    }
}
```

### Clojure core.async

```clojure
(require '[clojure.core.async :as a :refer [chan go <! >! <! go-loop]])

;; 创建 Channel
(def ch (chan 10))  ; 缓冲大小 10

;; go block（类似 goroutine）
(go
  (<! (a/timeout 1000))  ; 等待 1 秒
  (>! ch "Hello"))        ; 发送到 channel

;; 接收
(go
  (let [msg (<! ch)]
    (println "Received:" msg)))

;; 多路复用 (alts!)
(go
  (let [[v port] (a/alts! [ch (a/timeout 5000)])]
    (if (= port ch)
      (println "Got:" v)
      (println "Timeout!"))))

;; Pipeline 模式
(def input (chan))
(def output (chan))

(go-loop []
  (when-let [x (<! input)]
    (>! output (* x x))
    (recur)))

(go-loop [i 0]
  (when (< i 10)
    (>! input i)
    (recur)))

(go-loop []
  (when-let [result (<! output)]
    (println "Result:" result)
    (recur)))
```

## 四、Software Transactional Memory（STM）

STM 将数据库事务的概念引入内存管理，开发者通过事务操作共享内存，由运行时自动处理并发冲突。

```
┌────────────────────────────────────────────────┐
│              STM 工作流程                       │
│                                                │
│  1. 开始事务（读取内存快照）                      │
│  2. 执行操作（在事务内读/写）                     │
│  3. 验证：检查是否有冲突                         │
│     ├─ 无冲突 → 提交（原子写入）                  │
│     └─ 有冲突 → 回滚并重试                       │
│                                                │
│  优势：无需手动加锁，自动冲突检测                   │
│  劣势：重试成本，不适合 I/O 操作                   │
└────────────────────────────────────────────────┘
```

### Clojure STM

```clojure
;; Clojure 内置 STM 支持

;; Ref：可变引用，只能在事务中修改
(def balance1 (ref 1000))
(def balance2 (ref 500))

;; 事务转账
(defn transfer [from to amount]
  (dosync                        ; 开始 STM 事务
    (when (< @from amount)
      (throw (ex-info "Insufficient funds" {})))
    (alter from - amount)        ; 在事务中修改（自动重试冲突）
    (alter to + amount)))

;; 执行转账
(transfer balance1 balance2 200)
(println @balance1)  ; 800
(println @balance2)  ; 700

;; Atom（轻量级原子操作，无事务组合）
(def counter (atom 0))
(swap! counter inc)  ; 原子递增

;; commute（优化：最后提交时重新计算，减少冲突）
(defn add-credit [account amount]
  (dosync
    (commute account + amount)))  ; commute 代替 alter

;; Agent（异步更新）
(def logger (agent []))
(send logger conj "Log message")  ; 异步追加日志
```

### Scala STM（ScalaSTM 库）

```scala
import scala.concurrent.stm._

class Account(initialBalance: Int) {
  val balance = Ref(initialBalance)

  def deposit(amount: Int): Unit = atomic { implicit txn =>
    balance.set(balance.get + amount)
  }

  def withdraw(amount: Int): Unit = atomic { implicit txn =>
    if (balance.get < amount)
      throw new IllegalStateException("Insufficient funds")
    balance.set(balance.get - amount)
  }

  def getBalance: Int = single(balance.get)
}

object TransferService {
  def transfer(from: Account, to: Account, amount: Int): Unit = {
    atomic { implicit txn =>
      from.withdraw(amount)
      to.deposit(amount)
    }  // 事务自动提交或重试
  }
}
```

### Haskell STM

```haskell
import Control.Concurrent.STM
import Control.Concurrent

-- TVar：事务变量
type Account = TVar Int

main :: IO ()
main = do
    account1 <- newTVarIO 1000
    account2 <- newTVarIO 500

    -- atomically 执行 STM 事务
    atomically $ do
        transfer account1 account2 200

    bal1 <- readTVarIO account1
    bal2 <- readTVarIO account2
    putStrLn $ "Account1: " ++ show bal1  -- 800
    putStrLn $ "Account2: " ++ show bal2  -- 700

transfer :: Account -> Account -> Int -> STM ()
transfer from to amount = do
    balance <- readTVar from
    if balance < amount
        then retry  -- 自动重试（阻塞直到 from 变化）
        else do
            writeTVar from (balance - amount)
            modifyTVar' to (+ amount)

-- STM 的组合性：可以组合多个事务
-- orElse：尝试第一个事务，如果 retry 则尝试第二个
transferOrElse :: Account -> Account -> Account -> Int -> STM ()
transferOrElse primary backup to amount =
    transfer primary to amount `orElse` transfer backup to amount
```

## 模型对比

| 维度 | 线程与锁 | Actor | CSP | STM |
|------|---------|-------|-----|-----|
| **通信方式** | 共享内存+锁 | 异步消息 | Channel | 事务内存 |
| **状态管理** | 共享可变 | 私有可变 | 私有可变 | 事务隔离 |
| **死锁风险** | 高 | 低（无锁） | 低（无锁） | 中（重试成本） |
| **组合性** | 差 | 中 | 好 | 极好 |
| **分布式** | 不支持 | 天然支持 | 需扩展 | 不支持 |
| **代表语言** | Java/C++ | Erlang/Akka | Go/Clojure | Clojure/Haskell |
| **调试难度** | 高 | 中 | 中 | 中 |
| **学习曲线** | 陡（易出错） | 中 | 平缓 | 中 |
| **适用场景** | 通用 | 分布式系统 | 并发管道 | 复杂共享状态 |

## 最佳实践

### ✅ 通用推荐

1. **选择适合场景的模型**：不要用一把锤子解决所有问题
2. **避免共享可变状态**：优先使用不可变数据或消息传递
3. **减小临界区**：锁的持有时间越短越好
4. **使用高层抽象**：优先用 `CompletableFuture`/Channel/Actor 而非裸线程
5. **设置合理的超时**：防止永久阻塞

### 线程与锁

1. **始终在 finally 中释放锁**
2. **按固定顺序获取多个锁**（避免死锁）
3. **使用 `tryLock` 带超时**（避免死锁）
4. **优先使用 `java.util.concurrent` 工具类**

### Actor

1. **消息不可变**：Actor 间传递的消息应该是不可变的
2. **一个 Actor 只处理一种职责**
3. **使用监督策略处理子 Actor 故障**
4. **不要在 Actor 外部直接访问其状态**

### CSP

1. **明确 Channel 的所有权**
2. **使用有缓冲 Channel 解耦生产者和消费者**
3. **及时关闭 Channel**（由发送方关闭）
4. **使用 select 处理多路 Channel**
