# 技术债务管理

> 系统化地识别、评估、偿还和预防技术债务，保持代码库健康和团队生产力。

## 1. 概述

### 1.1 什么是技术债务

技术债务是指在软件开发过程中，为了短期利益（如快速交付）而做出的技术妥协，这些妥协会在未来产生额外的维护成本。

```
技术债务的隐喻：

借钱（快速交付）
    ↓
利息（持续的额外维护成本）
    ↓
本金（需要偿还的重构工作）
    ↓
债务危机（系统无法维护，必须重写）
```

### 1.2 技术债务的分类

```
                    技术债务分类
                        │
        ┌───────────────┼───────────────┐
        │               │               │
    ┌───▼───┐      ┌────▼────┐    ┌────▼────┐
    │ 故意的 │      │ 无意的   │    │ 环境的   │
    └───┬───┘      └────┬────┘    └────┬────┘
        │               │               │
   ┌────┴────┐    ┌─────┴─────┐   ┌─────┴─────┐
   │ 谨慎的   │    │ 谨慎的     │   │ 依赖过时   │
   │ (有计划) │    │ (不知道)   │   │ 平台变化   │
   └─────────┘    └───────────┘   │ 团队变化   │
   ┌─────────┐    ┌───────────┐   └───────────┘
   │ 鲁莽的   │    │ 鲁莽的     │
   │ (没时间) │    │ (不知道)   │
   └─────────┘    └───────────┘
```

| 类型 | 说明 | 示例 |
|------|------|------|
| **设计债务** | 架构设计不合理 | 模块耦合过紧、缺少抽象层 |
| **代码债务** | 代码质量问题 | 重复代码、过长函数、命名不清 |
| **测试债务** | 测试覆盖不足 | 缺少单元测试、集成测试不完整 |
| **文档债务** | 文档缺失或过时 | 缺少API文档、架构文档过时 |
| **依赖债务** | 第三方依赖风险 | 使用有漏洞的版本、依赖停止维护 |
| **基础设施债务** | 基础设施落后 | CI/CD不完善、监控缺失 |
| **知识债务** | 团队知识瓶颈 | 关键知识集中在少数人 |

## 2. 债务识别

### 2.1 识别方法

| 方法 | 适用场景 | 成本 | 效果 |
|------|----------|------|------|
| **代码分析工具** | 自动化扫描 | 低 | 发现代码级债务 |
| **架构评审** | 定期检查 | 中 | 发现设计级债务 |
| **开发者反馈** | 日常收集 | 低 | 发现实践级债务 |
| **故障复盘** | 事后分析 | 中 | 发现系统级债务 |
| **新人入职反馈** | 定期收集 | 低 | 发现文档/知识债务 |
| **性能分析** | 定期执行 | 中 | 发现性能级债务 |

## 3. 偿还策略

### 3.1 偿还模式

| 模式 | 说明 | 适用场景 | 风险 |
|------|------|----------|------|
| **持续偿还** | 每个迭代分配固定比例 | 债务均匀分布 | 可能被业务需求挤压 |
| **集中偿还** | 定期安排专项治理 | 债务集中区域 | 影响业务交付节奏 |
| **机会偿还** | 遇到时顺手修复 | 局部小问题 | 可能忽略大债务 |
| **重写偿还** | 整体重写受影响模块 | 债务严重到不可维护 | 成本高、风险大 |

### 3.2 20%规则

```markdown
## 技术债务偿还预算分配

### 原则
每个迭代分配 20% 的开发容量用于技术债务偿还和代码质量改进。

### 容量分配
假设团队 10 人，2 周迭代，总容量 100 人天

| 类别 | 比例 | 人天 | 说明 |
|------|------|------|------|
| 新功能开发 | 60% | 60 | 业务需求 |
| 缺陷修复 | 10% | 10 | Bug修复 |
| 技术债务偿还 | 20% | 20 | 质量改进 |
| 学习与创新 | 10% | 10 | 培训、技术调研 |

### 技术债务优先级排序
1. 影响当前迭代交付的债务
2. 影响系统稳定性的债务
3. 影响开发效率的债务
4. 长期架构改进
```

## 5. 质量度量

### 5.1 代码质量指标

| 指标 | 说明 | 目标值 | 测量工具 |
|------|------|--------|----------|
| **圈复杂度** | 函数/方法的复杂度 | ≤ 10 | SonarQube |
| **代码重复率** | 重复代码的占比 | ≤ 3% | SonarQube |
| **测试覆盖率** | 代码被测试覆盖的比例 | ≥ 80% | JaCoCo/coverage.py |
| **技术债务密度** | 每千行代码的技术债务时间 | ≤ 2h/KLOC | SonarQube |
| **代码行数** | 文件/函数的代码行数 | 文件≤500行，函数≤50行 | linter |
| **依赖漏洞数** | 第三方依赖的安全漏洞 | 0 高危 | Snyk/Dependabot |

### 5.3 质量门禁

```markdown
## 质量门禁规则

### 代码提交门禁
- 新增代码测试覆盖率 ≥ 80%
- 代码重复率 ≤ 5%
- 圈复杂度 ≤ 15
- 无新增高危安全漏洞

### 合并门禁
- 所有CI检查通过
- 至少1人代码评审通过
- 无新增代码异味（blocker/critical级别）

### 发布门禁
- 整体测试覆盖率 ≥ 75%
- 无未解决的高危漏洞
- 性能基准测试通过
```

## 6. 预防机制

### 6.1 债务预防策略

| 策略 | 说明 | 实施方式 |
|------|------|----------|
| **代码评审** | 合并前检查代码质量 | PR Review机制 |
| **编码规范** | 统一代码风格和实践 | Linter + 规范文档 |
| **架构守护** | 自动化检查架构约束 | ArchUnit/Deptrac |
| **持续重构** | 日常小规模改进 | 重构习惯 + 20%规则 |
| **知识共享** | 减少知识债务 | Code Review + 文档 + 分享 |

### 6.2 架构守护示例

```python
# ArchUnit 风格的架构约束检查（伪代码示例）

# 依赖方向检查：Controller → Service → Repository
check("Service层不应依赖Controller层"):
    no_class("service.*").should().depend_on("controller.*")

# 分层架构检查
check("Repository层不应依赖Service层"):
    no_class("repository.*").should().depend_on("service.*")

# 循环依赖检查
check("不应存在循环依赖"):
    no_classes().should().have_circular_dependency()
```

## 7. 最佳实践

### 7.1 童子军规则

> "让营地比你来时更干净。"

每次修改代码时，顺手改进周围的代码质量：
- 重命名不清晰的变量
- 提取重复代码
- 添加缺失的注释
- 补充缺失的测试

### 7.2 债务管理成熟度

```
Level 1: 无意识
  - 不知道存在技术债务
  - 代码质量问题不断累积
    ↓
Level 2: 被动应对
  - 问题出现时才处理
  - 没有系统化管理
    ↓
Level 3: 主动管理
  - 定期识别和评估
  - 有偿还预算和计划
    ↓
Level 4: 预防为主
  - 质量门禁完善
  - 持续重构文化
    ↓
Level 5: 质量内建
  - 质量是开发流程的核心
  - 技术债务持续维持在低位
```
