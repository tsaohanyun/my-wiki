# 技术团队管理

> 从团队建设、绩效管理、人才培养到技术文化建设的全方位技术团队管理指南。

## 1. 概述

优秀的技术团队不是简单地把优秀的工程师聚在一起，而是需要系统化的管理方法来激发集体智慧。本文档提供技术团队管理的核心框架和实践工具。

## 2. 团队建设

### 2.1 团队结构模型

| 模型 | 适用场景 | 优势 | 劣势 |
|------|----------|------|------|
| **职能型** | 技术深度优先 | 专业能力提升快 | 跨团队协作成本高 |
| **项目型** | 交付效率优先 | 响应速度快 | 技术沉淀不足 |
| **矩阵型** | 平衡型需求 | 灵活调配资源 | 双线汇报复杂 |
| **Spotify模型** | 大规模敏捷 | 自治与对齐兼顾 | 需要成熟的文化基础 |

### 2.2 团队规模指南

| 团队规模 | 管理策略 | 沟通方式 | 推荐架构 |
|----------|----------|----------|----------|
| 3-5人 | 扁平管理，技术Lead兼任 | 每日站会 | 单一功能团队 |
| 6-10人 | 设立Tech Lead + Scrum Master | 站会 + 周会 | Feature Team |
| 11-20人 | 分设小组，小组长制 | 分层会议 | Squad/Tribe |
| 20+人 | 组织架构化管理 | 体系化沟通 | 部门制 + 虚拟团队 |

### 2.3 角色与职责矩阵（RACI）

| 活动 | 技术总监 | 技术经理 | Tech Lead | 高级工程师 | 初级工程师 |
|------|----------|----------|-----------|------------|------------|
| 技术战略 | A | R | C | I | I |
| 架构设计 | C | A | R | C | I |
| 代码评审 | I | I | A | R | C |
| 技术选型 | A | R | R | C | I |
| 团队建设 | A | R | C | C | I |
| 代码开发 | I | I | C | R | R |

> R=Responsible（执行）, A=Accountable（负责）, C=Consulted（咨询）, I=Informed（知会）

### 3.2 绩效评估模型

#### 双维度评估矩阵

```
高 ┌─────────────┬─────────────┐
   │   培养区     │    明星区    │
   │             │             │
绩 │  高潜力      │  高绩效      │
效 │  低当前产出   │  高当前产出   │
   │             │             │
   ├─────────────┼─────────────┤
   │   观察区     │    稳定区    │
   │             │             │
   │  低潜力      │  中等潜力    │
   │  低当前产出   │  中等产出    │
   │             │             │
低 └─────────────┴─────────────┘
         低           高
              潜力
```

| 区域 | 策略 | 行动 |
|------|------|------|
| 明星区 | 保留激励 | 晋升、挑战性项目、导师角色 |
| 稳定区 | 发展提升 | 技能培训、轮岗、渐进式挑战 |
| 培养区 | 重点辅导 | 密切指导、定期反馈、明确成长路径 |
| 观察区 | 调整优化 | 明确期望、改进计划或岗位调整 |

## 4. 人才培养

### 4.1 技术能力模型

```
                    ┌───────────┐
                    │  技术专家  │  ← 深度 + 影响力
                    │  (P8+)    │
                ┌───┴───────────┴───┐
                │    高级工程师      │  ← 独立负责 + 技术深度
                │    (P6-P7)        │
            ┌───┴───────────────────┴───┐
            │      中级工程师            │  ← 独立完成 + 一定深度
            │      (P4-P5)              │
```

### 4.2 能力评估维度

| 维度 | P3（初级） | P5（中级） | P7（高级） | P8+（专家） |
|------|------------|------------|------------|-------------|
| **编码能力** | 完成明确任务 | 独立设计方案 | 代码质量把控 | 制定规范标准 |
| **系统设计** | 理解现有架构 | 模块级设计 | 系统级设计 | 架构演进规划 |
| **问题解决** | 解决已知问题 | 分析复杂问题 | 预防系统性问题 | 解决跨域难题 |
| **技术影响力** | 团队内分享 | 跨团队影响 | 部门级影响 | 公司/行业影响 |
| **业务理解** | 理解需求 | 理解业务背景 | 业务技术融合 | 技术驱动业务 |

### 4.4 导师制度

```markdown
## 导师制度规范

### 导师职责
1. 制定并跟踪学员的个人发展计划
2. 每两周至少一次一对一辅导
3. 代码评审与技术指导
4. 协助解决工作中的技术难题
5. 定期向管理层反馈学员进展

### 学员职责
1. 主动与导师沟通学习进展
2. 按计划完成学习任务
3. 积极实践导师建议
4. 定期复盘总结

### 匹配原则
- 技术方向相近
- 职级差距 ≥ 2级
- 性格互补
- 双方自愿
```

### 5.3 Code Review 文化

#### Code Review 规范

```markdown
## Code Review 规范

### 评审原则
1. **对事不对人**：评论针对代码而非作者
2. **建设性反馈**：指出问题的同时提供改进建议
3. **及时响应**：PR提交后 4 小时内开始评审
4. **小步提交**：单个 PR 变更控制在 400 行以内

### 评审检查清单
- [ ] 代码逻辑正确性
- [ ] 命名规范与可读性
- [ ] 边界条件处理
- [ ] 错误处理
- [ ] 性能影响
- [ ] 安全性考虑
- [ ] 测试覆盖
- [ ] 文档更新

### 评审等级
- **Approve**：代码可以合并
- **Comment**：有建议但不阻塞
- **Request Changes**：必须修改后才能合并
```

## 6. 最佳实践

### 6.1 管理者行为准则

1. **技术管理者首先要懂技术**：保持技术敏锐度，定期参与代码评审
2. **保护团队免受干扰**：过滤不合理需求，为团队创造专注环境
3. **以身作则**：遵守团队规范，展示期望的行为
4. **透明沟通**：信息充分共享，减少信息不对称
5. **授权而非微观管理**：给团队决策空间，关注结果而非过程

### 6.2 常见管理陷阱

| 陷阱 | 表现 | 应对 |
|------|------|------|
| **技术权威主义** | 所有技术决策都必须自己拍板 | 建立决策授权机制 |
| **英雄主义** | 关键问题总是自己解决 | 培养团队解决问题的能力 |
| **忙碌陷阱** | 用忙碌感替代成果导向 | 关注产出而非工时 |
| **避免冲突** | 回避困难对话 | 建立健康的反馈文化 |
| **过度承诺** | 接受所有需求 | 学会说不，管理期望 |
