# 技术架构评审

> 系统化的架构评审流程，确保技术决策的质量、一致性和可追溯性。

## 1. 概述

架构评审是技术治理的核心机制，其目标是：

1. **质量保障**：确保架构方案满足功能和非功能需求
2. **风险控制**：提前识别技术风险并制定应对策略
3. **知识沉淀**：记录决策过程，形成组织知识资产
4. **一致性维护**：确保系统架构的整体协调性

### 评审在研发流程中的位置

```
需求分析 → 方案设计 → 【架构评审】 → 开发实施 → 测试验证 → 上线发布
                          ↑
                    评审不通过则返回修改
```

## 2. 评审流程

### 2.1 评审流程总览

```
┌──────────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐
│  提交申请  │───→│  预审准备  │───→│  正式评审  │───→│  跟踪闭环  │
│          │    │          │    │          │    │          │
│ • 架构文档 │    │ • 材料分发 │    │ • 方案介绍 │    │ • 问题跟踪 │
│ • 设计方案 │    │ • 预审反馈 │    │ • 质疑讨论 │    │ • 修改确认 │
│ • 评估报告 │    │ • 补充材料 │    │ • 决议形成 │    │ • 文档归档 │
└──────────┘    └──────────┘    └──────────┘    └──────────┘
```

### 2.2 评审分级

| 级别 | 适用场景 | 评审组 | 时长 | 审批人 |
|------|----------|--------|------|--------|
| **L1 轻量级** | 模块内部设计变更 | Tech Lead + 2名工程师 | 30min | Tech Lead |
| **L2 标准级** | 跨模块架构调整 | 架构师 + 相关Tech Lead | 1h | 技术经理 |
| **L3 重大级** | 系统级架构变更 | CTO + 架构委员会 | 2h | CTO |

### 2.3 触发条件

以下场景**必须**进行架构评审：

- [ ] 新系统/新服务的架构设计
- [ ] 核心模块的重构方案
- [ ] 技术栈重大变更（如数据库迁移、框架升级）
- [ ] 引入新的中间件或基础设施组件
- [ ] 性能瓶颈的架构级优化方案
- [ ] 涉及数据模型重大变更
- [ ] 安全架构变更

## 3. 评审清单

### 3.1 架构设计文档清单

```markdown
## 架构设计文档要求

### 必须包含的内容

#### 1. 背景与目标
- [ ] 业务背景描述
- [ ] 核心问题定义
- [ ] 设计目标与约束
- [ ] 非功能需求（性能、可用性、扩展性）

#### 2. 方案设计
- [ ] 整体架构图
- [ ] 核心模块划分与职责
- [ ] 关键流程（核心业务流、数据流）
- [ ] 数据模型设计
- [ ] API设计（接口定义、协议选型）
- [ ] 依赖关系说明

#### 3. 技术选型
- [ ] 候选方案对比
- [ ] 选型理由与依据
- [ ] 技术风险评估

#### 4. 非功能设计
- [ ] 性能设计（预估QPS、延迟目标、容量规划）
- [ ] 高可用设计（故障模式、降级方案、容灾策略）
- [ ] 扩展性设计（水平/垂直扩展方案）
- [ ] 安全设计（认证授权、数据加密、审计日志）
- [ ] 可观测性设计（日志、监控、告警、链路追踪）

#### 5. 部署与运维
- [ ] 部署架构图
- [ ] 发布策略（灰度、蓝绿、金丝雀）
- [ ] 回滚方案
- [ ] 监控告警方案

#### 6. 风险与计划
- [ ] 技术风险清单与应对
- [ ] 开发计划与里程碑
- [ ] 依赖项与前置条件
```

### 3.2 评审检查清单

```markdown
## 架构评审检查清单

### 设计合理性
- [ ] 架构是否满足业务需求？
- [ ] 方案复杂度是否适度？（避免过度设计和欠设计）
- [ ] 是否遵循 SOLID 原则和相关设计原则？
- [ ] 模块划分是否职责清晰？
- [ ] 接口设计是否简洁、一致？

### 可靠性
- [ ] 是否分析了故障模式？
- [ ] 是否有降级和熔断机制？
- [ ] 数据一致性如何保证？
- [ ] 是否有数据备份和恢复方案？
- [ ] SLA目标是否明确且可达成？

### 可扩展性
- [ ] 是否支持水平扩展？
- [ ] 是否有热点问题？如何处理？
- [ ] 数据量增长的应对方案？
- [ ] 是否考虑了多活/异地部署？

### 性能
- [ ] 是否有性能预估和基准测试？
- [ ] 关键路径的延迟是否可接受？
- [ ] 是否有性能瓶颈？如何解决？
- [ ] 缓存策略是否合理？

### 安全性
- [ ] 认证和授权方案是否完善？
- [ ] 敏感数据是否加密存储和传输？
- [ ] 是否有防注入、防XSS等安全措施？
- [ ] 是否满足合规要求？

### 可运维性
- [ ] 部署方案是否自动化？
- [ ] 监控和告警是否完善？
- [ ] 日志是否充分且可检索？
- [ ] 是否便于问题排查？

### 成本
- [ ] 资源成本估算是否合理？
- [ ] 开发和维护成本评估？
- [ ] 是否有成本优化空间？
```

## 4. 架构决策记录（ADR）

### 4.1 ADR概述

ADR（Architecture Decision Record）是记录架构决策的轻量级文档格式。每个ADR记录一个架构决策的背景、选项和结论。

### 4.2 ADR模板

```markdown
# ADR-[编号]: [决策标题]

## 状态
[提议 | 已接受 | 已弃用 | 已替代（被ADR-XXX替代）]

## 日期
YYYY-MM-DD

## 背景
描述驱动这个决策的背景和问题。包括：
- 技术上下文
- 业务需求
- 约束条件
- 相关的前序决策

## 决策
清晰陈述做出的决策。

## 选项分析

### 选项1: [名称]
- 描述：
- 优势：
- 劣势：
- 成本：

### 选项2: [名称]
- 描述：
- 优势：
- 劣势：
- 成本：

### 选项3: [名称]
- 描述：
- 优势：
- 劣势：
- 成本：

## 决策理由
说明为什么选择当前方案，而非其他方案。

## 影响
描述这个决策的影响：
- 正面影响：
- 负面影响：
- 需要的后续行动：

## 参与者
- 决策者：
- 参与讨论：
```

### 4.3 ADR示例

```markdown
# ADR-001: 选择 PostgreSQL 作为主数据库

## 状态
已接受

## 日期
2026-06-27

## 背景
新项目需要选择关系型数据库作为核心数据存储。主要数据特征：
- 结构化业务数据，需要 ACID 事务支持
- 预计数据量 500GB-2TB
- 读写比约 3:1
- 需要支持 JSON 文档查询
- 团队有 MySQL 经验，无 PostgreSQL 经验

## 决策
选择 PostgreSQL 16 作为主数据库。

## 选项分析

### 选项1: MySQL 8.0
- 描述：团队现有技术栈，生态成熟
- 优势：团队熟悉、运维经验丰富、社区庞大
- 劣势：JSON支持较弱、扩展性不如PG
- 成本：低（无需学习成本）

### 选项2: PostgreSQL 16
- 描述：功能丰富的关系型数据库
- 优势：JSON支持强、扩展性好、功能丰富、MVCC成熟
- 劣势：团队需要学习、运维经验不足
- 成本：中（需1-2周学习和实践）

### 选项3: TiDB
- 描述：分布式NewSQL数据库
- 优势：水平扩展能力强、兼容MySQL协议
- 劣势：运维复杂度高、社区相对较小
- 成本：高（运维成本、学习成本）

## 决策理由
虽然团队有MySQL经验，但项目对JSON查询和扩展性有明确需求，PostgreSQL能更好满足。1-2周的学习成本在可接受范围内。

## 影响
- 正面：获得更好的JSON支持和扩展能力
- 负面：需要投入学习成本，初期可能有性能调优摸索
- 后续行动：
  - [ ] 安排团队 PostgreSQL 培训
  - [ ] 制定 PostgreSQL 运维手册
  - [ ] 建立性能基准测试

## 参与者
- 决策者：张三（技术总监）
- 参与讨论：李四（架构师）、王五（DBA）
```

## 5. 技术雷达

### 5.1 技术雷达概述

技术雷达是一种技术评估和沟通工具，将技术资产分为四个象限，每个象限内的技术按成熟度分层。

### 5.2 雷达维度定义

#### 象限（What）

| 象限 | 说明 | 示例 |
|------|------|------|
| **技术/平台** | 编程语言、框架、平台 | Go, Rust, K8s |
| **工具** | 开发和运维工具 | VS Code, Terraform |
| **技术** | 技术实践和方法 | 微服务, DDD, TDD |
| **平台** | 基础设施和平台服务 | AWS, Azure, K8s |

#### 环（Adopt/Trial/Assess/Hold）

| 环 | 含义 | 行动建议 |
|----|------|----------|
| **采纳（Adopt）** | 已在生产环境验证，强烈推荐 | 新项目默认使用 |
| **试验（Trial）** | 值得投入资源做POC | 在非关键项目中试用 |
| **评估（Assess）** | 值得探索和研究 | 做技术调研和原型 |
| **观望（Hold）** | 谨慎使用，暂不推荐 | 不在新项目中引入 |

### 5.3 技术雷达模板

```markdown

-

-

### 5.4 技术评估流程

```
新技术发现 → 初步调研 → 技术雷达讨论 → 纳入雷达
                    ↓
              评估报告
              • 技术概述

### 6.2 评审原则

1. **建设性**：评审目的是改进方案，而非批评个人
2. **聚焦关键**：优先讨论架构级问题，细节在会下讨论
3. **证据驱动**：基于数据和事实讨论，避免主观臆断
4. **时间盒**：严格控制评审时间，避免无休止讨论
5. **可追溯**：所有决策和行动项必须记录在案

## 7. 最佳实践

### 7.1 常见评审问题

| 问题 | 表现 | 应对 |
|------|------|------|
| **文档质量低** | 架构文档缺失关键内容 | 制定文档模板和检查清单 |
| **评审流于形式** | 走过场，不深入讨论 | 建立评审质量评分机制 |
| **决策不落地** | 评审结论没有跟踪 | 建立行动项跟踪机制 |
| **参与度不足** | 关键人员缺席 | 明确参与者要求，提前通知 |
| **讨论失焦** | 陷入技术细节争论 | 评审组长引导，控制讨论范围 |

### 7.2 评审成熟度模型

```
Level 1: 无正式评审
    ↓ 建立基本流程
Level 2: 有流程但执行不一致
    ↓ 标准化模板和检查清单
Level 3: 流程标准化，定期执行
    ↓ 量化评审效果
Level 4: 数据驱动，持续优化
    ↓ 文化内化
Level 5: 评审成为技术文化核心
```
