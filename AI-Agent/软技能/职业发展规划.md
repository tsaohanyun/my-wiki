# 职业发展规划

> 职业发展不是等来的，而是规划出来的。没有方向的努力，只是在原地打转。

## 一、职业路径规划

### 1.2 各级别核心能力模型

| 级别 | 技术深度 | 技术广度 | 影响力 | 领导力 |
|-----|---------|---------|--------|-------|
| **初级工程师** | 独立完成模块 | 了解相关技术栈 | 团队内分享 | 接受指导 |
| **中级工程师** | 独立设计系统 | 跨团队技术了解 | 团队内影响 | 指导新人 |
| **高级工程师** | 技术方案决策 | 跨领域技术视野 | 部门内影响 | 技术引领 |
| **Staff工程师** | 技术方向把控 | 跨部门协作 | 公司级影响 | 技术战略 |
| **首席工程师** | 行业技术前瞻 | 跨公司影响 | 行业影响 | 技术愿景 |

### 1.4 职业目标设定（OKR个人版）

```yaml
Objective: 在2年内成为高级工程师

KR1: 技术深度
  - 独立设计并主导完成2个以上核心系统重构
  - 在技术方案评审中提出被采纳的关键建议≥5次
  - 解决3个以上团队级技术难题

KR2: 技术广度
  - 掌握2个以上相邻技术领域（如前端+后端，或后端+基础设施）
  - 完成1个跨团队技术项目

KR3: 影响力
  - 在团队内做≥4次技术分享
  - 在公司级技术论坛做≥1次分享
  - 输出≥3篇高质量技术博客

KR4: 领导力
  - 指导1-2名初级工程师成长
  - 主导制定1个技术规范
  - 在技术选型中发挥决策作用
```

### 2.2 技能提升路径

#### 阶段一：入门期（0-6个月）

```
目标：建立基础，能独立完成任务
策略：
├── 系统学习：看官方文档 + 经典书籍
├── 动手实践：完成3-5个练手项目
├── 代码阅读：研究优秀开源项目
└── 请教他人：找mentor指导

产出：
☐ 独立完成模块开发
☐ 通过Code Review标准
☐ 理解团队技术栈
```

#### 阶段二：成长期（6-18个月）

```
目标：深入原理，能独立设计方案
策略：
├── 深入学习：阅读源码、理解设计思想
├── 解决难题：主动承担技术挑战
├── 分享输出：做技术分享、写博客
└── 拓展视野：了解相邻技术领域

产出：
☐ 主导完成1-2个技术方案
☐ 解决3个以上技术难题
☐ 输出3篇以上技术文章
```

#### 阶段三：突破期（18-36个月）

```
目标：建立影响力，成为技术骨干
策略：
├── 技术决策：参与技术选型和架构决策
├── 跨团队协作：推动跨团队技术项目
├── 行业交流：参加技术会议、社区贡献
└── 知识沉淀：建立个人知识体系

产出：
☐ 主导完成核心系统设计
☐ 建立技术规范或最佳实践
☐ 在公司级/行业级建立影响力
```

### 2.3 学习方法论

#### 费曼学习法

```
Step 1: 选择概念 → 明确要学习的主题
Step 2: 教授他人 → 用简单语言解释（假装教小白）
Step 3: 发现盲区 → 哪里说不清楚，哪里就是盲区
Step 4: 简化回顾 → 回到资料补充，简化理解
```

#### 刻意练习

| 要素 | 含义 | 实践 |
|-----|------|------|
| **明确目标** | 练习什么具体技能 | "提升系统设计能力"而非"变强" |
| **专注练习** | 在舒适区边缘 | 选择略高于当前水平的挑战 |
| **即时反馈** | 知道对错 | Code Review、测试、导师反馈 |
| **重复迭代** | 持续练习 | 每周固定时间练习 |

### 3.1 人脉圈层模型

```
                ┌───────────────────────┐
                │      弱关系网络        │

                │  │ （同事、合作伙伴） │  │
                │  │  ┌───────────┐  │  │
                │  │  │ 强关系网络 │  │  │
                │  │  │（导师/密友）│  │  │
                │  │  └───────────┘  │  │

### 3.2 人脉建设策略

#### 强关系（5-15人）

```
特征：深度信任、频繁互动、相互支持
来源：
├── 直接同事中的深度合作伙伴
├── 导师（Mentor）
├── 技术密友（共同成长）
└── 前同事中的好朋友

维护方式：
├── 定期1对1交流（每月/每季度）
├── 主动分享有价值的信息
├── 在对方需要时提供帮助
└── 真诚、长期、非功利
```

#### 中关系（50-150人）

```
特征：相互了解、偶尔互动、专业认可
来源：
├── 公司内其他团队的同事
├── 合作伙伴公司的对接人
├── 技术社区中的活跃成员
└── 会议/活动中认识的人

维护方式：
├── 社交媒体互动（点赞、评论）
├── 偶尔私信交流
├── 分享有价值的内容
└── 参加共同的活动
```

#### 弱关系（500+人）

```
特征：单向连接、信息流通、机会来源
来源：
├── 社交媒体关注者
├── 技术社区成员
├── 会议/活动参与者
└── 行业从业者

维护方式：
├── 持续输出高质量内容
├── 参与社区讨论
├── 分享行业信息
└── 保持活跃度
```

### 3.3 高效社交技巧

| 场景 | 技巧 | 示例 |
|-----|------|------|
| **初次见面** | 提供价值而非索取 | "我最近在研究X，发现了一个有趣的..." |
| **加好友** | 个性化备注 | "Hi，我是X会议上的张三，聊过Y话题" |
| **保持联系** | 定期触达 | 分享文章、节日问候、项目进展 |
| **求助他人** | 先给予再索取 | "我之前整理了一份X资料，顺便想请教..." |
| **介绍他人** | 做连接者 | "两位都在做X领域，我觉得你们可以聊聊" |

```

### 4.2 个人品牌定位

#### 找到你的独特定位

```
Step 1: 列出你的技能清单
  ├── 技术技能（编程、架构、运维...）
  ├── 软技能（沟通、写作、演讲...）
  └── 领域知识（电商、金融、AI...）

Step 2: 找到交叉点
  技术技能 ∩ 软技能 ∩ 领域知识 = 你的独特定位

Step 3: 定义一句话定位
  示例：
  "专注于高并发系统架构的技术博主"
  "擅长用可视化解释复杂技术的教育者"
  "深耕电商领域的全栈工程师"
```

## 五、职业发展常见问题

### 5.1 技术 vs 管理

| 维度 | 技术路线 | 管理路线 |
|-----|---------|---------|
| **核心能力** | 技术深度+广度 | 领导力+沟通 |
| **影响方式** | 通过技术方案 | 通过团队 |
| **成就感来源** | 解决技术难题 | 团队成长 |
| **风险** | 技术过时 | 脱离技术 |
| **适合人群** | 热爱技术、追求极致 | 喜欢与人合作、有大局观 |

**决策建议：**
- 不急于选择，先积累技术深度
- 尝试带新人、带项目，体验管理
- 选择让你更有热情的方向
- 两条路可以切换，不必一条路走到黑

### 5.2 跳槽决策框架

```
当前公司评估：
├── 成长空间：还能学到东西吗？
├── 薪资水平：是否低于市场价20%+？
├── 发展路径：晋升通道是否通畅？
├── 团队氛围：合作是否愉快？
└── 业务前景：公司/业务是否有前景？

跳槽收益评估：
├── 薪资涨幅：≥30%值得考虑
├── 技术成长：新技术栈/更大规模系统
├── 职位提升：是否有title提升
├── 行业前景：新公司所在行业如何
└── 生活平衡：通勤、加班、出差

决策矩阵：
高成长 + 高薪资 = 果断跳
高成长 + 低薪资 = 谨慎考虑
低成长 + 高薪资 = 短期可以
低成长 + 低薪资 = 果断跳
```

### 5.3 职业倦怠应对

| 症状 | 原因 | 应对策略 |
|-----|------|---------|
| 失去热情 | 重复性工作 | 申请新项目、学习新技术 |
| 效率下降 | 目标不清晰 | 重新设定OKR、拆解小目标 |
| 情绪低落 | 缺乏认可 | 主动寻求反馈、记录成就 |
| 身体疲惫 | 工作过载 | 学习时间管理、设定边界 |
| 迷茫焦虑 | 看不到未来 | 做职业规划、找mentor聊聊 |
