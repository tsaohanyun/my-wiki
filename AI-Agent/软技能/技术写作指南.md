# 技术写作指南

> 优秀的技术写作能力是工程师从"会做"到"会教"的关键跨越，也是影响力放大的核心杠杆。

## 一、技术文档写作

### 1.1 文档金字塔原则

```
            ┌─────────────┐
            │   概念概述    │  ← 为什么（Why）
            ├─────────────┤
            │   架构设计    │  ← 是什么（What）
            ├─────────────┤
            │  使用教程     │  ← 怎么用（How）
            ├─────────────┤
            │  API 参考     │  ← 细节（Details）
            └─────────────┘
```

### 1.2 四类技术文档

| 文档类型 | 目标读者 | 核心要素 | 示例 |
|---------|---------|---------|------|
| **教程（Tutorial）** | 新手 | 步骤清晰、可复现 | "从零搭建 Kubernetes 集群" |
| **指南（Guide）** | 有经验的用户 | 场景化、解决问题 | "如何配置高可用 MySQL" |
| **参考（Reference）** | 开发者 | 准确、完整、结构化 | API 文档、配置参数表 |
| **解释（Explanation）** | 所有人 | 概念清晰、有深度 | "微服务架构设计理念" |

### 1.3 API 文档写作模板

```markdown
## 接口名称：创建用户

### 概述
创建一个新用户账户。

### 请求

**URL:** `POST /api/v1/users`

**请求头：**
| Header       | 类型   | 必填 | 说明         |
|-------------|--------|------|-------------|
| Content-Type | string | 是   | application/json |
| Authorization | string | 是   | Bearer {token}   |

**请求体：**
```json
{
  "name": "张三",
  "email": "zhangsan@example.com",
  "role": "developer"
}
```

**参数说明：**
| 字段   | 类型   | 必填 | 说明                    |
|-------|--------|------|------------------------|
| name  | string | 是   | 用户姓名，2-50字符       |
| email | string | 是   | 邮箱地址，需唯一          |
| role  | string | 否   | 角色，默认 "developer"   |

### 响应

**成功响应（201 Created）：**
```json
{
  "code": 0,
  "data": {
    "id": "usr_abc123",
    "name": "张三",
    "email": "zhangsan@example.com",
    "created_at": "2026-06-27T10:00:00Z"
  }
}
```

**错误响应：**
| HTTP Status | code  | 说明           |
|------------|-------|---------------|
| 400        | 1001  | 参数校验失败    |
| 409        | 1002  | 邮箱已存在      |
| 401        | 2001  | 认证失败        |

### 示例代码

**cURL:**
```bash
curl -X POST https://api.example.com/api/v1/users \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer your_token" \
  -d '{"name": "张三", "email": "zhangsan@example.com"}'
```

**Python:**
```python
import requests

resp = requests.post(
    "https://api.example.com/api/v1/users",
    json={"name": "张三", "email": "zhangsan@example.com"},
    headers={"Authorization": "Bearer your_token"}
)
print(resp.json())
```
```

### 1.4 文档质量检查清单

- [ ] **准确性**：代码示例能否直接运行？
- [ ] **完整性**：是否覆盖了所有参数和错误码？
- [ ] **一致性**：术语、格式、命名风格是否统一？
- [ ] **可发现性**：是否容易搜索到？
- [ ] **时效性**：是否随代码更新同步维护？

## 二、技术博客写作

### 2.1 博客选题矩阵

```
          高价值
            │
   ┌────────┼────────┐
   │  深度   │  热门   │
   │  原理   │  实战   │  ← 优先写
   ├────────┼────────┤
   │  冷门   │  入门   │
   │  理论   │  教程   │  ← 按需写
   └────────┼────────┘
            │
          低价值
   低流量 ←──────→ 高流量
```

### 2.2 博客文章结构模板

```markdown
# 标题（具体、有吸引力）

## 引言（100-200字）
- 抛出问题或痛点
- 说明本文能解决什么
- 概述核心结论

## 背景/问题（200-300字）
- 什么场景下会遇到这个问题
- 不解决会有什么后果

## 解决方案（核心内容，800-1500字）
- 方案A：原理 + 代码 + 优缺点
- 方案B：原理 + 代码 + 优缺点
- 方案对比表格

## 实战演示（300-500字）
- 完整可运行的代码
- 效果展示（截图/GIF）

## 性能对比/注意事项
- 基准测试数据
- 常见陷阱

## 总结（100-200字）
- 核心要点回顾
- 最佳实践建议

## 参考资料
- 官方文档链接
- 相关论文/文章
```

### 2.3 标题写作技巧

| 模式 | 示例 | 适用场景 |
|-----|------|---------|
| 数字 + 技巧 | "5个让你代码性能提升10倍的优化技巧" | 技巧类 |
| 问题式 | "为什么你的 Docker 镜像总是那么大？" | 排错类 |
| 对比式 | "React vs Vue vs Svelte：2026年该选谁？" | 选型类 |
| 指南式 | "从零到一：构建企业级微服务架构完整指南" | 教程类 |
| 案例式 | "我们如何将 API 延迟从 2s 降到 50ms" | 实战类 |

## 三、报告撰写

### 3.1 技术报告结构

```
1. 执行摘要（Executive Summary）
   ├── 背景与目标
   ├── 核心结论（1-3句话）
   └── 关键建议

2. 问题分析
   ├── 现状描述
   ├── 数据支撑（图表）
   └── 根因分析

3. 方案设计
   ├── 方案对比
   ├── 推荐方案及理由
   └── 实施路线图

4. 风险评估
   ├── 技术风险
   ├── 资源风险
   └── 缓解措施

5. 附录
   ├── 详细数据
   ├── 参考资料
   └── 术语表
```

### 3.2 数据可视化原则

- **选择正确的图表类型**：趋势用折线图、对比用柱状图、占比用饼图、分布用散点图
- **少即是多**：每张图只传达一个核心信息
- **标注清晰**：标题、坐标轴、单位、数据来源
- **色彩克制**：主色不超过3种，用高亮色强调重点

## 四、演讲稿写作

### 4.1 演讲稿结构（18分钟法则）

```
开场（2分钟）
├── 钩子：故事/问题/数据
├── 自我介绍
└── 预告：今天讲什么

主体（14分钟）
├── 要点1（4分钟）→ 论据 + 案例
├── 要点2（4分钟）→ 论据 + 案例
├── 要点3（4分钟）→ 论据 + 案例
└── 互动/演示（2分钟）

结尾（2分钟）
├── 核心要点回顾
├── 行动号召（Call to Action）
└── 致谢 + Q&A
```

### 4.2 逐字稿 vs 要点稿

| 类型 | 适用场景 | 优点 | 缺点 |
|-----|---------|------|------|
| **逐字稿** | 重要演讲、TED风格 | 内容精确、时间可控 | 容易读稿、缺乏自然感 |
| **要点稿** | 技术分享、内部汇报 | 自然、有互动空间 | 可能遗漏要点、时间难控 |
| **混合稿** | 大多数场景 | 兼顾精确与自然 | 需要更多准备时间 |

### 4.3 演讲稿写作技巧

1. **口语化**：写完后大声读一遍，拗口的句子要改
2. **短句为主**：一句话不超过20个字
3. **重复核心观点**：重要的事说三遍
4. **设置记忆锚点**：每3-5分钟一个故事或案例
5. **准备过渡句**：段落之间用过渡句衔接

## 五、写作效率提升

### 5.1 写作工作流

```
选题 → 大纲 → 初稿 → 自审 → 他审 → 发布 → 迭代
 │      │      │      │      │      │      │
 1天    0.5天  2天    0.5天  1天    0.5天  持续
```

### 5.2 常见写作错误

| 错误 | 问题 | 改进 |
|-----|------|------|
| 开头太长 | 读者流失 | 3句话内进入正题 |
| 术语堆砌 | 受众看不懂 | 首次出现要解释 |
| 没有示例 | 抽象难理解 | 每个概念配代码/图 |
| 结构混乱 | 阅读体验差 | 先写大纲再填充 |
| 信息过载 | 记不住重点 | 一篇文章一个主题 |

## 七、最佳实践总结

### 写作原则（CUPID）

| 原则 | 含义 | 实践 |
|-----|------|------|
| **C**lear（清晰） | 读者能理解 | 避免术语、用短句 |
| **U**seful（有用） | 解决实际问题 | 面向场景写作 |
| **P**recise（精确） | 信息准确无误 | 代码可运行、数据有来源 |
| **I**nclusive（包容） | 适合不同水平读者 | 提供背景信息和进阶链接 |
| **D**iscoverable（可发现） | 容易找到 | SEO优化、清晰标题 |
