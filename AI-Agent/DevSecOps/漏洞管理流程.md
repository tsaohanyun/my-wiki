# 漏洞管理流程

> 漏洞管理不是单纯的「扫描 + 报告」，而是一个涵盖**发现 → 评估 → 分级 → 修复 → 验证 → 报告**的闭环流程。本页涵盖完整漏洞管理生命周期、CVSS 评分体系、修复 SLA 和优先级策略。

## 1. 漏洞管理生命周期

```
┌──────────────────────────────────────────────────────────────┐
│                   漏洞管理生命周期 (闭环)                       │
│                                                              │
│   ┌─────────┐    ┌─────────┐    ┌─────────┐    ┌─────────┐  │
│   │  发现    │───►│  评估    │───►│  分级    │───►│  修复    │  │
│   │ Discover │    │ Assess   │    │ Prioritize│   │  Remediate│ │
│   └─────────┘    └─────────┘    └─────────┘    └────┬────┘  │
│       ▲                                              │       │
│       │          ┌─────────┐    ┌─────────┐         │       │
│       └──────────│  报告    │◄───│  验证    │◄────────┘       │
│                  │ Report   │    │  Verify  │                │
│                  └─────────┘    └─────────┘                 │
└──────────────────────────────────────────────────────────────┘
```

| 阶段 | 目标 | 关键活动 |
|------|------|---------|
| **发现** | 全面发现资产中的漏洞 | 自动化扫描、渗透测试、漏洞悬赏 |
| **评估** | 确认漏洞真实性，评估影响 | 手工验证、PoC、影响分析 |
| **分级** | 确定修复优先级 | CVSS 评分 + 业务上下文 |
| **修复** | 实施修复方案 | 打补丁、配置修改、WAF 规则 |
| **验证** | 确认修复有效 | 重新扫描、回归测试 |
| **报告** | 生成合规报告 | 仪表盘、审计报告、趋势分析 |

### 2.5 自动化扫描脚本

```python
#!/usr/bin/env python3
"""
多工具漏洞扫描编排脚本
支持: Trivy (容器) + Semgrep (代码) + npm audit (依赖)
"""

import subprocess
import json
import sys
from datetime import datetime
from pathlib import Path

class VulnerabilityScanner:
    def __init__(self, project_dir: str = "."):
        self.project_dir = project_dir
        self.results = {
            "scan_date": datetime.now().isoformat(),
            "project": project_dir,
            "findings": []
        }

    def scan_container(self, image: str):
        """Trivy 容器镜像扫描"""
        print(f"[INFO] 扫描容器镜像: {image}")
        cmd = [
            "trivy", "image", "--format", "json",
            "--severity", "HIGH,CRITICAL",
            image
        ]
        result = subprocess.run(cmd, capture_output=True, text=True)
        data = json.loads(result.stdout)

        for target in data.get("Results", []):
            for vuln in target.get("Vulnerabilities", []):
                self.results["findings"].append({
                    "source": "trivy",
                    "type": "container",
                    "target": vuln.get("Target"),
                    "vuln_id": vuln.get("VulnerabilityID"),
                    "package": vuln.get("PkgName"),
                    "installed": vuln.get("InstalledVersion"),
                    "fixed": vuln.get("FixedVersion"),
                    "severity": vuln.get("Severity"),
                    "cvss": vuln.get("CVSS", {}).get("nvd", {}).get("V3Score"),
                    "title": vuln.get("Title"),
                    "description": vuln.get("Description")[:200]
                })

    def scan_code(self, path: str):
        """Semgrep 代码扫描"""
        print(f"[INFO] 扫描代码: {path}")
        cmd = [
            "semgrep", "scan", "--config", "auto",
            "--json", "--output", "semgrep-results.json",
            path
        ]
        subprocess.run(cmd, capture_output=True)
        data = json.loads(Path("semgrep-results.json").read_text())

        for finding in data.get("results", []):
            self.results["findings"].append({
                "source": "semgrep",
                "type": "code",
                "file": finding.get("path"),
                "line": finding.get("start", {}).get("line"),
                "rule_id": finding.get("check_id"),
                "severity": finding.get("extra", {}).get("severity"),
                "message": finding.get("extra", {}).get("message")
            })

    def scan_dependencies(self):
        """npm audit 依赖扫描"""
        print("[INFO] 扫描 npm 依赖...")
        cmd = ["npm", "audit", "--json"]
        result = subprocess.run(cmd, capture_output=True, text=True, cwd=self.project_dir)
        data = json.loads(result.stdout or "{}")

        for adv_id, adv in data.get("vulnerabilities", {}).items():
            self.results["findings"].append({
                "source": "npm-audit",
                "type": "dependency",
                "package": adv.get("name"),
                "severity": adv.get("severity"),
                "via": adv.get("via"),
                "fix_available": adv.get("fixAvailable")
            })

    def generate_report(self, output_file: str = "vuln-report.json"):
        """生成综合漏洞报告"""
        # 统计
        stats = {"critical": 0, "high": 0, "medium": 0, "low": 0}
        for f in self.results["findings"]:
            sev = f.get("severity", "unknown").lower()
            if sev in stats:
                stats[sev] += 1

        self.results["summary"] = stats
        self.results["total_findings"] = len(self.results["findings"])

        Path(output_file).write_text(
            json.dumps(self.results, indent=2, ensure_ascii=False)
        )
        print(f"\n[报告] 总计: {self.results['total_findings']} 个漏洞")
        print(f"  CRITICAL: {stats['critical']}")
        print(f"  HIGH:     {stats['high']}")
        print(f"  MEDIUM:   {stats['medium']}")
        print(f"  LOW:      {stats['low']}")

        # 门禁：存在 CRITICAL 则失败
        if stats["critical"] > 0:
            print(f"\n❌ 存在 {stats['critical']} 个严重漏洞，构建失败")
            sys.exit(1)

        print("\n✅ 无严重漏洞，构建通过")

# 使用示例
if __name__ == "__main__":
    scanner = VulnerabilityScanner(".")
    scanner.scan_container("myapp:v1.0.0")
    scanner.scan_code("src/")
    scanner.scan_dependencies()
    scanner.generate_report()
```

## 3. CVSS 评分体系

### 3.1 CVSS v3.1 评分模型

**CVSS (Common Vulnerability Scoring System)** 是业界标准的漏洞评分体系。

#### Base Score (基础分) — 漏洞内在特性

| 指标 | 选项 | 说明 |
|------|------|------|
| **Attack Vector (AV)** | Network / Adjacent / Local / Physical | 攻击路径 |
| **Attack Complexity (AC)** | Low / High | 攻击复杂度 |
| **Privileges Required (PR)** | None / Low / High | 所需权限 |
| **User Interaction (UI)** | None / Required | 是否需用户交互 |
| **Scope (S)** | Unchanged / Changed | 影响范围是否跨组件 |
| **Confidentiality (C)** | None / Low / High | 机密性影响 |
| **Integrity (I)** | None / Low / High | 完整性影响 |
| **Availability (A)** | None / Low / High | 可用性影响 |

#### 评分等级

| 分值 | 等级 | 颜色 | 说明 |
|------|------|------|------|
| 9.0 - 10.0 | **CRITICAL** | 🔴 | 严重 — 立即修复 |
| 7.0 - 8.9 | **HIGH** | 🟠 | 高危 — 尽快修复 |
| 4.0 - 6.9 | **MEDIUM** | 🟡 | 中危 — 计划修复 |
| 0.1 - 3.9 | **LOW** | 🟢 | 低危 — 酌情修复 |
| 0.0 | **NONE** | ⚪ | 无风险 |

### 3.2 CVSS 向量字符串

```
CVSS:3.1/AV:N/AC:L/PR:N/UI:N/S:U/C:H/I:H/A:H

解读:
  CVSS:3.1          → CVSS 版本
  AV:N              → Attack Vector: Network (网络可达)
  AC:L              → Attack Complexity: Low (低复杂度)
  PR:N              → Privileges Required: None (无需权限)
  UI:N              → User Interaction: None (无需交互)
  S:U               → Scope: Unchanged (不跨组件)
  C:H               → Confidentiality: High (高机密性影响)
  I:H               → Integrity: High (高完整性影响)
  A:H               → Availability: High (高可用性影响)

→ Base Score: 9.8 (CRITICAL) — 如 Log4Shell CVE-2021-44228
```

### 3.3 Temporal & Environmental Score

| 分数类型 | 含义 | 调整因素 |
|---------|------|---------|
| **Base** | 漏洞固有严重性 | 不可调整 |
| **Temporal** | 随时间变化的严重性 | Exploit 可用性、修复可用性、置信度 |
| **Environmental** | 特定环境的影响 | 资产重要性、安全控制补偿 |

```python
# 环境评分调整示例
# Log4Shell (Base: 9.8) 在内部管理系统中的环境分可能为 6.5
# 原因: 内网不可达 + 有 WAF + 数据非敏感

environmental_adjustments = {
    "base_score": 9.8,
    "asset_criticality": "low",      # 非核心资产
    "exposure": "internal",           # 仅内网可达
    "compensating_controls": ["WAF", "Network Segmentation"],
    "adjusted_score": 6.5,            # 环境评分
    "reason": "内网系统 + WAF 防护 + 非敏感数据"
}
```

### 4.3 优先级计算器

```python
#!/usr/bin/env python3
"""
漏洞修复优先级计算器
综合 CVSS + EPSS + 资产重要性 + 暴露面
"""

from enum import IntEnum
from dataclasses import dataclass

class Priority(IntEnum):
    P1_IMMEDIATE = 1   # 24 小时内
    P2_URGENT = 2      # 1 周内
    P3_SCHEDULED = 3   # 1 个月内
    P4_ROUTINE = 4     # 季度内

@dataclass
class Vulnerability:
    cve_id: str
    cvss_score: float
    epss_score: float          # 0.0 - 1.0
    asset_criticality: str      # low / medium / high / critical
    exposure: str               # internet / dmz / internal / isolated
    exploit_in_wild: bool       # 是否已有野外利用
    patch_available: bool       # 是否有可用补丁
    compensating_controls: list # 补偿控制措施

def calculate_priority(vuln: Vulnerability) -> Priority:
    """计算漏洞修复优先级"""

    score = 0

    # CVSS 权重 (0-40 分)
    if vuln.cvss_score >= 9.0:
        score += 40
    elif vuln.cvss_score >= 7.0:
        score += 30
    elif vuln.cvss_score >= 4.0:
        score += 15
    else:
        score += 5

    # EPSS 权重 (0-25 分)
    if vuln.epss_score >= 0.7:
        score += 25
    elif vuln.epss_score >= 0.3:
        score += 15
    elif vuln.epss_score >= 0.1:
        score += 8
    else:
        score += 2

    # 野外利用 (0-20 分)
    if vuln.exploit_in_wild:
        score += 20

    # 资产重要性 (0-10 分)
    asset_weights = {
        "critical": 10, "high": 8,
        "medium": 5, "low": 2
    }
    score += asset_weights.get(vuln.asset_criticality, 3)

    # 暴露面 (0-5 分)
    exposure_weights = {
        "internet": 5, "dmz": 4,
        "internal": 2, "isolated": 0
    }
    score += exposure_weights.get(vuln.exposure, 2)

    # 补偿控制减少分数
    control_credits = {
        "WAF": 5, "Network Segmentation": 3,
        "IPS/IDS": 3, "EDR": 2
    }
    for control in vuln.compensating_controls:
        score -= control_credits.get(control, 0)

    score = max(0, score)

    # 优先级映射
    if score >= 70:
        return Priority.P1_IMMEDIATE
    elif score >= 50:
        return Priority.P2_URGENT
    elif score >= 30:
        return Priority.P3_SCHEDULED
    else:
        return Priority.P4_ROUTINE

# 使用示例
vuln = Vulnerability(
    cve_id="CVE-2021-44228",
    cvss_score=9.8,
    epss_score=0.975,
    asset_criticality="high",
    exposure="internet",
    exploit_in_wild=True,
    patch_available=True,
    compensating_controls=["WAF"]
)

priority = calculate_priority(vuln)
print(f"{vuln.cve_id}: 优先级 = P{priority} (得分={priority})")
# 输出: CVE-2021-44228: 优先级 = P1 (24小时内修复)
```

## 5. SLA — 修复时限

### 5.1 标准 SLA 矩阵

| 严重程度 | CVSS | 默认 SLA | 互联网暴露 | 内部系统 | 关键系统 |
|---------|------|---------|-----------|---------|---------|
| **CRITICAL** | 9.0-10.0 | **7 天** | **24 小时** | 3 天 | 24 小时 |
| **HIGH** | 7.0-8.9 | **14 天** | **7 天** | 14 天 | 7 天 |
| **MEDIUM** | 4.0-6.9 | **30 天** | 14 天 | 30 天 | 14 天 |
| **LOW** | 0.1-3.9 | **90 天** | 30 天 | 90 天 | 30 天 |

### 5.2 SLA 自动化跟踪

```python
#!/usr/bin/env python3
"""
漏洞 SLA 跟踪系统 — 监控漏洞修复进度，触发告警
"""

from datetime import datetime, timedelta
from dataclasses import dataclass
from enum import Enum

class Severity(str, Enum):
    CRITICAL = "Critical"
    HIGH = "High"
    MEDIUM = "Medium"
    LOW = "Low"

class SLAStatus(str, Enum):
    ON_TRACK = "on_track"
    AT_RISK = "at_risk"       # 接近到期
    OVERDUE = "overdue"       # 已超期
    BREACHED = "breached"     # 严重超期

SLA_DAYS = {
    Severity.CRITICAL: 1,   # 互联网暴露的 CRITICAL: 24h
    Severity.HIGH: 7,
    Severity.MEDIUM: 30,
    Severity.LOW: 90
}

@dataclass
class VulnerabilityTicket:
    id: str
    cve_id: str
    severity: Severity
    discovered_at: datetime
    asset: str
    exposed_to_internet: bool
    assigned_to: str
    status: str  # open / in_progress / resolved

    def sla_deadline(self) -> datetime:
        days = SLA_DAYS[self.severity]
        if not self.exposed_to_internet:
            # 内部系统放宽
            days_map = {
                Severity.CRITICAL: 3,
                Severity.HIGH: 14,
                Severity.MEDIUM: 30,
                Severity.LOW: 90
            }
            days = days_map[self.severity]
        return self.discovered_at + timedelta(days=days)

    def sla_status(self) -> tuple[SLAStatus, int]:
        now = datetime.now()
        deadline = self.sla_deadline()
        days_remaining = (deadline - now).days

        if days_remaining < -3:
            return SLAStatus.BREACHED, days_remaining
        elif days_remaining < 0:
            return SLAStatus.OVERDUE, days_remaining
        elif days_remaining <= 2:
            return SLAStatus.AT_RISK, days_remaining
        else:
            return SLAStatus.ON_TRACK, days_remaining

# 使用示例
ticket = VulnerabilityTicket(
    id="VULN-001",
    cve_id="CVE-2024-1234",
    severity=Severity.CRITICAL,
    discovered_at=datetime(2026, 6, 25),
    asset="api-gateway-prod",
    exposed_to_internet=True,
    assigned_to="security-team",
    status="in_progress"
)

status, days = ticket.sla_status()
print(f"{ticket.cve_id}: {status.value} (剩余 {days} 天)")
```

### 6.2 操作系统补丁自动化 (Ansible)

```yaml
# patch-management.yml — Ansible 补丁自动化
---
- name: 安全补丁管理
  hosts: all
  become: yes
  vars:
    security_updates_only: true
    reboot_after_patch: true
    patch_window: "maintenance"

  tasks:
    # 1. 检查当前安全更新
    - name: 检查可用安全更新 (RHEL)
      yum:
        list: security
        state: latest
        update_cache: yes
      register: available_updates
      when: ansible_os_family == "RedHat"

    - name: 检查可用安全更新 (Debian)
      apt:
        update_cache: yes
        upgrade: no
      register: apt_update
      when: ansible_os_family == "Debian"

    # 2. 快照前状态
    - name: 记录补丁前状态
      copy:
        content: "{{ available_updates.results | to_nice_json }}"
        dest: /tmp/pre-patch-status.json
      when: available_updates is defined

    # 3. 应用安全补丁
    - name: 安装安全更新 (RHEL)
      yum:
        name: '*'
        state: latest
        security: yes
      register: patch_result_rhel
      when: ansible_os_family == "RedHat"

    - name: 安装安全更新 (Debian)
      apt:
        upgrade: safe
      register: patch_result_debian
      when: ansible_os_family == "Debian"

    # 4. 检查是否需要重启
    - name: 检查是否需要重启
      command: needs-restarting -r
      register: reboot_check
      failed_when: false
      changed_when: false
      when: ansible_os_family == "RedHat"

    - name: 重启服务器 (如果需要)
      reboot:
        reboot_timeout: 600
        test_command: uptime
      when:
        - reboot_after_patch
        - reboot_check.rc == 1

    # 5. 生成补丁报告
    - name: 生成补丁报告
      copy:
        content: |
          主机: {{ inventory_hostname }}
          补丁时间: {{ ansible_date_time.iso8601 }}
          补丁数: {{ (patch_result_rhel.results | default([]) | length) + (patch_result_debian.results | default([]) | length) }}
          重启: {{ reboot_check.rc | default(0) == 1 }}
        dest: /var/log/patch-report.txt
      delegate_to: localhost
```

### 6.3 Kubernetes 滚动更新补丁

```bash
#!/bin/bash
# k8s-node-patching.sh — Kubernetes 节点补丁脚本

set -euo pipefail

NODE=$1
DRAIN_TIMEOUT=300

echo "=== 开始补丁节点: $NODE ==="

# 1. 标记节点为不可调度
kubectl cordon "$NODE"

# 2. 排空节点（驱逐 Pod）
kubectl drain "$NODE" \
  --ignore-daemonsets \
  --delete-emptydir-data \
  --timeout="${DRAIN_TIMEOUT}s"

# 3. 执行补丁更新
ssh "root@$NODE" << 'EOF'
  # RHEL
  if command -v dnf &> /dev/null; then
    dnf update --security -y
  fi
  # Ubuntu
  if command -v apt &> /dev/null; then
    apt update && apt upgrade -y
  fi

  # 检查是否需要重启
  if [ -f /var/run/reboot-required ]; then
    echo "需要重启"
    reboot
  fi
EOF

# 4. 等待节点恢复
echo "等待节点恢复..."
until kubectl get node "$NODE" | grep -q "Ready"; do
  sleep 10
  echo "..."
done

# 5. 恢复调度
kubectl uncordon "$NODE"

echo "=== 节点 $NODE 补丁完成 ==="
```

### 6.4 容器镜像补丁策略

```dockerfile
    apt-get upgrade -y && \
    apt-get install -y --no-install-recommends \
      unattended-upgrades && \
    apt-get clean && \
    rm -rf /var/lib/apt/lists/*

HEALTHCHECK --interval=24h --timeout=5m \
  CMD apt list --upgradable 2>/dev/null | grep -i security || exit 0
```

### 7.2 集成扫描器到 DefectDojo

```python
#!/usr/bin/env python3
"""
将扫描结果上传到 DefectDojo
"""
import requests
import json
from datetime import datetime

DOJO_URL = "https://defectdojo.example.com"
API_KEY = "your-api-key"

def upload_scan_results(
    engagement_id: int,
    scan_type: str,
    file_path: str,
    active: bool = True,
    verified: bool = True
):
    """上传扫描结果"""
    url = f"{DOJO_URL}/api/v2/import-scan/"
    headers = {"Authorization": f"Token {API_KEY}"}

    files = {"file": open(file_path, "rb")}
    data = {
        "engagement": engagement_id,
        "scan_type": scan_type,     # e.g., "Trivy Scan", "Semgrep JSON"
        "active": active,
        "verified": verified,
        "scan_date": datetime.now().strftime("%Y-%m-%d"),
        "minimum_severity": "Low",
        "close_old_findings": True,  # 自动关闭不再存在的漏洞
        "push_to_jira": True,        # 推送到 Jira
    }

    response = requests.post(url, headers=headers, files=files, data=data)
    if response.status_code == 201:
        print(f"✅ 扫描结果已上传: {response.json()['test']}")
    else:
        print(f"❌ 上传失败: {response.text}")

# 使用示例
upload_scan_results(
    engagement_id=1,
    scan_type="Trivy Scan",
    file_path="trivy-results.json"
)
```

## 8. 最佳实践

### ✅ 推荐做法

| 实践 | 说明 |
|------|------|
| **资产清单优先** | 无法保护你不知道的资产，维护准确的 CMDB |
| **基于风险的优先级** | 不只看 CVSS，结合 EPSS + 资产重要性 + 暴露面 |
| **SLA 驱动** | 为每个严重级别设定明确修复时限 |
| **补丁先测后上** | 所有补丁先在 Staging 验证，防止引入故障 |
| **虚拟补丁** | 无法立即打补丁时，先用 WAF/IPS 规则临时防护 |
| **闭环跟踪** | 发现 → 修复 → 验证 → 关闭，每一步可追溯 |
| **趋势分析** | 关注 MTTR 和积压趋势，而非单次快照 |
| **例外管理** | 无法修复的漏洞需有书面例外和补偿控制 |

### ❌ 避免

| 反模式 | 风险 |
|--------|------|
| 仅看 CVSS 评分 | 忽略了业务上下文和实际可利用性 |
| 一刀切 SLA | 内部系统和互联网系统用相同 SLA |
| 扫描但不修复 | 产生「扫描疲劳」，团队忽视告警 |
| 手工跟踪漏洞表 | Excel 表格无法扩展，容易遗漏 |
| 不追踪 MTTR | 无法衡量安全运营效率 |
| 跳过补丁测试 | 补丁可能破坏生产系统 |
