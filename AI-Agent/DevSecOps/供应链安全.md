## 1. 核心概念

| 概念 | 说明 |
|------|------|
| **SCA** (Software Composition Analysis) | 分析开源依赖中的已知漏洞和许可证风险 |
| **SBOM** (Software Bill of Materials) | 软件组件清单，记录所有依赖及其版本、来源 |
| **依赖扫描** | 自动检测项目依赖中的安全漏洞 |
| **Sigstore / Cosign** | 容器镜像签名与验证工具链，确保镜像完整性 |
| **依赖锁定** | 锁定依赖版本，防止供应链注入攻击 |

### SLSA 框架

**SLSA (Supply-chain Levels for Software Artifacts)** 是保护软件供应链的渐进式框架：

| 级别 | 要求 | 说明 |
|------|------|------|
| L1 | 构建过程有文档 | 构建来源可追溯 |
| L2 | 托管构建服务 | 构建在受控环境中进行 |
| L3 | 构建平台隔离 | 构建隔离，防篡改 |
| L4 | 双人审核 | 变更需两人审核 |
| L5 | 可审计的构建 | 构建过程可独立验证 |

## 2. SCA — 软件组成分析

### 2.1 Trivy — 一体化扫描工具

```bash
# 安装 Trivy
curl -sfL https://raw.githubusercontent.com/aquasecurity/trivy/main/contrib/install.sh | sh

# 扫描项目依赖
trivy fs --severity HIGH,CRITICAL .

# 扫描容器镜像
trivy image nginx:1.25

# 扫描 IaC 文件
trivy config ./terraform/

# 生成 SBOM (CycloneDX 格式)
trivy fs --format cyclonedx -o sbom.json .

# 输出为 JSON 并忽略未修复的漏洞
trivy fs --format json --ignore-unfixed -o results.json .
```

### 2.2 Trivy 配置文件

```yaml
# trivy.yaml
scan:
  scanners:
    - vuln
    - secret
    - license
  severity:
    - HIGH
    - CRITICAL

vulnerability:
  ignore-unfixed: true
  ignore-policy: .trivyignore.yaml

secret:
  config: trivy-secret.yaml

license:
  forbidden:
    - GPL-3.0
    - AGPL-3.0

db:
  skip-update: false
  update-interval: 24h

output:
  format: table
  # 也可选 json, sarif, cyclonedx, spdx
```

### 2.3 Grype — Anchore 漏洞扫描器

```bash
# 安装 Grype
curl -sSfL https://raw.githubusercontent.com/anchore/grype/main/install.sh | sh

# 扫描容器镜像
grype nginx:1.25

# 扫描 SBOM 文件
grype sbom:./sbom.json

# 只显示可修复的漏洞
grype nginx:1.25 --only-fixed

# 输出 SARIF（GitHub 集成）
grype nginx:1.25 -o sarif > results.sarif
```

## 3. SBOM — 软件物料清单

### 3.1 Syft — SBOM 生成工具

```bash
# 安装 Syft
curl -sSfL https://raw.githubusercontent.com/anchore/syft/main/install.sh | sh

# 从容器镜像生成 SBOM (CycloneDX)
syft nginx:1.25 -o cyclonedx-json > sbom.json

# 从目录生成 SBOM (SPDX)
syft . -o spdx-json > sbom-spdx.json

# 从目录生成 SBOM (Syft Table 格式)
syft . -o table
```

### 3.2 SBOM 格式对比

| 格式 | 标准组织 | 特点 | 适用场景 |
|------|---------|------|---------|
| **CycloneDX** | OWASP | JSON/XML，支持组件依赖树 | 漏洞管理、许可证分析 |
| **SPDX** | Linux Foundation | JSON/RDF/Tag-Value，ISO 标准 | 合规审计、许可证合规 |
| **SWID** | ISO/IEC 19770-1 | XML，软件标识 | 资产管理 |

  "metadata": {

    "tools": [
      {
        "vendor": "Anchore",

      }

    "component": {

### 3.4 在 CI 中自动生成 SBOM

```yaml
# .gitlab-ci.yml
stages:
  - build
  - sbom
  - scan

generate-sbom:
  stage: sbom
  image: anchore/syft:latest
  script:
    - syft dir:. -o cyclonedx-json > sbom.json
  artifacts:
    paths:
      - sbom.json
    expire_in: 90 days

scan-sbom:
  stage: scan
  image: anchore/grype:latest
  needs:
    - generate-sbom
  script:
    - grype sbom:./sbom.json --fail-on high -o json > vuln-report.json
  artifacts:
    paths:
      - vuln-report.json
```

## 5. Sigstore 与 Cosign — 镜像签名

### 5.1 Sigstore 架构

```
┌─────────────────────────────────────────────────┐
│                   Sigstore 生态                   │
├──────────────┬───────────────┬───────────────────┤
│   Cosign     │   Rekor       │   Fulcio          │
│  (签名工具)   │  (透明日志)    │  (证书颁发)        │
│  签名/验证    │  存储签名记录   │  签发短期证书       │
│  镜像/制品    │  不可篡改      │  基于OIDC身份       │
├──────────────┼───────────────┼───────────────────┤
│   Policy     │   HashiCorp   │   Kubernetes       │
│  Controller  │   Vault KMS   │   Admission        │
│  策略执行      │  密钥管理      │  准入控制          │
└──────────────┴───────────────┴───────────────────┘
```

### 5.2 Cosign 安装与基本使用

```bash
# 安装 Cosign
curl -O -L "https://github.com/sigstore/cosign/releases/latest/download/cosign-linux-amd64"
mv cosign-linux-amd64 /usr/local/bin/cosign
chmod +x /usr/local/bin/cosign

# 生成密钥对（有密码保护）
cosign generate-key-pair
# 输出: cosign.key (私钥) 和 cosign.pub (公钥)

# 使用密钥对签名镜像
COSIGN_PASSWORD=<your-password> \
  cosign sign --key cosign.key myregistry.com/myapp:v1.0.0

# 验证镜像签名
cosign verify --key cosign.pub myregistry.com/myapp:v1.0.0
```

### 5.3 Keyless 签名（免密钥 — 推荐）

```bash
# 使用 OIDC 身份进行 Keyless 签名（无需管理密钥）
cosign sign --yes myregistry.com/myapp:v1.0.0

# 验证 Keyless 签名
cosign verify myregistry.com/myapp:v1.0.0 \
  --certificate-identity-regexp ".*@mycompany\.com" \
  --certificate-oidc-issuer "https://token.actions.githubusercontent.com"
```

### 5.4 CI/CD 中使用 Cosign (GitHub Actions)

```yaml
# .github/workflows/build-and-sign.yml
name: Build and Sign Image

on:
  push:
    branches: [main]
    tags: ["v*"]

permissions:
  contents: read
  id-token: write   # OIDC token for keyless signing
  packages: write

jobs:
  build-sign:
    runs-on: ubuntu-latest
    steps:
      - name: Checkout
        uses: actions/checkout@v4

      - name: Set up Docker Buildx
        uses: docker/setup-buildx-action@v3

      - name: Login to Registry
        uses: docker/login-action@v3
        with:
          registry: ghcr.io
          username: ${{ github.actor }}
          password: ${{ secrets.GITHUB_TOKEN }}

      - name: Build and Push
        id: build
        uses: docker/build-push-action@v5
        with:
          context: .
          push: true
          tags: |
            ghcr.io/${{ github.repository }}:${{ github.sha }}
            ghcr.io/${{ github.repository }}:latest

      - name: Install Cosign
        uses: sigstore/cosign-installer@v3

      - name: Sign Image (Keyless)
        run: |
          cosign sign --yes \
            ghcr.io/${{ github.repository }}:${{ github.sha }}

      - name: Generate SBOM
        uses: anchore/sbom-action@v0
        with:
          image: ghcr.io/${{ github.repository }}:${{ github.sha }}
          format: cyclonedx-json
          output-file: sbom.json

      - name: Attach SBOM to Image
        run: |
          cosign attach sbom --sbom sbom.json \
            ghcr.io/${{ github.repository }}:${{ github.sha }}

      - name: Sign SBOM
        run: |
          cosign sign --yes \
            --attachment sbom \
            ghcr.io/${{ github.repository }}:${{ github.sha }}

      - name: Verify Signature
        run: |
          cosign verify \
            --certificate-identity-regexp "https://github.com/${{ github.repository }}/.github/.+" \
            --certificate-oidc-issuer "https://token.actions.githubusercontent.com" \
            ghcr.io/${{ github.repository }}:${{ github.sha }}
```

### 5.5 Kubernetes 中强制镜像签名验证

```yaml
# Kyverno ClusterPolicy — 拒绝未签名镜像
apiVersion: kyverno.io/v1
kind: ClusterPolicy
metadata:
  name: verify-image-signatures
spec:
  validationFailureAction: Enforce
  rules:
    - name: verify-cosign-signature
      match:
        resources:
          kinds:
            - Pod
      verifyImages:
        - imageReferences:
            - "ghcr.io/mycompany/*"
          attestors:
            - entries:
                - keyless:
                    subject: "https://github.com/mycompany/*"
                    issuer: "https://token.actions.githubusercontent.com"
```

```yaml
# OPA Gatekeeper Constraint — 镜像签名验证
apiVersion: constraints.gatekeeper.sh/v1beta1
kind: K8sAllowedRepos
metadata:
  name: require-signed-images
spec:
  match:
    kinds:
      - apiGroups: [""]
        kinds: ["Pod"]
  parameters:
    repos:
      - "ghcr.io/mycompany"
```

## 6. 依赖锁定

### 6.1 npm — package-lock.json

```bash
# 生成锁文件
npm install --package-lock-only

# 使用 npm ci（严格使用锁文件安装）
npm ci

# 审计锁文件中的漏洞
npm audit --audit-level=high

# 自动修复
npm audit fix
```

### 6.2 pip — Pipfile.lock / requirements.txt 哈希

```bash
# 使用 pip-tools 锁定依赖
pip install pip-tools

# 编译依赖（生成带哈希的 requirements.txt）
pip-compile --generate-hashes requirements.in

# 安装时验证哈希
pip install --require-hashes -r requirements.txt
```

### 6.4 Docker — 镜像摘要锁定

```dockerfile
# 不安全的写法（tag 可变）

# 安全的写法（锁定到摘要）

```

## 7. CI/CD 集成完整示例

```yaml
# .github/workflows/supply-chain-security.yml
name: Supply Chain Security

on: [push, pull_request]

jobs:
  # 1. SCA 扫描
  sca-scan:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Run Trivy
        uses: aquasecurity/trivy-action@master
        with:
          scan-type: fs
          scan-ref: .
          severity: HIGH,CRITICAL
          format: sarif
          output: trivy-results.sarif
      - name: Upload SARIF
        uses: github/codeql-action/upload-sarif@v3
        with:
          sarif_file: trivy-results.sarif

  # 2. SBOM 生成
  generate-sbom:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Generate SBOM
        uses: anchore/sbom-action@v0
        with:
          format: cyclonedx-json
          output-file: sbom.json
      - name: Upload SBOM Artifact
        uses: actions/upload-artifact@v4
        with:
          name: sbom
          path: sbom.json
          retention-days: 365

  # 3. SBOM 漏洞扫描
  scan-sbom:
    needs: generate-sbom
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Download SBOM
        uses: actions/download-artifact@v4
        with:
          name: sbom
      - name: Scan SBOM
        uses: anchore/scan-action@v3
        with:
          sbom: sbom.json
          fail-build: true
          severity-cutoff: high

  # 4. 镜像签名
  sign-image:
    if: github.ref == 'refs/heads/main'
    needs: [sca-scan, scan-sbom]
    runs-on: ubuntu-latest
    permissions:
      contents: read
      id-token: write
      packages: write
    steps:
      - uses: actions/checkout@v4
      - name: Build Image
        uses: docker/build-push-action@v5
        with:
          context: .
          push: true
          tags: ghcr.io/${{ github.repository }}:${{ github.sha }}
      - name: Install Cosign
        uses: sigstore/cosign-installer@v3
      - name: Sign Image
        run: cosign sign --yes ghcr.io/${{ github.repository }}:${{ github.sha }}
```

## 8. 最佳实践

### ✅ 推荐做法

| 实践 | 说明 |
|------|------|
| **生成并存储 SBOM** | 每次构建生成 SBOM，作为漏洞响应的基础数据 |
| **锁定依赖版本** | 使用锁文件和镜像摘要，避免 tag 可变风险 |
| **CI 中集成 SCA 扫描** | 在 PR 合并前阻断高危依赖 |
| **使用 Keyless 签名** | 无需管理密钥，基于 OIDC 身份签名 |
| **验证签名后再部署** | 在 Kubernetes 准入层强制签名验证 |
| **订阅安全公告** | 订阅 GitHub Security Advisory、OSV、NVD |
| **定期更新基础镜像** | 设定基线（如每周更新），使用 Renovate/Dependabot 自动化 |
| **最小化依赖** | 定期审查无用依赖，减少攻击面 |

### ❌ 避免

| 反模式 | 风险 |
|--------|------|
| 使用 `latest` tag | 镜像内容不可预测，无法审计 |
| 不验证锁文件哈希 | 依赖可被中间人篡改 |
| 忽略 LICENSE 风险 | 可能违反 GPL/AGPL 等 copyleft 许可证 |
| 仅扫描一次 | 漏洞库持续更新，需定期重新扫描 |
| 未存储构建出处 | 无法证明制品来源可信 |
