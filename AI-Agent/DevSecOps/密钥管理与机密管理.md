## 1. 概念与架构

### 1.1 密钥 vs 机密

| 概念 | 说明 | 示例 |
|------|------|------|
| **密钥** | 加解密/签名用的密钥材料 | TLS 私钥、API Signing Key |
| **机密** | 应用运行时需要的敏感数据 | 数据库密码、API Token、SSH Key |
| **配置** | 非敏感的配置数据 | 端口号、超时时间 |

### 1.2 工具对比

| 工具 | 类型 | 适用场景 | 特点 |
|------|------|---------|------|
| **HashiCorp Vault** | 动态机密管理 | 企业级、多租户 | 动态密钥生成、租约、自动轮转 |
| **Sealed Secrets** | GitOps 机密存储 | Kubernetes | 加密机密存 Git，控制器解密 |
| **SOPS** | 加密文件管理 | GitOps、配置文件 | 部分加密（值加密，key 明文） |
| **AWS KMS** | 云密钥管理 | AWS 生态 | 托管密钥，IAM 集成 |
| **External Secrets Operator** | K8s 外部机密同步 | Kubernetes + 外部 Vault | 自动同步外部机密到 K8s Secret |

### 1.3 架构总览


### 2.2 Vault 基本操作

```bash
# 启用 KV 机密引擎 v2 (版本化)
vault secrets enable -path=secret -version=2 kv

# 写入机密
vault kv put secret/myapp/database \
  username="db_user" \
  password="SuperSecret123!" \
  host="db.internal" \
  port="5432"

# 读取机密
vault kv get secret/myapp/database

# 读取特定字段
vault kv get -field=password secret/myapp/database

# 读取 JSON 格式
vault kv get -format=json secret/myapp/database

# 列出机密
vault kv list secret/myapp/

# 删除机密（软删除，保留版本历史）
vault kv delete secret/myapp/database

# 恢复机密
vault kv undelete -versions=1 secret/myapp/database

# 永久删除特定版本
vault kv destroy -versions=1 secret/myapp/database
```

### 2.3 动态数据库凭据

```bash
# 启用数据库机密引擎
vault secrets enable database

# 配置 PostgreSQL 连接
vault write database/config/my-postgresql \
  plugin_name=postgresql-database-plugin \
  connection_url="postgresql://{{username}}:{{password}}@db.internal:5432/myapp?sslmode=disable" \
  allowed_roles="readonly,readwrite" \
  username="vault_admin" \
  password="VaultAdminPassword123!"

# 创建只读角色 (动态凭据，TTL=1小时)
vault write database/roles/readonly \
  db_name=my-postgresql \
  creation_statements=@/vault/policies/pg-readonly.sql \
  default_ttl="1h" \
  max_ttl="24h"

# 创建读写角色
vault write database/roles/readwrite \
  db_name=my-postgresql \
  creation_statements=@/vault/policies/pg-readwrite.sql \
  default_ttl="30m" \
  max_ttl="4h"

# 生成动态凭据
vault read database/creds/readonly
# 输出:
# Key                Value
# lease_id           database/creds/readonly/f3u2k...
# lease_duration     1h
# username           v-token-readonly-abc123
# password           random-generated-password
```

```sql
-- /vault/policies/pg-readonly.sql
CREATE ROLE "{{name}}" WITH LOGIN PASSWORD '{{password}}' VALID UNTIL '{{expiration}}';
GRANT SELECT ON ALL TABLES IN SCHEMA public TO "{{name}}";
ALTER DEFAULT PRIVILEGES IN SCHEMA public GRANT SELECT ON TABLES TO "{{name}}";
```

### 2.4 Kubernetes 认证

```bash
# 启用 Kubernetes 认证方法

# 配置 Kubernetes 认证
vault write auth/kubernetes/config \
  kubernetes_host="https://kubernetes.default.svc:443" \
  kubernetes_ca_cert=@/var/run/secrets/kubernetes.io/serviceaccount/ca.crt \
  token_reviewer_jwt=@/var/run/secrets/kubernetes.io/serviceaccount/token \

# 创建策略

  capabilities = ["read"]
}

  capabilities = ["read"]
}

# 为 Kubernetes ServiceAccount 创建角色
vault write auth/kubernetes/role/myapp \
  bound_service_account_names=myapp-sa \
  bound_service_account_namespaces=production \
  policies=myapp-policy \
  ttl=1h
```

### 2.5 Vault Agent Sidecar 注入

```yaml

apiVersion: apps/v1
kind: Deployment
metadata:
  name: myapp

          export DATABASE_URL="postgresql://{{ .Data.username }}:{{ .Data.password }}@db.internal:5432/myapp"

    spec:
      serviceAccountName: myapp-sa

        - name: myapp
          image: myregistry/myapp:v1.0.0
          env:
            - name: DB_CONFIG_PATH
              value: /vault/secrets/db-config
```

## 3. Kubernetes Sealed Secrets

### 3.1 安装 Sealed Secrets

```bash
# 安装控制器
helm repo add sealed-secrets https://bitnami-labs.github.io/sealed-secrets
helm install sealed-secrets sealed-secrets/sealed-secrets \
  --namespace kube-system \
  --set fullnameOverride=sealed-secrets-controller

# 安装 kubeseal CLI
wget https://github.com/bitnami-labs/sealed-secrets/releases/download/v0.26.3/kubeseal-0.26.3-linux-amd64.tar.gz
tar xzf kubeseal-0.26.3-linux-amd64.tar.gz
mv kubeseal /usr/local/bin/
chmod +x /usr/local/bin/kubeseal
```

### 3.2 创建 Sealed Secret

```bash
# 1. 创建普通 Secret
echo -n 'SuperSecretPassword123!' | kubectl create secret generic db-password \
  --dry-run=client \
  --from-file=password=/dev/stdin \
  -o yaml > secret.yaml

# 2. 加密为 SealedSecret
kubeseal --format=yaml < secret.yaml > sealed-secret.yaml

# 3. 查看 SealedSecret（加密的，可安全提交到 Git）
cat sealed-secret.yaml

# 4. 应用 SealedSecret
kubectl apply -f sealed-secret.yaml

# 5. 验证解密后的 Secret
kubectl get secret db-password -o yaml
```

```yaml
# sealed-secret.yaml — 可以安全存入 Git
apiVersion: bitnami.com/v1alpha1
kind: SealedSecret
metadata:
  name: db-password
  namespace: default
spec:
  encryptedData:
    password: AgA1B2C3...加密内容...
  template:
    metadata:
      name: db-password
      namespace: default
    type: Opaque
```

### 3.3 Sealed Secret 作用域

```bash
# strict (默认) — 名称和命名空间变更后无法解密
echo -n 'secret' | kubectl create secret generic test \
  --dry-run=client --from-file=key=/dev/stdin -o yaml | \
  kubeseal --format=yaml --scope=strict > sealed-strict.yaml

# namespace-wide — 命名空间内可迁移
echo -n 'secret' | kubectl create secret generic test \
  --dry-run=client --from-file=key=/dev/stdin -o yaml | \
  kubeseal --format=yaml --scope=namespace-wide > sealed-ns-wide.yaml

# cluster-wide — 全集群可用（最灵活，最不安全）
echo -n 'secret' | kubectl create secret generic test \
  --dry-run=client --from-file=key=/dev/stdin -o yaml | \
  kubeseal --format=yaml --scope=cluster-wide > sealed-cluster-wide.yaml
```

### 3.4 备份私钥（灾难恢复）

```bash
# 导出 Sealed Secrets 私钥（安全存储！）
kubectl get secret -n kube-system \
  -l sealedsecrets.bitnami.com/sealed-secrets-key=active \
  -o yaml > sealed-secrets-key-backup.yaml

# 恢复私钥（新集群灾难恢复）
kubectl apply -f sealed-secrets-key-backup.yaml
```

### 4.2 使用 age 加密

```bash
# 生成 age 密钥对
age-keygen -o keys.txt
# 输出公钥: age1xxx... (用于加密)
# 私钥在 keys.txt 中 (用于解密)

# 设置环境变量
export SOPS_AGE_KEY_FILE=$(pwd)/keys.txt

# 创建配置文件
cat > .sops.yaml << 'EOF'
creation_rules:
  # 使用 age 加密
  - path_regex: secrets/.*\.yaml$
    age: age1xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx
  # 使用 AWS KMS 加密
  - path_regex: secrets/aws/.*\.yaml$
    kms: arn:aws:kms:us-east-1:123456789012:key/abcdef01-2345-6789-abcd-ef0123456789
  # 使用 GCP KMS
  - path_regex: secrets/gcp/.*\.yaml$
    gcp_kms: projects/my-project/locations/global/keyRings/my-ring/cryptoKeys/my-key
  # 使用 Vault
  - path_regex: secrets/vault/.*\.yaml$
    hc_vault_transit_uri: "https://vault.example.com:8200/v1/transit/keys/sops"
EOF
```

### 4.3 加密与解密

```bash
# 创建机密文件
cat > secrets/database.yaml << 'EOF'
database:
  host: db.internal
  port: 5432
  username: admin
  password: SuperSecretPassword123!
  api_key: ak_live_1234567890abcdef
EOF

# 加密（值被加密，key 保持明文）
sops --encrypt --in-place secrets/database.yaml

# 查看加密结果
cat secrets/database.yaml
# database:
#   host: ENC[AES256_GCM,data:abc...,iv:def...,tag:ghi...,type:str]
#   port: 5432
#   username: ENC[AES256_GCM,data:jkl...,iv:mno...,tag:pqr...,type:str]
#   password: ENC[AES256_GCM,data:stu...,iv:vwx...,tag:yz0...,type:str]
#   api_key: ENC[AES256_GCM,data:123...,iv:456...,tag:789...,type:str]

# 解密
sops --decrypt secrets/database.yaml

# 编辑加密文件（自动解密 → 编辑 → 重新加密）
sops secrets/database.yaml

# 只加密特定 key（如不加密 host 和 port）
sops --encrypt --encrypted-regex '(password|api_key|secret)' secrets/database.yaml

# 旋转密钥（重新加密，更换数据密钥）
sops --update-keys secrets/database.yaml
```

## 5. 云 KMS 集成

### 5.1 AWS KMS

```bash
# 创建 KMS 密钥
aws kms create-key \
  --description "Application encryption key" \
  --key-usage ENCRYPT_DECRYPT \
  --key-spec SYMMETRIC_DEFAULT \
  --tags '[{"TagKey":"Environment","TagValue":"production"}]'

  --key-id alias/my-app-key \
  --plaintext-blob fileb://<(echo -n 'SuperSecret123!') \
  --output text --query CiphertextBlob | base64 --decode > encrypted.dat

aws kms decrypt \
  --ciphertext-blob fileb://encrypted.dat \
  --output text --query Plaintext | base64 --decode
```

## 6. 密钥轮转策略

### 6.1 轮转策略矩阵

| 密钥类型 | 轮转频率 | 轮转方式 | 工具 |
|---------|---------|---------|------|
| 数据库密码 | 30-90 天 | 动态凭据 (Vault) | Vault Dynamic DB |
| API Token | 90 天 | 双 Token 并行 | 手动/Automation |
| TLS 证书 | 90 天 | ACME 自动续签 | cert-manager |
| 加密密钥 (KMS) | 90-365 天 | 自动轮转 | KMS 原生 |
| SSH 密钥 | 180 天 | 自动生成+分发 | Vault SSH CA |
| 服务账号密钥 | 30 天 | 短期 Token | OIDC/Workload Identity |

### 6.2 Vault 自动轮转配置

```bash
# 数据库动态凭据 — 自动轮转

  db_name=my-postgresql \
  creation_statements=@/vault/policies/pg-app.sql \
  default_ttl="1h" \

# 为现有密码设置轮转

# AWS 动态凭据

  secret_key=$AWS_SECRET_ACCESS_KEY \

vault write aws/roles/my-app-role \
  credential_type=iam_user \
  policy_document=- <<EOF
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": ["s3:GetObject"],
      "Resource": ["arn:aws:s3:::my-app-bucket/*"]
    }
  ]
}
EOF

# 生成 AWS 动态凭据
vault read aws/creds/my-app-role
```

### 6.4 双 Token 轮转方案

```python
#!/usr/bin/env python3
"""
双 Token 轮转 — 实现 API Token 无停机轮换
原理: 同时维护新旧两个 Token，新 Token 创建后等待所有服务切换，
      再撤销旧 Token。
"""

import time
import logging
from dataclasses import dataclass

logging.basicConfig(level=logging.INFO)
logger = logging.getLogger("token-rotation")

@dataclass
class TokenPair:
    """双 Token 对"""
    primary: str
    secondary: str
    primary_created: float
    secondary_created: float

class TokenRotator:
    def __init__(self, vault_client, secret_path: str):
        self.vault = vault_client
        self.secret_path = secret_path

    def rotate(self):
        """执行双 Token 轮转"""
        # 1. 读取当前 Token
        current = self.vault.read(self.secret_path)
        old_token = current['data']['token']

        # 2. 生成新 Token
        new_token = self._generate_token()
        logger.info(f"新 Token 已生成: {new_token[:8]}...")

        # 3. 写入双 Token (primary=新, secondary=旧)
        self.vault.write(self.secret_path, **{
            'token': new_token,         # 新 Token (primary)
            'previous_token': old_token, # 旧 Token (secondary)
            'rotation_time': int(time.time()),
            'rotation_status': 'in_progress'
        })

        # 4. 等待所有服务刷新（宽限期）
        grace_period = 300  # 5 分钟
        logger.info(f"等待宽限期 {grace_period}s...")
        time.sleep(grace_period)

        # 5. 撤销旧 Token
        self._revoke_token(old_token)
        logger.info(f"旧 Token 已撤销: {old_token[:8]}...")

        # 6. 更新状态
        self.vault.write(self.secret_path, **{
            'token': new_token,
            'previous_token': '',
            'rotation_time': int(time.time()),
            'rotation_status': 'completed'
        })
        logger.info("Token 轮转完成")

    def _generate_token(self) -> str:
        """通过 Vault 生成新 Token"""
        result = self.vault.write('auth/token/create', policies=['app-policy'])
        return result['auth']['client_token']

    def _revoke_token(self, token: str):
        """撤销旧 Token"""
        self.vault.write(f'auth/token/revoke', token=token)
```

## 7. CI/CD 机密管理

### 7.1 GitHub Actions 机密

```yaml
# 推荐方式：使用 OIDC + 短期凭据（无静态密钥）
name: Deploy to AWS

permissions:
  id-token: write   # OIDC
  contents: read

jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      # 使用 OIDC 获取 AWS 凭据（无需存储 Access Key）
      - uses: aws-actions/configure-aws-credentials@v4
        with:
          role-to-assume: arn:aws:iam::123456789012:role/github-actions-deploy
          aws-region: us-east-1
      - name: Deploy
        run: |
          aws s3 sync ./dist s3://my-bucket/
```

### 7.2 Vault GitHub Actions 集成

```yaml
# .github/workflows/vault-secrets.yml
name: Read Secrets from Vault

on: [push]

jobs:
  deploy:
    runs-on: ubuntu-latest
    permissions:
      id-token: write
      contents: read
    steps:
      - uses: actions/checkout@v4

      - name: Import Secrets from Vault
        uses: hashicorp/vault-action@v3
        with:
          url: https://vault.example.com
          role: github-actions
          method: jwt
          # 从 Vault 读取多个机密并设为环境变量
          secrets: |
            secret/data/myapp/database username | DB_USER ;
            secret/data/myapp/database password | DB_PASSWORD ;
            database/creds/readonly username | DB_DYNAMIC_USER ;
            database/creds/readonly password | DB_DYNAMIC_PASSWORD

      - name: Use Secrets
        run: |
          echo "DB_USER: $DB_USER"
          # 机密已在环境中，安全使用
          ./deploy.sh
```

## 8. 最佳实践

### ✅ 推荐做法

| 实践 | 说明 |
|------|------|
| **动态优于静态** | 使用 Vault 动态凭据，按需生成、自动过期 |
| **零信任** | 不信任网络边界，每次访问都需认证授权 |
| **最小权限** | 每个应用/服务只能访问自己的机密 |
| **审计一切** | 记录谁、何时、访问了什么机密 |
| **定期轮转** | 制定并执行密钥轮转计划 |
| **OIDC 替代长期 Token** | CI/CD 使用 OIDC 短期凭据 |
| **GitOps 兼容** | Sealed Secrets / SOPS 让机密可存入 Git |
| **加密传输和存储** | TLS 传输 + AES-256 存储 |

### ❌ 避免

| 反模式 | 风险 |
|--------|------|
| 硬编码密钥在代码中 | Git 历史泄露，无法审计 |
| 使用 .env 文件提交到 Git | 意外泄露 |
| 共享同一密钥给多个服务 | 泄露后影响面大 |
| 永不过期的 Token | 泄露后永久有效 |
| 明文存储在 CI/CD 变量中 | CI/CD 日志可能泄露 |
| 无访问审计 | 无法追踪泄露源头 |
