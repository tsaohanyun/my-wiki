# 数据库迁移策略

## 概述

数据库迁移是管理数据库Schema变更的过程，包括表结构修改、数据迁移、索引变更等。良好的迁移策略确保数据库变更可追踪、可重复、可回滚。

## 1. 迁移工具

### 1.1 Flyway（Java生态）

```bash
# 安装Flyway
brew install flyway
# 或使用Docker
docker pull flyway/flyway

# 目录结构
# db/migration/
# ├── V1__Create_user_table.sql
# ├── V2__Add_email_column.sql
# └── V3__Create_order_table.sql

# 迁移文件示例
# V1__Create_user_table.sql
CREATE TABLE users (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    username VARCHAR(50) NOT NULL UNIQUE,
    email VARCHAR(100) NOT NULL,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP
);

CREATE INDEX idx_users_email ON users(email);

# V2__Add_phone_column.sql
ALTER TABLE users ADD COLUMN phone VARCHAR(20);
ALTER TABLE users ADD COLUMN status ENUM('active', 'inactive') DEFAULT 'active';

# 执行迁移
flyway -url=jdbc:mysql://localhost:3306/mydb \
       -user=root \
       -password=secret \
       migrate

# 查看迁移状态
flyway info

# 回滚（需要专业版）
flyway undo

# 修复迁移历史
flyway repair
```

### 1.3 Alembic（Python/SQLAlchemy）

```bash
# 初始化
alembic init alembic

# 配置 alembic.ini
sqlalchemy.url = postgresql://user:pass@localhost/mydb

# 创建迁移
alembic revision --autogenerate -m "create users table"

# 生成的迁移文件 alembic/versions/xxx_create_users_table.py
```

```python
# alembic/versions/xxx_create_users_table.py
"""create users table

Revision ID: abc123
Revises: 
Create Date: 2024-01-15 10:30:00.000000
"""
from alembic import op
import sqlalchemy as sa

revision = 'abc123'
down_revision = None
branch_labels = None
depends_on = None

def upgrade():
    op.create_table(
        'users',
        sa.Column('id', sa.BigInteger(), nullable=False),
        sa.Column('username', sa.String(50), nullable=False),
        sa.Column('email', sa.String(100), nullable=False),
        sa.Column('created_at', sa.DateTime(), server_default=sa.text('now()')),
        sa.PrimaryKeyConstraint('id'),
        sa.UniqueConstraint('username')
    )
    op.create_index('idx_users_email', 'users', ['email'])

def downgrade():
    op.drop_index('idx_users_email', table_name='users')
    op.drop_table('users')
```

```bash
# 执行迁移
alembic upgrade head

# 回滚一步
alembic downgrade -1

# 查看历史
alembic history

# 查看当前版本
alembic current
```

### 1.4 Django Migrations

```python
# models.py
from django.db import models

class User(models.Model):
    username = models.CharField(max_length=50, unique=True)
    email = models.EmailField()
    phone = models.CharField(max_length=20, blank=True, null=True)
    created_at = models.DateTimeField(auto_now_add=True)
    updated_at = models.DateTimeField(auto_now=True)

    class Meta:
        db_table = 'users'
        indexes = [
            models.Index(fields=['email']),
        ]
```

```bash
# 创建迁移
python manage.py makemigrations

# 查看迁移SQL
python manage.py sqlmigrate app_name 0001

# 执行迁移
python manage.py migrate

# 回滚到特定版本
python manage.py migrate app_name 0002

# 回滚所有
python manage.py migrate app_name zero
```

### 2.3 Git工作流

```bash
# 分支策略
# main: 生产环境迁移
# develop: 开发环境迁移
# feature/*: 新功能迁移

# 创建迁移分支
git checkout -b feature/add-user-roles

# 创建迁移文件
alembic revision --autogenerate -m "add user roles"

# 提交迁移
git add alembic/versions/
git commit -m "feat(db): add user roles migration"

# 合并前确保迁移可重入
alembic upgrade head
alembic downgrade -1
alembic upgrade head
```

## 3. 零停机迁移

### 3.1 扩展-收缩模式（Expand-Contract）

```sql
-- 阶段1：扩展（添加新列，保持旧列）
-- 步骤1：添加新列
ALTER TABLE users ADD COLUMN email_new VARCHAR(255);

-- 步骤2：双写（应用同时写入新旧列）
-- 应用代码修改：同时写入 email 和 email_new

-- 步骤3：回填历史数据
UPDATE users SET email_new = email WHERE email_new IS NULL;

-- 步骤4：添加约束
ALTER TABLE users ALTER COLUMN email_new SET NOT NULL;

-- 阶段2：切换（更新应用使用新列）
-- 步骤5：更新应用代码读取 email_new

-- 阶段3：收缩（移除旧列）
-- 步骤6：停止写入旧列
-- 步骤7：删除旧列
ALTER TABLE users DROP COLUMN email;
ALTER TABLE users RENAME COLUMN email_new TO email;
```

### 3.2 在线DDL（Online DDL）

```sql
-- MySQL 5.6+ 支持Online DDL
ALTER TABLE users ADD COLUMN status VARCHAR(20) DEFAULT 'active', 
ALGORITHM=INPLACE, LOCK=NONE;

-- 使用pt-online-schema-change（Percona工具）
pt-online-schema-change \
  --alter "ADD COLUMN status VARCHAR(20) DEFAULT 'active'" \
  --user=root \
  --password=secret \
  --host=localhost \
  D=mydb,t=users \
  --execute

-- 使用gh-ost（GitHub工具）
gh-ost \
  --host=localhost \
  --database=mydb \
  --table=users \
  --alter="ADD COLUMN status VARCHAR(20) DEFAULT 'active'" \
  --allow-on-master \
  --execute
```

### 3.3 蓝绿部署策略

```bash
# 1. 创建新表结构
CREATE TABLE users_v2 (
    id BIGINT PRIMARY KEY,
    username VARCHAR(50) NOT NULL,
    email VARCHAR(255) NOT NULL,
    status VARCHAR(20) DEFAULT 'active',
    created_at TIMESTAMP,
    updated_at TIMESTAMP
);

# 2. 同步数据
INSERT INTO users_v2 SELECT * FROM users;

# 3. 设置触发器同步增量
CREATE TRIGGER sync_users_to_v2
AFTER INSERT OR UPDATE OR DELETE ON users
FOR EACH ROW EXECUTE FUNCTION sync_to_v2();

# 4. 切换应用到新表
# 应用代码：users -> users_v2

# 5. 清理旧表
DROP TABLE users;
ALTER TABLE users_v2 RENAME TO users;
```

## 4. 回滚策略

### 4.1 可逆迁移

```sql
-- 迁移文件包含up和down
-- V2__Add_email_column.sql
-- UP
ALTER TABLE users ADD COLUMN email VARCHAR(100);
CREATE INDEX idx_users_email ON users(email);

-- DOWN
DROP INDEX idx_users_email;
ALTER TABLE users DROP COLUMN email;
```

```python
# Alembic可逆迁移
def upgrade():
    op.add_column('users', sa.Column('email', sa.String(100)))
    op.create_index('idx_users_email', 'users', ['email'])

def downgrade():
    op.drop_index('idx_users_email', table_name='users')
    op.drop_column('users', 'email')
```

### 4.2 不可逆迁移处理

```sql
-- 数据转换迁移（不可逆）
-- V3__Migrate_user_data.sql
-- UP
UPDATE users SET email = LOWER(TRIM(email));
UPDATE users SET status = 'active' WHERE status IS NULL;

-- 记录变更日志
INSERT INTO migration_log (version, description, executed_at)
VALUES ('V3', 'Migrated user data', NOW());

-- DOWN（无法完全回滚，只能记录）
-- 回滚说明：数据转换无法完全回滚
-- 请从备份恢复或手动处理
```

### 4.3 备份策略

```bash
# 迁移前备份
#!/bin/bash
BACKUP_DIR="/backup/pre-migration"
DATE=$(date +%Y%m%d_%H%M%S)
MIGRATION_VERSION=$1

# PostgreSQL备份
pg_dump -h localhost -U postgres -d mydb \
  -f "${BACKUP_DIR}/mydb_${MIGRATION_VERSION}_${DATE}.sql"

# MySQL备份
mysqldump -h localhost -u root -p \
  --single-transaction \
  --routines \
  --triggers \
  mydb > "${BACKUP_DIR}/mydb_${MIGRATION_VERSION}_${DATE}.sql"

# 压缩备份
gzip "${BACKUP_DIR}/mydb_${MIGRATION_VERSION}_${DATE}.sql"

echo "Backup completed: mydb_${MIGRATION_VERSION}_${DATE}.sql.gz"

# 回滚脚本
#!/bin/bash
BACKUP_FILE=$1

# PostgreSQL恢复
psql -h localhost -U postgres -d mydb < "$BACKUP_FILE"

# MySQL恢复
mysql -h localhost -u root -p mydb < "$BACKUP_FILE"
```

## 5. 生产环境迁移流程

### 5.1 CI/CD集成

```yaml
# .github/workflows/db-migration.yml
name: Database Migration

on:
  push:
    paths:
      - 'db/migrations/**'

jobs:
  migrate:
    runs-on: ubuntu-latest
    
    steps:
    - uses: actions/checkout@v3
    
    - name: Setup Python
      uses: actions/setup-python@v4
      with:
        python-version: '3.11'
    
    - name: Install dependencies
      run: pip install alembic psycopg2-binary
    
    - name: Run tests
      run: |
        python -m pytest tests/migrations/
    
    - name: Backup production DB
      run: |
        ./scripts/backup_db.sh production
    
    - name: Run migration
      env:
        DATABASE_URL: ${{ secrets.PROD_DATABASE_URL }}
      run: |
        alembic upgrade head
    
    - name: Verify migration
      run: |
        python scripts/verify_migration.py
```

## 6. 最佳实践

### 6.1 迁移文件原则

1. **原子性**：每个迁移只做一件事
2. **可逆性**：尽可能提供回滚脚本
3. **幂等性**：迁移可重复执行
4. **向后兼容**：新旧版本应用可同时运行

### 6.2 大表迁移策略

```sql
-- 分批更新避免锁表
-- 错误方式
UPDATE orders SET status = 'completed' WHERE status = 'old_status';

-- 正确方式：分批处理
DO $$
DECLARE
  batch_size INT := 1000;
  affected_rows INT;
BEGIN
  LOOP
    UPDATE orders 
    SET status = 'completed' 
    WHERE id IN (
      SELECT id FROM orders 
      WHERE status = 'old_status' 
      LIMIT batch_size
      FOR UPDATE SKIP LOCKED
    );
    
    GET DIAGNOSTICS affected_rows = ROW_COUNT;
    EXIT WHEN affected_rows = 0;
    
    PERFORM pg_sleep(0.1);  -- 短暂休眠减少负载
    COMMIT;
  END LOOP;
END $$;
```
